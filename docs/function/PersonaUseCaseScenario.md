# LedgerLeap ペルソナ、ユースケース、シナリオ

**日付:** 2025年9月27日

**最終更新日:** 2026年3月10日

## 0. この文書の位置づけ

この文書は、LedgerLeap の利用者像と業務シナリオを、人が読んでイメージしやすい粒度で整理するための基礎資料です。

- **目的:** 利用者が「どんな状況で」「何に困り」「どの流れで LedgerLeap を使うか」を共有する
- **役割:** ペルソナの厚み、現場背景、判断軸、ユースケースの具体像を保持する
- **関連:** Sprint 2 で整理した capability 名や初期 skill セットとの対応は `docs/work/llm-integration/2026-03-10_Client_Facing_Capability_Taxonomy.md` を参照する

> この文書は、client-facing capability を設計するための**物語的・業務的な根拠資料**です。
>
> capability 名や初期 skill セットは taxonomy 側を正本としつつ、こちらでは人物像や現場の文脈を端折らずに保持します。

## 1. ペルソナ

LedgerLeapの利用者を以下のペルソナに分類し、それぞれの特徴とニーズを定義します。

### 1.1. 実務担当者 (Operational Staff)
*   **特徴:** 日々の業務で台帳へのデータ入力や参照を行う。ITリテラシーは様々。
*   **ニーズ:**
    *   直感的で分かりやすいUIで、迷わずデータ入力ができること。
    *   必要な情報を素早く検索し、参照できること。
    *   ワークフローに沿って、自分のタスク（作成、点検、承認）を把握し、実行できること。
    *   重要な情報が更新された際に通知を受け取れること。
*   **現場背景:**
    *   記録作業は本業の合間に行うことが多く、入力や確認に長い説明は求めていない。
    *   「まず今やるべきことが分かる」「迷ったら一覧から辿れる」ことを重視する。
    *   添付資料や過去記録を参照しながら、その場で判断したい場面が多い。
*   **判断軸・成功条件:**
    *   必須項目を漏らさず入力できること。
    *   必要な過去記録を数手で見つけられること。
    *   自分に割り当てられた確認タスクを優先度順に処理できること。
*   **初期 skill セットとの対応:** `ledger-search`, `ledger-create`, `workflow-review`

### 1.2. 管理者 (Administrator / Manager)
*   **特徴:** 部門長、プロジェクトマネージャー、情報システム担当者など。システム全体の管理、メンバー管理、権限設定、監査などを行う。
*   **ニーズ:**
    *   ユーザー、ロール、フォルダ、台帳定義などを柔軟に管理できること。
    *   各リソースへのアクセス権限を細かく設定し、セキュリティを確保できること。
    *   システム全体の活動状況（アクティビティログ）を追跡し、監査できること。
    *   ワークフローの進捗を把握し、必要に応じて介入できること。
    *   システム設定をカスタマイズできること。
*   **現場背景:**
    *   個別レコードの処理よりも、全体の滞留・不備・偏りを把握する役割が大きい。
    *   「誰が・いつ・何をしたか」を説明できる証跡が必要になる。
    *   問題が起きた時には、一覧から異常箇所を絞り込み、必要なレコードだけ掘り下げたい。
*   **判断軸・成功条件:**
    *   台帳定義や運用ルールを過不足なく整えられること。
    *   監査や問い合わせに対して根拠を示せること。
    *   件数・進捗・滞留から、介入が必要な箇所を判断できること。
*   **初期 skill セットとの対応:** `ledger-search`, `workflow-review`, `activity-audit`, `analytics-report`

### 1.3. 現場リーダー/作業班長 (Team Leader / Foreman)
*   **特徴:** チームメンバーの業務を監督し、情報共有や指示確認を行う。実務担当者と管理者の両方の側面を持つ。
*   **ニーズ:**
    *   チーム内の情報（台帳、添付ファイル）を効率的に共有・管理できること。
    *   メンバーの代理でデータ入力やワークフロー操作ができること。
    *   チームメンバーの活動状況を把握できること。
*   **現場背景:**
    *   現場の急な変更や確認依頼に対して、代理での更新や差し戻し対応が発生しやすい。
    *   文章だけでなく、写真・手順書・帳票などの添付資料も一緒に見て判断する場面が多い。
    *   チームの共通認識を揃えるため、検索・確認・コメントの往復が多い。
*   **判断軸・成功条件:**
    *   必要な記録をすぐ見つけて、チームへ共有できること。
    *   代理修正が必要な時に、状態変化を把握しながら安全に更新できること。
    *   差し戻しや再提出の理由をチームに伝えられること。
*   **初期 skill セットとの対応:** `ledger-search`, `ledger-update`, `workflow-review`

### 1.4. 開発者 (Developer)
*   **特徴:** LedgerLeapシステムの開発、保守、機能拡張を行う。
*   **ニーズ:**
    *   システムの設計思想、アーキテクチャ、技術スタックを理解できること。
    *   コーディング規約や開発ガイドラインが明確であること。
    *   テストコードが充実しており、効率的に開発・テストができること。
    *   API仕様が明確であり、外部システム連携が容易であること。
*   **補足:** 開発者は Sprint 2 の client-facing 初期 skill セットの対象ではなく、developer-facing 文書の主対象です。

## 1.5. capability / 初期 skill セット対応表

| ペルソナ | 代表的な目的 | 初期 skill セット | 最初に見せたい導線 |
|---|---|---|---|
| 実務担当者 | 記録を作る・探す・自分のタスクを処理する | `ledger-search`, `ledger-create`, `workflow-review` | 一覧 → 詳細 → 保存 / 承認 |
| 管理者 | 監査する・状況を把握する・ルールを整える | `ledger-search`, `workflow-review`, `activity-audit`, `analytics-report` | 集計 → 絞り込み → 詳細確認 |
| 現場リーダー | 代理更新する・共有資料を確認する・差し戻しを回す | `ledger-search`, `ledger-update`, `workflow-review` | 検索 → 詳細 → 更新 / 差し戻し |
| 開発者 | 実装・保守・調査を行う | client-facing ではなく developer-facing | 設計資料 → 実装資料 → テスト |

## 2. ユースケースとシナリオ

各ペルソナがLedgerLeapを利用する際の具体的なユースケースとシナリオを記述します。

### 2.1. 実務担当者

#### ユースケース1: 日報の作成と提出
*   **シナリオ:**
    1.  実務担当者は、日報台帳を開く。
    2.  今日の業務内容、進捗、特記事項などを入力する。
    3.  関連する写真や資料を添付する。
    4.  「作成完了」ボタンをクリックし、日報を提出する（ワークフローが有効な場合、点検者に通知される）。
    5.  提出後、日報の詳細画面で「保存しました」というトーストが表示される。
    6.  別ウィンドウで開いている日報一覧画面が自動的に更新される。
*   **このシナリオで大事なこと:**
    *   必須項目が分かりやすいこと
    *   提出後に状態がどう変わったかすぐ分かること
    *   添付資料込みで1件の記録として完結していること
*   **対応する capability:** `ledger-create`, `workflow-review`

#### ユースケース2: 過去の業務記録の検索と参照
*   **シナリオ:**
    1.  実務担当者は、特定の顧客に関する過去の業務記録を検索したい。
    2.  検索ボックスに顧客名やキーワードを入力する。
    3.  検索結果から関連する台帳レコードを見つけ、詳細を参照する。
    4.  添付されている資料（Word, Excel, PDFなど）の内容も検索対象となっているため、キーワードでファイル内容も検索できる。
    5.  スキャンPDFの内容も確認対象に含めて探せる。
*   **このシナリオで大事なこと:**
    *   まず一覧で候補を見てから詳細へ進めること
    *   記録本文だけでなく添付資料も含めて判断できること
    *   表記ゆれや記憶違いがあっても探し直しやすいこと
*   **対応する capability:** `ledger-search`

#### ユースケース3: 承認待ちタスクの確認と処理
*   **シナリオ:**
    1.  実務担当者は、マイポータルで自分に割り当てられた承認待ちタスクがあることを確認する。
    2.  承認待ちタスクの一覧から、特定の台帳レコードの詳細を開く。
    3.  内容を確認し、「承認」または「差し戻し」（編集）を行う。
    4.  差し戻しの場合、理由を入力してステータスを「作成中」に戻す。
*   **このシナリオで大事なこと:**
    *   自分が処理すべき件数をすぐ把握できること
    *   優先度や期限が近いものから見ていけること
    *   差し戻し理由を次の人に分かる形で残せること
*   **対応する capability:** `workflow-review`

#### ユースケース4: 添付資料を見ながら判断材料を探す
*   **シナリオ:**
    1.  実務担当者は、請求書や契約書が添付された記録を探したい。
    2.  キーワード検索で候補を一覧表示する。
    3.  候補の中から、必要な記録だけ詳細を開く。
    4.  本文と添付資料をあわせて確認し、必要な記述や金額、期限を見直す。
*   **このシナリオで大事なこと:**
    *   添付資料を別管理せず、記録と一緒に見られること
    *   一覧から詳細へ自然に進めること
*   **対応する capability:** `ledger-search`

### 2.2. 管理者

#### ユースケース1: 新規ユーザーの追加と権限設定
*   **シナリオ:**
    1.  管理者は、新しい従業員をシステムに追加したい。
    2.  ユーザー管理画面で新規ユーザーの情報を入力する。
    3.  ユーザーに適切なロール（例: 「一般ユーザー」）を割り当てる。
    4.  特定のフォルダに対して、そのロールのアクセス権限（例: 「書き込み」）を設定する。
    5.  グローバル通知の設定も行う。
*   **このシナリオで大事なこと:**
    *   使い始める前に必要な権限と通知を揃えられること
    *   運用開始後に「誰が何を見られるか」を説明できること
*   **対応する capability:** 主に developer-facing / 管理機能寄り。Sprint 2 では `ledger-search`・`analytics-report`・`activity-audit` の文脈に接続する補助シナリオとして扱う。

#### ユースケース2: 台帳定義の作成と公開
*   **シナリオ:**
    1.  管理者は、新しい業務プロセスに対応する台帳を作成したい。
    2.  台帳定義管理画面で、新しい台帳のタイトル、説明を設定する。
    3.  必要な項目（列）を追加し、入力形式、必須入力、重複禁止などの属性を設定する。
    4.  ワークフローを有効にするか設定し、推奨担当者を設定する。
    5.  台帳定義を保存し、実務担当者が利用できるように公開する。
*   **このシナリオで大事なこと:**
    *   実務担当者が迷わず使える台帳定義にできること
    *   保存先や状態管理を含めて運用設計できること
*   **対応する capability:** `ledger-search`, `analytics-report`

#### ユースケース3: システムの活動状況の監査
*   **シナリオ:**
    1.  管理者は、特定の期間におけるシステム内の活動状況を監査したい。
    2.  アクティビティログ画面で、期間やユーザー、操作内容でフィルタリングを行う。
    3.  特定の台帳レコードに対する変更履歴や、ファイルダウンロードの履歴などを確認する。
    4.  不審なアクティビティがないかを確認し、必要に応じて対応する。
*   **このシナリオで大事なこと:**
    *   監査の起点を期間・ユーザー・保存先から切り替えられること
    *   一覧から異常な記録だけを掘り下げられること
*   **対応する capability:** `activity-audit`

#### ユースケース4: 件数と進捗の偏りを把握する
*   **シナリオ:**
    1.  管理者は、今月の台帳作成件数や承認待ち件数を把握したい。
    2.  集計期間を決める。
    3.  台帳の種類や状態ごとに件数を見る。
    4.  滞留しているものや偏りがあるものを確認し、必要なレコードへ進む。
*   **このシナリオで大事なこと:**
    *   件数を見るだけでなく、次に確認すべきレコードへ繋げられること
    *   「どこに介入すべきか」を判断できる粒度で集計できること
*   **対応する capability:** `analytics-report`, `workflow-review`

### 2.3. 現場リーダー/作業班長

#### ユースケース1: チームメンバーの代理で台帳を編集
*   **シナリオ:**
    1.  現場リーダーは、チームメンバーが作成した台帳レコードに修正が必要なことを確認する。
    2.  その台帳レコードの詳細を開き、内容を編集する。
    3.  編集後、「保存」ボタンをクリックし、必要に応じて修正理由をコメントとして残す。
    4.  ワークフローが有効な場合、ステータスが「作成中」に戻り、点検者に再度通知される。
*   **このシナリオで大事なこと:**
    *   代理更新でも、状態変化を見失わないこと
    *   修正理由を次の確認者や作成者に伝えられること
*   **対応する capability:** `ledger-update`, `workflow-review`

#### ユースケース2: チーム内の情報共有と確認
*   **シナリオ:**
    1.  現場リーダーは、チームメンバーが共有した新しい業務手順書（添付ファイル）を確認したい。
    2.  関連するフォルダを開き、新しい業務手順書を見つける。
    3.  ドキュメントの内容を閲覧し、必要に応じてコメントを残す。
    4.  全文検索機能を使って、過去の類似事例や関連情報を検索する。
*   **このシナリオで大事なこと:**
    *   チームで参照すべき資料をすぐ共有できること
    *   類似事例や過去の判断を横断的に参照できること
*   **対応する capability:** `ledger-search`, `workflow-review`

#### ユースケース3: 差し戻し案件の整理と再提出支援
*   **シナリオ:**
    1.  現場リーダーは、差し戻しになった記録を確認する。
    2.  差し戻し理由を読み、必要な修正箇所を整理する。
    3.  メンバー本人が対応するか、代理で修正するか判断する。
    4.  修正後、再提出まで見届ける。
*   **このシナリオで大事なこと:**
    *   差し戻し理由が具体的に残っていること
    *   チーム内で次の対応者を決めやすいこと
*   **対応する capability:** `workflow-review`, `ledger-update`

### 2.4. 開発者

#### ユースケース1: 新機能の実装
*   **シナリオ:**
    1.  開発者は、新しい機能（例: 外部システム連携）をLedgerLeapに追加するよう依頼される。
    2.  `docs/architecture/overview.md` や `docs/README.md` を参照し、システム全体のアーキテクチャと技術スタックを理解する。
    3.  `docs/development/coding_standards.md` を参照し、コーディング規約を確認する。
    4.  関連するモデルやサービス (`docs/models/*.md`, `docs/services/*.md`) を参照し、既存の実装を把握する。
    5.  新機能の設計を行い、コードを実装する。
    6.  `docs/function/Test.md` や既存のテストコードを参考に、テストコードを作成し、実行する。

#### ユースケース2: 既存機能のバグ修正
*   **シナリオ:**
    1.  開発者は、特定の機能（例: 台帳の検索）でバグが報告される。
    2.  `docs/function/Search.md` を参照し、検索機能の実装詳細を理解する。
    3.  関連するサービス (`App\Services\SearchService`) やモデル (`App\Models\Ledger`) のコードを調査する。
    4.  バグの原因を特定し、修正コードを実装する。
    5.  既存のテストコードを実行し、バグが修正され、他の機能に影響がないことを確認する。

### 2.5. 断片的な記憶からの検索（共通シナリオ）

部分一致検索が必要になるのは、正式名を覚えていないが、タグ・フォルダ・台帳名の断片だけ思い出せる場面です。以下のシナリオは、`ledger-search` の必須導線として扱います。

さらに、`folder_id` や `ledgerDefineId` を知らない場合は、先にフォルダ / 台帳定義の候補を探して ID を得てから検索を組み立てる lookup-first が前提です。

#### 実務担当者
*   **シナリオ:**
    1.  実務担当者は、タグの一部「請求」や「要確認」だけ覚えている。
    2.  部分一致で候補一覧を出し、必要な記録を見つける。
    3.  候補の中から、本文と添付資料を見て詳細へ進む。
*   **このシナリオで大事なこと:**
    *   正式なタグ名を覚えていなくても再検索できること
    *   候補一覧から詳細へ自然に進めること
    *   まず広く拾い、そこから絞り込めること
*   **対応する capability:** `ledger-search`

#### 管理者
*   **シナリオ:**
    1.  管理者は、フォルダ名の一部「営業」「監査」などを手がかりに、権限内の候補を探したい。
    2.  一覧から対象フォルダや関連台帳を絞り込む。
    3.  台帳名の断片もあわせて見て、保存先・担当者・更新状況を確認する。
*   **このシナリオで大事なこと:**
    *   権限外の候補を出さないこと
    *   フォルダ名の断片からでも候補に辿り着けること
    *   監査や問い合わせに使える説明ができること
*   **対応する capability:** `ledger-search`, `activity-audit`

#### 現場リーダー / 作業班長
*   **シナリオ:**
    1.  現場リーダーは、共有した台帳名の一部やタグ断片だけを頼りに、チーム資料を探したい。
    2.  検索結果から、同じ案件の記録を横断して確認する。
    3.  必要なら関連レコードへ進み、差し戻しや再提出の判断をする。
*   **このシナリオで大事なこと:**
    *   名称の一部だけでも、共有資料にすぐ辿り着けること
    *   候補が複数あっても、チーム内で説明しやすいこと
    *   関連レコードへの横断が途切れないこと
*   **対応する capability:** `ledger-search`, `workflow-review`

## 3. Sprint 2 との整合メモ

Sprint 2 では、この文書を **人物像と現場文脈を保持する資料** として使い、client-facing capability 名や初期 skill セットの正本は taxonomy 側で管理します。

### client-facing に引き継ぐ観点
- 一覧 → 詳細 の二段階導線
- 優先度 / 期限 / 滞留 を見ながら処理順を決める流れ
- 添付資料を本文とあわせて確認する流れ
- 差し戻し理由を残して再提出へ繋げる流れ
- 集計や監査から個別レコードへ掘り下げる流れ

### developer-facing に残す観点
