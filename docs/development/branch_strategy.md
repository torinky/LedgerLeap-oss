# Gitブランチ戦略とコミット規約

## はじめに
このドキュメントは、LedgerLeapプロジェクトにおけるGitのブランチ運用戦略とコミットメッセージの規約を定めます。これらのルールに従うことで、開発プロセスの透明性を高め、複数人での共同作業をスムーズにし、安定したリリース管理を実現することを目的とします。

## ブランチ戦略

LedgerLeapでは、Gitflowをベースとしたブランチ戦略を採用します。これにより、機能開発、リリース準備、本番環境の保守を並行して効率的に行うことを目指します。

### 基本的なブランチ構成

*   **`main`**
    *   目的: 本番環境にリリースされている最新の安定バージョンを反映するブランチ。
    *   運用:
        *   このブランチへの直接的なコミットは固く禁止します。
        *   `release/*` ブランチまたは `hotfix/*` ブランチからのマージのみ許可されます。
        *   マージ時にはタグ付け (`vX.Y.Z`) を行います。
*   **`develop`**
    *   目的: 次期リリースに向けた開発の統合ブランチ。最新の開発状況を反映します。
    *   運用:
        *   `feature/*` ブランチや `hotfix/*` ブランチのマージ先となります。
        *   定期的に `release/*` ブランチがこのブランチから分岐します。
        *   不安定な状態になることもありますが、CI/CDによる自動テストで品質を担保します。
*   **`feature/<issue-id>-<feature-name>`**
    *   目的: 新機能の開発や、リリース計画に含まれるバグ修正を行うためのブランチ。
    *   命名規則:
        *   `<issue-id>`: 関連する課題管理システムのID (例: `123`, `gh-45`)。無い場合は省略可。
        *   `<feature-name>`: 機能の内容を簡潔に示す英語のケバブケース (例: `user-authentication`, `ledger-search-enhancement`)。
    *   運用:
        *   `develop` ブランチから分岐します。
        *   作業完了後、`develop` ブランチへのプルリクエストを作成し、レビューを経てマージされます。
        *   ローカルでの作業ブランチとして扱い、不要になったら削除します。
*   **`release/<version>`**
    *   目的: リリース準備専用のブランチ。バグ修正、ドキュメント作成、最終テストなど、リリースに必要な作業を行います。
    *   命名規則: `<version>`: リリースバージョン (例: `v1.0.0`, `v1.2.3-beta`)。
    *   運用:
        *   `develop` ブランチから分岐します。
        *   このブランチでは、リリースに関連する軽微なバグ修正やバージョン番号の更新、ドキュメント修正のみを行います。新たな機能追加は行いません。
        *   リリース準備完了後、`main` ブランチと `develop` ブランチの両方にマージされます。`main` ブランチへのマージ時にはリリースバージョンでタグ付けを行います。
        *   マージ後はブランチを削除します。
*   **`hotfix/<issue-id>-<fix-name>`**
    *   目的: 本番環境で発生した緊急のバグを修正するためのブランチ。
    *   命名規則:
        *   `<issue-id>`: 関連する課題管理システムのID。
        *   `<fix-name>`: 修正内容を簡潔に示す英語のケバブケース。
    *   運用:
        *   `main` ブランチの該当するタグから分岐します。
        *   修正作業完了後、`main` ブランチと `develop` ブランチの両方にマージされます。`main` ブランチへのマージ時には新たなパッチバージョンでタグ付けを行います。
        *   マージ後はブランチを削除します。

### プルリクエスト (マージリクエスト) の運用ルール

*   **対象ブランチ**: `develop`, `main` ブランチへのマージは、必ずプルリクエストを経由して行います。
*   **作成**:
    *   `feature/*` -> `develop`
    *   `release/*` -> `main`, `develop`
    *   `hotfix/*` -> `main`, `develop`
*   **レビュー**:
    *   原則として、1名以上のレビュアーによる承認が必要です。レビュアーはプロジェクトリーダーまたは他の開発者が担当します。
    *   レビューでは、コーディング規約の遵守、ロジックの妥当性、テストコードの有無、ドキュメントの更新などを確認します。
    *   セルフレビューも推奨されます。
*   **マージ条件**:
    *   レビュアーによる承認が得られていること。
    *   自動テスト (CI) が全て成功していること。
    *   コンフリクトが解消されていること。
    *   (可能であれば) 関連するIssueがクローズされること。
*   **マージ後のブランチ削除**: `feature/*`, `release/*`, `hotfix/*` ブランチは、マージ後に速やかに削除します。

## コミットメッセージ規約

LedgerLeapでは、コミットメッセージの可読性と履歴の追跡しやすさを向上させるため、**Conventional Commits** 形式を強く推奨します。コミットメッセージは原則として**日本語**で記述します。

### 形式

```
<type>(<scope>): <subject>

<body>

<footer>
```

*   **ヘッダー (`<type>(<scope>): <subject>`)**: 必須。1行目に記述します。
    *   **`<type>`**: コミットの種類を示す識別子。以下を参考にしてください。
        *   `feat`: 新機能の追加
        *   `fix`: バグ修正
        *   `docs`: ドキュメントのみの変更
        *   `style`: コードの整形やフォーマット変更（コードの動作に影響しないもの）
        *   `refactor`: リファクタリング（バグ修正や機能追加を含まないコード構造の変更）
        *   `perf`: パフォーマンス改善
        *   `test`: テストコードの追加・修正
        *   `build`: ビルドシステムや外部依存関係の変更（例: composer.json, package.json）
        *   `ci`: CI/CD設定ファイルの変更
        *   `chore`: 上記いずれにも当てはまらない雑多な変更（例: .gitignore の更新）
        *   `revert`: 以前のコミットの取り消し
    *   **`(<scope>)`**: コミットの変更範囲を示す短い記述（任意）。影響範囲が明確な場合に記述します。
        *   例: `(auth)`, `(ledger-api)`, `(user-profile)`
    *   **`<subject>`**: 変更内容の簡潔な説明。
        *   50文字以内を推奨します。
        *   体言止め、または「〜する」「〜した」のような簡潔な動詞で記述します。
        *   例: `ユーザー認証機能を追加`, `台帳検索時のN+1問題を修正`

*   **ボディ (`<body>`)**: （任意）ヘッダーだけでは説明が不十分な場合に、変更の理由や詳細、背景などを記述します。
    *   1行あたり72文字程度で改行することを推奨します。
    *   変更前後の振る舞いの違いや、この変更が解決する問題点などを具体的に記述します。

*   **フッター (`<footer>`)**: （任意）破壊的変更 (Breaking Changes) の記述や、関連する課題管理システムのIssue番号などを記述します。
    *   **破壊的変更**: `BREAKING CHANGE:` で始まるセクションを設け、変更内容と移行方法などを記述します。
    *   **Issue参照**: `Closes #123`, `Fixes #456` のように記述すると、GitHubなどで自動的にIssueがクローズされることがあります。

### コミットメッセージの例

```
feat(auth): ユーザー登録機能のAPIエンドポイント実装

ユーザーがメールアドレスとパスワードで新規登録できるためのAPIを追加。
登録成功時にはユーザートークンを返却する。

Closes #42
```

```
fix(ledger): 台帳一覧表示時の日付フォーマット誤りを修正

特定の条件下で日付が正しくフォーマットされていなかった問題を修正。
表示ライブラリのタイムゾーン設定を見直し。
```

```
docs(readme): 開発環境構築手順を更新

Node.jsの推奨バージョン情報を追記。
```

```
refactor(service): UserServiceの重複ロジックを共通メソッドに集約

可読性と保守性の向上のため、`getUserData`と`getAdminData`メソッド内の
重複していた権限チェック処理をプライベートメソッド`checkPermissions`に切り出した。
```

この規約は、チームメンバー全員が変更履歴を効率的に理解し、プロジェクトの品質を維持するために重要です。不明な点や改善提案があれば、積極的に共有してください。
