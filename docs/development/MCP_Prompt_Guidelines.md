## 概要

LedgerLeapシステムのMCPサーバーを効率的に活用するためのプロンプトガイドラインです。

> **📖 技術的な背景:** MCPプロトコルの詳細な動作メカニズムや`instructions`プロパティの技術的な役割については、[MCP アーキテクチャと動作フロー](./MCP_Architecture_and_Flow.md)を参照してください。

## 基本指針

### 1. 応答形式の最適化と情報の取捨選択

**`__summary__` フィールドの優先:**
- MCPツールからの応答に `__summary__` が含まれている場合、それは「システムがユーザーに最も伝えたい要約」です。まずこれをそのまま、あるいは自然に加工して応答の冒頭に使用してください。

**`__display_fields__` に基づく表形式表示:**
- リスト形式のデータ（検索結果やタスク一覧）には `__display_fields__` が含まれています。
- これらを使用して、ユーザーが見やすい Markdown 表形式や箇条書きで情報を整理してください。
- 概要形式 (`format="summary"`) では、各レコードに詳細画面への `link` が含まれることが多いため、これもユーザーに提示してください。

### 2. 検索戦略（最重要）

LedgerLeapの検索は、キーワード検索（Mroonga）と意味検索（Vector Search）のハイブリッドです。以下のワークフローを厳守してください。

**最優先検索ルール: キーワード検索 → 意味検索 → 活動ログによる手がかり探し**

1. **Step 1: キーワード検索 (Keyword Search)**
   - 固有名詞、会社名、製品コードなどが明確な場合に使用します。
   - `mode="count"` を併用することで、詳細を取得する前に「まず何件あるか」を素速く確認することも有効です。
2. **Step 2: 意味検索 (Semantic Search)**
   - **Step 1 で期待した結果が得られない場合、必ず実行してください。**
   - 概念、文脈、あるいは自然言語の質問で検索します。
   - 指定方法: `order_by="semantic_score"` をセットし、`q` に自然な文章を入力します。
   - 例: `SearchLedgers(q="A社との商談に関する過去の議事録", order_by="semantic_score")`
3. **Step 3: 他のツールによる補助（Pivot）**
   - それでも見つからない場合、`GetActivityLog` で最近の操作履歴を確認し、特定のファイル名や操作コメントからキーワードを拾って Step 1 に戻ります。

### 3. データ構造の理解

**台帳コンテンツ (`content`):**
- 台帳の本文データは、**数値インデックスの配列**です（例: `[0 => 'タイトル', 1 => '2025-01-01']`）。
- 各インデックスの意味を知るには、必ず `GetLedgerDefines` で対応する台帳定義の `column_define` を参照してください。
- インデックス 0 は通常「タイトル」や「件名」に対応することを覚えておくと便利ですが、厳密には定義を確認してください。

**メタデータ (`meta`):**
- 検索結果には、関連するエンティティ情報が正規化されて含まれています。
- `meta.users`: 作成者・更新者の詳細情報（名前、ID等）。
- `meta.folders`: フォルダの階層パス情報。
- `meta.ledger_defines`: 各カラム（content[n]）のラベルや型の定義。
- 「作成者の田中さんは〜」といった回答をする際は、レコードの `creator_id` を元に `meta.users` を参照してください。

**VLM抽出テキスト (`vlm_markdown`):**
- 添付ファイル情報には、VLMによって構造化された Markdown 形式のテキストが含まれます。ファイルを開かずに内容を要約したり、RAGのソースとして活用したりできます。

### 4. ワークフロー（承認・点検）の操作手順

業務タスクの処理は以下の流れで行います。

1. **タスクの発見**: `GetPendingApprovals` で自分に割り当てられた未処理タスクを取得します。
2. **背景の確認**: `GetWorkflowHistory` で、その申請の過去のステップや、差し戻し時のコメント、滞留時間等を確認します。
3. **担当の引き受け**: 自分に直接割り当てられていないが権限があるタスク（点検者ロール等）を実行したい場合は、`ClaimWorkflowTask` で自分の担当にします。
4. **決裁の実行**: `ExecuteApproval` を使用します。
   - **承認**: `action="approve"`
   - **差し戻し/却下**: `action="return"`。この際、**必ず具体的な理由を `comment` パラメータに含めてください**。

### 5. 統計とレポート

`GetLedgerStats`, `GetUserActivityStats`, `GetFolderStats` を使用して分析を行います。

- **期間指定 (`period`):** `today`, `yesterday`, `this_week`, `last_week`, `this_month`, `last_month`, `this_quarter`, `last_year`, `last_7_days`, `last_30_days` 等。
- **表示形式:** `format="summary"` を使用すると、人間にとって直感的な要約メッセージを取得できます。

### 6. 便利な制約とTips

- **日本語/マルチバイト対応:** `q` や `tags` には日本語をそのまま渡して問題ありません。MCPプロトコルが自動的にエンコーディングを処理します。
- **日付形式:** すべての日付フィルタは `YYYY-MM-DD` 形式で指定してください。
- **検索の粒度:** 多数の候補がある場合は `limit` を小さくし、`include_content=false` に設定してメタデータのみを先に閲覧することで、トークン消費を抑えつつ高速にブラウジングできます。

## 実装例

### ユーザー: "昨日私が作成した日報を見せて"

**1. ツール呼び出し:**
```json
{
  "tool": "SearchLedgers",
  "arguments": {
    "creator_id": 1,
    "created_from": "2025-01-18",
    "created_to": "2025-01-18",
    "q": "日報", 
    "format": "summary"
  }
}
```

**2. LLM応答:**
```
あなたが昨日作成した台帳は2件です。

📋 **見つかった日報:**
• **件名:** 2025年1月18日営業日報
  - **ステータス:** 承認待ち
  - **更新日時:** 2025年1月18日 18:30

• **件名:** 2025年1月18日作業報告
  - **ステータス:** 下書き  
  - **更新日時:** 2025年1月18日 20:15

他にも何かお手伝いできることはありますか？
```

### ユーザー: "私が承認すべきタスクはある？"

**1. ツール呼び出し:**
```json
{
  "tool": "GetPendingApprovals",
  "arguments": {
    "limit": 5
  }
}
```

**2. LLM応答:**
```
はい、現在あなたが承認・点検すべきタスクが3件あります。

🚩 **未処理タスク:**
1. **[点検待ち] 2025年01月分経費精算** (作成者: 田中)
2. **[承認待ち] 新製品開発プロジェクト計画書** (作成者: 佐藤)
3. **[点検待ち] 備品購入申請** (作成者: 鈴木)

詳細を確認して承認または差し戻しを行いますか？
```

### ユーザー: "最近のA社に関する動きを教えて"

**1. 思考:** 
キーワード検索を行い、レコードが見つかればその内容と活動ログ (`GetActivityLog`) を組み合わせて概要を作る。

**2. ツール呼び出し:**
```json
{
  "tool": "SearchLedgers",
  "arguments": {
    "q": "A社",
    "limit": 5,
    "format": "summary"
  }
}
```

**3. ステップアップ（もし見つからなければ意味検索）:**
```json
{
  "tool": "SearchLedgers",
  "arguments": {
    "q": "A社との最近の取引や交渉、課題について",
    "order_by": "semantic_score",
    "limit": 5
  }
}
```

## 注意事項

1. **日付の処理**: `created_from` と `created_to` は YYYY-MM-DD 形式で指定
2. **creator_id**: ユーザーが「私の」「自分の」と言った場合は現在のユーザーIDを使用
3. **format パラメータ**: 必ず `"summary"` を使用して見やすい応答を取得
4. **エラー処理**: ツールからエラーが返された場合は、ユーザーにとって理解しやすい形で説明
