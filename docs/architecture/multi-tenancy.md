# マルチテナント・アーキテクチャ

**関連ドキュメント:** [新マルチテナント実装計画書](/docs/work/2025-08-30_new-multi-tenant-implementation-plan-final.md)

## 1. 概要

LedgerLeapは、複数のテナント（企業、部署、プロジェクトなど）が単一のアプリケーションインスタンスを共有しながら、それぞれのデータを論理的に分離して管理できるマルチテナント・アーキテクチャを採用しています。

本ドキュメントは、その技術的な実装方針と設計上の意思決定について解説します。

## 2. 採用アーキテクチャ: 「シングルDB・`tenant_id`方式」

本プロジェクトでは、安定性と将来の拡張性を両立するため、`stancl/tenancy` ライブラリが公式にサポートする **「シングルDB・`tenant_id`方式」** をアーキテクチャの基盤としています。

### 2.1. 基本概念

-   **単一データベース:** 全てのテナントのデータは、物理的に単一のデータベース内に格納されます。
-   **論理的なデータ分離:** テナントに帰属するデータを持つテーブルには `tenant_id` カラムを追加します。アプリケーションは、クエリ実行時にこの `tenant_id` を `WHERE` 句に含めることで、他のテナントのデータにアクセスできないように制御します。
-   **自動スコープ:** `stancl/tenancy` が提供する `BelongsToTenant` トレイトをEloquentモデルに適用することで、上記の `WHERE` 句の付与が自動的に行われ、開発者はテナント分離を常に意識することなく、安全に開発を進めることができます。

### 2.2. テーブルの分類

本アーキテクチャでは、データベース内のテーブルを以下の2種類に分類して管理します。

-   **中央テーブル (Central Tables):**
    -   テナントに依存せず、システム全体で共有されるデータを格納します。
    -   これらのテーブルには `tenant_id` カラムは存在しません。
    -   **該当テーブル:** `tenants`, `domains`, `users`, `organizations`, `roles`, `permissions`, `model_has_roles`, `role_has_permissions`, `jobs` など。

-   **テナントテーブル (Tenant Tables):**
    -   各テナントに固有のデータを格納します。
    -   必ず `tenant_id` カラムを持ち、`tenants` テーブルと外部キー制約で関連付けられます。
    -   対応するEloquentモデルには `BelongsToTenant` トレイトが適用されます。
    -   **該当テーブル:** `folders`, `ledger_defines`, `ledgers`, `ledger_diffs`, `attached_files`, `tags`, `taggables`, `activity_log` など。

-   **ハイブリッドな関連を持つテーブル:**
    -   `role_folder_permissions` のように、中央テーブルのレコード（`roles`）とテナントテーブルのレコード（`folders`）を紐付ける中間テーブルも存在します。このようなテーブルは、`folder_id` を通じて間接的にテナントが特定されるため、テーブル自体はテナントテーブルとして扱われます。

## 3. 設計の意思決定プロセス

現在のアーキテクチャに至るまで、複数のアプローチが検討されました。その経緯を理解することは、将来の技術選定において重要です。

### 3.1. 検討された他のアプローチ

1.  **マルチDB方式:**
    -   **概要:** テナントごとに物理的に異なるデータベースを用意する方式。
    -   **課題:** 開発初期段階で試みたものの、「中央DBの`users`」と「テナントDBのデータ」を繋ぐリレーションの実装が非常に複雑化しました。特に `stancl/tenancy` の「リソース同期」機能に依存しようとした結果、外部キー制約エラーが頻発し、実装が頓挫しました。

2.  **テーブルプレフィックス方式:**
    -   **概要:** 単一DB内で、テナントごとにテーブル名にプレフィックス（例: `tenant_a_`）を付与してテーブルを物理的に分離する方式。
    -   **課題:** `stancl/tenancy` の標準機能ではない方法で実装を試みた結果、ルーティングの競合という深刻な副作用が発生しました。ライブラリの想定から外れた実装は、将来のバージョンアップで破綻するリスクが高いと判断し、破棄しました。

### 3.2. 「シングルDB・`tenant_id`方式」の採用理由

-   **安定性と信頼性:** `stancl/tenancy` が公式にサポートする、最も実績のある堅牢な方式であるため。
-   **開発効率:** Eloquentのリレーションがほぼ通常通り機能し、開発者はテナント分離の複雑さを意識する必要がほとんどありません。
-   **段階的な拡張性:** 将来、特定のテナントのデータ量が増大し、物理的な分離が必要になった場合でも、`stancl/tenancy` の機能を活用して、**対象のテナントのみを専用データベースに移行する「ハイブリッド構成」へ段階的に移行することが可能**です。

この「まずは堅牢な論理分離基盤を構築し、必要に応じて物理分離へ移行する」という段階的アプローチが、本プロジェクトのリスクとコストを最小化する最適解であると結論付けました。
