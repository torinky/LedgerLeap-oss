# マルチテナント境界

## 概要

LedgerLeap は単一のシステム上で複数の組織やプロジェクトを安全に分離するマルチテナントアーキテクチャを採用しています。このページでは、テナント間の境界がどのように機能するか、何が分離され何が共有されるか、そしてテナント境界がルーティングやデータアクセスにどう影響するかを説明します。

## なぜマルチテナントか

LedgerLeap を利用する組織は、自社だけの専用システムを必要とするとは限りません。複数の部署、プロジェクト、または取引先が 1 つのシステムインスタンスを共有しながら、互いのデータにアクセスできないようにする必要があります。

マルチテナント方式により、次のメリットが得られます。

- **運用コストの削減**: 1 つのシステムで複数の組織をホスティングできる。
- **管理の集中**: ユーザーアカウントやロールの管理はシステム全体で一元化できる。
- **クロステナント制御の防止**: 各テナントのデータは論理的に分離され、他のテナントから読み取ることができない。

## テナント境界のモデル

LedgerLeap は、単一データベース内で `tenant_id` による論理的なデータ分離を行う方式を採用しています。

### 分離されるデータ

次のデータはテナントごとに完全に分離されます。

- フォルダ階層とフォルダ権限
- 台帳定義（テンプレート）
- 台帳レコードとその差分履歴
- 添付ファイルとファイル処理状態
- タグ
- アクティビティログ
- キャッシュ（テナントスコープのキャッシュキーには `tenant_id` を含む）

これらのテーブルにはすべて `tenant_id` カラムが存在し、クエリ実行時に自動的に現在のテナントでフィルタリングされます。

### テナント間で共有されるデータ

次のデータはテナントに依存せず、システム全体で共有されます。

- ユーザーアカウント
- ロール（役割）とパーミッション（機能権限）
- 組織（所属）
- システム設定

1 人のユーザーは複数のテナントに所属でき、テナントごとに異なるロールを割り当てられます。

### 境界を越えるアクセス

テナント間で共有されるデータ（ユーザー、ロールなど）は、中央コンテキストで管理されます。テナントスコープのコードからこれらのデータにアクセスする必要がある場合は、`tenancy()->central()` を使用して一時的に中央コンテキストに切り替えます。

## テナントコンテキストの確立

テナントコンテキストは、次の 2 つの方法で確立されます。

### パスベースのテナント解決

ユーザーがテナントを選択すると、以降の URL にはテナント識別子がパスとして含まれます。

```
https://example.com/{tenant}/...
```

Laravel のルーティングでは `InitializeTenancyByPath` ミドルウェアがこのパスパラメータを読み取り、リクエスト処理の開始時にテナントコンテキストを初期化します。

### リクエストデータベースのテナント解決（API / MCP）

API や MCP のリクエストでは、パスによるテナント特定ができない場合があります。その場合は `InitializeTenancyByRequestData` ミドルウェアがリクエストパラメータ（`snapshot.memo.tenant_id`）からテナントを解決します。

### Livewire コンポーネントのテナント復元

Livewire のコンポーネントは、`InitializesTenantContext` トレイトを使用してテナントコンテキストを管理します。コンポーネントの `mount()` または `boot()` で `tenantId` プロパティを設定し、テナントコンテキストが失われた場合に再初期化します。これにより、`#[Lazy]` コンポーネントや非同期リクエストでも安定してテナントを特定できます。

## トレードオフ

### 単一データベース方式の利点

- **運用のシンプルさ**: データベースは 1 つでよく、バックアップやマイグレーションも単純。
- **クロステナントクエリが可能**: 集計や管理目的で、中央コンテキストから全テナントのデータにアクセスできる。
- **段階的な移行が可能**: 将来、特定のテナントだけを専用データベースに移行するハイブリッド構成に拡張できる。

### 単一データベース方式の制約

- **テナント間の物理的な分離はない**: 論理分離のため、バグや設定ミスによるデータ漏洩リスクは完全には排除できない。
- **大規模テナントの影響**: 1 つのテナントの大規模クエリが他のテナントのパフォーマンスに影響を与える可能性がある。
- **テナント単位のリソース制限が困難**: テナントごとのストレージや CPU 使用量の制限はアプリケーションレベルで実装する必要がある。

### キャッシュ戦略

テナントスコープのキャッシュは、キャッシュキーにテナント ID を含めることで分離しています。

```php
$cacheKey = "folder_tree:{$tenantId}";
Cache::tags(['tenant_access'])->remember($cacheKey, ...);
```

テナントアクセス権限の変更があった場合は、`TenantAccessService::clearAllCache()` で関連キャッシュをクリアします。
