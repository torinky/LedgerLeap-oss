# 権限とフォルダアクセスモデル

## 概要

LedgerLeap は **2 層のアクセス制御モデル** を採用しています。システム全体の機能権限（Spatie Permission）とフォルダ単位のアクセス権限（RoleFolderPermission）が組み合わさることで、きめ細かいアクセス制御を実現します。このページでは、2 層モデルの設計理由、各層の責務、キャッシュ戦略、およびテナント境界を考慮した権限管理の仕組みを説明します。

## なぜ 2 層なのか

単一の権限レイヤーでは、次の要件を同時に満たせません。

- 組織全体として「台帳を作成できる」という機能レベルの制御
- フォルダごとに「このフォルダの台帳は読み取りのみ」というデータレベルの制御

LedgerLeap は、台帳がフォルダに属する階層構造を持ち、かつユーザーは複数の組織やテナントにまたがって業務を行うため、機能権限とフォルダ権限を分離する必要がありました。

## 2 層の権限モデル

### 第 1 層: Spatie Permission（システム権限）

システム全体の機能に対する権限を管理します。ロールに紐づき、ユーザーはロールを通じて間接的に権限を取得します。

| 区分 | 例 |
|---|---|
| 台帳操作 | `view_ledgers`, `create_ledgers`, `update_ledgers`, `delete_ledgers` |
| フォルダ操作 | `view_folders`, `create_folders`, `update_folders`, `delete_folders` |
| ユーザー管理 | `view_users`, `create_users`, `update_users` |
| 組織管理 | `view_organizations`, `create_organizations` |

ユーザーの実効的なシステム権限は、割り当てられた全ロールの権限の和集合です。`User` リソースからユーザーに直接権限を付与することも可能です。

### 第 2 層: RoleFolderPermission（フォルダ権限）

フォルダ単位のアクセスレベルを管理します。ロールに対してフォルダごとに 3 段階の権限を設定します。

| 権限レベル | できること |
|---|---|
| `read` | フォルダと配下の台帳の閲覧 |
| `write` | read + 台帳の作成・編集・削除 |
| `manage` | write + サブフォルダの作成・編集・削除・権限設定 |

フォルダ権限は **親フォルダから子フォルダへ継承** されます。子フォルダに直接権限を設定すると、親からの継承を上書きできます。

### 2 層の関係

台帳に対する操作には、**両方の層の権限が必須** です。

```
台帳を閲覧するには:
  1. Spatie Permission: view_ledgers 権限が必要
  2. RoleFolderPermission: 対象フォルダへの read 以上の権限が必要

どちらか一方でも欠けているとアクセスできません。
```

この 2 段階チェックにより、例えば「システム上では台帳の作成は許可するが、特定のフォルダでは作成させない」といった制御が可能になります。

## 権限の評価フロー

```
ユーザーがフォルダ F の台帳 L を閲覧する
  ├─ Spatie: can('view_ledgers')?
  │   └─ No → 403
  ├─ RoleFolderPermission: F に対する read 以上を持っているか?
  │   ├─ 直接権限がある → Yes
  │   ├─ 親フォルダの継承がある → Yes
  │   └─ どちらもない → 403
  └─ OK → 台帳 L を表示
```

書き込み操作の場合は、`view_ledgers` の代わりに対応する `create_ledgers` / `update_ledgers` / `delete_ledgers` 権限と、フォルダの `write` または `manage` 権限が必要です。

## キャッシュ戦略

権限情報はパフォーマンスのためにキャッシュされます。3 種類のキャッシュが関与します。

| キャッシュ | スコープ | 保持内容 | クリア方法 |
|---|---|---|---|
| ユーザー権限キャッシュ | ユーザー単位 | Spatie の全判定結果 | `flushAllUserPermissionsCache()` |
| テナントアクセスキャッシュ | ユーザー単位 | アクセス可能なテナント一覧 | `TenantAccessService::clearAllCache()` |
| WritableFolder キャッシュ | ユーザー単位 | 書き込み可能なフォルダ ID 一覧 | `WritableFolderRepository::clearAllCache($user)` |

権限変更（ロールへの権限追加/削除、フォルダ権限の変更、ユーザーのロール変更）が発生した場合は、**必ず全キャッシュをクリア** する必要があります。

```php
// ロール変更後のキャッシュクリア例
$user->flushAllUserPermissionsCache();
TenantAccessService::clearAllCache();
WritableFolderRepository::clearAllCache($user);
```

キャッシュキーにはテナント ID を含めることで、テナント間のキャッシュ混在を防止しています。

```php
$cacheKey = "folder_tree:{$tenantId}";
Cache::tags(['tenant_access'])->remember($cacheKey, ...);
```

## テナント境界と権限

ロールとシステム権限は **テナント間で共有** される中央データです。一方、フォルダ権限はテナントスコープのフォルダを参照します。

- あるロールに割り当てられたシステム権限は、全テナントで有効です。
- フォルダ権限はテナントごとに独立して設定されます。
- ユーザーはテナントを切り替えることで、異なるフォルダ権限のもとで作業できます。

テナントスコープのコードから中央データ（ユーザー、ロールなど）にアクセスする必要がある場合は、`tenancy()->central()` で一時的に中央コンテキストに切り替えます。

## トレードオフ

### 2 層モデルの利点

- **きめ細かい制御**: 機能レベルとデータレベルの権限を独立して管理できる。
- **ロールベースの管理**: ユーザー個別ではなくロール単位で権限を設定するため、管理コストが低い。
- **継承による効率化**: フォルダ権限の継承により、大規模なフォルダ階層でも最小限の設定で済む。

### 2 層モデルの制約

- **権限判断の複雑さ**: どの操作にどの権限が必要かの理解コストが高い。利用者には「権限がありません」とだけ表示され、どちらの層が原因かわかりにくい。
- **キャッシュ依存**: 権限変更後のキャッシュクリア漏れが原因で、実際の権限とシステムの判定がずれるリスクがある。変更時は必ず 3 種類のキャッシュすべてをクリアする必要がある。
- **直接権限のリスク**: ユーザーに直接付与した Spatie 権限はフォルダ権限をバイパスできないが、システム権限の直接付与は管理画面から見えにくい。
