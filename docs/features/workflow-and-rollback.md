# ワークフローとロールバック

## 概要

LedgerLeap のワークフロー機能は、台帳の作成や更新に対して点検・承認のプロセスを追加する仕組みです。台帳定義ごとにワークフローの有無を設定でき、有効な場合はステータスが遷移しながらデータの品質と承認経路を管理します。ロールバック機能を使うと、承認済みのレコードを下書きに戻すこともできます。

## ワークフローが必要な理由

情報の品質を保つには、誰でも自由に編集できる状態と、承認を受けてから確定する状態を使い分ける必要があります。ワークフローが有効な台帳では、次のような課題を解決します。

- **未確認の情報が公開されるのを防ぐ**: 点検者が内容を確認してから次の工程に進む。
- **承認プロセスを可視化する**: 誰がいつ点検・承認したかを記録として残す。
- **差し戻しと再提出を可能にする**: 承認者が不備を指摘した場合、作成者は修正して再提出できる。
- **承認済みデータの変更を追跡する**: 承認後の編集はロールバックを経由するため、変更履歴が確実に残る。

## ステータス遷移

ワークフローが有効な台帳のステータスは、次のように遷移します。

```text
下書き ──提出──► 点検待ち ──点検──► 承認待ち ──承認──► 承認済み
                 ──差し戻し─► 下書き            ──差し戻し─► 下書き
承認済み ──ロールバック──► 下書き
```

### 各ステータスの意味

| ステータス | 説明 |
|---|---|
| 下書き（DRAFT） | 作成または編集後の状態。点検・承認はまだ行われていない。 |
| 点検待ち（PENDING_INSPECTION） | 提出済みで、点検者の確認を待っている状態。 |
| 承認待ち（PENDING_APPROVAL） | 点検が完了し、承認者の判断を待っている状態。 |
| 承認済み（APPROVED） | 承認が完了し、データが確定した状態。 |
| なし（NONE） | ワークフローが無効な台帳定義で使用される。ステータス管理は行われない。 |

## ワークフローの各操作

### 提出（Submit）

下書きの台帳を点検待ちに進めます。提出時には、必要に応じて点検者や承認者を指定できます。

### 点検（Inspect）

点検待ちの台帳を確認し、問題がなければ承認待ちに進めます。問題がある場合は、コメントを添えて下書きに差し戻せます。

### 承認（Approve）

承認待ちの台帳を確認し、問題がなければ承認済みにします。複数承認者が必要な設定では、全員の承認が完了するまで承認待ちのままです。問題がある場合は差し戻せます。

### 差し戻し（Reject）

点検待ちまたは承認待ちの台帳を、コメントを添えて下書きに戻します。作成者は指摘事項を修正したあと、再度提出できます。

### ロールバック（Rollback）

承認済みの台帳を下書きに戻します。承認後のデータを修正する場合は、この操作を行ってから編集します。ロールバックの理由はコメントとして記録されます。

## 操作の使い分け

| 操作 | 実行できるステータス | 操作者 | 遷移先 |
|---|---|---|---|
| 提出 | 下書き | 作成者 | 点検待ち |
| 点検 | 点検待ち | 点検者 | 承認待ち / 下書き |
| 承認 | 承認待ち | 承認者 | 承認済み / 下書き |
| 差し戻し | 点検待ち / 承認待ち | 点検者 / 承認者 | 下書き |
| ロールバック | 承認済み | 権限のあるユーザー | 下書き |
| 編集保存 | 下書き | 作成者 | 下書き（維持） |

## 点検・承認タスクの確認

自分に割り当てられた点検・承認タスクは、次の場所で確認できます。

- **My Portal**: ログイン後のダッシュボードに、承認待ちタスクの件数が表示されます。
- **点検・承認待ち一覧**: 専用の一覧画面で、すべてのタスクを確認・操作できます。

## 注意点と制約

- ワークフローが有効な台帳定義で作成された台帳は、承認済みになるまでは確定データとして扱われません。
- 承認済みの台帳を編集するには、必ずロールバック操作が必要です。ロールバックなしで編集することはできません。
- 点検者や承認者は、台帳定義の設定または提出時に指定します。権限のないユーザーは操作できません。
- ワークフローを無効に設定した台帳定義では、ステータスが「なし（NONE）」になり、作成・更新が即座に反映されます。
