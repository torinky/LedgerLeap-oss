# テナントセットアップコマンド実装計画書

**日付:** 2025年8月31日
**作成者:** Gemini
**ステータス:** 計画中
**関連ドキュメント:** [新マルチテナント実装計画書](/docs/work/2025-08-30_new-multi-tenant-implementation-plan-final.md)

## 1. 目的

テナントの新規作成と、それに付随するデータベースのマイグレーション、初期データのシーディング、デフォルトのフォルダ構造作成、管理者ユーザーの割り当てといった一連の初期設定作業を自動化する、単一のArtisanコマンド `app:setup-tenant` を実装する。

これにより、手作業による設定ミスを防ぎ、誰でも一貫性のあるテナント環境を迅速に構築できるようにすることを目的とする。

## 2. コマンド仕様

*   **コマンド名:** `app:setup-tenant`
*   **引数:**
    1.  `tenant_id`: 作成するテナントのID（必須）。URLのパスや、DBのプレフィックスなどに使用される。
    2.  `admin_email`: テナントの管理者に設定するユーザーのメールアドレス（必須）。
*   **実行例:** `php artisan app:setup-tenant acme admin@example.com`

## 3. 実装ステップ

### ステップ1: Artisanコマンドの雛形作成

*   **目的:** `make:command` を使用して、基本的なファイル構造を作成する。
*   **タスク:**
    1.  `vendor/bin/sail artisan make:command SetupTenant` を実行する。
*   **検証方法:**
    *   `app/Console/Commands/SetupTenant.php` ファイルが作成されていることを確認する。
        *   **結果:** `INFO  Console command [app/Console/Commands/SetupTenant.php] created successfully.` が表示され、ファイルが作成されたことを確認済み。
    *   **完了:** ステップ1は完了しました。

### ステップ2: コマンドのシグネチャと説明の定義

*   **目的:** 作成したコマンドクラスに、正しいコマンド名、引数、および説明を設定する。
*   **タスク:**
    1.  `app/Console/Commands/SetupTenant.php` を開く。
    2.  `$signature` プロパティを `app:setup-tenant {tenant_id : The ID of the tenant} {admin_email : The email of the admin user}` に変更する。
    3.  `$description` プロパティを `Create a new tenant and run all necessary setup processes` に変更する。
*   **検証方法:**
    *   `vendor/bin/sail artisan app:setup-tenant --help` を実行し、設定した引数と説明が表示されることを確認する。
        *   **結果:** `Description:` と `Arguments:` に設定通りの内容が表示されることを確認済み。
    *   **完了:** ステップ2は完了しました。

### ステップ3: テナント作成とドメイン紐付け処理の実装

*   **目的:** コマンドの中核となる、テナントとドメインをデータベースに作成するロジックを実装する。
*   **タスク:**
    1.  `SetupTenant.php` の `handle()` メソッド内に、引数で受け取った `tenant_id` を使用して `App\Models\Tenant::create()` を実行する処理を記述する。 **(当初の実装 `DB::insert` から修正)**
    2.  作成したテナントに対し、`{tenant_id}.localhost` という形式のドメインを `domains()->create()` で紐付ける処理を記述する。
    3.  テナント作成前に、テナントIDとドメインが既に存在しないかチェックする処理を追加する。
    4.  処理の途中でエラーが発生した場合に、作成したテナントを削除するロールバック処理を `try-catch` ブロックに追加する。
*   **検証方法:**
    *   コマンドをテスト実行し、`tenants` テーブルと `domains` テーブルにレコードが正しく作成されることを確認する。
    *   再度コマンドを実行し、重複エラーが正しくハンドリングされることを確認する。
*   **結果:**
    *   `vendor/bin/sail artisan app:setup-tenant test-tenant test@example.com` を実行し、コマンドが正常に完了することを確認済み。
    *   `DB::table('tenants')` および `DB::table('domains')` でレコードが作成されていることを確認済み。
        *   *特記事項: `tinker` 環境から `Tenant::find()` でモデルを直接検索できなかったが、コマンド内の重複チェックは正常に機能した。これはコンソール実行時と `tinker` でのコンテキストの違いによるものと推測される。*
    *   重複実行時に `Tenant with ID 'test-tenant' already exists.` というエラーが表示されることを確認済み。
*   **完了:** ステップ3は完了しました。

### ステップ4: DBマイグレーションとシーディング処理の実装

*   **目的:** テナント作成に続き、そのテナント専用のDB設定を行う処理を実装する。
*   **タスク:**
    1.  `handle()` メソッド内に、`Artisan::call('tenants:migrate', ['--tenants' => [$tenant->id]])` を実行する処理を追加する。
    2.  `handle()` メソッド内に、`Artisan::call('tenants:seed', ['--tenants' => [$tenant->id]])` を実行する処理を追加する。
    3.  シーディング処理の前後で `activity()->disableLogging()` と `activity()->enableLogging()` を呼び出し、シーディング中のアクティビティ記録を一時的に無効化する。 **(シーディング中のエラー回避のため追加)**
*   **検証方法:**
    *   コマンドをテスト実行し、コンソールにマイグレーションとシーディングの実行結果がエラーなく表示されることを確認する。
*   **結果:**
    *   コマンド実行時に、マイグレーションとシーディングが正常に完了することを確認済み。
*   **完了:** ステップ4は完了しました。

### ステップ5: 初期フォルダと管理者割り当て処理の実装

*   **目的:** テナント固有のアプリケーションデータと、管理者権限の初期設定を行う。
*   **タスク:**
    1.  `handle()` メソッド内で、`tenancy()->run()` を使用して、作成したテナントのコンテキスト内で処理を実行するクロージャを定義する。
    2.  クロージャ内で、ルートフォルダ (`/`) を作成するロジックを記述する。
    3.  引数で受け取った `admin_email` を元に `User` を検索（存在しない場合はエラーメッセージを表示）し、そのユーザーに `Admin` ロールを付与する処理を記述する。
*   **検証方法:**
    *   コマンドをテスト実行し、`folders` テーブルと `model_has_roles` テーブルにレコードが正しく作成されることを確認する。

### ステップ6: 完了メッセージとエラーハンドリング

*   **目的:** コマンドの実行結果をユーザーに分かりやすくフィードバックする。
*   **タスク:**
    1.  すべての処理が正常に完了した場合に、成功メッセージと作成されたテナントの情報を表示する。
    2.  各ステップでエラーが発生した場合（例: ユーザーが存在しない、DB接続に失敗した等）に、処理を停止し、適切なエラーメッセージを表示する。
*   **検証方法:**
    *   コマンドが正常に終了すること、および意図的にエラーを発生させた場合に適切なメッセージが表示されることを確認する。

## 4. 実装中の課題と解決策

当初の計画では単純な実装を想定していたが、シーディングの過程で複数の問題が発生したため、以下の通り対応を行った。

1.  **シーダーによるテナントの二重作成:**
    *   **問題:** `DatabaseSeeder` 内で `Tenant::create()` が呼ばれており、コマンド本体での作成と重複していた。
    *   **解決策:** `DatabaseSeeder` からテナント作成ロジックを削除した。

2.  **シーダー実行時のDB接続コンテキスト問題:**
    *   **問題:** `tenants:migrate --seed` や `db:seed` を実行した際、シーダーがテナントDBではなく中央DBに接続してしまい、`users` テーブルでユニークキー制約違反が発生した。
    *   **解決策:** `tenants:seed` コマンドを使用するように修正した。

3.  **アクティビティログによる通知サービスの予期せぬトリガー:**
    *   **問題:** シーディング中にユーザー等が作成されるとアクティビティが記録され、`NotificationService` が呼び出された。しかし、コンソール実行のコンテキストでは通知に必要な情報（フォルダ等）が不足しており、`null` アクセスエラーが発生した。
    *   **解決策:** シーディング処理の前後で `activity()->disableLogging()` と `activity()->enableLogging()` を呼び出し、この間のアクティビティ記録を一時的に無効化することで問題を回避した。

これらの対応により、コマンドは安定して動作するようになった。
