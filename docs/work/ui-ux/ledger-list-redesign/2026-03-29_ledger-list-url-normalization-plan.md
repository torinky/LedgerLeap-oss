# 台帳一覧URL正規化 計画書 (2026-03-29)

**作成日:** 2026-03-29  
**関連 Issue:** [#121 台帳一覧URLの正規化: 共有リンクと検索/階層状態の整理](https://github.com/torinky/LedgerLeap/issues/121)

---

## 1. 概要

台帳一覧画面および詳細画面の表示状態を再現可能な形で正規化し、QRコード・クリップボード・画面遷移で使うリンクを単一のルールに揃える。

現状は `q` / `l` / `f` / `cf` / `dl` / `sem` / `highlight` などの状態が分散しており、URLの意味が読み取りにくい。`highlight` はユーザーシナリオ上重要な共有対象であり、詳細画面では独立した URL クエリとして再現できる必要がある。さらに、階層位置や表示中の台帳文脈が複数のキーやルートに散らばっているため、URL生成の責務が複雑化している。

また、表示レベル、検索オプション、ソート列、列フィルタのように、現時点では query 化されていない表示状態も含めて再現可能性を整理する必要がある。少なくとも台帳リストと詳細画面の表示状態は、再訪時に完全に網羅できるようにする。

この計画では、**既存互換性を考慮せず**、新規生成URLを canonical 形式へ一本化する方針を定める。

---

## 2. 背景と目的

### 2.1 背景

- 台帳一覧画面は検索、フォルダ階層、台帳定義、表示レベル、並び順など複数の状態を持つ
- QRコードやクリップボード共有のたびに、同じ状態を別々のクエリで表現している
- 短縮クエリは便利だが、要素が増えるほど URL の意味が分かりづらくなる
- 共有URLは人間が読む・貼る・説明する前提なので、可読性と再現性が重要

### 2.2 目的

- URL の意味を一貫させる
- 共有リンクの生成ルールを一本化する
- 一覧の文脈と検索状態の責務を分離する
- 一覧・詳細それぞれの表示状態を、再現対象と除外対象に分けて整理する
- テスト可能な URL 仕様に整理する

## 4. ペルソナ・ユーザーシナリオ要件

`docs/function/PersonaUseCaseScenario.md` の内容を踏まえ、共有URLに必要な要件を次のように整理する。

### 4.1 実務担当者

- 今見ている一覧をすぐ共有したい
- 再訪時に同じ検索条件・絞り込み・階層文脈を復元したい
- 長すぎる URL は避けたい

### 4.2 現場リーダー / 作業班長

- フォルダ階層や台帳定義の範囲を、リンクだけで説明できる必要がある
- 代理で見た一覧をそのまま共有できる必要がある
- 詳細画面の補助状態を一覧URLに混ぜたくない

### 4.3 管理者

- 監査や確認で、再現性のある URL を使いたい
- 絞り込みの意味が読めることを重視する
- 複数の状態がある場合でも、どの状態が URL に残るかを理解しやすいことが重要

### 4.4 表示状態再現の対象

この計画では、現時点で query 化されていない表示状態も調査対象に含める。

#### 台帳リスト画面

- 表示レベル
- 検索オプション
- ソート列
- 昇順 / 降順
- 列フィルタ
- フォルダ / 台帳定義の選択状態
- 現在の階層位置
- 一覧上の表示切替や補助表示

#### 台帳詳細画面

- 表示タブ / セクションの選択状態
- 折りたたみ / 展開状態
- 関連表示の切替状態
- 一覧へ戻ったときの文脈再現に必要な補助状態

> 目的は「今どの画面を見ているか」だけでなく、「その画面がどの設定で見えていたか」まで再現できるようにすること。

### 4.5 localStorage に残す状態の候補

以下は、現時点では localStorage の方が自然な状態として扱う。

- 台帳リストのフォルダツリーの展開 / 折りたたみ
- 台帳詳細のセクション折りたたみ / 展開
- 画面をまたいでも個人の読みやすさだけを維持したい補助状態

ただし、**URL を正本としつつ localStorage にもミラーする** 併用はあり得る。
この場合は、次のルールを Sprint 0 で固定する。

- URL に値がある場合は URL を優先する
- URL が空のときだけ localStorage を初期値として使う
- ユーザー操作があった場合は URL と localStorage を同時更新する
- 共有リンクの生成時は、localStorage ではなく URL の canonical 値のみを使う

現時点のペルソナ / ユーザーシナリオでは、ツリー展開や詳細セクションのような **純 UI 状態** は localStorage 優先でよい。一方、表示レベルや比較状態のように再訪時の見え方を変える状態は、URL 正本を維持しながら localStorage を補助的に使う余地がある。

## 5. 正規URL方針

### 5.1 基本方針

- 一覧スコープは **パス** で表現する
- 一覧状態を表すクエリは **最小限** に絞る
- `l` / `f` / `cf` は一覧の選択状態として **共有URLへ保持する**
- `highlight` は詳細画面の共有対象として URL に含める
- 詳細画面や単一レコード文脈は一覧URLから切り離す
- 表示状態のうち、再訪時の見え方に必要なものは query でもパスでも表現候補として評価する
- localStorage は補助記憶として使ってよいが、共有URLの正本にはしない

### 5.2 仕様の考え方

- **パスで表すもの**
  - 一覧の大枠スコープ
  - どのフォルダ配下か
  - どの台帳定義配下か
- **query で表すもの**
  - 検索語
  - ソート
  - 表示レベル
  - 必要最小限のフラグ
- **調査対象として扱う表示状態**
  - 一覧の表示レベル、検索オプション、ソート、列フィルタ、選択状態
  - 詳細画面のタブ / セクション / 比較状態 / 補助表示
- **一覧URLに含めないもの**
  - 一覧の一時的なハイライト
  - 詳細画面の localStorage 管理が妥当な補助状態（セクション折りたたみ等）
  - 生成元固有の内部状態
  - ※ただし `highlight` は詳細共有URLの対象として別扱い

### 5.3 想定する整理イメージ

- `ledger.index` は一覧の基準点
- `ledgersByFolderId` / `ledgersByDefineId` はスコープ指定の入口として整理する
- 一覧の選択状態は `l` / `f` / `cf` で共有し、詳細の補助状態と混ぜない

> 注: この計画では、最終的な route / query の命名は実装段階で確定する。
> ただし、**「短縮キーを増やさない」** ことと **「共有URLを単純化する」** ことは先に固定する。

#### 6.4.1 Sprint 4-1: 状態設計

- FileInspector の `open` / `selectedTab` / 展開・プレビュー系の状態はローカル状態として扱う
- `FileInspector::$fileId` はコンポーネント内部キー、`selectedFileId` はその共有URLキーとして扱う
- `selectedLedgerId` は `selectedFileId` から導出されるフォーカス対象 ID とし、別の永続キーとしては持たない
- `selectedColumnId` は `selectedFileId` と `column_id` の組み合わせから導出する補助状態として扱う
- `isFileInspectorOpen` は `selectedFileId` の有無から導出できる補助状態として扱う
- 共有状態は URL に載せる最小単位を `selectedFileId` に絞り、他は Livewire の一時状態に留める

| 状態 | 所有先 | URL 化 | 役割 |
| :--- | :--- | :---: | :--- |
| `open` | FileInspector | - | ファイルインスペクタの開閉 |
| `fileId` | FileInspector | - | 現在開いている添付ファイルの内部キー |
| `selectedTab` | FileInspector | - | content / details / history / permissions の表示切替 |
| `selectedFileId` | URL / 一覧の一時状態 | ✅ | 共有URLと一覧復帰に使う添付ファイルID |
| `selectedLedgerId` | 一覧の一時状態 | - | 画面内へ戻す台帳レコード |
| `selectedColumnId` | 一覧の一時状態 | - | 強調すべき台帳カラム |
| `isFileInspectorOpen` | 一覧の一時状態 | - | ファイルインスペクタ開閉の同期フラグ |

**決定理由:**
- 実務担当者・現場リーダーは「添付資料を見ながら判断する」「チーム内の情報（台帳、添付ファイル）を共有する」ため、ファイル単位の共有導線が必要だから
- `selectedLedgerId` / `selectedColumnId` はファイル→台帳・列の逆引き結果なので、永続キーではなく導出状態として扱うのが自然だから
- `isFileInspectorOpen` は `selectedFileId` の有無で復元できるため、独立した共有キーにする必要がないから

**再評価メモ:**
- `selectedFileId` を共有しない場合、包含する台帳レコードの特定に追加操作が必要になる
- 共有したい本体は「ファイル」だが、台帳リストへ戻すための最小共有単位としても `fileId` が必要
- したがって、Sprint 4-1 では `selectedFileId` を URL 化し、`selectedLedgerId` / `selectedColumnId` / `isFileInspectorOpen` は派生状態として扱う

#### 6.4.2 Sprint 4-2: イベントと復元方針

- `open-file-inspector` の payload は `id` を必須、`column_id` と `search` を任意とする
- `selectedFileId` は `id` から復元し、`selectedLedgerId` は `AttachedFile::ledger_id` から導出する
- `selectedColumnId` は payload の `column_id` を優先し、未指定なら `AttachedFile::column_id` から導出する
- close 時は FileInspector のローカル状態を解放しつつ、一覧側へ「復元対象の破棄」を通知する
- 一覧側は `selectedFileId` を受けて、対象行が画面内にあればスクロール・強調、未描画なら状態だけ保持する

**決定理由:**
- 複数添付・複数カラムに対応するため、ファイル単位の選択だけでなく列情報も復元に必要だから
- ただし列情報はファイルの属する文脈であり、永続URLキーとして独立させず、導出状態として扱う方が責務が明確だから
- close は「開閉の終了」であり、共有状態の破棄と一覧側の復元解除を同時に行うのが自然だから
- `search` は FileInspector の補助表示であり、選択復元の主キーにはしない

**最小復元イベントの考え方:**
- open: `id`, `column_id`, `search`
- close: `selectedFileId` の破棄通知のみ
- 一覧側復元: `selectedFileId` から `selectedLedgerId` / `selectedColumnId` を導出して行・セルへ反映

#### 6.4.3 Sprint 4-3: 一覧可視化

- `RecordsTable` / `table-row` に、対象レコードの強調と画面内フォーカスを反映する
- 一覧内に存在する場合は自動スクロール、外れる場合は案内表示にするかを決める
- 既存 canonical state と衝突しない補助表示として実装する

#### 6.4.4 Sprint 4-4: テスト整備

- FileInspector の open / close で共有状態が更新されるテストを追加する
- 一覧側で対象レコードが強調・スクロール対象になるテストを追加する
- モック添付ファイルと実データの両方で回帰を確認する

## 7. テスト計画

### 7.1 URL生成テスト

- canonical URL が単一のルールで生成されること
- `l` / `f` / `cf` に依存しないこと
- 同じ状態から同じ URL が返ること

### 7.2 状態復元テスト

- フォルダ文脈が正しく復元されること
- 台帳定義文脈が正しく復元されること
- 検索語・表示レベルが期待通りに反映されること
- 空・単一・複数条件の差分を判定できること
- 列フィルタ、ソート列、検索オプションが期待通りに反映されること
- 台帳詳細画面の表示状態（タブ、折りたたみ、補助表示など）が再現対象として扱われること
- `highlight` クエリから詳細ハイライトが再現されること

### 7.3 共有導線テスト

- QRコードの URL が canonical 形式になること
- クリップボードにコピーされる URL が canonical 形式になること
- 一覧画面から生成するリンクと、他画面から生成するリンクが一致すること

### 7.4 回帰テスト

- 詳細画面や補助導線の状態が一覧URLへ混入しないこと
- `highlight` が詳細画面の独立クエリとして共有されること
- 直接アクセス、再読み込み、戻る/進むで破綻しないこと
- 無効な ID を含む場合のフォールバックが定義どおりであること

> 実装後は、URL文字列そのものの検証だけでなく、一覧の表示状態が一致していることまで確認する。

## 8. リスクと非対象

### 8.1 リスク

- 互換性を追わないため、旧URLは利用できなくなる
- URL 正規化の過程で、一覧状態の復元ロジックが崩れる可能性がある
- 共有導線が複数あるため、1か所の修正漏れでルールが再分岐する恐れがある

### 8.2 非対象

- 検索アルゴリズムの変更
- 権限モデルの変更
- 一覧 UI の再設計
- 既存URL互換の維持
- Mroonga / semantic search の挙動変更
- 表示状態の再現対象として採用しない UI 補助要素の大幅な再設計
