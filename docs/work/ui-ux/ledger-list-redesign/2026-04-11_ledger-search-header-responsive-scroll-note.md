# 台帳一覧検索ヘッダー レスポンシブ/スクロール境界メモ (2026-04-11)

**作成日:** 2026-04-11  
**対象:** `resources/views/components/ledger/search.blade.php` / `resources/views/livewire/ledger/index-manager.blade.php`

---

## 1. 正としたデザイン

このタスクでは、台帳一覧の検索ヘッダーを「結果一覧を邪魔しない、縦方向に薄い導線」として再構成した。

最終的に正とした要点は次のとおり。

- ヘッダー上側は短いブランド帯としてまとめ、縦方向の余白を極小化する
- 検索キーワード入力はラベルなしの主役として大きめに扱う
- `sort_by` と `per_page` は、完全に1列へ落ちる前の幅では横並びのまとまりを維持する
- 普段使わないオプションは `details` で折りたたみ、必要時のみ展開する
- 中間ブレークポイントでは横並びの情報密度を保ち、最終的に狭い画面で縦並びへ落ちるのは許容する

## 2. 何を学んだか

### 2.1 上側カードの空きは、ヘッダーで吸収してよい

初期案ではカードの上下パディングが大きく、結果一覧より検索カードが目立ちすぎていた。
最終的には、ヘッダー帯を短くまとめることで、カード本体の `pt` / `pb` をかなり薄くしても見た目が成立した。

**学び:**
- ヘッダーが十分に情報を持っていれば、カード上側は `pt-0` 近辺まで詰めてもよい
- ただし、折りたたみの summary や入力欄が窮屈になるなら、上下余白よりも先にヘッダー文量を削る

### 2.2 横並びのまとまりは「完全な1列崩れ」の前に保つ

`sort_by` と `per_page` は、画面幅が中間の段階では別々に落とすのではなく、同じグループとして横並びを維持した方が自然だった。

**学び:**
- 途中段階のレスポンシブでは、`sm` / `md` で横並びのまとまりを残す
- 最終的に狭い幅で縦並びになること自体は問題ない
- ただし、1つだけ先に縦へ落ちると「制御が散っている」印象になりやすい

### 2.3 キーワード入力は Google 的な主役に寄せる

検索欄はラベルを省略し、プレースホルダーとサイズ感で意味を補う方が、一覧の導線として自然だった。

**学び:**
- ラベルを消す代わりに、入力欄を大きくして視認性を上げる
- 余白よりも「入力のしやすさ」を優先する
- 画面の顔はヘッダーでも、実際の操作の主役は検索欄

### 2.4 スクロール時の「どこまで隠れるか」を見落とさない

今回の検索カードは sticky で上部に残るため、スクロール後に下の本体コンテンツがどこまで見えているかを考える必要があった。

**学び:**
- sticky な検索ヘッダーは、見た目の高さだけでなく「スクロール後にコンテンツを何 px 隠すか」を確認する
- 余白を詰めると、結果一覧の初見領域が広がり、一覧主導の画面に戻せる
- ブレークポイント調整は、単純な列数変更ではなく「隠れる領域の合計」で評価する

## 3. 未来の保守者向けメモ

### 3.1 このパターンを再調整するときのチェック項目

- `sm` / `md` / `lg` / `xl` で、`sort_by` と `per_page` がいつ横並びから崩れるか
- sticky ヘッダーの高さが、結果一覧の初期視認領域をどの程度削るか
- `details` を開いたときに、ヘッダー全体が長くなりすぎないか
- 検索欄がラベルなしでも、placeholder とサイズだけで十分に理解できるか
- オプション群が折りたたまれた状態でも、状態 badge で内容を把握できるか

### 3.2 skill 化の結果

この学びは、台帳一覧の検索ヘッダーに強く結びついた feature-local な調整から、再利用可能な UI レビュー skill へ昇格した。

昇格先:

- `.github/skills/search-header-responsive-layout/SKILL.md`

昇格した判断軸:

- sticky / responsive / collapse をまたぐ検索ヘッダーのレイアウト判断
- breakpoint 途中で `sort_by` / `per_page` の横並びを維持するかどうか
- スクロール時にどれだけ結果一覧を隠すかという occlusion 評価

`docs/work` のこの記録は、skill の evidence として残しつつ、今後の再調整時の参照先として維持する。
