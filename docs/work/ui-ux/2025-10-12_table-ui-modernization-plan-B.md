## 📋 概要

### 1. 目的

当初計画の議論を経て、**「データ本文の可読性」**よりも**「メタ情報（スコア、ステータス）の俯瞰・一覧性」**を優先するという新たな要件が明確になった。
本ドキュメントは、この要件に基づき、当初計画とは異なるアプローチでUI改善を実現することを目的とする。具体的には、アクションの発見しやすさを確保しつつ、スコアとステータスを常時確認できるUIを目指す。

### 2. 背景

[当初計画](./2025-10-12_table-ui-modernization-plan.md)では、ホバー時にアクションやメタ情報を表示する「データファースト」な設計を推奨していた。しかし、議論の結果、以下のニーズがより重要であることが判明した。

- 複数の行のスコアやステータスを、ホバー操作なしで一覧比較したい。
- データ本文が多少隠れても、詳細はボタンから確認できるため許容できる。
- 台帳ごとに内容が異なるデータカラムのソート/フィルタは、統合ツールバーにまとめず、従来通り各カラムヘッダーで行いたい。

これらの要求を満たすため、本B案を策定する。

### 3. 設計思想

1.  **メタ情報ファースト:** テーブルの役割を「データを読む場所」から、「メタ情報で全体を俯瞰し、当たりをつけるインデックス」と再定義する。スコアとステータスの常時可視性を最優先する。
2.  **アクションの常時可視化:** 編集・詳細ボタンを常に表示し、ユーザーが操作に迷わないようにする。
3.  **操作場所の論理的分割:** 全台帳で「共通」の項目（スコア、ステータス）の操作は統合ツールバー、「個別」の台帳データカラムの操作は各カラムヘッダー、と役割を明確に分離する。

## 🏆 最終仕様

1.  **アクションボタン (編集/詳細):**
    *   **場所:** 行の先頭（左端）に配置。
    *   **表示:** **常時表示**し、カラム幅は最小限（例: `w-20`）に固定する。
2.  **スコア & ステータス:**
    *   **カラム:** テーブル本体からは削除する。
    *   **表示:** 各行の**右端揃え**で、**透過度30%程度の大きめのラベルを常時オーバーレイ表示**する。
    *   **視覚効果:** ラベルには `backdrop-blur` （背景ぼかし）を適用し、デザイン性と可読性を補助する。
    *   **インタラクション:**
        *   ユーザーが行にホバーすると、その行のラベルの**不透明度が100%に上がる**。
        *   同時に、行全体の背景色も少し濃く（`group-hover`）し、操作対象を明確にする。
3.  **ソート & フィルタ:**
    *   **スコア/ステータス:** **統合ツールバー**で操作する。
    *   **台帳データカラム:** 従来通り、**各カラムヘッダー**で操作する。

## 📐 実装計画と結果

### Phase 1: 統合ツールバーの実装 (スコア・ステータス専用)

**目的:** スコアとステータスに関するソート・フィルタ機能を、テーブル上部のツールバーに実装する。

#### Step 1.1: `RecordsTable.php` の拡張

-   **当初計画:** スコア・ステータス用のフィルタリングプロパティ（`$filterScoreMin`, `$filterStatus` 等）をLivewireコンポーネントに追加し、`render()` メソッド内でクエリを構築するロジックを追加する。
-   **実装結果と変更点:**
    *   `$filterStatus` は既存で存在し、`render()` メソッド内でクエリに適用済み。
    *   スコアソートのUIを統合ツールバーに実装するため、以下のプロパティとメソッドを追加・修正した。
        *   `public string $orderByLabel = '';`: 現在のソート対象の表示名を保持するプロパティ。
        *   `private function getStandardSortLabel(string $columnName): string`: `composite_score`, `created_at`, `updated_at` といった標準ソートキーに対応する翻訳済みラベルを返すヘルパーメソッド。
        *   `mount()` メソッド: 初期表示時に `$orderBy` の値に基づいて `$orderByLabel` を設定するロジックを追加。
        *   `sort($columnName, $columnLabel = null)` メソッド:
            *   シグネチャを `$columnLabel` を受け取れるように変更。
            *   `$this->orderByLabel` を `$columnLabel` または `getStandardSortLabel` の結果で設定するように変更。
        *   `updatedOrderBy($value)` ライフサイクルフック:
            *   ツールバーのソートドロップダウンからの変更時に `$this->orderByLabel` を `getStandardSortLabel($value)` で更新。
            *   **重要:** カスタムカラムソート中に、ツールバーのドロップダウンで「現在ソート中のカラム名」のオプションを再度選択した場合、ユーザーが標準ソートに戻したい意図があると解釈し、`$this->orderBy` を `composite_score` (デフォルトのスコアソート) に戻すロジックを追加した。
        *   `isStandardSort` プロパティおよび `render()` メソッド内の関連ロジックは、`orderByLabel` の導入により不要となったため削除した。
-   **成功基準:** プロパティの変更に応じて、表示されるレコードが正しくフィルタリング・ソートされる。`$orderByLabel` が現在のソート状態を正しく反映する。

#### Step 1.2: 統合ツールバーコンポーネントの作成

-   **当初計画:** スコアとステータスのソート・フィルタ機能のみを持つ、新しいBladeコンポーネント (`resources/views/components/ledger/meta-toolbar.blade.php`) を作成する。
-   **実装結果と変更点:**
    *   新しいコンポーネントは作成せず、既存の `resources/views/components/ledger/search.blade.php` を統合ツールバーとして拡張した。
    *   `search.blade.php` の修正内容:
        *   `@props(['hasWorkflowEnabled', 'orderBy', 'orderByLabel'])` を追加し、Livewireコンポーネントから必要なプロパティを受け取れるようにした。
        *   ソート順を選択する `<select>` ボックスを追加。
        *   この `<select>` ボックスは常に表示され、`wire:model.live="orderBy"` で `RecordsTable` の `$orderBy` プロパティにバインドされる。
        *   `$orderByLabel` が空でない場合（＝カスタムカラムソート中の場合）、そのラベルを `value="{{ $orderBy }}" selected` の `<option>` としてセレクトボックスの先頭に動的に追加する。これにより、ユーザーは現在のソート状態を認識しつつ、他の標準ソートを選択して戻ることが可能になった。
        *   ステータスフィルタの `<option value="">{{ __('ledger.all') }}</option>` に対応する日本語翻訳キー `ledger.all` を `lang/ja.json` に追加した。
        *   「表示件数」のラベル部分を、DaisyUI の `label` コンポーネントのスタイルに揃えるように修正し、統合ツールバー内のUI要素の視覚的な一貫性を向上させた。
-   **成功基準:** ツールバーのUIが正しく表示され、Livewireプロパティと連携して動作する。ソート状態がツールバーに正しく反映され、変更可能である。

#### Step 1.3: テストの実装

-   **作業内容:** スコア範囲やステータスでフィルタした際に、期待通りのレコードのみが表示されることを確認するフィーチャーテストを作成する。
-   **成功基準:** ツールバー関連のテストがすべてパスする。
-   **現状:** 未実施。

-   **成功基準:** 仕様通りにオーバーレイラベルが表示され、ホバー時のインタラクションが正しく動作する。スコアとステータスの意味がアイコンによって視覚的に補強される。

## 💡 当初計画からの変更点と判断理由

### 1. 統合ツールバーのソートUI

-   **当初計画:** 「スコアソートのスイッチ」を統合ツールバーに追加。
-   **変更点:** スイッチではなく、セレクトボックスによるソートキー選択UIを実装。
-   **判断理由:**
    *   **拡張性:** スコアだけでなく、作成日、更新日など複数の標準ソートキーに対応するため、スイッチよりもセレクトボックスの方が拡張性が高いと判断した。
    *   **UIの一貫性:** ソート関連の操作を一つのUI要素に集約することで、ユーザーが操作に迷いにくくなる。

### 2. カラムソート時の統合ツールバーの表示と操作性

-   **当初計画:** カラムソート時は統合ツールバーのソートUIを `disabled` にし、別のバッジでソート中であることを示す。
-   **変更点:** 統合ツールバーのセレクトボックスを常に表示し、カスタムカラムソート中は、そのカラム名を `disabled` ではない `<option>` としてセレクトボックスの先頭に動的に追加する方式に変更。
-   **判断理由:**
    *   **ユーザー理解度とUXの向上:**
        *   当初案の「カラムID表示」ではユーザーが理解できないというフィードバックがあったため、`table-header.blade.php` からカラムの表示名 (`$column_define->name`) を `sort()` メソッドに渡すように変更し、ツールバーにはその表示名を表示するようにした。
        *   カスタムカラムソート中にセレクトボックスが `disabled` になると、標準ソートに戻す手段が分かりにくくなるという課題があった。
        *   セレクトボックス内に現在のカスタムソート状態を `selected` オプションとして表示することで、ユーザーは現在の状態を認識しつつ、他の標準ソートを直接選択してソートを切り替えることが可能になり、UXが向上すると判断した。
    *   **Livewire 3 の挙動への対応:**
        *   `wire:model` と `selected disabled` オプションの組み合わせがLivewireの内部的な選択ロジックと競合し、意図通りに機能しない可能性があった。
        *   `disabled` 属性を外し、`updatedOrderBy` メソッドでカスタムソートオプションが再度選択された場合にデフォルトソートに戻すロジックを実装することで、Livewireのリアクティビティを最大限に活用しつつ、ユーザーの意図（標準ソートに戻したい）を汲み取るようにした。

### 3. `isStandardSort` プロパティの廃止と `orderByLabel` の導入

-   **当初計画:** `isStandardSort` プロパティで標準ソートかカスタムソートかを判定し、UIを切り替える。
-   **変更点:** `isStandardSort` プロパティを廃止し、`orderByLabel` プロパティと `getStandardSortLabel` ヘルパーメソッドを導入。
-   **判断理由:**
    *   `orderByLabel` が空文字列であるかどうかでカスタムソート中であることを判断できるため、`isStandardSort` という別途のフラグが不要になった。
    *   `orderByLabel` が直接表示名を持つことで、コードの可読性が向上し、ビューでの処理も簡素化された。

### 4. `WorkflowStatus` Enum へのアイコンメソッド追加

-   **背景:** ステータスラベルにアイコンを追加する要望があり、当初はBlade側で `match` 式を用いてアイコンを決定していた。
-   **変更点:** `app/Enums/WorkflowStatus.php` に `icon()` メソッドを追加し、各ステータスに対応する Font Awesome アイコンクラスを返すようにした。
-   **判断理由:** アイコンの管理を一元化し、コードの可読性と保守性を向上させるため。Blade 側での `match` 式によるアイコン決定ロジックを Enum 側に移譲した。

### 5. 統合ツールバーのセレクトボックスの表示統一

-   **背景:** 「表示件数」と「並び順」のセレクトボックスのラベルの見え方が異なっており、DaisyUI のスタイルに揃えたいという要望があった。
-   **変更点:** `resources/views/components/ledger/search.blade.php` の「表示件数」のラベル部分を、DaisyUI の `label` コンポーネントのスタイルに揃えるように修正した。具体的には、`<legend class="fieldset-legend">` を `<label class="label"><span class="label-text">` に変更し、親要素を `div class="form-control"` で囲んだ。
-   **判断理由:** 統合ツールバー内のUI要素の視覚的な一貫性を保ち、ユーザーエクスペリエンスを向上させるため。

## 🔍 リスクと対策

### リスク1: データの可読性低下

-   **内容:** ラベルが重なることで、下のデータが読みにくくなる。
-   **対策:**
    1.  **受容:** 本計画では「詳細ボタンへの誘導」を前提とし、ある程度の可読性低下は許容する。
    2.  **緩和:** ラベルを右端に配置し、`backdrop-blur` を適用することで、影響を最小限に抑える。

### リスク2: 操作場所の分散

-   **内容:** ソート/フィルタの場所が2箇所に分かれるため、ユーザーが混乱する可能性がある。
-   **対策:**
    1.  **受容:** 「共通 vs 個別」という論理的な分離であり、ユーザーの慣れによって解決されると判断する。
    2.  **将来的な補助:** 必要に応じて、初回利用時に操作方法を案内するチュートリアル機能の導入を検討する。
