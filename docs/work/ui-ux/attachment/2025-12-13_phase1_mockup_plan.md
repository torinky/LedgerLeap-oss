## 1. 概要と目的

本ドキュメントは、添付ファイルUI刷新プロジェクトにおける **Phase 1: モックアップとUI仕様の確定** に特化した詳細計画です。
最新のSaaS UIトレンドを取り入れ、多様なファイル形式に対応しつつ、ユーザーが直感的に操作できる「インスペクター・ドロワー」形式のUIを具体化します。

### 1.1. ゴール
*   実装手戻りを防ぐため、Blade/Tailwind CSSによる高精度なモックアップを作成する。
*   特に、**既存の台帳一覧画面 (`RecordsTable`) 上に新しい添付ファイル表示カラムをダミーで組み込む**ことで、実際の業務画面での見え方をリアルに体感できるようにする。
*   ステークホルダーによるレビューを経て、UI仕様を確定させる。

## 2. 機能の網羅的調査と選定

ペルソナ（実務担当者、管理者、リーダー）のユースケース（監査、権限管理、チーム共有）に基づき、添付ファイルに関連する機能・情報を網羅的に洗い出し、新UIへの配置を決定します。

### 2.1. ファイル種別ごとの考慮事項

| ファイル種別 | 特徴 | ユーザーの関心事 | UI対応方針 | アクセシビリティ |
| :--- | :--- | :--- | :--- | :--- |
| **画像** (JPG, PNG, WEBP) | 視覚情報が主。OCR/VLM対象。 | 画像そのものの確認、抽出されたテキストの正確性。 | リスト: サムネイル表示<br>ドロワー: 大きなプレビュー表示 | `alt`属性必須。画像読込失敗時の代替テキスト。 |
| **PDF** (Native/Scan) | 複数ページ、文書。OCR対象。 | 内容の通読、テキスト検索、元ファイルとOCR版の使い分け。 | リスト: PDFアイコンor1ページ目サムネ<br>ドロワー: PDFビューワーまたはリンク | ページ数表示。ファイルサイズ明示。 |
| **Office** (Word, Excel) | 編集可能文書。 | 内容確認、ダウンロードして編集。 | リスト: アイコン表示<br>ドロワー: メタデータ中心 | ファイル形式を明示（.docx, .xlsx）。 |
| **その他** (Zip, Bin) | 閲覧不可。 | 保存場所としての利用。 | リスト: 汎用アイコン<br>ドロワー: 基本情報のみ | ファイルタイプと用途の説明。 |

| 項目 | 概要 | 関連ペルソナ | 重要度 | 配置: リスト | 配置: ドロワー | キーボード操作 | 備考 |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- | :--- |

| **権限情報** | 自分の操作可能範囲 | 全員 | 中 | 非表示 | フッター/情報タブ | - | 「読取専用」等の明示 |

## 3. UI構成と動線設計 (DaisyUI/MaryUIベース)

### 3.1. デザインコンセプト: "Context-Aware Drawer"
最近のWeb UIトレンド（Linear, Notion, Google Drive等）を踏襲し、**「一覧性を維持したまま、コンテキストを失わずに詳細へダイブする」** 体験を提供します。

*   **リスト表示:** グリッドレイアウト（カード）を採用し、視認性を高める。
*   **ドロワー:** 画面右側からオーバーレイ（またはコンテンツ押し出し）で表示。背景を完全に隠さず、元のリストが見える状態を維持する。

### 3.2. コンポーネント構成案

#### A. 詳細/編集用リスト表示 (`x-ledger.attachment-list mode="full"`)
詳細画面・編集画面で使用するリッチな表示。

*   **レイアウト:** `grid grid-cols-2 md:grid-cols-3 lg:grid-cols-4 gap-4`
*   **カード構成 (`x-mary-card` カスタム):**
    *   **Image/Icon Area:** アスペクト比固定の領域。中央にサムネイルまたは `fa-file-*` アイコン。
    *   **Overlay Actions:** ホバー時に浮かび上がるアクション（ダウンロード、クイックプレビュー）。
    *   **Status Indicator:** 右上にDaisyUI `indicator` で処理状態（緑:完了, 黄:処理中, 赤:エラー）を表示。
    *   **Footer:** ファイル名を `truncate` で表示。クリックでドロワー展開。
    *   **Loading State:** 画像読み込み中や処理中のスケルトンスクリーン (`animate-pulse`) を考慮。
        *   実装例: `<div class="animate-pulse bg-gray-300 rounded h-48 w-full"></div>`
        *   処理中ステータスの場合は、アイコンも回転アニメーション (`animate-spin`) を追加。
    *   **Error State:** サムネイル読込失敗時の代替表示。
        *   背景色をグレーに変更し、中央に `fa-exclamation-triangle` アイコンを表示。
        *   `alt`属性にファイル名を設定し、画像が表示されない場合もテキストで識別可能にする。

#### B. 一覧用簡易リスト表示 (`x-ledger.attachment-list mode="compact"`)
台帳一覧画面 (`RecordsTable`) で使用する省スペース表示。

*   **レイアウト:** `flex flex-wrap gap-1` (インラインに近い形式)
*   **表示内容:**
    *   小さなファイルアイコン (`w-6 h-6`) の羅列。
    *   ホバー時にツールチップでファイル名を表示。
    *   ファイル数が多い場合は「+3」のようなバッジで省略。
    *   クリックでドロワー（または詳細画面）へ遷移。

#### C. インスペクター・ドロワー (`FileInspector`)
詳細情報を表示する専用コンポーネント。

*   **コンテナ:** Alpine.js + Tailwind CSS で独自実装（`x-mary-drawer`は存在しないため）。
    *   配置: 右側 (`fixed right-0 top-0 h-full w-96 md:w-1/3 transform transition-transform`)。
    *   オーバーレイ: `z-39` (既存モーダルは `z-50` のため競合回避)。
    *   ドロワー本体: `z-40`。
*   **ヘッダー:**
    *   ファイル名（2行まで表示、`line-clamp-2`）。
    *   主要アクション（ダウンロード、外部で開く、リンクコピー）。
        *   **権限制御:** 削除や再処理ボタンは、権限がない場合 `disabled` (グレーアウト) または非表示にするデザインを含める。
        *   **視覚的フィードバック:** ボタンホバー時に `tooltip` で説明を表示。
    *   閉じるボタン (`x-mary-button icon="o-x-mark"`).
    *   **キーボードショートカット:**
        *   `Esc`: ドロワーを閉じる (`@keydown.escape.window="close()"`)。
        *   `Tab`: フォーカス可能な要素間を移動。
        *   `Shift+Tab`: 逆方向にフォーカス移動。
*   **アクセシビリティ属性:**
    *   `role="dialog"`
    *   `aria-modal="true"`
    *   `aria-labelledby="drawer-title"` (ファイル名のIDを参照)
    *   `aria-describedby="drawer-description"` (任意: 詳細説明)
*   **コンテンツ (`x-mary-tabs`):**
    1.  **解析 (Analysis):** 
        *   OCR/VLMテキストを表示。`textarea` (readonly) + コピーボタン。
        *   信頼度スコア表示（VLMの場合）。
        *   **ローディング状態:** テキストがまだ処理中の場合は `animate-pulse` のスケルトンスクリーンを表示。
    2.  **情報 (Info):** 
        *   `x-mary-list-item` を使用したメタデータ一覧。
        *   権限情報（「削除不可」等）もここに表示。
        *   **ファイルサイズの人間可読形式:** `1.2 MB` のように表示。
    3.  **履歴 (History):** **【専用作り込み】** 縦型のタイムラインUI。
        *   **処理ログ:** 処理ステップ（アップロード→Tika→OCR→VLM）を時系列で表示。
            *   各ステップの所要時間を表示（例: `2.3秒`）。
            *   エラー時はエラーメッセージを展開可能なアコーディオンで表示。
        *   **アクティビティログ:** 「2025-12-13 10:00 田中太郎がダウンロード」のようなアクセス履歴も統合して表示。
            *   **プライバシー配慮:** 閲覧履歴は管理者権限のみ表示する設計を考慮。

## 5. UI/UX 評価項目 (チェックリスト)

計画の品質を担保するため、モックアップ完成時に以下の項目を評価します。

### 5.1. 基本的なUX
- [ ] **視認性:** サムネイル/アイコンだけでファイルの種類が判別できるか？
- [ ] **状態把握:** エラーが発生しているファイルがひと目で分かるか？
- [ ] **操作性:** ドロワーを開くアクションと、ダウンロードするアクションが明確に区別されているか？（誤クリック防止）
- [ ] **情報密度:** ドロワー内のテキスト情報（OCR結果）は読みやすいか？コピーしやすいか？
- [ ] **レスポンシブ:** モバイル画面でもドロワーが操作可能か？（全画面化などの調整ができているか）
- [ ] **一覧性:** 一覧画面用の簡易表示モードが、スペースを圧迫せずに情報を伝えているか？

### 5.2. アクセシビリティ
- [ ] **キーボード操作:** すべての操作が`Tab`キーとマウスの両方で実行可能か？
- [ ] **フォーカス表示:** 現在フォーカスされている要素が視覚的に明確か？（`outline`が適切に表示される）
- [ ] **スクリーンリーダー:** `aria-label`, `role`, `aria-modal` などの属性が適切に設定されているか？
- [ ] **色覚多様性:** エラー・警告・成功などの状態が色だけでなくアイコンでも識別可能か？
- [ ] **フォントサイズ:** 最小フォントサイズは14px以上で、拡大しても崩れないか？

### 5.3. エラーハンドリング
- [ ] **ローディング状態:** ファイル読み込み中や処理中にスケルトンスクリーンが表示されるか？
- [ ] **エラー表示:** サムネイル読込失敗時に代替テキストやアイコンが表示されるか？
- [ ] **エラーメッセージ:** 処理失敗時のエラーメッセージは明確で、次のアクションが提示されているか？
- [ ] **タイムアウト:** 長時間処理が完了しない場合の表示は適切か？

### 5.4. 権限とセキュリティ
- [ ] **権限:** 操作できないボタン（例：削除）が適切に非活性化/非表示化される想定になっているか？
- [ ] **監査性:** アクティビティログが時系列で確認できるか？
- [ ] **プライバシー:** 閲覧履歴などのセンシティブな情報は適切に制御されているか？

### 5.5. 技術的要件
- [ ] **RPA互換性:** DOM上に静的なダウンロードリンクが存在するか？
- [ ] **多言語対応:** すべてのテキストが`__('...')` 形式で多言語化されているか？
- [ ] **パフォーマンス:** 大量ファイル（50件以上）表示時にスクロールや操作が遅延しないか？

## 5.6. Phase 1 特有の懸念事項と対応策

### 懸念1: モックアップが実装の制約を無視している 🔴
**懸念:**  
モックアップ段階で理想的なデザインを作成しても、実際の実装時に技術的制約（Livewireの制限、DaisyUIのコンポーネント仕様など）により実現不可能な場合がある。

**対応策:**
- モックアップ作成時から、既存のMaryUI/DaisyUIコンポーネントのドキュメントを参照し、実現可能な範囲内でデザインする。
- タスク1.3でドロワーの基本動作を先に確認し、不可能な場合は代替案（全画面モーダル化など）を検討する。
- 実装者（開発者）を早期にレビューに巻き込み、技術的フィードバックを得る。

### 懸念2: 既存画面への組み込みによるレイアウト崩れ 🟡
**懸念:**  
`RecordsTable` にダミーカラムを追加することで、既存の列の幅バランスが崩れ、一覧画面全体のUIが破綻する可能性がある。

**対応策:**
- タスク1.1で、添付ファイルカラムの幅を固定（`w-32`など）にし、他のカラムに影響を与えないようにする。
- レスポンシブ対応として、モバイル画面では添付ファイルカラムを非表示にするなどの調整を行う。
- 既存のテーブルレイアウトが`table-auto`か`table-fixed`かを確認し、適切なクラスを使用する。

### 懸念3: ダミーデータの現実性不足 🟡
**懸念:**  
モックアップ用のダミーデータが、実際の業務データの複雑さ（大量ファイル、長いファイル名、特殊文字など）を反映していない場合、実装後に問題が発覚する。

**対応策:**
- タスク1.1で作成するダミーデータに、以下のエッジケースを含める:
    - ファイル名が50文字以上の長いケース
    - 日本語・英語・数字・記号が混在するファイル名
    - 処理中、エラー、完了の各ステータス
    - 画像、PDF、Office、その他の各ファイルタイプ
    - 1つのカラムに10件以上のファイルが添付されているケース
- 実際のステージング環境のデータをサンプルとして参照する。

### 懸念4: アクセシビリティ検証の不足 🟡
**懸念:**  
モックアップ段階でキーボード操作やスクリーンリーダー対応を検証しないと、後のフェーズで大幅な手戻りが発生する。

**対応策:**
- タスク1.5の評価段階で、実際にキーボードのみでドロワーを操作してみる（`Tab`, `Shift+Tab`, `Enter`, `Esc`）。
- ブラウザの開発者ツールで`Accessibility Tree`を確認し、適切な`role`や`aria-*`属性が設定されているか検証する。
- 可能であれば、スクリーンリーダー（macOSの`VoiceOver`、Windowsの`NVDA`）で読み上げ動作を確認する。

### 懸念5: ドロワーとモーダルの混在による混乱 🟡
**懸念:**  
既存のシステムにはVLMプレビューモーダルなど、複数のモーダルが存在する。新たにドロワーを導入することで、ユーザーが「どこに何の情報があるか」を混乱する可能性がある。

**対応策:**
- モックアップ段階で、ドロワーとモーダルの使い分けルールを明確にする:
    - **ドロワー:** ファイルの詳細情報（常に右側からスライドイン）
    - **モーダル:** 確認ダイアログ、編集フォーム（中央に表示）
- 既存のVLMプレビューモーダルは、Phase 3でドロワー統合を検討する。
- ドロワー内にも「VLMプレビューを見る」ボタンを配置し、一貫性を保つ。

### 懸念6: モバイル対応の後回し 🟡
**懸念:**  
デスクトップ優先でモックアップを作成し、モバイル対応を後回しにすると、レスポンシブ対応が困難になる。

**対応策:**
- タスク1.2, 1.3でTailwindの`md:`, `lg:`などのブレークポイントを意識したクラス設計を行う。
- タスク1.5で、ブラウザの開発者ツールでモバイルビューに切り替えて動作確認する。
- モバイルでは、ドロワーを全画面表示（`w-full h-full`）にし、下部に閉じるボタンを配置する案を検討する。

### 懸念7: パフォーマンスの未検証 🟢
**懸念:**  
モックアップ段階では少数のファイルしか表示しないため、大量ファイル時のパフォーマンス問題が見過ごされる。

**対応策:**
- タスク1.5で、ダミーデータを50件以上に増やし、スクロール時の動作速度を確認する。
- 仮想スクロール（Virtual Scroll）の導入が必要か判断する材料を得る。
- Phase 5（パフォーマンス最適化）で本格対応するため、現時点では問題の有無を確認するのみ。
