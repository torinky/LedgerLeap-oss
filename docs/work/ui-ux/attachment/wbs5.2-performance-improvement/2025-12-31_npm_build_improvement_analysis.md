## 🔍 npm run dev vs npm run build の違い

### npm run dev（開発モード）

**特徴:**
- Viteの開発サーバーが起動
- HMR（Hot Module Replacement）が有効
- ソースマップ生成
- **大量のJavaScriptオーバーヘッド**

**影響:**
- Alpine.jsの初期化が遅い
- イベントリスナーの登録が遅い
- DOMの操作が遅い
- **UIがブロックされやすい** → フォーカス遅延

### npm run build（本番モード）

**特徴:**
- JavaScriptがミニファイされる
- ツリーシェイキング
- コード分割
- **最適化されたバンドル**

**効果:**
- Alpine.jsが高速に動作
- イベント処理が即座
- DOMの操作が軽快
- **UIがスムーズ** → フォーカス即座

## 🎯 残る問題: キーワード検索の1500ms

### なぜ検索だけ改善しないのか

**理由:** 検索は**Livewireのサーバーサイドレンダリング**が主因

```
ユーザー入力 → Livewireリクエスト → サーバー処理 → Blade全体レンダリング(1500ms) → 表示
```

**Viteの最適化とは無関係:**
- サーバー側でBladeテンプレートを再生成
- Livewireが全タブのHTMLを生成
- これはJavaScriptの問題ではない

### 解決策

#### Option A: wire:ignore（最も効果的）

**実装:**
```blade
<div wire:ignore x-data="{
    previewText: @js($this->getPreviewText()),
    keyword: @entangle('searchKeyword')
}">
    <!-- Alpine.jsのみで検索 -->
</div>
```

**効果:** 1500ms → <50ms

#### Option B: Livewireの差分レンダリング

**現状:**
```php
public function updatedSearchKeyword()
{
    $this->clearPreviewCache();  // 全体再レンダリング
}
```

**改善:**
```php
public function updatedSearchKeyword()
{
    $this->skipRender();  // レンダリングをスキップ
}
```

**効果:** Livewireリクエストなし → 即座

## 💡 重要な発見

### 開発環境でのパフォーマンス問題

**発見:**
- `npm run dev`は**大量のオーバーヘッド**がある
- フロントエンド（Alpine.js）の問題に見えたが、実際はViteのオーバーヘッド
- 本番環境（`npm run build`）では大幅に改善

**教訓:**
- **パフォーマンス測定は本番モード（npm run build）で実施すべき**
- 開発モードの遅さをLivewireのせいにしていた
- フォーカス遅延はViteのHMRが原因だった

### Livewireとフロントエンドの分離

**改善したもの（フロントエンド起因）:**
- フォーカス遅延 → Viteのオーバーヘッド
- 画像プレビュー → JavaScriptの実行速度
- UIのブロック → イベント処理の遅延

**改善しないもの（サーバー起因）:**
- キーワード検索 → Livewireのレンダリング
- ドロワー開閉（部分的） → Livewireのレンダリング
