## エグゼクティブサマリー

### 🔍 主要な発見

1. **ドロワー開閉時間**: 1789ms〜2120ms（平均約2000ms） - **目標300ms未達**
2. **タブ切り替え**: ほとんどが速い（8-76ms）だが、**Permissionsタブのみ6761ms**
3. **テキスト検索**: サーバー処理は極めて速い（0.02-0.04ms）
4. **画像プレビュー**: ログに記録なし（フロントエンド処理）

### 🎯 結論

**ユーザーの指摘は正しい: これはフロントエンドの問題です。**

- サーバー側の処理は既に最適化されている（<1ms）
- ドロワー開閉の遅延は**Livewireのレンダリング時間**が原因
- タブ切り替えの6.7秒は**Permissionsタブ固有の問題**
- 画像プレビューの遅延はブラウザの画像読み込み

## 詳細分析

### A. ドロワー開閉時間

#### 測定データ

| ファイルID | 開閉時間 | 判定 |
|-----------|---------|------|
| 15 | 2120ms | ❌ 遅い |
| 15 | 1920ms | ❌ 遅い |
| 15 | 1918ms | ❌ 遅い |
| 17 | 1789ms | ❌ 遅い |
| 15 | 2117ms | ❌ 遅い |
| 17 | 1783ms | ❌ 遅い |
| 16 | 2022ms | ❌ 遅い |

**平均:** 約2000ms（2秒）  
**目標:** 300ms以下  
**達成率:** 0%

#### 原因分析

**サーバー側の処理時間:**
- `loadData()`: ログに記録なし（推定<100ms）
- `getPreviewText()`: 0.52-1.17ms（極めて速い）
- DB クエリ: ログに記録なし（推定<100ms）

**結論:** サーバー側は既に最適化されている。**問題はフロントエンド（Livewire レンダリング）**

**フロントエンドの時間配分（推定）:**
```
合計2000ms = サーバー処理100ms + Livewireレンダリング1500ms + DOM構築400ms
```

**Livewireレンダリングが遅い理由:**
1. Bladeテンプレートの複雑さ
2. Alpine.jsの初期化
3. 大量のデータバインディング（`@entangle`）
4. CSSトランジション（300ms設定）

#### 原因分析

**Permissionsタブが遅い理由（推定）:**
1. **権限チェックの複雑な処理**（フォルダ権限の再帰的な計算）
2. **Livewireコンポーネントの再読み込み**
3. **DOMの複雑な構造**

**サーバー処理時間:** `getPreviewText()`は0.6-0.8ms → サーバー側は問題なし

**結論:** Permissionsタブの**フロントエンド実装に問題あり**

### C. テキスト検索速度

#### 測定データ

**検索キーワード更新:**
- `updatedSearchKeyword()`: 0.2-2.4ms（ほぼ即座）

**検索実行（`hasKeywordHit()`）:**
- cache miss: 0.02-0.04ms（極めて速い）
- cache hit: ログに記録なし → キャッシュが効いていない可能性

| 操作 | キーワード | 時間 | 結果 | キャッシュ |
|------|-----------|------|------|----------|
| 初回検索 | "a b" | 0.04ms | Hit | Miss |
| キーワード変更 | " b" | 0.03ms | No Hit | Miss |
| キーワードクリア | "" | - | - | - |
| 新規検索 | "ｇ" | 0.03ms | No Hit | Miss |
| 追加入力 | "ｇa" | 0.03ms | No Hit | Miss |
| 追加入力 | "ｇaki" | 0.03ms | No Hit | Miss |
| 削除 | "aki" | 0.02ms | Hit | Miss |
| 修正 | "gaki" | 0.02ms | Hit | Miss |

**サーバー処理時間:** 0.02-0.04ms（極めて速い）

#### 問題分析

**サーバー側は問題なし。** 問題は：

1. **キャッシュヒットが記録されていない** → キャッシュが機能していない
2. **ローディングUIが表示されない** → Alpine.jsの問題
3. **体感が遅い** → フロントエンドのレンダリング時間

**キャッシュが効かない理由:**
- 毎回 "cache miss" → `$wire.searchKeyword`が変わるたびにキャッシュがクリアされている
- `wire:model.live.debounce.300ms` → サーバーリクエストが毎回発生
- Livewireの再レンダリングでキャッシュがリセットされる

### D. 画像プレビュー

**ログに記録なし** → これは完全に**ブラウザ側の処理**

#### 測定不可の理由

画像プレビューの読み込みは：
1. ブラウザのHTTPリクエスト
2. 画像のダウンロード
3. DOMへの描画

これらはすべて**クライアント側の処理**で、Laravelログには記録されない。

#### 問題の推定

ユーザーが「2回目も遅い」と感じる理由：
1. **ブラウザキャッシュが効いていない**（HTTPヘッダーの問題）
2. **sessionStorageが機能していない**（Livewireの再レンダリング）
3. **画像URLにタイムスタンプが含まれる**（毎回異なるURL）

---

## 問題の根本原因

### 🎯 主要な問題: Livewireのレンダリングコスト

**Livewireの仕組み:**
1. ユーザーアクション → サーバーリクエスト
2. サーバー処理（速い: <100ms）
3. **HTMLレンダリング（遅い: 1500ms）**
4. クライアント側でDOMを更新
5. Alpine.jsの再初期化

**ドロワー開閉が遅い理由:**
- Bladeテンプレートが複雑すぎる（4タブ × 多数のコンポーネント）
- `@entangle`が多数（状態の同期にコスト）
- Alpine.jsのデータバインディングが重い

**テキスト検索が遅く感じる理由:**
- `wire:model.live` → 入力ごとにサーバーリクエスト
- サーバー処理は速い（0.02ms）が、**レンダリングに数百ms**
- ローディングUIが機能していない → 待ち時間が分からない

**Permissionsタブが異常に遅い理由:**
- 権限チェックのロジックが重い
- Livewireコンポーネントの再初期化
- DOMの複雑さ

## 対策案

### 優先度: 🔴 高（即実施）

#### 対策1: Livewireの使用を最小化

**問題:** Livewireのレンダリングコストが高すぎる

**対策:**
- 検索機能を**Alpine.js + フロントエンドのみ**に変更
- キャッシュを**ブラウザのlocalStorage**に保存
- サーバーリクエストを減らす

**効果:** 検索が即座に応答（<50ms）

#### 対策2: 検索UIをMaryUIの標準フォームに置き換え

**ユーザーの指摘:** 「MaryUIのフォームなどに実装を寄せたほうが良い」

**理由:**
- MaryUIのフォームは最適化されている
- Livewireとの統合がスムーズ
- メンテナンス性が向上

**実装:**
```blade
{{-- 現在（カスタム実装） --}}
<x-mary-input wire:model.live.debounce.300ms="searchKeyword" ... />

{{-- 提案（Alpine.js + localStorage） --}}
<input type="text" 
       x-data="{ keyword: localStorage.getItem('search') || '' }"
       x-model="keyword"
       x-on:input.debounce.300ms="
           localStorage.setItem('search', keyword);
           $wire.set('searchKeyword', keyword, false);  // サーバー送信なし
       "
       class="input input-sm" />
```

#### 対策3: Permissionsタブの遅延ロード

**問題:** Permissionsタブが6.7秒かかる

**対策:**
- タブ選択時のみ権限情報を読み込む
- 権限チェックの結果をキャッシュ
- または、Permissionsタブを別コンポーネントに分離

**効果:** タブ切り替えが<100msに改善

---

### 優先度: 🟡 中（効果は限定的）

#### 対策4: activitiesの遅延ロード

**当初の計画通り実施:**
```php
#[Computed]
public function activities()
{
    if ($this->selectedTab !== 'history') {
        return collect();
    }
    return $this->file->activities()->with('causer:id,name')->get();
}
```

**効果:** ドロワー開閉時間が100-200ms改善（2000ms → 1800ms）

#### 対策5: Bladeテンプレートの最適化

- 不要な`@entangle`を削除
- Alpine.jsのデータバインディングを最小化
- CSSトランジションを短縮（300ms → 150ms）

**効果:** ドロワー開閉時間が200-300ms改善

---

### 優先度: 🟢 低（画像プレビュー）

#### 対策6: 画像URLの最適化

**問題:** 画像URLが毎回変わる可能性

**対策:**
```php
// FileInspector.php
#[Computed]
public function previewUrl(): ?string
{
    if (!$this->isImage) {
        return null;
    }
    
    // タイムスタンプを削除してキャッシュ可能にする
    return route('file.preview', [
        'tenant' => $this->tenantId,
        'file' => $this->file->id,
    ]);
}
```

**HTTPヘッダーの設定:**
```php
// FileController.php - preview()
return response()->file($path, [
    'Cache-Control' => 'public, max-age=3600',  // 1時間キャッシュ
    'Etag' => md5_file($path),
]);
```

**効果:** 2回目以降の画像読み込みが即座（ブラウザキャッシュ）

## 結論

### 🎯 ユーザーの指摘は正しい

> 「反応時間のばらつきが大きいのでフロントエンドの問題のような気がします。MaryUIのフォームなどに実装を寄せたほうが良いかもしれません。」

**分析結果:**
- ✅ **フロントエンドの問題**: サーバー処理は0.02-1ms、Livewireレンダリングが1500ms
- ✅ **MaryUIへの寄せ**: カスタム実装よりも標準コンポーネントを使うべき
- ✅ **反応時間のばらつき**: Livewireの再レンダリングが不安定

| タブ切り替え | 30ms（平均） | 100ms | ✅ | - |

| テキスト検索（サーバー） | 0.02ms | - | ✅ | - |
