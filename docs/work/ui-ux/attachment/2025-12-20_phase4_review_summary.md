## エグゼクティブサマリー

Phase 4計画の精査により、**6つの重大な懸念事項**と**10の不足タスク**を特定しました。総見積工数を**30h → 36h（+20%）**に修正し、リスク緩和策と成功基準を大幅に強化しました。

### 主要な発見事項
1. ✅ **Phase 2/3の実装品質は高い**: `AttachedFile` モデル、`MimeTypeHelper`、`attachment-list` コンポーネントは計画通り実装済み
2. ⚠️ **UI分岐の網羅性不足**: モックアップは24パターンの処理状態組み合わせを網羅していない
3. ⚠️ **権限管理の複雑さ**: `AttachedFilePolicy` が空実装のため、`LedgerPolicy` 経由の間接的なチェックが必要
4. ⚠️ **アクセシビリティ未検証**: WCAG 2.1 AA準拠の体系的な検証計画が不足

**背景:** Phase 1モックアップは以下を網羅していない可能性が高い：
- 処理状態: 最終化前/後 × Tika/VLM/OCR成功/失敗/未実施（24パターン）
- MIMEタイプ: Phase 3で40種類以上定義されたが、全てのUIパターンは未実装
- エラー状態: 部分的成功、タイムアウト等

**追加タスク:**
- **4.1.6**: 処理フロー図作成と全分岐パターン洗い出し
- **4.2.3-4.2.4**: 頻出ケースの優先実装とエラーフォールバック

- Phase 5で体系的な実装を計画

## 2. 新規追加されたリスク項目

### 2.1. UI分岐の網羅性リスク（高）
**影響度:** 高（ユーザー体験に直結）  
**発生確率:** 高（モックアップの検証範囲が限定的）

**具体的なリスク:**
- 想定外の処理状態でUIが崩れる
- エラーメッセージが表示されない
- ユーザーが再処理方法を理解できない

**緩和策:**
1. 4.1.6で処理フロー図を作成し、全分岐を可視化
2. 4.2.3-4.2.4で頻出ケースを優先実装
3. 4.6.5で検証計画を策定
4. Phase 4完了後にレビュー会を開催

### 2.2. 権限管理の複雑さ（中）
**影響度:** 中（セキュリティリスクあり）  
**発生確率:** 中（`AttachedFilePolicy` 未実装）

**具体的なリスク:**
- N+1クエリの発生（`$file->ledger->define->folder` の連鎖）
- Soft Delete済み台帳に属するファイルの扱いが不明確
- 管理者専用機能（VLM再処理）の権限判定ロジックが複雑化

**緩和策:**
1. 4.1.4で専用ヘルパーメソッドを実装
2. Eager Loadingを徹底
3. Phase 5で `AttachedFilePolicy` を完全実装

### 2.3. データ整合性リスク（中）
**影響度:** 中（データ表示エラー）  
**発生確率:** 低（Phase 5/6実装は堅牢）

**具体的なリスク:**
- `hashedbasename` と `content_attached` のキー不一致
- Phase 5以前の旧データとの互換性

**緩和策:**
1. 4.2.1で `previewable_text` アクセサの動作を検証
2. 4.7.4で旧データを模したテストケースを追加

### 2.4. アクセシビリティリスク（中）
**影響度:** 中（法的コンプライアンス）  
**発生確率:** 中（体系的な検証未実施）

**具体的なリスク:**
- キーボード操作でタブ切り替えができない
- スクリーンリーダーでバッジが読み上げられない
- WCAG 2.1 AA違反（コントラスト比等）

**緩和策:**
1. 4.1.7でARIA属性を適切に配置
2. 4.6.7でaxe DevToolsスキャンと手動テスト

## 7. 結論

Phase 4計画の精査により、以下が明確になりました：

✅ **強み:**
- Phase 2/3の実装品質が高く、基盤は堅牢
- Eager Loading戦略が確立されており、N+1クエリ対策は良好
- MimeTypeHelperとattachment-listコンポーネントの実装が優秀

⚠️ **課題:**
- UI分岐の網羅性が不足（24パターン未実装）
- 権限管理が複雑（AttachedFilePolicy空実装）
- アクセシビリティの体系的検証が不足

📋 **対策:**
- 総見積工数を36hに修正（+20%）
- 10の不足タスクを追加
- 6項目のリスク緩和策を明記
- 25項目の成功基準を定義
- Phase 5以降への引き継ぎ事項を明確化

**推奨:** 本精査結果を踏まえ、Phase 4実装を開始することを推奨します。特に4.1.6（UI分岐検討）と4.6.5（UI分岐検証）は、Phase 5計画の品質を左右する重要なマイルストーンです。
