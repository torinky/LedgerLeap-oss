# 管理者お知らせ機能の最終スコープメモ

- date: 2026-04-28
- scope: 上部強制表示を主UIにした MVP / 運用拡張 / 将来拡張
- goal: 別機能として見える登録管理 UI を持たせつつ、通知センターを統合先にする前提で切り分ける

## 1. 前提

既存の通知機能には、通知データ、未読管理、既読処理、グローバルな通知一覧、テナント付きの通知ページ、ワークフロー通知の表示導線がすでにあります。
そのため、管理者お知らせはこれらを裏側の統合先として活用できます。

ただし、ユーザーに着意させる主経路はページ上部の強制表示です。
通知センターは履歴、再確認、既読管理の補助経路として使い、主役にしない方がよいです。

## 2. MVP

MVP では次を満たせば十分です。

- ページ上部に管理者お知らせを強制表示できる
- 表示開始日時と終了日時を持てる
- info / warning / critical の3段階を持てる
- リンクを1つ持てる
- 閉じる可否をレベル別に決められる
- テナント共通か全体共通かを切り替えられる
- 管理者が登録・公開・停止を管理画面で行える

MVP であえて外すものは次です。

- 通知センターのみの従来運用
- 複数件の積み上げ表示
- ユーザー既読の永続保存
- 役割別 / ユーザー別の対象制御
- 履歴一覧の管理画面
- 管理者プレビュー

## 3. 運用拡張

MVP の後に必要になりやすいのは次です。

- 通知センターにも同じ告知を統合表示する
- 複数件を優先度順に表示する
- 対象範囲を role / user まで広げる
- 閉じた後の再表示条件をセッションとユーザー既読で分ける
- 画面上の見た目を半透明ステッカー寄りに調整する
- 管理画面で開始・終了・優先度を一覧編集できるようにする
- 閲覧ログや操作ログを残す

## 4. 既存機能で流用できるもの

- NotificationType による通知タイプ管理
- GenericNotification / WorkflowSummaryNotification による database 通知
- NotificationService による配信先判定と未読管理
- notifications.index の通知センター画面
- NotificationList の既読 / 全既読 / ページネーション
- ルート付きの通知導線とテナントコンテキスト

## 5. 既存機能で足りないもの

- 予約公開と終了時刻の管理
- 管理者向けの配信予約画面
- 全画面上部の常駐表現
- 優先度ごとの強調表示
- 表示対象を tenant / role / user で切り替える送信 UI

## 6. 将来拡張

将来必要になる可能性があるが、現時点では後回しにするものです。

- 条件分岐つきの配信ルール
- ユーザー属性や組織属性に応じた出し分け
- 既読率やクリック率の分析
- お知らせのテンプレート化
- 国際化や多言語配信
- 重要通知の強制表示モード
- 通知センター中心に戻す案

## 7. 実装方針の骨子

1. 最初は上部強制表示を含む新しい管理 UI から始める
2. 表示条件は starts_at / ends_at / priority / scope に限定する
3. UI は既存コンポーネントだけで構成する
4. 表示レベルは info / warning / critical を先に決める
5. 管理画面は一覧編集より先に単一編集でよい
6. 通知センター統合は裏側の反映として後追いで合わせる

## 8. 仕様決定のチェックポイント

- 上部強制表示で着意させる位置づけが必要か
- 閉じる可否をレベル別に分けるか
- テナント単位で足りるか
- 全体共通とテナント共通を両方必要とするか
- 履歴を最初から管理画面に出すか
