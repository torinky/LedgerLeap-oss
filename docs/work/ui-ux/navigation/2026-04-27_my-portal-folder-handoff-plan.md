# マイポータルから台帳一覧へのフォルダ遷移計画

**日付:** 2026-04-27  
**ステータス:** 計画段階
**対象:** tenant 側マイポータル、フォルダツリー、台帳一覧のフォルダ選択状態

## 1. 目的

マイポータルは、ログイン直後にユーザーが自分の所属、役割、権限、担当範囲を素早く把握するための overview 画面として維持する。そのうえで、担当フォルダやアクセス可能フォルダを起点に、台帳一覧のフォルダ選択状態へ自然に遷移できる導線を追加する。

今回の再構成では、マイポータルを台帳一覧の代替ブラウザにしない。あくまで「状況把握」と「次の作業場所への入口」を担わせ、実作業は台帳一覧側で行う方針とする。

## 2. 背景と課題

現状のマイポータルは、役割・権限・担当フォルダ・詳細フォルダツリーを一画面で確認できる一方、フォルダツリーから台帳一覧へ移る動線が弱い。結果として、ユーザーは「どのフォルダを開けばよいか」を把握しても、その状態のまま台帳一覧へ移行しづらい。

この課題は、特に次のユーザーに影響が大きい。

- 初回利用者: まず自分の状況を知り、その後どこから作業を始めるかを迷いたくない。
- 日常業務ユーザー: 担当フォルダから台帳一覧へ素早く入りたい。
- 権限確認ユーザー: 自分に見える範囲を把握したあと、実際の一覧へ遷移したい。
- 管理・運用ユーザー: 役割や担当範囲を確認しながら、すぐ台帳へ移りたい。

## 3. 設計方針

### 3.1 ポータルの役割を維持する

マイポータルは以下を満たす overview に留める。

- ユーザーの所属・役割・権限を短時間で把握できる
- 担当フォルダや主要なアクセス先を確認できる
- 詳細フォルダツリーは必要時にだけ開く

### 3.2 ツリーは「選択」ではなく「遷移の入口」でもある

ポータル上のフォルダツリーは、単なる閲覧用ツリーではなく、台帳一覧の初期フォルダ状態へつながる入口として扱う。

- フォルダ行から台帳一覧へ進める
- 台帳一覧側で対象フォルダが選択済み状態になる
- 既存の `currentFolderId` / `selectedFolderIds` の契約を再利用する
- 新しい選択モデルや別系統の state は作らない

### 3.3 初度ポータルの目的を壊さない

ポータルの目的は「迷わず状況を把握し、次の行動を決められること」である。
そのため、次の操作は台帳一覧へ委ねる。

- 台帳の閲覧
- 台帳の検索
- 台帳の表示密度変更
- 台帳の詳細操作

### 3.4 デザイン方針

- アイコンは意味を補う補助として使い、ラベルなしで理解させる役割は持たせない
- 所属、役割、権限、担当フォルダのような関連情報は近接させて、ひとまとまりで読めるようにする
- 担当フォルダと台帳一覧への導線はセットで見せ、ユーザーが次に何をするかを迷わないようにする
- badge は状態や件数などの短いメタ情報に限定し、主説明は本文や見出しで読めるようにする
- tooltip は補足説明に使い、主メッセージを隠す場所にはしない
- 文字サイズとアイコンサイズは固定の小サイズに閉じ込めず、desktop で読める標準サイズを優先する
- 専門的な内輪表現より、一般的な業務画面で多くの人が理解しやすい最大公約数的な見せ方を採る
- 1 画面で多くの情報を扱う場合でも、主役は 1 つ、準主役は 1〜2 個に絞る

## 4. ペルソナ別の設計要件

### 4.1 初回利用者

- 自分の所属、役割、主要な権限が見てすぐ分かること
- 担当フォルダから台帳一覧へ進めること
- 詳細フォルダツリーは補助情報として扱われること

### 4.2 日常業務ユーザー

- よく使う担当フォルダから、ワンクリックで台帳一覧へ入れること
- 余計な説明を読まずに、行動へ移れること
- 一覧へ移ったあとも、どのフォルダを開いたかが明示されること

### 4.3 権限確認ユーザー

- 自分の権限と閲覧可能範囲を overview で確認できること
- 詳細ツリーを開けば、見える範囲の全体像を把握できること
- 一覧遷移後に、選択済みフォルダがそのまま確認できること

### 4.4 管理・運用ユーザー

- フォルダを起点に作業へ入るまでのステップが短いこと
- 権限や担当範囲の確認から、台帳一覧の実作業にすぐ切り替えられること
- テナント文脈やフォルダ文脈が遷移後も崩れないこと

## 5. ユーザーシナリオ別の検討

### 5.1 ログイン直後に状況を把握する

ユーザーはマイポータルで、自分の所属・役割・権限・担当範囲を確認する。ここでは、詳細操作よりも理解のしやすさを優先する。

### 5.2 担当フォルダから作業を開始する

ユーザーはマイポータルのフォルダツリーから担当フォルダを選び、その状態のまま台帳一覧へ移る。遷移後は、対象フォルダが既に開かれているため、すぐに台帳の閲覧や検索へ進める。

### 5.3 詳細範囲を確認してから一覧へ進む

ユーザーは折りたたみの詳細ツリーを開き、アクセス可能な全フォルダを確認したうえで、必要なフォルダを選んで台帳一覧へ移る。

### 5.4 権限や所属を再確認してから作業する

ユーザーは overview で自分の役割や権限を見直し、担当フォルダを起点に台帳一覧へ入る。ポータルは「説明」、一覧は「作業」の役割分担を保つ。

## 6. 実装方針

### 6.1 マイポータル側

- `app/Livewire/MyPortal.php` と `resources/views/livewire/my-portal.blade.php` を中心に、overview と遷移導線の役割を整理する
- フォルダツリーの各行に、台帳一覧へ進むための明示的な導線を持たせる
- テナント文脈を失わない URL 生成を行う
- ポータル内の情報量は増やしすぎず、詳細は折りたたみに寄せる

### 6.2 フォルダツリー側

- `resources/views/components/folder/tree.blade.php` と `app/Livewire/Folder/Tree.php` の state 表現を確認し、選択状態の責務を明確にする
- フォルダ行のクリックで、台帳一覧に渡すべきフォルダ ID を明示する
- `currentFolderId` と `selectedFolderIds` の既存の意味を崩さない
- ポータル用と台帳一覧用で同じツリー表現を使いつつ、行動の意味だけを切り替える

### 6.3 台帳一覧側

- `app/Livewire/Ledger/IndexManager.php` の `currentFolderId`、`selectedFolderIds`、`selectedLedgerDefineIds` の既存フローを再利用する
- `resources/views/livewire/ledger/index-manager.blade.php` で、遷移直後に対象フォルダが選択済みで見えることを前提にする
- `resources/views/components/ledger/search.blade.php` は、一覧の状態把握を担う役割を維持する
- 検索語や表示レベルなどの一覧固有 state は、最初はフォルダ遷移と分離して考える

### 6.4 スプリント分解

- Sprint 1: ポータルで残す情報と台帳一覧へ渡す情報を確定し、アイコン / badge / tooltip / CTA の役割分担を決める
- Sprint 2: マイポータルの overview を再構成し、フォルダツリーから台帳一覧へ進む導線を実装する
- Sprint 3: 台帳一覧の state handoff を固定し、ポータル経由と直接アクセスの両方で回帰テストを通す

## 8. 変更の境界

### 含める

- ポータルから台帳一覧へ移る導線の再構成
- フォルダ選択状態の受け渡し
- 初度ポータルの目的に沿った情報整理
- 必要最小限の文言見直し

### 含めない

- 台帳一覧の検索・表示レベル・ページングの再設計
- グローバルのテナント選択画面の刷新
- 権限ロジックやフォルダ権限の再実装
- ポータルを台帳一覧ブラウザへ変質させる拡張

## 9. 検証方針

### 9.1 自動検証

- ポータルのレンダリング確認を行い、overview と CTA が共存していることを確認する
- ツリーから台帳一覧へ遷移する Feature / Livewire テストを追加し、対象フォルダが `currentFolderId` として反映されることを確認する
- 必要に応じて、`selectedFolderIds` がフォルダ配下の状態を正しく表すことを確認する
- 既存の台帳一覧テストで、直接アクセスとポータル経由アクセスの両方を固定する

### 9.2 手動検証

- 初回利用者想定で、ポータルから台帳一覧へ迷わず入れることを確認する
- 日常業務ユーザー想定で、担当フォルダからの遷移が短いことを確認する
- 権限確認ユーザー想定で、詳細ツリーと一覧の状態が一致することを確認する
- tenant context が失われず、遷移先で `tenant()` が維持されることを確認する

## 10. 完了条件

- マイポータルが overview として成立している
- フォルダツリーから台帳一覧へ自然に遷移できる
- 遷移後の台帳一覧で、対象フォルダが選択済みとして見える
- 既存の初度ポータルの目的が満たされ続ける
- tenant 文脈と state 契約が崩れていない
