# テナント分離テスト計画書

**日付:** 2025年9月4日
**作成者:** Gemini
**ステータス:** 計画完了・実装待ち

## 1. 目的

本ドキュメントは、全体計画書「[新マルチテナント実装計画書](./2025-08-30_new-multi-tenant-implementation-plan-final.md)」のステップ7「総合テストとテナント運用確認」を完全に完了させるため、テナント間のデータ分離を網羅的に検証する自動テストの具体的な計画を定義するものである。

## 2. 調査結果

### 2.1. 既存テストの確認

`tests/Feature` ディレクトリ配下のテストコードを調査した結果、以下のシナリオが既にカバーされていることを確認した。

-   **データ閲覧の分離:** `TenantIsolationTest`にて、あるテナントが別のテナントのデータを閲覧できないこと。
-   **直接URLアクセスの防止:** `TenantIsolationTest`にて、URLを直接指定して別テナントのリソースにアクセスできないこと。
-   **テナント切り替えUI:** `TenantSwitcherTest`にて、UI上の表示ロジックが正しいこと。
-   **テナント作成:** `SetupTenantCommandTest`にて、Artisanコマンドが正常に動作すること。

### 2.2. ペルソナとユースケースの確認

`docs/function` 配下の仕様ドキュメントを基に、以下の観点からテストシナリオを検討した。

-   ユーザーは複数の組織（テナント）に所属できる。
-   フォルダ単位での詳細な権限設定が存在する。
-   ユーザーはマイポータルから自身の所属テナントやタスクにアクセスする。

## 3. テストケースの過不足分析と追加仕様

### 3.1. 不足しているテストシナリオ

既存のテストはデータ**閲覧**の分離が中心であり、より堅牢性を高めるためには以下のシナリオを検証するテストが不足している。

1.  **書き込み操作の分離:** 作成・更新・削除といった書き込み操作がテナント間で完全に分離されていることの保証。
2.  **テナント横断リレーションの防止:** データ作成・更新時に、別のテナントに属するIDを指定して不正なリレーションを作成できないことの確認。
3.  **複数テナント所属ユーザーのコンテキスト保証:** 複数テナントに所属するユーザーが、テナントを切り替えた際に操作コンテキストが正しく切り替わることの保証。

### 3.2. 追加するテストケースの仕様

上記の不足シナリオをカバーするため、`tests/Feature/TenantIsolationTest.php` に以下のテストメソッドを追加する。

| テストメソッド名                                                                 | 検証内容                                                                                                                                                           |
| :------------------------------------------------------------------------------- | :----------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `user_cannot_update_data_in_another_tenant()`                                    | `tenant-B` のユーザーとして認証し、`tenant-A` に属する台帳データの更新リクエストを送信。リクエストが `403 Forbidden` または `404 Not Found` で失敗することを確認する。 |
| `user_cannot_delete_data_in_another_tenant()`                                    | `tenant-B` のユーザーとして認証し、`tenant-A` に属する台帳データの削除リクエストを送信。リクエストが失敗し、対象のデータがデータベースから削除されていないことを確認する。 |
| `validation_prevents_creating_relations_across_tenants()`                        | `tenant-A` のユーザーとして認証し、台帳作成処理において `tenant-B` に属する `folder_id` を指定してリクエストを送信。バリデーションエラーとなり、データが作成されないことを確認する。 |
| `user_belonging_to_multiple_tenants_can_switch_context_and_operate_correctly()` | 1. 特定のユーザーを `tenant-A` と `tenant-B` の両方に所属させる。<br>2. `tenant-A` のコンテキストで、`tenant-A` のデータが操作でき、`tenant-B` のデータが見えないことを確認。<br>3. `tenant-B` にコンテキストを切り替え、`tenant-B` のデータが操作でき、`tenant-A` のデータが見えないことを確認。 |
