## 1. 戦略改訂の背景

当初のフェーズ3計画では、文書全体ベクトルとは別に「キーワードベクトル」専用のカラム (`embedding_keyword`) を作成し、複数のベクトルを併用するハイブリッド検索を構想していた。

しかし、導入済みの最新埋め込みモデル **Ruri v3 (cl-nagoya/ruri-v3-310m)** の性能特性を再評価した結果、単一のベクトルカラムでより効率的かつ柔軟な実装が可能であることが判明した。これにより、データベーススキーマを簡素化しつつ、検索精度を向上させる「エンハンスドベクトルインデックス戦略」へと移行する。

---

## 2. 技術基盤の検証：Ruri v3の実力

### 2.1. モデル仕様と優位性
現在システムで稼働中のモデル `cl-nagoya/ruri-v3-310m` は、以下の仕様を持つ。

*   **最大シーケンス長:** 8192トークン (旧来の512トークンから大幅拡張)
*   **次元数:** 768次元
*   **インストラクション追従:** クエリとパッセージに特定のプレフィックス（`検索クエリ:`, `検索文書:`）を付与することで、検索意図を汲んだベクトル生成が可能。

### 2.2. 懸念事項の解消 (Token Limit & Metadata)
*   **懸念:** チャンク化する際、本文に加えて抽出キーワードなどのメタデータを埋め込むと、コンテキスト長（以前は512トークン）を超過し、情報が切り捨てられるリスクがあった。
*   **検証結果:** v3モデルの **8192トークン** 対応により、このリスクは完全に解消された。現在のチャンクサイズ設定（2000文字）に、数百文字のメタデータを付与しても十分な余裕がある。

## 3. エンハンスドベクトルインデックス・アーキテクチャ

### 3.1. 戦略の核心：メタデータ注入型チャンキング

専用のベクトルカラムを追加する代わりに、**チャンクテキスト自体に構造化メタデータを注入**してからベクトル化を行う。

#### チャンク構成イメージ
Ruriモデルに対して、以下のような構造化テキストを入力とする。

```text
検索文書: 【メタデータ】
キーワード: 請求書, 株式会社A商事, 2025年11月, サーバー保守費用, 定期契約
ファイル名: invoice_202511.pdf
ページ番号: 1
文書タイプ: 帳票/請求書

【本文】
下記の通りご請求申し上げます。
件名: 11月度サーバー保守費用
金額: 550,000円
... (以下、VLM/OCR抽出テキスト) ...
```

**効果:**
1.  **単一ベクトルでの表現力:** 「請求書」や「A商事」といったキーワードの意味と、本文の文脈の両方を1つのベクトル（`embedding`）に凝縮できる。
2.  **スキーマの簡素化:** `embedding_keyword` カラムの追加が不要となり、既存のMroongaベクトルインデックスをそのまま活用できる。

### 3.2. インデックス粒度の最適化 (Partial Re-indexing)

運用コストを最小化するため、台帳（Ledger）単位ではなく、**添付ファイル（AttachedFile）単位**でチャンクを管理・更新できる構造へ変更する。

#### データベーススキーマ変更 (`ledger_chunks`)

| カラム名 | 型 | 説明 | 変更内容 |
| :--- | :--- | :--- | :--- |
| `id` | bigInt | PK | - |
| `ledger_id` | bigInt | FK (ledgers) | - |
| **`attached_file_id`** | **bigInt** | **FK (attached_files)** | **新規追加 (Nullable, Index付)** |
| `chunk_text` | text | ベクトル化元のテキスト | メタデータ注入済みのテキストを格納 |
| `embedding` | vector | ベクトルデータ | 既存カラムを継続利用 |

#### 更新フロー
特定の添付ファイルのOCR処理が再実行された場合：
1.  `DELETE FROM ledger_chunks WHERE attached_file_id = ?`
2.  キーワード抽出・メタデータ生成・ベクトル化
3.  `INSERT INTO ledger_chunks ...`

## 5. 完了条件

1.  `ledger_chunks` テーブルに `attached_file_id` カラムが追加されている。
2.  生成されたチャンクテキストの冒頭に `検索文書: 【メタデータ】...` という形式でキーワードが含まれている。
3.  特定のファイルを更新した際、関連するチャンクのみがDELETE/INSERTされていることがログで確認できる。
4.  検索画面で、キーワード完全一致の文書と、類義語を含む文書の両方が検索結果に現れる。
