## 🏗️ アーキテクチャの明確化

### 直接実行 vs サーバー経由

#### パターンA: 直接実行（現在の試行）❌
```
LedgerLeap App (CPU) → PaddleOCR-VL (CPU)
```
結果: **不可能**（safetensors互換性問題）

#### パターンB: vLLMサーバー経由 ✅
```
LedgerLeap App (CPU) → HTTP API → vLLM Server (GPU) → PaddleOCR-VL
```
結果: **可能**

### なぜサーバー経由なら動作するか

1. **推論処理の分離**
   - クライアント: CPU環境で動作可能
   - サーバー: GPU環境でモデル実行
   
2. **API通信**
   - HTTP/gRPC経由でリクエスト送信
   - GPU処理結果を受信
   - safetensorsエラーはサーバー側で発生しない
   
3. **メモリ管理**
   - vLLMが動的メモリ管理
   - 安定したGPU使用率
   - OOMエラーの削減

## 🔧 vLLMサーバー実装の検討

### 必要なリソース

**サーバー側:**
- NVIDIA GPU搭載サーバー
- CUDA 11.2+
- 8GB+ VRAM
- Docker対応

**クライアント側:**
- CPU環境（現状維持）
- HTTP/gRPCクライアント
- ネットワーク接続

### 実装パターン

#### パターンA: クラウドGPU
```
AWS/GCP/Azure GPU Instance
  ├── vLLM Server (Docker)
  └── PaddleOCR-VL 0.9B
       ↓
  HTTP API
       ↓
LedgerLeap (オンプレミス/クラウド)
```

**メリット:**
- スケーラブル
- 管理が容易
- コスト最適化可能

**デメリット:**
- ランニングコスト
- ネットワーク遅延
- データ転送量

#### パターンB: オンプレミスGPU
```
社内GPUサーバー
  ├── vLLM Server
  └── PaddleOCR-VL
       ↓
  内部ネットワーク
       ↓
LedgerLeap
```

**メリット:**
- 低遅延
- データセキュリティ
- 初期投資のみ

**デメリット:**
- ハードウェア調達
- 運用・保守
- スケーリング制限
