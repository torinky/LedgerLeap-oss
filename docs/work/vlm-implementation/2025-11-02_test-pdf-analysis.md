### 2. Key-Valueペア検出の問題点

**検出された51個のKey-Valueペア:**
```
1. [1.000] 16: 28         ← 時刻（誤検出）
2. [1.000] 今日10: 19     ← 時刻（誤検出）
3. [0.992] 10: 17         ← 時刻（誤検出）
4. [0.998] 19: 45         ← 時刻（誤検出）
...
```

**問題:**
- ❌ **ほぼ全てが時刻の誤検出**
- 時刻表示（例: `10:17`）がコロン区切りと認識される
- 実際の意味のあるKey-Valueペアは含まれていない

**原因:**
- 現在の検出ロジックが単純なコロン区切りのみをチェック
- コンテキストや意味を考慮していない

---

### 3. 表構造認識の制約

**文書の実際の構造:**
```
┌─────────────────┬──────────┬────────┬────────┬──────────┐
│ ファイル名      │ 更新日時 │ 時刻   │ サイズ │ 種類     │
├─────────────────┼──────────┼────────┼────────┼──────────┤
│ workflow-...md  │ 2025/7/22│ 10:17  │ 99KB   │ Markdown │
│ access-ctrl...  │ 2025/7/22│ 10:17  │ 38KB   │ Markdown │
│ ...             │ ...      │ ...    │ ...    │ ...      │
└─────────────────┴──────────┴────────┴────────┴──────────┘
```

**PaddleOCR 2.8の結果:**
```
{
  "tables": [],  // ❌ 空配列
  "text_blocks": [
    {"content": "workflow-...md"},
    {"content": "2025/7/22"},
    {"content": "10:17"},
    {"content": "99KB"},
    {"content": "Markdown"},
    // 構造情報は失われる
  ]
}
```

**制約:**
- ❌ 表の構造は認識されない
- ❌ セル間の関係性は保持されない
- ❌ 列・行の情報は失われる
- ✅ テキストは全て抽出される（順不同）

## 改善提案

### 1. Key-Valueペア検出の改善

**現在の問題:**
```python
# 単純なコロン区切り検出（時刻も誤検出）
if ':' in text or '：' in text:
    # ...
```

**改善案:**
```python
# より高度な検出ロジック
# 1. 時刻パターンを除外
if re.match(r'^\d{1,2}:\d{2}$', text):
    continue  # 時刻は除外

# 2. コンテキストを考慮
if len(key) > 2 and len(value) > 2:
    # 意味のあるKey-Valueのみ

# 3. LLM連携
# 抽出後にLLMで構造化
```

### 2. 表構造認識

**オプション1: PaddleOCR-VL**
- PP-StructureV3による表認識
- GPU推奨
- 構造化データとして出力

**オプション2: Marker/MinerU**
- PDF→Markdown変換
- 表はMarkdown形式で保持
- CPU環境で動作

**オプション3: LLM後処理**
```python
# PaddleOCRで抽出したテキストをLLMで構造化
extracted_text = "\n".join([b['content'] for b in text_blocks])
prompt = f"以下はファイルリストです。表形式に構造化してJSON形式で出力してください:\n{extracted_text}"
structured_data = llm.complete(prompt)
```

## 結論

### PaddleOCR 2.8の能力評価（test.pdf基準）

| 項目 | 評価 | 詳細 |
|------|------|------|
| テキスト抽出 | ✅ **優秀** | 500ブロック、精度0.951 |
| Key-Value検出 | ⚠️ **制約あり** | 単純なロジック、誤検出多い |
| 表構造認識 | ❌ **未対応** | 構造情報は失われる |
| 処理速度 | ⚠️ **やや遅い** | 37秒（大きなファイル） |

### 推奨される使用シーン

**✅ 適している:**
- 単純なテキスト抽出
- 請求書・領収書のシンプルなKey-Value
- 全文検索用のインデックス作成

**❌ 適していない:**
- 複雑な表構造の保持
- ファイルリストなどの構造化データ
- 複雑なフォーム認識

### 次のステップ

1. **短期:** Key-Value検出ロジックの改善
   - 時刻パターンの除外
   - 最小文字数の設定

2. **中期:** 表構造が必要な場合の対応
   - PaddleOCR-VLの評価
   - Marker/MinerUの検証

3. **長期:** LLM連携
   - 抽出テキストの後処理
   - 意味理解に基づく構造化
