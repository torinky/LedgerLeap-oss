# 台帳表示・入力UIの改善計画

## 1. 目的と背景

LedgerLeapは、柔軟な台帳定義により多様な業務に対応できる一方、一つの台帳に含まれるカラム（項目）が数十個に及ぶケースも想定されます。現状のUIでは、カラムや入力項目が多くなると以下のような課題が生じ、システムのポテンシャルを最大限に引き出せていません。

*   **リスト表示の視認性低下:**
    *   台帳一覧画面で全カラムが表示されるため、情報が横に長くなり、重要な項目がノイズに埋もれてしまいます。
    *   ユーザーは水平スクロールを多用する必要があり、複数レコードの主要な値を比較・把握することが困難です。
*   **入力フォームの操作性低下:**
    *   台帳の作成・編集画面で全項目が縦一列に並ぶため、フォームが長大になりがちです。
    *   ユーザーは目的の入力項目にたどり着くまでに多くのスクロールを要し、入力作業の効率が低下します。また、入力項目の全体像を把握しにくく、入力ミスや漏れの原因にもなり得ます。

本計画では、これらの課題を解決するため、**リスト表示の段階的省略機能**と**入力フォームのグループ化・折りたたみ機能**を導入します。これにより、情報の視認性とデータ入力の操作性を向上させ、あらゆるユーザーがストレスなくシステムを利用できる環境を構築することを目的とします。

## 2. 基本方針

**「重要度（表示レベル）」という単一の概念を導入し、それをリスト表示とフォーム表示の両方で活用する。**

*   **設定は一元化:** 管理者は、台帳定義の各カラムに対して「**表示レベル**」（例: レベル1, 2, 3）と「**グループ名**」を設定するだけ。
*   **表示は段階的:** ユーザーは、リスト表示画面と入力フォーム画面の両方で、表示レベルを切り替えるボタン（例: 「概要」「詳細」「全項目」）を操作し、表示される情報の粒度を動的に変更する。

この方針により、設定のシンプルさと、UIの柔軟性を両立させます。

## 3. ユーザーシナリオと機能要件

### 3.1. ペルソナ

本機能の検討にあたり、役割と関心事が異なる以下の2つのペルソナを想定します。

*   **ペルソナA: 管理者 / 部門長 (佐藤さん)**
    *   **役割・関心事:**
        *   複数の部署やプロジェクトの状況を横断的に把握したい。
        *   個々のデータ入力の正確性よりも、全体の進捗、ステータス、KPI、問題の有無といったサマリー情報を重視する。
        *   部下が作成した台帳定義の構成が、管理目的に沿っているかを確認する。

*   **ペルソナB: 実務担当者 / 現場リーダー (田中さん)**
    *   **役割・関心事:**
        *   日々の業務データを正確かつ効率的に入力・更新することが最優先。
        *   特定の機器のパラメータや、過去の作業記録など、特定の項目を重点的に確認・比較することが多い。
        *   入力フォームが業務の流れに沿って整理されていると、入力ミスが減り、作業効率が上がる。

### 3.2. ユースケース

#### **ケース1: 台帳リスト表示の視認性向上**

##### **佐藤さんのユースケース: 部門全体のプロジェクト進捗確認**

*   **状況:**
    佐藤さんは、部門で進行中の全プロジェクトの状況を一覧で確認したいと考えています。各プロジェクト台帳には、詳細な技術パラメータや予算実績、リスク評価など多数のカラムが存在しますが、佐藤さんがまず見たいのは「**プロジェクト名**」「**現在のステータス**」「**担当リーダー**」「**最終更新日**」といった主要な管理項目だけです。
*   **課題:**
    現状のUIでは全カラムが表示されるため、重要な情報がノイズに埋もれてしまい、一目で状況を把握できません。水平スクロールを多用する必要があり、非常に非効率です。
*   **理想の体験:**
    1.  台帳一覧画面を開くと、デフォルトではあらかじめ「重要」と設定されたカラムだけが表示されている（**概要表示**）。
    2.  必要に応じて、ボタン一つで「主要項目」や「全項目」といったように、表示する情報の粒度を段階的に切り替えられる。
    3.  さらに表示を絞り込み、「プロジェクト名」と「ステータス」だけを並べて比較する、といったことも可能だと嬉しい。

##### **田中さんのユースケース: 担当設備のパラメータ比較**

*   **状況:**
    田中さんは、自分が担当する複数の設備のコンディションを日々チェックしています。一覧表示で、各設備の「**設備ID**」「**温度**」「**圧力**」「**振動数**」の4つのカラムだけを並べて比較し、異常値がないか素早く確認したいと考えています。他の「前回の点検者」や「設置場所」といった情報は、この作業では不要です。
*   **課題:**
    多くのカラムが表示されていると、比較したいカラムが隣接しておらず、視線を左右に大きく動かす必要があります。これにより、見落としや判断の遅れが生じる可能性があります。
*   **理想の体験:**
    1.  表示するカラムを自分で自由に選択・カスタマイズできる。
    2.  「設備パラメータチェック用」のように、カスタマイズした表示状態を"ビュー"として保存し、次回以降もワンクリックでその表示状態を呼び出せる。

#### **ケース2: 台帳入力フォームの操作性向上**

##### **田中さんのユースケース: 日々の点検データの効率的な入力**

*   **状況:**
    田中さんは、1日に何度も点検データを入力します。入力項目は20項目以上ありますが、業務の流れに沿って「**基本情報**（点検日時、担当者）」「**測定値**（温度、圧力など）」「**状態確認**（異音、異臭の有無など）」「**特記事項**」の4つのグループに分けられます。
*   **課題:**
    全項目が縦に長く並んでいると、どこまで入力したか分かりにくく、スクロール量も多くなります。特に入力頻度の低い「特記事項」が常に表示されているのは邪魔に感じます。
*   **理想の体験:**
    1.  入力項目が、業務の流れに沿ったグループ（例: 「基本情報」「測定値」）にまとめられている。
    2.  各グループはアコーディオンのように開閉でき、普段あまり使わない「特記事項」グループはデフォルトで閉じていてほしい。
    3.  入力必須項目があるグループは、分かりやすくマーク（例: `*`印）が表示されていると入力漏れを防げる。

##### **佐藤さんのユースケース: 台帳定義の構成レビュー**

*   **状況:**
    佐藤さんは、部下が作成した新しい台帳定義の内容を確認しています。この定義が、業務プロセスと管理要件を正しく反映しているかをレビューする必要があります。
*   **課題:**
    全てのカラム設定がフラットに並んでいると、定義全体の構造や意図を理解するのに時間がかかります。
*   **理想の体験:**
    1.  入力フォームと同様に、定義するカラムがグループ分けされていると、どのような情報を収集しようとしているのかが一目瞭然になります。
    2.  「このグループは申請者が入力」「このグループは承認者が入力」といったように、業務プロセスに沿ったグループ化がされていると、定義の妥当性を評価しやすくなります。

## 4. 機能要件とUI/UXデザイン

### 4.1. 機能要件

1.  **台帳定義:**
    *   各カラムに「**表示レベル**」という属性を追加できる。値は数値（例: 1, 2, 3）で、小さいほど重要度が高い。
    *   各カラムに「**グループ名**」という属性を追加できる。
2.  **リスト表示 (`RecordsTable`):**
    *   表示レベルを切り替えるボタン（「概要」「詳細」「全項目」）を設置する。
    *   「概要」選択時は、レベル1のカラムのみ表示する。
    *   「詳細」選択時は、レベル1とレベル2のカラムを表示する。
    *   「全項目」選択時は、すべてのカラムを表示する。
    *   この表示状態は、ユーザーごとに（またはブラウザのセッションで）記憶されることが望ましい。
3.  **入力フォーム/詳細表示 (`CreateColumn`, `ModifyColumn`, `Show`):**
    *   カラムを「グループ名」でグループ化し、折りたたみ可能なパネルで表示する。
    *   表示レベルを切り替えるボタンを設置する。
    *   選択された表示レベルに応じて、表示/非表示になるカラムを制御する。
    *   必須項目が含まれるグループには、それが分かるようにインジケーターを表示する。

### 4.2. UI/UXデザイン

#### **台帳定義画面 (`ModifyColumn`)**

各カラムの設定項目に、以下の2つを追加します。

1.  **表示レベル:** `1`, `2`, `3` から選択するシンプルなドロップダウン。
2.  **グループ名:** テキスト入力フィールド。

#### **リスト表示画面 (`RecordsTable`)**

テーブルのヘッダー部分に、表示レベルを切り替えるボタン（トグルボタンやボタングループ）を設置します。

![リスト表示UI](https://storage.googleapis.com/gemini-agent-images/20250802/UI_List.png)

#### **入力フォーム/詳細表示画面**

フォームの上部に、表示レベルを切り替えるボタンを設置します。各カラムは、設定された「グループ名」ごとに折りたたみパネルでまとめられます。

## 5. ユースケースシミュレーションと考察

### 5.1. シミュレーション

#### **ペルソナA: 管理者 / 部門長 (佐藤さん)**

*   **進捗確認:**
    台帳一覧画面を開くと、デフォルト（概要レベル）では「プロジェクト名」「ステータス」等の主要カラムのみが表示されている。ボタン一つで「詳細」レベルに切り替え、「予算」「実績」カラムを追加で表示し、状況を多角的に把握する。
*   **定義レビュー:**
    部下が作成した「障害報告書」の定義を確認する際、カラムが「発生状況」「技術情報」「対応記録」のグループに分かれているため、定義の構造と意図を即座に理解できる。「エラーログ」の表示レベルが「2（詳細）」に設定されているのを見て、普段は非表示にしつつ、必要な時に確認できる設計が妥当だと判断する。

#### **ペルソナB: 実務担当者 / 現場リーダー (田中さん)**

*   **データ入力:**
    「設備点検記録」の入力フォームを開くと、項目が「基本情報」「測定値」などのグループに分かれ、折りたたまれている。必要なグループだけを開いて入力するため、スクロール量が減り、入力ミスなく迅速に作業を完了できる。
*   **データ比較:**
    担当設備の一覧画面で、デフォルトの「概要」表示から「詳細」表示に切り替えることで、「温度」「圧力」「振動数」といったカラムを表示させ、複数の設備の数値を効率的に比較・確認する。

### 5.2. 潜在的な問題と対策

1.  **問題点: グループ名の乱立**
    *   **懸念:** 各ユーザーが思い思いのグループ名を付けてしまうと、表記が揺れて統一感が失われる。
    *   **対策:**
        *   **フェーズ1（今回）:** 自由テキスト入力とし、運用ガイドラインで代表的なグループ名を例示する。
        *   **フェーズ2（将来）:** グループ名のサジェスト機能や、管理者が「グループ名マスタ」を登録できる機能を検討する。

2.  **問題点: 表示レベル設定の形骸化**
    *   **懸念:** 管理者が全カラムを「レベル1」に設定してしまうと、段階的表示機能が意味をなさなくなる。
    *   **対策:** 台帳定義画面のUIで、ヘルプテキストやツールチップを充実させ、設定の意図を明確に伝える。

3.  **問題点: ユーザーごとの表示カスタマイズ**
    *   **懸念:** 「自分専用のビューを保存したい」という高度な要求には、今回の実装では応えられない。
    *   **対策:** 今回は「管理者があらかじめ設定した共通ルール」を提供することに焦点を当てる。ユーザーごとの詳細なカスタマイズは、将来の機能拡張候補とする。

### 5.3. 結論

シミュレーションの結果、提案されたUI改善は、ペルソナが抱える課題を効果的に解決できると判断します。潜在的な問題は許容可能、あるいは将来的な改善で対応可能な範囲です。

## 6. 実装計画

### **ステップ1: データ構造の拡張**

*   **目的:** カラム定義に「表示レベル」と「グループ名」という2つの新しい概念を追加するための、堅牢かつ後方互換性のあるデータ構造を設計する。
*   **設計方針:**
    *   **影響の最小化:** 既存のデータベーススキーマ（テーブル構造）には変更を加えず、JSON型である `ledger_defines.column_define` カラムの内部構造を拡張するに留まります。
    *   **後方互換性の確保:** 新しい属性（`display_level`, `group`）を持たない既存の台帳定義データも、システムがエラーなく処理できるようにします。
    *   **責務の明確化:** データ構造の解釈やデフォルト値の適用は、`ColumnDefine` モデルクラス内で行い、ロジックの散逸を防ぎます。
*   **`column_define` の新しいデータ構造:**
    `column_define` JSON配列内の各カラムオブジェクトに、以下の2つのキーを追加します。
    ```json
    [
      {
        "id": 1,
        "name": "プロジェクト名",
        "type": "text",
        "order": 1,
        "required": true,
        "unique": false,
        "options": null,
        "display_level": 1,
        "group": "基本情報"
      },
      {
        "id": 2,
        "name": "技術要素",
        "type": "text",
        "order": 2,
        "display_level": 2,
        "group": "詳細パラメータ"
      },
      {
        "id": 3,
        "name": "補足",
        "type": "textarea",
        "order": 3,
        "display_level": 3,
        "group": null
      }
    ]
    ```
*   **各属性の詳細仕様:**
    *   **`display_level`**:
        *   **目的:** カラムの重要度を定義し、段階的な表示切り替えの基準とする。
        *   **データ型:** `integer` (整数)
        *   **値の定義:** `1` (概要), `2` (詳細), `3` (全項目)
        *   **デフォルト値:** `3`。キーが存在しない、または値が不正な場合に適用。これにより、管理者が意識的に設定しない限りカラムは「補足情報」として扱われ、機能の形骸化を防ぎます。
    *   **`group`**:
        *   **目的:** 入力フォームや詳細表示画面で、関連するカラムをまとめるためのグループ名を定義する。
        *   **データ型:** `string` (文字列)
        *   **デフォルト値:** `null`。キーが存在しない、または値が `null` や空文字列の場合、UI上では「その他」グループとして扱われます。
*   **実装への落とし込み:**
    *   **データベースマイグレーション:** スキーマ変更は**不要**。
    *   **モデル (`app/Models/ColumnDefine.php`) の改修:**
        *   `ColumnDefine` クラスに `public int $display_level;` と `public ?string $group;` プロパティを追加。
        *   コンストラクタ内で、渡されたデータにこれらのキーが存在しない場合にデフォルト値を設定するロジックを実装する。
*   **状態:** **完了**
*   **成果物:**
    *   `app/Models/ColumnDefine.php` の修正。
*   **確認結果:**
    *   `tests/Unit/Models/ColumnDefineTest.php` に追加したテストケース（`test_column_define_initializes_with_default_display_level_and_group`、`test_column_define_initializes_with_specified_display_level_and_group`、`test_column_define_handles_existing_data_without_new_properties`）が全てパスしたことを確認しました。
    *   これにより、`ColumnDefine` モデルが新しい `display_level` と `group` プロパティを正しく扱い、デフォルト値の適用や既存データとの互換性も問題ないことが確認できました。

### **ステップ2: 台帳定義画面の改修**

*   **目的:** 管理者が台帳定義のカラムごとに「表示レベル」と「グループ名」を設定できるUIを提供します。これにより、後続のリスト表示や入力フォームでの段階的表示・グループ化の基盤を構築します。
*   **対象ファイル:**
    *   `app/Livewire/LedgerDefine/ModifyColumn.php` (Livewireコンポーネント)
    *   `resources/views/livewire/ledger-define/modify-column.blade.php` (Bladeビュー)
    *   `lang/ja/ledger.php` (翻訳ファイル)
*   **詳細設計:**
    #### 3.1. `app/Livewire/LedgerDefine/ModifyColumn.php` の改修

    このLivewireコンポーネントは、台帳定義のカラム設定を管理する主要なロジックを担います。

    *   **`$columns` プロパティ:**
        *   `$columns` 配列は、各カラムの定義を保持しています。ステップ1で `ColumnDefine` モデルに `display_level` と `group` プロパティが追加されたため、Livewireはこれらの値を自動的にハイドレート（復元）します。
        *   **変更内容:** `mount()` メソッド内の `$this->columns` 初期化ロジックを修正し、`ColumnDefine` オブジェクトから `display_level` と `group` の値も `columns` 配列に含めるようにしました。
    *   **`addColumn()` メソッド:**
        *   新しいカラムを追加する際に、`display_level` と `group` のデフォルト値を明示的に設定します。これにより、新規作成されるカラムが常に正しい初期値を持つことを保証します。
        ```php
        // 既存のaddColumn()メソッド内
        public function addColumn()
        {
            // ... 既存のロジック ...

            $maxId = collect($this->columns)->max('id') ?? -1;

            $newColumn = [
                'id' => $maxId + 1,
                'name' => '',
                'type' => 'text',
                'order' => count($this->columns) + 1,
                'useOptions' => false, // Default value
                'options' => [], // Default value
                'required' => false, // Default value
                'unique' => false, // Default value
                'sortBy' => false, // Default value
                'hint' => '',
                'file' => [],
                'display_level' => 3, // 追加: デフォルトの表示レベル
                'group' => null,      // 追加: デフォルトのグループ名
                'is_collapsed' => true, // 新規追加時は開いた状態にする
            ];

            $this->columns[] = $newColumn;
            // ... 既存のロジック ...
        }
        ```
    *   **`rules()` メソッド:**
        *   `display_level` と `group` フィールドに対するバリデーションルールを追加します。これにより、管理者が不正な値を入力することを防ぎます。
        ```php
        // 既存のrules()メソッド内
        protected function rules(): array
        {
            $rules = [
                // ... 既存のルール ...
                'columns.*.display_level' => ['required', 'integer', 'min:1', 'max:3'], // display_levelのバリデーション
                'columns.*.group' => ['nullable', 'string', 'max:255'], // groupのバリデーション
            ];
            // ... 既存のルール ...
            return $rules;
        }
        ```

    #### 3.2. `resources/views/livewire/ledger-define/modify-column.blade.php` の改修

    #### 3.1. `app/Livewire/Ledger/RecordsTable.php` の改修

    このLivewireコンポーネントは、台帳レコードの表示ロジックを管理します。

    *   **プロパティの追加:**
        *   `public int $displayLevel = 1;`
            *   現在の表示レベルを保持するプロパティ。デフォルト値は `1` (概要) とします。
            *   このプロパティは、ユーザーの選択に応じて更新され、表示するカラムをフィルタリングするために使用されます。
        *   `protected $queryString = ['displayLevel'];`
            *   `displayLevel` プロパティをURLのクエリ文字列に含めることで、ページリロード時やURL共有時に表示状態を維持できるようにします。

    *   **メソッドの追加:**
        *   `public function setDisplayLevel(int $level): void`
            *   ユーザーがボタンをクリックした際に呼び出され、`$this->displayLevel` を更新します。
            *   バリデーションを追加し、`$level` が `1`, `2`, `3` のいずれかであることを保証します。
            ```php
            public function setDisplayLevel(int $level): void
            {
                if (!in_array($level, [1, 2, 3])) {
                    // 不正なレベルが指定された場合は何もしないか、エラーをログに記録
                    return;
                }
                $this->displayLevel = $level;
            }
            ```

    *   **`mount()` メソッドの改修:**
        *   初期化時に、URLクエリ文字列から `displayLevel` を読み込みます。もしクエリ文字列に存在しない場合や不正な値の場合は、デフォルト値 `1` を設定します。
        *   `mount()` メソッド内で、`$this->displayLevel` が有効な値であることを確認するロジックを追加します。
            ```php
            public function mount(LedgerDefine $ledgerDefine): void
            {
                $this->ledgerDefine = $ledgerDefine;
                // ... 既存のロジック ...

                // displayLevelがURLクエリ文字列から設定されている場合、その値を使用
                // そうでない場合、または不正な値の場合はデフォルトの1を使用
                if (!in_array($this->displayLevel, [1, 2, 3])) {
                    $this->displayLevel = 1;
                }
            }
            ```

    *   **`getColumnsProperty()` またはカラム取得ロジックの改修:**
        *   `ledgerDefine` から取得したカラム定義 (`$this->ledgerDefine->column_define`) をループし、各カラムの `display_level` が現在の `$this->displayLevel` 以下である場合にのみ、そのカラムをテーブル表示用のカラムリストに含めます。
        *   これにより、ビューに渡されるカラムデータがフィルタリングされます。
        ```php
        // 例: getColumnsProperty() メソッド内でフィルタリング
        public function getColumnsProperty(): Collection
        {
            return collect($this->ledgerDefine->column_define)
                ->filter(function ($column) {
                    // display_levelが設定されていないカラムはデフォルトで3として扱う
                    $columnDisplayLevel = $column['display_level'] ?? 3;
                    return $columnDisplayLevel <= $this->displayLevel;
                })
                ->sortBy('order'); // 必要に応じてソート
        }
        ```
