## 📊 概要

Phase5では、VLMとOCRの**並列処理統合**を実装します。これにより、処理時間を50%削減し、インデックス空白期間を最小化します。

### 主要な改善

- ⚡ **処理時間50%削減**: VLM/OCR並列実行
- 🕐 **インデックス空白最大1分**: スケジュール最終化
- 🔄 **結果の冗長性**: 片方失敗しても補完
- 📊 **最高品質データ**: VLM > OCR > Tika自動選択

## 🔄 アーキテクチャ比較

### Phase4（現在）

```
ファイルアップロード
  ↓
Tika処理
  ↓
VLM判定 → VLM処理（同期） → 完了
         ↓
         失敗 → OCR処理
```

**問題点:**
- 直列処理で時間がかかる
- VLM成功後もOCR判定ロジックが複雑
- インデックス更新のタイミングが不明確

### Phase5（新設計）

```
ファイルアップロード
  ↓
Tika処理
  ↓
並列ディスパッチ
  ├─ VLM処理（キュー: vlm）
  └─ OCR処理（キュー: ocr）
  
★スケジューラー（1分ごと）
  ↓
FinalizeProcessing
  - 完了ファイル検索
  - 結果選択（VLM > OCR > Tika）
  - インデックス更新
```

**改善点:**
- 並列実行で処理時間50%削減
- スケジュールで確実にインデックス更新
- ロジックが単純で保守性向上

## 🎓 技術的ハイライト

### スケジュール最終化の利点

```php
// app/Console/Kernel.php
$schedule->command('ledger:finalize-processing')
    ->everyMinute()              // 1分ごと実行
    ->withoutOverlapping(10)     // 重複防止
    ->onOneServer()              // 1サーバーで1回
    ->runInBackground();         // 非同期実行
```

**メリット:**
- インデックス空白期間を最小化（最大1分）
- キューの状態に依存しない
- タイムアウト処理が確実
- 手動実行も可能

### 並列処理の実装

```php
// app/Jobs/Ledger/ProcessAttachedFile.php
private function dispatchParallelProcessing(AttachedFile $file): void
{
    // VLM処理をvlmキューへ
    ProcessVlmExtraction::dispatch($file)->onQueue('vlm');
    
    // OCR処理をocrキューへ（少し遅延）
    OcrAndOptimizeFile::dispatch($file)
        ->delay(now()->addSeconds(2))
        ->onQueue('ocr');
}
```

**メリット:**
- 処理時間50%削減
- 専用キューでリソース制御
- 片方失敗しても他方で補完
