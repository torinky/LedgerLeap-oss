

# プレゼンテーション構成案：生成AI時代の「AI駆動開発」とアーキテクチャの実践

**ターゲット聴衆:** 製造業向けツールを請負でウォーターフォール開発している技術者
**目的:** 現在のLedgerLeapのアーキテクチャと、生成AIを活用した開発手法の変化を説明し、積極的な開発プロセスの変革（AIエージェントの活用など）を推奨すること。

## 2. 背景：Generative AIとLedgerLeapの誕生
- **開発のきっかけ:**
  - 自作ツールが7000名以上が利用する基幹システムに成長、10年以上の運用で後継の課題。仕様の明確化のためにプライベートでプロジェクト開始 
  - ソフトウェア工学を学んでいない「現場の技術者」が、生成AIの台頭（2023年6月）を機に挑んだ「趣味での全工程構築」。
  - 「作法を知らない」からこそ、AIとの対話だけで仕様策定から実装までを完結させるという、極めて泥臭いアプローチを選択。
- **直面した葛藤と壁:**
  - 開発そのものよりも、**「周囲の認識とのギャップ」**との戦い。誰もやっていない手法に対する理解を得るための、社内合意形成や方向性の明示という「開発以外の重労働」。
- **衝撃の原体験（AI革命）:**
  - 「コードを書く」のではなく、**「チャットとの対話」だけで仕様が形になり、動くプログラムが生成された瞬間の衝撃**。
- **問いかけ（共感ポイント）:**
  - 「面倒くさくないですか？」：仕様の抜け漏れチェック、ドキュメント作成、テストコードの記述、エビデンスの整理……。
  - プロジェクトの成功を左右するのに、後回しにされがちなこれらのタスクを、**「自分たちがやる必要はあるのか？」**。AIという最強の事務員を雇う準備はできていますか？

## 3. 現在のシステム・アーキテクチャ（全体像）
- **全体像:**
  - *（ここに図を挿入。複雑に見えるが、実は「動くパーツ」の組み合わせに過ぎないことを強調）*
- **技術スタック選択の真意（Windows開発者への翻訳）:**
  - **Laravel/Livewire:** 「Web画面開発の工数を極限まで減らすためのフレームワーク」。複雑なJava EEや重厚なフロントエンド環境を組まなくても、サーバサイドの処理だけでインタラクティブなUIを実現できる。
  - **Mroonga:** 「高速全文検索エンジン」。検索サーバーを別途構築することなく、使い慣れたRDBの中に高速検索機能が組み込まれている。
- **アーキテクチャの「現場の勝利」ポイント:**
  - **「1、2歩楽になる」徹底したユーザー目線:** 
    - 例えば、フォームを開いた瞬間に連番・氏名・日時が自動入力済み。さらにその状態からワンクリックでQRコードを生成・配布。ExcelでVBAを組むよりも遥かに簡単で、かつ管理しやすい「現場のためのUX」。
  - **維持・運用の最小化:** 設定項目を極力減らし、何かあったときに「どこを見ればいいか」がすぐ分かる構成。
  - **ガバナンスへの配慮:** 管理者が本当に知りたい情報を自然に収集できる構造。レコード以外のメタ情報も網羅し、運用の足跡を確実に残す。
  - **小さな工夫の積み重ね:** 巨大な機能を一度に作るのではなく、現場の小さな課題を一つずつコンポーネントとして繋ぎこむことで、運用負荷を上げずに機能拡張を継続。

## 4. 開発方法の変化：従来型（ウォーターフォール）との比較
- **従来（ウォーターフォール）の「空転」:** 
  - **会話の不一致:** 現場の事情を知らない「要求者」と、技術スタックが異なる「エンジニア」が、互いに別のレイヤーで会話するため、仕様確認が堂々巡りする。
  - **フィードバックの欠如:** 「動くもの」が手元に届かないまま議論が進むため、焦点が定まらず、結果として「言ったはず」「そんなつもりはない」の応酬になる。
- **現在（AI駆動開発）の「合意形成」:**
  - **「AIは高性能な辞書・処理器」と割り切る:** 魔法の杖ではなく、高度な道具として扱う。誰が、どのような環境で、どう使うかを対話の中で明確にすることで、全ての成果物のフォーカスが一点に定まる。
  - **「プロトタイプ駆動」の加速:** アイディアをAIに即座に形にさせることで、「動くもの」を前提とした議論へ強制的に転換させる。
- **明日からの「小さな変化」：まずは「ペルソナとシナリオ」から:**
  - いきなり仕様書を書くのではなく、まずは「その道具を使う人はどんな人で、どんな場面で困っているのか？」という **ユーザーの日常（前後・周辺のペルソナ・シナリオ）** の解像度をAIと上げる作業を先行する。
  - 共通認識をテキストで固めてからコードを生成させることで、議論の循環を断ち切る。

## 5. AIを指揮するコントロールセンター: `.github` ディレクトリの構造と役割
- **役割:** 単なる設定置き場ではなく、AIに「LedgerLeapのルールと技」を教え込み、自律的に動かすための**プロジェクトの脳（知能）**。
- **構造の「標準化」:**
  - **`.github/instructions/` (標準作業手順書: SOP):** Laravel/Livewire/Testなどの技術スタックごとの「守るべき作法」を定式化したもの。ここを見れば誰でも（AIでも）同じ品質でコードを書ける。
  - **`.github/prompts/` (AIへの作業指示書):** バグ調査、HAR解析、Gitコミットなど、よくある業務を「最短で最大の成果を出すための手順」に落とし込んだテンプレート。
  - **`.github/skills/` (熟練工の技・治具):** 特定の技術課題（権限モデル、テストの最適化、UI構築など）に対して、成功事例や参考情報を集約した「AI用の専用治具」。個別のスキルの中に具体的な手順（SKILL.md）と参照データ（references/）がセットで格納されている。
- **進化のサイクル（ヒヤリハット報告からの改善）:**
  - 開発現場で起きた不具合やボトルネックを単なる「失敗」で終わらせず、その知見を `skills/` や `instructions/` へフィードバック。
  - 過去の失敗をAIの「チェックリスト」に昇華させることで、二度と同じミスを繰り返さない「組織的な強さ」を自動で積み上げている。

## 6. 実践：生成AI（エージェント）を前提とした実際の開発プロセス
- **プロセス例1: 機能開発・改善（Issue起票から実装まで）**
    - 例: `docs/work/ui-ux/2026-03-21_issue-116-sprint2-plan.md` のようなMarkdown形式の実装計画書を作成。これは「人間への説明用」ではなく「AIへの指示書兼ログ」として機能する。
    - 実装・テストはAIエージェントが自律的に実行し、エンジニアは生成されたコードのレビューとマージに集中する。
- **プロセス例2: 高度なパフォーマンス改善や不具合調査**
    - 例: Issue #111〜113（UI描画の遅延）でのHAR（HTTP Archive）解析。AIにネットワークログを渡し、ボトルネック（Livewireの不要な再レンダリング等）を特定・修正させる。
    - 調査結果を報告書として残し、同じ調査をAIが再現・再利用できるよう `.github/skills/` へ定型ワークフローとして組み込む。
- **「メタ開発」：AIへの指示体系（ルール）自体の恒常的なアップデート**
    - 例: テスト用のDBが意図せず汚染されたインシデントを受け、リカバリ手順を `docs` に記録し、汎用ルール（`copilot-instructions.md`等）へ即座に反映。AIが「一度したミスを構造的に繰り返さない」仕組みを構築。
- **現場の知見・教訓:**
    - **AIとの「立ち位置」の変化:** 初期は「部下（指示出し）」としてコンテキストを叩き込み、実装後は「相棒（ペアプロ）」や「先生」として活用。
    - **現実の「綻び」との向き合い方:** 管理成果物（ドキュメント等）の整合性崩壊は前提とする。AIが好む「独自のパターン」を尊重し、無理に意図を押し付けず折り合いをつける柔軟性が重要。
    - **「小さな手触り」からのスタート:** 大規模な導入や完璧なやり方を探すのではなく、手元のルーチンワークから実験を始める。「やり方を固めず、変化に順応し続けること」が最強のスキルとなる。

## 7. 現在の課題と今後の展望（AI駆動開発のリアル）
- **AI駆動開発が直面する「新たな負債」:**
  - **コンテキストの迷宮:** システムが成長するにつれ、関連情報の断片（ドキュメント、Issue、コード、テスト、ヒヤリハット記録）が `docs/` や `.github/` 配下に分散。AIが「正解」に辿り着くための「文脈の把握」に要するコストが増大している。
  - **「自動化」の影（テストの網羅性）:** AIによる爆速実装の恩恵を受ける一方で、複雑なビジネスロジックの「仕様の抜け漏れ」や「人間がレビューしきれないエッジケース」が新たなリスクに。特にマルチテナント環境における権限モデルのテスト網羅が最重要課題。
- **展望：自律的・組織的な進化へ:**
  - **知識の自己組織化:** 指示書やスキルが「追加」されるだけでなく、矛盾を検知して「クリーンアップ（リファクタリング）」をAI自らが自律的に行う仕組みの導入。
  - **「AIが読むためのアーキテクチャ」:** 人間にとっての読みやすさ以上に、AIエージェントにとっての「理解しやすさ・操作しやすさ（疎結合化、モジュール化）」を最優先にしたアーキテクチャへの再設計。

## 8. 製造業・請負開発への提言：プロセス変革の第一歩
- **「コードを書く人」から「システムを育てる設計者」へ:** 
  - プログラマの役割は「コーディング（作業）」から「環境・ルール・テストを管理する（監督）」へと完全に移行する。
- **「現場をわかっている」ドキュメントの再定義:**
  - Excelでの完璧な仕様書作成を諦め、AIが入力として処理できる「ペルソナ・ユーザーシナリオ・ヒヤリハット記録」をベースにした、生きたドキュメント文化への脱却。
- **スモールスタートの王道:**
  - いきなり大規模開発にAIを持ち込まず、まず「エビデンス作成」や「テストコードの雛形」「ログの調査」といった、**彼ら自身の「日々の面倒な業務（ゴミ掃除）」をAIにやらせる**ことから始める。

## 9. まとめと質疑応答
- **技術者としての武器の磨き方:**
  - 明日からの武器は「新しい言語の習得」ではなく、「AIを相棒として使いこなし、現場の泥臭い課題をいかに言語化・構造化できるか」。
- **LedgerLeapの証明:**
  - 個人や少人数でも、AIを「脳（指示書）」として使いこなすことで、請負開発の現場レベルからエンタープライズ級の複雑性に立ち向かえる。
- **未来の作り方:**
  - 明日、最初に行うのは「面倒な業務を一つ、AIと一緒にやってみる」こと。その小さな成功が、明日からの請負開発の風景を少しずつ変えていく。
