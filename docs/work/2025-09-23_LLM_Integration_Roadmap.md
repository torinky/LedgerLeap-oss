# LedgerLeap LLM連携機能 開発ロードマップ

**日付:** 2025年9月23日

## 1. はじめに

### 1.1. 目的と戦略

本ドキュメントは、台帳管理システム LedgerLeap に LLM (大規模言語モデル) を連携させるための、包括的な開発ロードマップを定義するものである。

本戦略は、以下の2つのフェーズで構成される。

1.  **フェーズ1: API基盤の構築（短期計画）**
    LedgerLeapを、Gemini CLIやDifyのような外部LLMアプリケーションから機能を呼び出せる「ヘッドレスなデータストア/機能プロバイダー」として確立する。これにより、アプリケーション本体に大規模な改修を加えることなく、迅速に高度なLLM連携の実験・検証を開始できる「素地」を整える。

2.  **フェーズ2: API活用による高度な機能実現シナリオ（将来展望）**
    フェーズ1で構築したAPIを外部ツールと組み合わせることで、よりリッチでユーザーフレンドリーな機能を実現する。LedgerLeapはあくまで「使われる側のソリッドな仕組み」を維持し、UIやLLMの頭脳は外部の専門ツールに委ねる、疎結合なアーキテクチャを基本方針とする。

## 2. 機能要件（全体像）

本ロードマップ全体を通じて実現を目指す機能要件は、以下の通りである。

### 2.1. 【検索・参照】局面：情報の発見と理解を高速化する

- **要件1: 自然言語による高度な情報検索**
    - **シナリオ:** ユーザーが「A社の〇〇トラブルの時の対応手順と最終的な顧客への報告書を要約して」といった自然言語で質問する。
    - **機能:** システムが関連情報を全文検索で取得（Retrieval）し、LLMがそれを元に要約や回答を生成（Generation）して提示する。
    - **提供価値:** 従来のキーワード検索と比較し、情報発見の速度と精度を大幅に向上させ、ユーザーの自己解決を促進する。

### 2.2. 【作成・登録】局面：高品質な情報入力を支援・自動化する

- **要件2: AIによる台帳作成支援**
    - **シナリオ:** ユーザーが台帳を作成する際に、過去の類似情報を参考にしたい、または定型的な内容の入力を効率化したい。
    - **機能:**
        - **テンプレート生成:** 「〇〇の週次報告書を作成したい」という指示に対し、過去の類似報告書をRAGで検索し、LLMが記載すべき項目を含んだテンプレートを生成する。
        - **内容のサジェスト:** 台帳作成中に、入力内容と関連する過去の台帳や添付ファイルをRAGで検索し、「この項目には、以前の△△のケースを参考に記載してはいかがですか？」とLLMが具体的な記述をサジェストする。
        - **添付ファイルの自動要約:** WordやPDFファイルを添付した際に、LLMがその内容を自動で読み取り、台帳の「概要」欄などに要約を自動入力する。
    - **提供価値:** 品質の高い情報の入力を促し、ナレッジの属人化を防ぐ。入力の手間を削減する。

- **要件3: 対話による台帳レコードの直接作成**
    - **シナリオ:** ユーザーがチャット形式の対話を通じて、議事録や作業報告などをシステムに登録したい。
    - **機能:** ユーザーが「今日の〇〇プロジェクトの定例議事録を作成して。参加者はAさんとBさん、決定事項はDです。」のように自然言語で指示した内容をLLMが解釈し、適切な台帳定義を選択し、内容をマッピングして、台帳レコードを自動で作成する。
    - **提供価値:** フォーム画面を開くことなく、対話の流れで直感的にデータを登録できる。モバイル環境など、フォーム入力が煩雑な状況での利便性が向上する。

### 2.3. 【整理】局面：情報の適切な分類を自動化する

- **要件4: AIによる台帳整理支援**
    - **シナリオ:** ユーザーが作成した台帳を、どのフォルダに保存すべきか迷ってしまう。
    - **機能:**
        - **自動フォルダ提案:** LLMが作成された台帳の内容を理解し、過去の類似台帳がどのフォルダに格納されているかをRAGで検索。「この台帳は『/プロジェクトA/仕様書』フォルダへの保存を推奨します」のように、最適な保存先を提案する。
        - **タグの自動生成:** LLMが台帳内容から関連キーワードを抽出し、タグとして自動で付与する。
    - **提供価値:** 手動での分類作業をなくし、一貫性のある情報整理を実現する。情報のサイロ化を防ぎ、検索性を向上させる。

## 3. フェーズ1: API基盤の構築 (短期計画)

### 3.1. アーキテクチャ方針

- **認証:** 外部アプリケーションからの安全なアクセスのため、Laravel Sanctumを利用したAPIトークン認証を導入する。
- **エンドポイント:** API用のルートは `routes/api.php` に定義し、`/api/v1/` のプレフィックスを付与する。
- **レスポンス形式:** レスポンスはJSON形式とし、ページネーション情報などを含む場合はJSON:API標準に準拠した形式を基本とする。

### 3.2. 段階的開発計画

#### ステップ1.1: API認証基盤の構築 (Sanctum)

- **目的:** 外部アプリケーションが安全にAPIを利用するための認証メカニズムを確立する。
- **タスク:**
    1.  **Sanctumのインストールと設定:** `composer require laravel/sanctum` を実行し、基本的な設定（マイグレーション、ミドルウェアの登録）を行う。
    2.  **`User`モデルへの対応:** `App\Models\User` モデルに `HasApiTokens` トレイトを追加する。
    3.  **トークン管理UIの実装:** 管理者がユーザーごとにAPIトークンを発行・管理できるよう、Filamentの`UserResource`管理画面にトークン管理機能を追加する。
- **確認できること:** 管理画面から特定のユーザーに対してAPIトークンを発行・削除でき、発行されたトークンを使ってAPIにアクセスできること。

#### ステップ1.2: 検索APIの実装 (RAGのRetrieval機能)

- **目的:** 外部のLLMアプリケーションが、LedgerLeap内の情報を検索・取得（Retrieval）できるようにする。これはRAGを実現するための最も重要なAPIとなる。
- **エンドポイント:** `GET /api/v1/search`
- **パラメータ:**
    - `q` (string, 必須): 検索キーワード。
    - `limit` (integer, 任意, デフォルト: 10): 取得する最大件数。
- **処理概要:**
    1.  Sanctumミドルウェアで認証を行う。
    2.  `LedgerService::searchLedgers()` を呼び出し、キーワードに一致する台帳レコードを取得する。
    3.  取得したレコードの `content` と `content_attached` を含む、整形されたJSONを返す。
- **確認できること:** APIトークンをヘッダーに付与してリクエストを送信すると、キーワードに一致する台帳データがJSON形式で返ってくること。

#### ステップ1.3: 台帳定義リストAPIの実装

- **目的:** 外部のLLMが、どのような種類の台帳を作成できるかを動的に把握できるようにする。
- **エンドポイント:** `GET /api/v1/ledger-defines`
- **処理概要:**
    1.  認証を行う。
    2.  `LedgerDefine` モデルの一覧を取得する。
    3.  各台帳定義の「名前」「説明」「カラム構造（カラム名、型など）」を含むJSONを返す。
- **確認できること:** システムに登録されている台帳定義の一覧が、構造化されたJSONとして返ってくること。

#### ステップ1.4: 台帳作成APIの実装

- **目的:** 外部のLLMアプリケーションとの対話の結果、新しい台帳レコードをLedgerLeapにプログラム経由で登録できるようにする。
- **エンドポイント:** `POST /api/v1/ledgers`
- **リクエストボディ (例):**
    ```json
    {
        "ledger_define_id": 1,
        "content": {
            "0": "議事録のタイトル",
            "1": "参加者A, 参加者B",
            "2": "決定事項の内容..."
        }
    }
    ```
- **処理概要:**
    1.  認証を行う。
    2.  リクエストボディの内容をバリデーションする。
    3.  `Ledger::create()` を実行して、新しい台帳レコードを作成する。
    4.  作成されたリソースの情報をJSONで返す。
- **確認できること:** API経由で送信したデータで、新しい台帳レコードが正しく作成されること。

## 4. フェーズ2: API活用による高度な機能実現シナリオ (将来展望)

### 4.1. はじめに

フェーズ1で構築したAPI基盤を、Difyのような外部LLMアプリケーション開発プラットフォームや、カスタムのフロントエンドアプリケーションと組み合わせることで、以下の「機能要件」で定義したような高度なユーザー体験を実現することを目指す。

将来的には、LLMがより汎用的なミドルウェアとなった場合、UI上のツールチップやサジェストといった**補助的な機能**として、LedgerLeap内に直接組み込む可能性も考えられる。しかし、検索や作成といった**コアな機能連携**は、APIを介した疎結合なアーキテクチャを基本方針とする。

### 4.2. API活用シナリオ

#### シナリオ2.1: Q&Aプロトタイプの実現
- **ゴール:** 外部ツール上で、ユーザーの質問に対し、LedgerLeap内の情報を参照してLLMが回答を生成する基本的なRAGフローを構築する。
- **利用するLedgerLeap API:** `GET /api/v1/search`
- **外部ツールでの実装イメージ:**
    1.  ユーザーからの入力を受け付けるチャットUIを構築する。
    2.  ユーザーの質問をキーワードとして、LedgerLeapの検索APIを呼び出す。
    3.  APIから返された検索結果（JSON）を「参考情報」としてプロンプトに組み込む。
    4.  LLM APIを呼び出し、回答を生成してUIに表示する。

#### シナリオ2.2: 作成支援（サジェスト）機能の実現
- **ゴール:** 外部の台帳作成フォーム等で、入力内容に応じた関連情報を提示し、ユーザーの入力を支援する。
- **利用するLedgerLeap API:** `GET /api/v1/search`
- **外部ツールでの実装イメージ:**
    1.  台帳作成フォームの特定の入力欄（例: タイトル、概要）の変更をトリガーとする。
    2.  入力されたテキストをキーワードとして、LedgerLeapの検索APIを非同期で呼び出す。
    3.  返された検索結果を「参考情報」としてLLMに渡し、「追記すべき項目」や「関連する過去の記述」を生成させる。
    4.  生成されたサジェストを、フォームの横や下に提示する。

#### シナリオ2.3: 整理支援（フォルダ提案）機能の実現
- **ゴール:** 外部ツールで台帳が作成された後、最適な保存先フォルダをユーザーに提案する。
- **利用するLedgerLeap API:** `GET /api/v1/search`
- **外部ツールでの実装イメージ:**
    1.  ユーザーが台帳の作成を完了したタイミングをトリガーとする。
    2.  作成された台帳の内容（タイトルや本文）をキーワードとして、LedgerLeapの検索APIを呼び出す。
    3.  返された検索結果（類似の過去の台帳）が最も多く含まれるフォルダを特定する。
    4.  特定したフォルダを「推奨保存先」としてユーザーに提示する。

#### シナリオ2.4: 対話による台帳作成機能の実現
- **ゴール:** 外部のチャットUI上で、ユーザーとの対話を通じて台帳レコードをLedgerLeapに作成する。
- **利用するLedgerLeap API:**
    - `GET /api/v1/ledger-defines`
    - `POST /api/v1/ledgers`
- **外部ツールでの実装イメージ（エージェントの振る舞い）:**
    1.  ユーザーからの「〇〇の議事録を作成して」といった指示をトリガーとする。
    2.  まず、台帳定義リストAPIを呼び出し、利用可能な台帳の種類を把握する。
    3.  ユーザーの指示と台帳定義リストをLLMに渡し、最も適切と思われる台帳定義を選択させる。
    4.  次に、選択した台帳定義のカラム構造とユーザーの指示をLLMに渡し、各カラムに設定すべき値をJSON形式で生成させる。
    5.  最後に、生成されたJSONをリクエストボディとして、台帳作成APIを呼び出し、LedgerLeapにレコードを登録する。
    6.  登録成功後、作成された台帳へのリンクをチャットUIに表示し、ユーザーにフィードバックする。

## 5. 全体テスト計画

- **フェーズ1 (API):**
    - 各APIエンドポイントに対して、認証、認可、正常系、異常系（バリデーションエラー等）を網羅するフィーチャーテストを作成する。
- **フェーズ2 (API活用シナリオ):**
    - LedgerLeap側のテストはフェーズ1で完了しているため、外部ツール側で、API連携が正しく行えるかの結合テストを実施する。

## 6. 今後の展望

本ロードマップの完了後、以下の高度な機能についても検討を進める。
- **ベクトル検索の導入**: キーワード検索に加え、文章の意味的な類似度で検索できるベクトル検索を導入し、RAGの検索精度をさらに向上させる。
- **自律エージェント化**: ユーザーの指示に基づき、LLMが複数のステップ（情報検索→要約→新規台帳作成→関係者へ通知）を自律的に実行するエージェント機能の開発。
