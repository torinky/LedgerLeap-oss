## 1. Day 1-3: 環境構築と基本機能テスト実施記録

### 1.1. 実施サマリー

VLMコンテナの環境構築、ビルド、起動を実施しました。
初期設定で複数の問題が発生しましたが、依存関係の調査と修正を繰り返し行い、最終的にコンテナの正常起動と、JPGおよびPDFファイルに対する基本的なOCR機能の動作を確認しました。

### 1.2. 計画からの主な変更点（トラブルシューティング記録）

当初の計画通りには進まず、以下の問題解決のための変更を行いました。

#### 変更点1: Dockerfileのシステム依存関係を修正

-   **問題:** `apt-get install`実行時に`libgl1-mesa-glx`パッケージが見つからず、コンテナのビルドに失敗しました。
-   **原因:** ベースイメージである`python:3.10-slim`（Debian Trixie）のリポジトリでは、パッケージ名が変更されていました。
-   **対応:** パッケージ名を`libgl1`に修正することで解決しました。

#### 変更点2: Dockerfileの起動コマンドを修正

-   **問題:** コンテナが起動直後に終了し、ログも出力されませんでした。
-   **原因:** 計画書記載の`CMD ["python", "app.py"]`では、FastAPIアプリケーションがASGIサーバーとして正しく起動していませんでした。
-   **対応:** 起動コマンドを`CMD ["uvicorn", "app:app", "--host", "0.0.0.0", "--port", "8000"]`に変更し、正常にサーバーとして起動するように修正しました。

#### 変更点3: `requirements.txt` の `PyMuPDF` バージョンを調整 (最重要)

-   **問題:** JPGファイルのOCRは成功するものの、PDFファイルを処理しようとすると`'Document' object has no attribute 'pageCount'`というエラーが発生し、失敗しました。
-   **調査:** インターネットでの調査の結果、このエラーは`paddleocr==2.7.0`と`PyMuPDF`の新しいバージョンとの間にAPIの非互換性があることが原因であると判明しました。
-   **対応:** `paddleocr`が要求する古いAPIと互換性のある`PyMuPDF==1.19.0`にバージョンをダウングレードすることで、PDFの処理に成功しました。

#### 変更点4: ポートフォワーディングの変更

-   **問題:** 既存の`embedding`サービスがコンテナ内で`8000`番ポートを使用しており、将来的な衝突の可能性がありました。
-   **対応:** VLMサービスのホスト側ポートを`8000`から`8001`に変更し（`8001:8000`）、衝突を回避しました。

## 2. Day 4: 負荷テストとリソース測定 (2025-10-25実施)

### 2.1. 実施サマリー

同時実行数5、総リクエスト数50回で負荷テストを実施しました。
テストスクリプトの環境依存（macOSの`date`コマンド）の問題など、いくつかのトラブルがありましたが、スクリプトを修正し、最終的にすべてのリクエストを正常に処理できることを確認しました。

### 2.2. 計画からの主な変更点（トラブルシューティング記録）

#### 変更点1: 負荷テストスクリプトの修正

-   **問題:** `top`コマンドが存在しない、`xargs`でコマンドラインが長すぎる、`date +%s%3N`がmacOSで動作しない、など複数の問題でスクリプトが正常に動作しませんでした。
-   **対応:**
    -   リソース確認を`docker stats`コマンドに変更しました。
    -   `curl`コマンドを関数化し、`xargs`から安全に呼び出せるようにしました。
    -   ミリ秒単位の時間計測を、環境依存の少ない`python3`コマンドを使用する方法に変更しました。
    -   `curl`のレスポンスからHTTPステータスコードを安全に切り出すロジックに改善しました。

## 4. 補足調査: 日本語構造化データ抽出の探求 (2025-10-26実施)

当初のPoC計画には含まれていなかったが、VLMの価値を最大化するため、プレーンテキスト抽出に加えて、テーブルなどの構造化データを抽出する可能性を調査・検証した。

### 4.1. キャッチアップ方法 (要約)

-   **忙しい人向け (30秒で理解):**
    1.  `PaddleOCR`の構造化抽出機能(`PP-Structure`)を試したが、**日本語の文字認識との両立が困難**であることが判明した（レイアウト解析モデルが日本語非対応のため、OCR結果が中国語に引っ張られる）。
    2.  ライブラリをハックして解決を試みたが、根本的な解決には至らなかった。
    3.  代替案として日本語に特化した`YomiToku`を調査したが、**ライセンス(AGPL-3.0)が厳しく、商用利用は困難**と判断。
    4.  **結論:** `PaddleOCR`単体での日本語構造化抽出は断念。次のステップとして、PDFからMarkdownへの変換に特化した**`Marker`**（MITライセンス）を検証する方針に転換する。

-   **詳細を追いたい人向け:**
    以下のセクション4.2〜4.4の試行錯誤の経緯を参照してください。`PaddleOCR`の`lang`パラメータと内部コンポーネントの挙動、ライセンス問題の調査など、技術選定の背景が理解できます。

### 4.2. 試行1: `PP-Structure`の有効化と失敗

-   **アプローチ:** `PaddleOCR`の初期化時に`use_structure=True`を設定し、構造化データ抽出用のエンドポイント`/extract_structured`を実装した。
-   **結果:** 失敗。`PP-Structure`は`lang='japan'`をサポートしておらず、コンテナが起動しなかった。
    -   **エラーログ:** `ppocr ERROR: lang japan is not support, we only support dict_keys(['en', 'ch']) for layout models`

### 4.3. 試行2: `lang='ch'`での代用とOCR精度の問題

-   **アプローチ:** レイアウト解析モデルを動作させるため、`PP-Structure`の初期化パラメータを`lang='ch'`に変更してテストした。
-   **結果:** テーブルの**構造（HTMLタグ）**は抽出できたが、テーブル内の**日本語テキストが中国語として誤認識**され、文字化けのような状態になった。
    -   **考察:** `lang`パラメータがレイアウト解析だけでなく、OCRの言語設定にも影響を与えてしまっている。

### 4.4. 試行3: OCRエンジンの注入 (ハック) と失敗

-   **アプローチ:** `PP-Structure`の`ocr_engine`パラメータに着目。レイアウト解析用に`lang='ch'`で`PP-Structure`を初期化しつつ、OCR処理には`lang='japan'`で別途初期化した`PaddleOCR`インスタンスを注入する、という高度な方法を試した。
-   **結果:** 失敗。結果は試行2と変わらず、日本語テキストが中国語として認識された。
    -   **結論:** `PP-Structure`の内部実装により、注入したOCRエンジンの言語設定が無視され、親である`PP-Structure`の`lang='ch'`設定が強制されている可能性が高い。ライブラリのソースコードレベルの改造なしに、この問題を回避するのは極めて困難と判断。

### 4.5. 代替案の調査と方針転換

-   **`YomiToku`の調査:** 日本語に特化した有望なOSSとして`YomiToku`を調査したが、ライセンスが**AGPL-3.0**であることが判明。本プロジェクトの要件とは合致しないため、採用を見送った。
-   **方針転換:** `PaddleOCR`単体での日本語構造化抽出は困難と結論。次のPoCステップとして、PDFからMarkdownへの変換に特化した以下のOSSを検証する方針に転換する。
    1.  **最有力候補:** **`Marker`** (MITライセンス) - 導入が容易で実績も豊富。まずはこれを試す。
    2.  **次点候補:** **`MinerU`** (Apache-2.0ライセンス) - 日本語対応を明記しており、Markerで精度が出ない場合の代替案。
