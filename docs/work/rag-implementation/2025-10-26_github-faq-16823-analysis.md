## 🔑 重要な情報（抜粋・翻訳）

### 1. CPU/NPU対応状況 ⚠️

**質問:** CPU または HUAWEI NPU 上でのデプロイは可能か？

**公式回答:**
```
現在のバージョンは主に NVIDIA GPU での推論をサポートしています。
x64 CPU および Ascend 910B NPU への対応作業は現在高優先度で進行中です。
今後のアップデートにご期待ください。
```

**重要ポイント:**
- ✅ CPU対応は開発中（高優先度）
- ✅ 将来的に対応予定あり
- ⚠️ 現時点ではNVIDIA GPU必須
- 📅 リリース時期は未定

**LedgerLeapへの影響:**
```
短期: GPU必須（変わらず）
中期: CPU対応リリース待ち（高優先度開発中）
長期: CPU環境での直接実行が可能に
```

### 2. 最小ハードウェア要件

**質問:** 最小ハードウェア構成は？

**公式回答:**
```
最小動作確認済み構成:
- GPU: RTX 3060 (12 GB)
- Compute Capability: 7.0以上（例: V100）

メモリ使用量と推論速度はハードウェアにより大きく異なります。
```

**テスト済みハードウェア:**
- ✅ RTX 3060 (12GB) - 最小構成
- ✅ V100 - Compute Capability 7.0
- ✅ T4 - サポート（dtype問題解決済み）
- ✅ RTX 3090 Ti - コミュニティ報告あり

### 3. OOM（メモリ不足）問題

**質問:** 推論中にOOMエラーが発生する

**公式回答:**
```
デフォルトの PaddlePaddle 動的グラフ推論モードでは
メモリ使用量の変動が大きく、複雑な画像処理時に
ピークメモリが高くなる可能性があります。

より安定したメモリ使用のためには、vLLM や SGLang などの
専用推論加速フレームワークでのデプロイを推奨します。
```

**推奨対策:**
1. **vLLMデプロイ（最推奨）**
   - 安定したメモリ管理
   - ピーク使用量の削減
   - OOMエラーの回避

2. **SGLangデプロイ**
   - vLLMの代替
   - 同様の安定性

3. **画像前処理**
   - 解像度の最適化
   - バッチサイズの調整

### 4. dtype不一致問題（解決済み）

**質問:** Compute Capability < 8.5 のGPU（T4、V100）で
dtype不一致エラーが発生する

**公式回答:**
```
現在（2025年10月24日時点）、PaddleOCR-VLのデフォルト推論方式は
Compute Capability ≥ 7.0 のGPUをサポートしています。

既にインストール済みの場合は、以下のコマンドで
PaddleX をアップグレードしてください：

python -m pip install -U paddlex
```

**解決状況:**
- ✅ 2025年10月24日に解決
- ✅ PaddleXのアップグレードで対応
- ✅ Compute Capability 7.0以上をサポート

### 2. vLLM/SGLangデプロイが公式推奨

**明確な推奨:**
- デフォルトのPaddlePaddle動的グラフ: ⚠️ 不安定
- vLLM/SGLangフレームワーク: ✅ 推奨

**理由:**
1. 安定したメモリ使用
2. OOMエラーの削減
3. 本番環境での信頼性

### 3. Compute Capability 7.0以上は完全サポート

**対応GPU:**
- V100（Compute Capability 7.0）
- T4（Compute Capability 7.5）
- RTX 2000シリーズ（Compute Capability 7.5）
- RTX 3000シリーズ（Compute Capability 8.6）
- RTX 4000シリーズ（Compute Capability 8.9）
- A100（Compute Capability 8.0）

**非対応GPU:**
- GTX 1000シリーズ（Compute Capability 6.1）
- GTX 900シリーズ（Compute Capability 5.2）
