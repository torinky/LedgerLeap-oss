## 1. 調査の目的

先のVLM動作検証PoCにより、`Marker`がPDF/画像からのMarkdown構造化抽出において有効であることが確認された。しかし、技術選定を確定するには、いくつかの未解決の疑問点が残っている。

本調査は、これらの疑問点を解消し、`LedgerLeap`のVLM機能として長期的に最適なツールセットを判断するための追加情報を得ることを目的とする。

具体的には、以下の3つの疑問点を調査・検証する。

1.  **PaddleOCRの将来性:** 現状では日本語の構造化抽出が困難だが、将来的にサポートされる可能性はあるか？
2.  **手書き文字認識性能:** `PaddleOCR`は、業界標準の`Tesseract`と比較して、手書き文字認識において優位性があるか？
3.  **Markerの手書き文字対応:** `Marker`は手書き文字をどの程度テキストとして認識できるのか、あるいは全くできないのか？

## 2. 調査項目と実施計画

### 2.1. PaddleOCRの日本語構造化抽出の将来性調査

-   **背景:** PoCでは、`PaddleOCR`の構造化抽出機能(`PP-Structure`)が日本語OCRと両立できない問題があった。もし将来的にこれが解決されるのであれば、`PaddleOCR`は依然として有力な選択肢となりうる。
-   **調査方法:**
    1.  **GitHubリポジトリのIssue検索:** `PaddleOCR`の公式リポジトリにて、「Japanese」「structure」「layout model」「PP-Structure」等のキーワードで過去のIssueとPull Requestを検索し、日本語対応に関する議論や開発状況を追跡する。
    2.  **公式ドキュメントとロードマップの確認:** 公式ドキュメントや公開されているロードマップに、多言語対応（特に日本語）の構造化抽出に関する記述がないかを確認する。
    3.  **関連論文・技術ブログの調査:** `PP-Structure`の論文や、開発者による技術ブログ等から、アーキテクチャの制約や将来の展望に関する情報を収集する。
-   **評価基準:**
    -   日本語サポートに関する具体的な開発計画やマイルストーンが公式に言及されているか。
    -   コミュニティ内で日本語対応に関する要望や議論が活発に行われているか。
-   **調査結果:**
    -   PoCで発生した問題は、古いライブラリバージョン(`paddleocr==2.7.0`)で`PPStructure`クラスを`lang='japan'`で初期化したことによる、既知の制約が原因であった。
    -   最新版の`PaddleOCR`ではアーキテクチャが変更され、`PaddleOCR`クラス自体に構造化抽出機能が統合された。`structure_version='PP-StructureV2'`のように引数を指定して初期化することで、言語 (`lang='japan'`) と構造化抽出の併用が可能になっている。
-   **結論:**
    -   **「将来性あり」**。PoCでの失敗は旧バージョンの制約によるものであり、最新版では問題なく日本語の構造化抽出が実行可能である。
    -   具体的な実装方法は、以下のガイドドキュメントを参照。
        -   **[PaddleOCR最新版による日本語構造化抽出 実装ガイド](./2025-10-26_paddleocr-latest-impl-guide.md)**

### 2.2. 手書き文字認識性能の比較 (PaddleOCR vs. Tesseract)

-   **背景:** `PaddleOCR`は手書き文字認識に強いという仮説がある。これが事実であれば、手書きのメモや帳票を扱う可能性がある`LedgerLeap`にとって重要な判断材料となる。
-   **調査方法:** 当初、`Tesseract`の実行環境を別途構築する計画だったが、プロジェクトに既に`ocrmypdf`（内部でTesseractを利用）が組み込まれているため方針を変更。Web上の既存の性能比較調査と、`ocrmypdf`の挙動をTesseractの性能とみなし、机上調査で比較を行う。
-   **調査結果:**
    -   Web上の複数の性能比較記事やベンチマークにおいて、**PaddleOCRがTesseractよりも手書き文字認識精度で優れている**ことが一貫して示されている。
    -   PaddleOCRは深層学習ベースのアルゴリズム、特にアジア言語の手書き文字への最適化が進んでいる。
    -   Tesseractは元々、印刷された活字の認識を主目的としており、手書き文字、特に崩れた文字の認識精度は低いとされる。精度を向上させるには、追加の画像前処理やカスタム学習が必要となるケースが多い。
-   **結論:**
    -   **PaddleOCRの方が手書き日本語認識性能において優位性が高い。**
    -   LedgerLeapのように、手書きのメモや帳票を含む不特定のドキュメントを扱うシステムにおいては、Tesseract(ocrmypdf)よりもPaddleOCRを選択する方が、高いテキスト化精度を期待できる。

### 2.3. Markerの手書き文字認識能力の検証

-   **背景:** PoCのログでは`hand_writing_01.png`の処理に成功しているが、これが「手書き文字をテキストとして認識した」結果なのかは不明。`Marker`が手書きに非対応の場合、その役割は活字文書の構造化に限定される。
-   **調査方法:** PoCの成果物である`storage/test/vlm-poc/results/hand_writing_01_result.md`を分析し、`Marker`が内部で利用しているOCRエンジンを特定。そのエンジンの手書き対応状況をWebで調査する。
-   **調査結果:**
    -   成果物のJSONから、`Marker`が内部でテキスト抽出（OCR）に使用しているのは **`Surya`** というライブラリであることが判明した。 (`"text_extraction_method": "surya"`)
    -   `Surya`の公式ドキュメントや関連情報を調査した結果、このライブラリは**主に印刷されたテキストのOCRのために設計**されており、手書き文字認識には最適化されていないことが確認された。
    -   ベンチマークにおいても、`Surya`の手書き文字認識精度は、印刷されたテキストに対する精度よりも低い値となっている。
-   **結論:**
    -   **Markerは手書き文字認識を主目的としていない。**
    -   `Marker`の強みは、あくまで**印刷されたドキュメント**のレイアウトを解析し、Markdownとして構造化する点にある。手書き文字を高精度にテキスト化する目的には適していない。

## 4. VLMモデル切り替え構成の検討

### 4.1. 目的と要件

-   **目的:** 将来的に新しいVLMモデル（例: `Nougat`, `LayoutLM`など）が登場した際に、迅速な検証と比較を行える柔軟な開発環境を構築する。
-   **要件:**
    -   開発者は、簡単な操作（設定ファイルの変更やコマンドの引数指定など）で、`Marker`, `PaddleOCR`, その他のVLMモデルを切り替えてSail環境を起動できること。
    -   Laravelアプリケーション側は、呼び出すVLMのエンドポイントを意識せず、常に単一のエンドポイント（例: `http://vlm:8000`）にリクエストを送信できることが望ましい。
    -   各VLMモデルの依存関係はコンテナ単位で完全に分離されていること。

### 4.2. 検討方案

#### 方案A: Docker Compose Override を利用する方式

-   **概要:** ベースとなる`docker-compose.yml`に加え、モデルごとの設定を記述した`docker-compose.marker.yml`や`docker-compose.paddle.yml`のようなオーバーライドファイルを用意する。
-   **利用方法:**
    ```bash
    # Markerを起動
    vendor/bin/sail up -d --build

    # PaddleOCRを起動 (オーバーライドファイルを指定)
    docker-compose -f docker-compose.yml -f docker-compose.paddle.yml up -d --build
    ```
-   **`docker-compose.yml` (ベース):**
    ```yaml
    services:
      # Laravel, MySQL, Redisなど基本サービス
      vlm:
        build:
          context: ./docker/marker # デフォルトはMarker
        ports:
          - "8001:8000"
        # ... その他設定
    ```
-   **`docker-compose.paddle.yml` (オーバーライド):**
    ```yaml
    services:
      vlm:
        build:
          context: ./docker/paddle # ビルドコンテキストを上書き
        # 必要に応じてポートなども上書き
    ```
-   **メリット:**
    -   Docker Composeの公式機能であり、標準的で理解しやすい。
    -   設定の差分がファイル単位で明確になる。
-   **デメリット:**
    -   起動時のコマンドが長くなる。Makefileやシェルスクリプトでのラップが推奨される。
    -   サービス名を共通（例: `vlm`）にするため、`docker-compose.yml`にデフォルトのvlmサービスを定義する必要がある。

#### 方案B: 環境変数 (`.env`) を利用する方式

-   **概要:** `.env`ファイルに`VLM_SERVICE_CONTEXT=./docker/marker`のような変数を定義し、`docker-compose.yml`でそれを参照する。
-   **利用方法:**
    1.  `.env`ファイルを編集: `VLM_SERVICE_CONTEXT=./docker/paddle`
    2.  通常通り`sail up`を実行: `vendor/bin/sail up -d --build`
-   **`.env`:**
    ```dotenv
    # 利用したいVLMのDockerコンテキストパスを指定
    VLM_SERVICE_CONTEXT=./docker/marker
    ```
-   **`docker-compose.yml`:**
    ```yaml
    services:
      vlm:
        build:
          context: ${VLM_SERVICE_CONTEXT:-./docker/marker} # 環境変数を参照、未設定ならmarkerをデフォルトに
        ports:
          - "8001:8000"
        # ...
    ```
-   **メリット:**
    -   開発者は`.env`ファイルを変更するだけで済み、操作が非常に簡単。
    -   起動コマンドは常に`vendor/bin/sail up`で統一される。
-   **デメリット:**
    -   どのモデルが利用可能か、`docker-compose.yml`の記述だけでは分かりにくい場合がある。
    -   環境変数の管理が煩雑になる可能性がある。

### 4.3. 推奨アプローチ

現段階では、開発者の利便性と設定のシンプルさから**方案B: 環境変数 (`.env`) を利用する方式**を第一候補として推奨する。

`.env.example`に`VLM_SERVICE_CONTEXT`の項目とコメントを追記しておくことで、新しい開発者でも容易に設定を理解できる。

**次のステップ:**
1.  方案Bをベースに、`docker-compose.yml`と`.env.example`を修正し、実際に`Marker`と`PaddleOCR`の切り替えが可能か技術検証を行う。
2.  検証結果を元に、開発手順書を更新する。

## 5. 総合結論と最終推奨

### 5.1. 各ツールの評価まとめ

| ツール | 構造化抽出 | 手書き認識精度 | 得意なこと | 不得意なこと |
| :--- | :--- | :--- | :--- | :--- |
| **PaddleOCR** | ⭕ (最新版) | 🥇 **高い** | 手書き文字を含む、多様なドキュメントの高精度なテキスト化と構造解析 | - |
| **Marker** | 🥇 **高い** (Markdown) | 🥉 **低い** | 印刷されたドキュメントのレイアウトを維持したMarkdownへの変換 | 手書き文字の認識 |
| **Tesseract** | ❌ (非対応) | 🥉 **低い** | 印刷された活字のテキスト化 | 手書き文字の認識、複雑なレイアウト解析 |

### 5.2. 最終推奨

LedgerLeapのVLM機能の要件である「多様なドキュメント（手書きを含む可能性）から構造化された情報を抽出する」という点を踏まえると、以下のハイブリッド戦略を推奨します。

1.  **デフォルトのOCRエンジンとして `PaddleOCR` を採用する。**
    -   **理由:** 手書き文字認識性能が高く、構造化抽出にも対応しているため、最も広範囲のドキュメントに対応できる。
    -   **アクション:** [実装ガイド](./2025-10-26_paddleocr-latest-impl-guide.md)に基づき、`paddle`コンテナを更新し、デフォルトのVLMサービスとして設定する。

2.  **補助的なツールとして `Marker` を維持・活用する。**
    -   **理由:** 印刷されたドキュメントを、より可読性の高いMarkdown形式で出力したい場合に依然として非常に強力である。
    -   **アクション:** VLMモデル切り替え構成を実装し、開発者が必要に応じて`Marker`に切り替えて利用できる環境を整備する。

この方針により、各ツールの長所を活かし、幅広いニーズに対応できる柔軟なVLM基盤を構築します。

## 6. VLM実装の試行錯誤の記録 (2025-10-26)

上記5.2の推奨アクションに基づき、`paddle`コンテナの更新作業に着手したが、深刻な依存関係の問題に直面し、解決に長時間を要した。これは将来の技術選定における重要な教訓となるため、以下に経緯を記録する。

### 6.1. 発生した主なエラー

当初、`paddleocr==2.7.3` とその関連ライブラリのバージョンを固定することで安定動作を目指したが、以下のエラーが連鎖的に発生した。

1.  **`ImportError: cannot import name 'sorted_layout_boxes'`**: `paddleocr`のパッケージ構造が想定と異なり、インポートに失敗。
2.  **`NameError: name 'CCompiler' is not defined`**: `paddlex`が依存する古い`scikit-learn`が、さらに古い`numpy`のソースビルドを試み、失敗。
3.  **`ImportError: numpy.core.multiarray failed to import`**: 最終的に、`paddle`系のライブラリが要求する古いNumPy 1.x系と、他のライブラリがインストールしたNumPy 2.x系の間でABI（Application Binary Interface）の非互換性問題が発生。

### 6.2. 判明した根本原因と教訓

-   **依存関係の複雑性:** `PaddleOCR`、特に構造化抽出機能は、`paddlepaddle`, `paddlex`, `numpy`, `opencv`, `safetensors`など多数のライブラリに密接に依存している。これらのライブラリ間のバージョン互換性は非常に繊細であり、`pip`による依存関係解決がしばしば失敗する。
-   **`PaddleOCRVL`へのアーキテクチャ転換:** デバッグの過程でユーザーより提供された最新の公式ドキュメントにより、`PaddleOCR`は`PaddleOCRVL`という新しい統合パイプラインに移行していることが判明した。これは、我々が試みていたアプローチが既に古いものであることを示唆していた。
-   **公式イメージの限界:** `PaddleOCRVL`用の公式Dockerイメージも試したが、これは推論サーバーとして動作することが前提であり、ライブラリとして直接インポートする用途には適していなかった。

### 6.3. 結論と方針の再修正

クリーンな`python:slim`イメージから`pip install`で依存関係を構築するアプローチは、`PaddleOCR`の複雑なエコシステムにおいては非常に不安定であり、多大な時間を浪費するリスクがある。

この経験に基づき、実装アプローチを再度見直し、**更新された実装ガイド**に基づいて再挑戦することが最も合理的であると結論付けた。

### 8.4. 調査結果のまとめ

本調査（Phase 0追加調査）の3つの疑問点に対する最終回答:

#### 1. PaddleOCRの将来性
**結論:** ⚠️ **PaddleOCRVLは現時点では使用困難**
- 最新版にsafetensors互換性問題あり
