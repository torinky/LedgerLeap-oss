## 🐛 発見した課題と解決

### 1. データベーススキーマの相違
- **問題**: Seeder内で使用したカラム名が実際のテーブル構造と異なる
- **解決**:
  - `organizations.name` は存在せず、`organizations.title` を使用すべき → 実際は`name`が正しい
  - `organizations.description` が存在しない → `org_id`をUUIDで設定
  - `folders.name` → `folders.title` に修正
  - `tags.description` が存在しない → 削除
  - `ledger_defines.description` → `create_description` に修正
  - `ledger_defines.column_define` が必須 → 空配列で初期化
  - `role_folder_permissions.permission_type` → `permission` に修正
  - `ledgers.folder_id` が存在しない → 削除

### 2. ColumnDefineコンストラクタ引数順序
- **問題**: 引数の順序が間違っていた
- **正しい順序**: `(id, name, typeIdentifier, order, options, required, ...)`
- **解決**: 全てのColumnDefine生成コードを修正

### 3. InputType識別子
- **問題**: `'ymd'` および `'checkbox'` が無効
- **正解**: `'YMD'` (大文字) および `'chk'`
- **解決**: Seederファイル内で一括置換

### 4. タグとledger_define_idの関係
- **問題**: タグは`ledger_define_id`が必須で、台帳定義に紐づく
- **解決**: 台帳定義作成後にタグを作成・付与する順序に変更

### 5. ⭐ Mroongaのベクターカラム処理による二重JSON化（重要）
- **問題**: シーダーで作成された経費申請の台帳のみcontentが空（null）になる
- **原因**: Mroongaのベクターカラム前処理において、数値キーのJSON配列内に整数値がある場合、その配列をさらにJSON配列としてエンコードする副作用
  ```
  整数を含む配列: ["EXP-0001", "2025-10-11", "交通費", 1000, "説明", []]
  → Mroongaの処理により二重配列化: ["[\"EXP-00","01\",...]"]
  → Eloquentでの取得時にJSON decodeエラーでnullに
  ```
- **解決**: `AsColumnArrayJson::setContent()`メソッドで整数・浮動小数点数を自動的に文字列に変換
  ```php
  // Mroongaのベクターカラム処理の副作用を回避
  if (is_int($item) || is_float($item)) {
      return (string) $item;
  }
  ```
- **メリット**:
  - シーダーやテストで個別対応が不要
  - 一箇所で一元管理され、メンテナンスが容易
  - UIとの整合性（フォーム入力は自動的に文字列）
  - 将来の問題防止（開発者がMroongaの副作用を意識する必要なし）
- **ドキュメント**: `AsColumnArrayJson`クラスに詳細なコメントを追加

## 💡 学んだ教訓

1. **事前のスキーマ確認が重要**
   - Seeder作成前に全テーブルのカラム構造を確認すべき
   - `tinker`での確認コマンド: `\Schema::getColumnListing('table_name')`

2. **モデルのコンストラクタ仕様の確認**
   - 特にValue Objectのような複雑なコンストラクタは事前調査が必須

3. **InputType識別子の確認**
   - `InputTypeFactory::$typeMap` を事前に確認すべき
   - 大文字小文字の区別に注意

4. **タグの設計理解**
   - タグは台帳定義に紐づく設計であることを理解
   - 横断検索用途とレコード個別タグの使い分け

5. **段階的な実装とテスト**
   - 大規模Seederは段階的に作成し、各Stepでテストすべき
   - エラー修正サイクルが長くなりすぎないように

6. **⭐ Mroongaの特性理解（最重要）**
   - Mroongaはベクターカラム前処理で整数を含むJSON配列を二重配列化する
   - この問題は`AsColumnArrayJson`レベルで対処することで、全シーダー・テストで解決
   - 一箇所での対応により、開発者が個別に対応する必要がなくなる
   - ドキュメント化により、将来の開発者も同じ問題に遭遇しない

7. **手動テストとログの重要性**
   - UIからの手動登録と比較することで問題の本質が見えた
   - 詳細なログ出力により、問題箇所を特定できた
   - データ型（整数 vs 文字列）の違いが原因と判明
