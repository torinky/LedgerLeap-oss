# 新マルチテナント実装計画書 (最終版)

**日付:** 2025年8月30日
**作成者:** Gemini

## 1. はじめに

### 1.1. 背景

本ドキュメントは、LedgerLeapプロジェクトにマルチテナント機能を導入するための最終的な実装計画を定義するものである。

当初、既存システムからの移行要件として、大規模なデータ量とテナント数を考慮し、シングルDB構成における以下の懸念が提示された。

*   **IDの枯渇:** 将来的な主キーの枯渇リスク。
*   **パフォーマンス低下:** 大規模テナントが他のテナントに影響を及ぼす「ノイジーネイバー問題」。
*   **リソースの集中:** 単一DBサーバーへの負荷集中。
*   **コンプライアンス要件:** 将来的にデータの物理的な分離が求められる可能性。

これらの課題を解決するため、アーキテクチャの検討を重ねてきた。

### 1.2. 目的

本計画の目的は、**安定性と将来の拡張性を両立するマルチテナント・アーキテクチャを確立し、段階的かつ安全に実装を完了させること**である。具体的には、まず堅牢な論理データ分離の基盤を構築し、将来的に必要となった場合にのみ、物理的なデータ分離へ安全に移行できる道筋を確保する。

## 2. アーキテクチャ選定の経緯と最終決定

最適なアーキテクチャを決定する過程で、複数のアプローチを検討・試行し、重要な教訓を得た。

### 2.1. 過去の試みと教訓

*   **試み1: 最初からのマルチDB化（失敗）**
    *   **アプローチ:** `stancl/tenancy` を利用し、開発初期から全テナントを物理的に異なるDBに分離しようと試みた。
    *   **結果:** 「中央DBの`users`」と「テナントDBのデータ」を繋ぐリレーションの問題に直面。解決策として`stancl/tenancy`の**「リソース同期」機能**に依存したが、技術的複雑性から外部キー制約エラーなどが頻発し、実装が頓挫した。
    *   **教訓:** アプリケーションのビジネスロジックが固まっていない段階で、インフラの複雑な問題（DB分散）に同時に取り組むのはリスクが極めて高い。

*   **試み2: テーブルプレフィックス方式（破棄）**
    *   **アプローチ:** `TenancyServiceProvider` をカスタマイズし、`TenancyInitialized` イベントで `config(['database.connections.mysql.prefix' => "tenant_{$tenant->getTenantKey()}_"])` を実行することで、リクエスト中に動的にテーブルプレフィックスを適用するロジックを実装した。
    *   **結果:** ルーティングの競合という深刻な副作用に直面した。具体的には、`makeTenancyMiddlewareHighestPriority()` メソッドが意図せず `InitializeTenancyByPath` ミドルウェアを**グローバル**に適用してしまい、テナントに属さない中央ルートへのアクセス時にテナントが見つからず、一貫性のないルーティングエラー（404, 500）が頻発した。
    *   **結論:** この複雑なデバッグ作業を経て、我々のアプローチが `stancl/tenancy` の想定する標準的な利用方法から外れた**非公式なハック**であると結論付けた。公式ドキュメントの再調査により、この方法が将来のバージョンアップで破綻するリスクが極めて高いと判断し、破棄した。
    *   **教訓:** ライブラリが公式にサポートする方法から逸脱した実装は、短期的な解決策に見えても、長期的には技術的負債となる。

### 2.2. 調査と再検討によるブレークスルー

過去の失敗を踏まえ、`stancl/tenancy`の公式ドキュメントとコミュニティを徹底的に再調査した結果、以下の重要な発見があった。

1.  **公式のシングルDB戦略:** `stancl/tenancy`が推奨するシングルDB戦略は、テーブルプレフィックスではなく、**`tenant_id`カラムと`BelongsToTenant`トレイトによる論理分離**であることが判明した。これは、クエリレベルでテナントのスコープを自動的に適用する、堅牢で安定した方法である。

2.  **ハイブリッド構成の可能性:** `stancl/tenancy`は、テナントごとにDB接続を動的に切り替えるBootstrapperのカスタマイズを許容している。これにより、**「一部のテナントは中央DB、特定のテナントは専用DB」というハイブリッド構成が可能**であり、シングルDBからマルチDBへの段階的な移行が実現できることがわかった。

### 2.3. 最終アーキテクチャ決定: 「段階的移行」アプローチ

以上の経緯と発見に基づき、本プロジェクトのマルチテナント・アーキテクチャとして、以下の段階的移行アプローチを正式に採択する。

*   **フェーズ1（今回実装）:** まずは公式に推奨されている**「シングルDB・`tenant_id`方式」**を採用し、安定的で堅牢な論理分離基盤を構築する。
*   **フェーズ2（将来の拡張）:** パフォーマンスやコンプライアンス要件が顕在化した際に、特定のテナントから順次**「物理DB分離」**へ移行する。

このアプローチは、過去の失敗要因（リソース同期、非公式なハック）を完全に排除し、かつ当初の懸念事項（ID枯渇、パフォーマンス等）を将来的に解決する、最も合理的で安全な最適解である。

### ステップ3: マイグレーション修正とモデル対応

*   **目的:** 全てのテナント用データ構造を `tenant_id` 方式に対応させ、将来の拡張に備える。
*   **タスク:**
    1.  **テナントテーブルの特定:** 以下をテナントに属するテーブルとして定義する。
        *   `folders`, `ledger_defines`, `ledgers`, `ledger_diffs`, `attached_files`, `tags`, `taggables`, `role_folder_permissions`, `activity_log`
    2.  **マイグレーションファイルの修正:** 上記テーブルの既存のマイグレーションファイルを修正し、`tenant_id` カラムと外部キー制約を追加する。
        ```php
        // 例: ledgersテーブルのマイグレーション
        $table->string('tenant_id');
        $table->foreign('tenant_id')->references('id')->on('tenants')->onUpdate('cascade')->onDelete('cascade');
        ```
    3.  **モデルへのトレイト適用:** 上記テーブルに対応する全てのEloquentモデルに `use Stancl\Tenancy\Database\Concerns\BelongsToTenant;` を追加する。
    4.  **【強く推奨】主キーのUUID化:** 将来のID枯渇・衝突リスクを完全に排除するため、`ledgers` など主要なテナントテーブルの主キーをUUIDに変更する新しいマイグレーションを作成する。
        ```php
        // 新しいマイグレーションファイルを作成
        Schema::table('ledgers', function (Blueprint $table) {
            // 既存の外部キー制約を一時的に削除
            // ...
            $table->dropPrimary();
            $table->renameColumn('id', 'old_id');
            $table->uuid('id')->primary()->first();
            // データの移行ロジック
            // ...
            // 外部キー制約の再設定
        });
        ```
        *注意: 既存データの移行には注意が必要です。まずは`ledgers`など主要なものから着手することを推奨します。*
*   **検証方法:**
    *   `vendor/bin/sail artisan migrate:fresh --seed` がエラーなく完了すること。
    *   データベースクライアントで各テーブルのスキーマを確認し、`tenant_id` カラムとUUIDの主キーが正しく設定されていることを確認する。
        *   **結果:** `2025_08_30_063606_add_tenant_id_to_tenant_tables.php` マイグレーションにより、対象の全テーブルに `tenant_id` カラムが追加済み。また、対応する各モデルには `BelongsToTenant` トレイトが適用済み。これにより、論理的なデータ分離の基盤が確立された。
        *   **主キーのUUID化については、現状未着手**であり、`ledgers` テーブルの主キーは `bigIncrements` のままであることを確認。
    *   **完了:** ステップ3の検証が完了し、`tenant_id` 方式の基盤が構築されました。

## 4. 将来の展望: フェーズ2 - 物理分離への道

本計画（フェーズ1）が完了し、アプリケーションが安定稼働した後、将来的に特定のテナントの物理分離が必要になった場合は、以下の手順で安全に移行が可能です。

1.  **移行準備:** `tenants`テーブルにDB接続情報を格納するカラムを追加し、データ移行用のArtisanコマンドと、DB接続を動的に切り替えるカスタムBootstrapperを実装する。
2.  **移行実行:** 対象テナントのデータを新DBに移行し、`tenants`テーブルの接続情報を更新する。
3.  **運用:** `stancl/tenancy`がリクエストに応じて自動でDB接続を切り替え、論理分離と物理分離が共存するハイブリッド構成での運用が開始される。

この拡張パスにより、本計画が将来のビジネス要件の変化にも柔軟に対応できる、持続可能なアーキテクチャであることが保証されます。
