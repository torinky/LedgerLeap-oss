## 📋 Issue概要

### 要件

1. **複数種類の台帳を表示したとき、それぞれの基準列で並ぶようにする**
   - 現在は複数台帳表示時に全体で単一のソート設定しか適用できない
   - 各台帳定義ごとに設定された基準列（デフォルトソート）で並べたい

2. **基準列が分かるようにする**
   - どの列が基準列（ソート優先列）なのかを視覚的に明示する
   - ユーザーが現在のソート状態を理解しやすくする

## ❌ 課題の詳細分析

### 課題1: アーキテクチャの制約

**現状:**
- 全レコードに対して単一の`$orderBy`でソートを適用
- 台帳定義AとBが混在している場合:
  - Aの基準列（例: 主番）とBの基準列（例: 商品コード）は異なるカラムID
  - 単一の`content->{column_id}`では両方に対応できない

**必要な変更:**
- 台帳定義ごとに**個別のソート設定**を適用する仕組み
- 各台帳グループ内でのみ、その台帳のデフォルトソートを適用

### 課題2: パフォーマンスとクエリ構築

**現在の実装 (`RecordsTable.php` 330-646行目):**
- データベースクエリでソート（`orderByRaw()`など）
- ページネーション後にコレクション操作

**検討が必要な点:**
1. **データベースレベルでのソート vs コレクションレベルでのソート**
   - DB: 単一のORDER BY句では複数台帳の異なる基準列に対応できない
   - Collection: ページネーション後に各グループ内でソートする方式？

2. **ページネーションとの整合性**
   - 現在のページネーションは全体に対して実行
   - 台帳ごとにソートした場合、ページネーションの挙動をどうするか

## 💡 技術的検討ポイント

### 検討1: ソート処理の実装方法

#### オプションA: コレクションレベルでのグループ別ソート

**フロー:**
```
1. データベースから全レコードを取得（ソートなしorデフォルトソート）
2. ページネーション実行
3. 取得したコレクションを台帳定義IDでグループ化
4. 各グループ内で、その台帳のdefaultSortColumnsを適用してソート
5. 表示
```

**メリット:**
- 実装が比較的シンプル
- 既存のページネーション機構を活用可能

**デメリット:**
- ページネーション後のソートのため、全体の順序は保証されない
- 各台帳グループ内でのソート順のみ保証

##### パフォーマンス分析（実規模ベース）

**システム想定規模（`docs/README.md`より）:**
- 台帳レコード総数: **数百万件規模**
- ユーザー: 数人～数千人
- 台帳定義: 数千種類

**オプションAの処理フロー詳細:**

1. **データベースクエリ（既存）**
   - WHERE句でフィルタリング（tenant_id、folder_id、検索条件など）
   - 既存のORDER BY（composite_score、created_atなど）でソート
   - **DBレベルでページネーション実行（LIMIT 50 OFFSET X）**
   - → 結果: 50件のレコード（最悪ケース: すべて異なる台帳定義）

2. **PHP側での処理**
   - 50件を台帳定義IDでグループ化（`groupBy('ledger_define_id')`）
   - 各グループに対してPHP `Collection::sortBy()` を複数回適用
   - 例: 5つの異なる台帳定義 → 5回のソート処理
     - グループA: 10件をソート
     - グループB: 15件をソート
     - グループC: 8件をソート
     - グループD: 12件をソート
     - グループE: 5件をソート

**パフォーマンス試算:**

| 項目 | 最悪ケース | 標準的ケース | 備考 |
|------|-----------|-------------|------|
| ページサイズ | 50件 | 50件 | デフォルト設定 |
| 台帳定義数/ページ | 50種類 | 3-5種類 | 最悪=全件異なる定義 |
| PHP側ソート回数 | 50回 | 3-5回 | グループ数に依存 |
| 最大グループサイズ | 50件 | 10-20件 | 1グループあたり |
| PHP Collection::sortBy 処理時間（50件） | ~0.1ms | ~0.05ms | ベンチマーク想定 |
| **合計PHP処理時間** | **~5ms** | **~0.15-0.25ms** | 50回 × 0.1ms or 5回 × 0.05ms |

**DB処理時間との比較:**
- DBクエリ実行時間（インデックス最適化済み）: 10-100ms（数百万件から50件取得）
- PHP側追加処理: 0.15-5ms
- **比率: PHP処理はDBクエリの1-10%程度**

**結論:**
- ✅ **最悪ケースでもPHP側の追加コストは5ms程度で許容範囲**
- ✅ **標準的なケース（3-5種類の台帳定義混在）では0.2ms程度で無視できるレベル**
- ✅ **ボトルネックはDB クエリであり、PHP側ソートの影響は軽微**
- ⚠️ **ただし、ページ内の台帳定義数が極端に多い場合（20種類以上）は要注意**

**スケーラビリティ評価:**

| データ規模 | 評価 | 理由 |
|----------|------|------|
| 1万件未満 | ⭐⭐⭐⭐⭐ | 全く問題なし |
| 1万～10万件 | ⭐⭐⭐⭐⭐ | ページネーション後の処理のため影響なし |
| 10万～100万件 | ⭐⭐⭐⭐☆ | DB最適化が重要だがPHP側は問題なし |
| **100万件以上** | ⭐⭐⭐☆☆ | DB負荷増。オプションD検討推奨 |

**パフォーマンス最適化の余地:**

1. **Collection::sortBy の最適化**
   - 必要に応じてネイティブ `usort()` に置き換え可能
   - 期待効果: 処理時間を30-50%削減

2. **グループ化とソートの一体化**
   - 現在: グループ化 → 各グループソート
   - 改善: グループ化とソート同時実行
   - 期待効果: 処理時間を10-20%削減

3. **キャッシュ活用**
   - `defaultSortColumns`の取得結果をキャッシュ
   - 期待効果: 台帳定義アクセスの削減

**推奨事項:**
- ✅ Phase 1ではオプションAで実装開始
- ✅ パフォーマンスメトリクスを計測（Laravel Telescope等）
- ✅ 10万件を超えたタイミングで実測評価
- ⚠️ 100万件到達前にPhase 2（オプションD）移行を検討


#### オプションB: 複雑なクエリ構築

**フロー:**
```
1. 各台帳定義のdefaultSortColumnsを取得
2. 台帳定義ごとにサブクエリを構築（CASE文やUNIONを使用）
3. 統合されたクエリでデータベースレベルでソート
4. ページネーション実行
5. 表示
```

**メリット:**
- データベースレベルで正確なソート
- パフォーマンスが良い可能性

**デメリット:**
- クエリが非常に複雑になる
- 保守性が低下

#### オプションC: 台帳定義ごとに個別取得

**フロー:**

```text
1. 選択された各台帳定義に対して個別のクエリを実行
2. それぞれのdefaultSortColumnsでソート
3. 各結果をマージ
4. ページネーション（全体 or 台帳ごと？）
```

**メリット:**
- 各台帳で明確にソートが適用される
- ロジックが分かりやすい

**デメリット:**
- 複数クエリによるパフォーマンス懸念
- ページネーションの扱いが難しい

#### オプションD: 非正規化カラム方式（提案）

**概要:**
- `ledgers`テーブルに非表示のソート用カラム（`default_sort_value`など）を追加
- `sort_index`で設定された優先度順にカラムの値を詰め込んで保存
- 基準列ソートを選択した場合はそのカラムで単純に`ORDER BY`
- カラムの更新はキューで非同期処理

**フロー:**

```text
1. [事前準備] マイグレーション: ledgersテーブルにdefault_sort_value TEXT追加
2. [保存時] Ledger保存・更新時にObserverまたはJobで以下を実行:
   a. LedgerDefineからsort_indexが設定されたカラムを取得
   b. 優先順位順にcontentから値を取得し、区切り文字で連結
      例: "001|A-123-456|2025-12-01" (主番|副番|日付)
   c. default_sort_valueカラムに保存
3. [表示時] RecordsTableで:
   - orderBy === 'default' の場合、ORDER BY default_sort_value ASC
   - 通常のページネーション実行
4. [カラム定義変更時] 全レコードのdefault_sort_valueを再計算（Job）
```

**詳細設計:**

**1. カラム設計:**
```sql
ALTER TABLE ledgers ADD COLUMN default_sort_value TEXT NULL;
CREATE INDEX idx_ledgers_default_sort ON ledgers(ledger_define_id, default_sort_value);
```

**2. 値の生成ロジック:**
```php
// 疑似コード
public function generateDefaultSortValue(): string
{
    $ledgerDefine = $this->ledgerDefine;
    $sortColumns = collect($ledgerDefine->column_define)
        ->whereNotNull('sort_index')
        ->sortBy('sort_index');
    
    $values = [];
    foreach ($sortColumns as $column) {
        $value = $this->content[$column->id] ?? '';
        
        // 型に応じて正規化
        $normalizedValue = match ($column->type) {
            'number', 'auto_number' => str_pad($value, 20, '0', STR_PAD_LEFT), // 数値は0埋め
            'date', 'YMD' => $value, // 日付はそのまま（YYYY-MM-DD形式想定）
            default => $value, // 文字列はそのまま
        };
        
        $values[] = $normalizedValue;
    }
    
    return implode('|', $values); // パイプ区切りで連結
}
```

**3. Observer/Job:**
```php
// app/Observers/LedgerObserver.php
public function saved(Ledger $ledger): void
{
    // 非同期でソート値を更新
    dispatch(new UpdateLedgerDefaultSortValue($ledger->id));
}

// app/Jobs/UpdateLedgerDefaultSortValue.php
public function handle(): void
{
    $ledger = Ledger::find($this->ledgerId);
    $ledger->default_sort_value = $ledger->generateDefaultSortValue();
    $ledger->saveQuietly(); // イベント発火を防ぐ
}
```

**4. クエリ:**
```php
// RecordsTable.php
if ($this->orderBy === 'default') {
    $query->orderBy('default_sort_value', 'asc');
}
```

**メリット:**
- ✅ **データベースレベルでの高速なソート**（インデックス利用可能）
- ✅ **既存のページネーション機構をそのまま利用可能**
- ✅ **クエリが非常にシンプル**（単純なORDER BYで済む）
- ✅ **複数台帳混在時でも一貫したソート順を保証**
- ✅ **パフォーマンスが最も良い**（特に大量データ時）

**デメリット:**
- ❌ **データの非正規化によるストレージ増加**（レコード数 × 数百バイト程度）
- ❌ **データ整合性の維持が必要**
  - カラム定義変更時に全レコードの再計算が必要
  - LedgerDefineのsort_index更新時の対応
- ❌ **キュー処理による遅延**
  - 保存直後は新しいソート値が反映されていない可能性
  - キューワーカーが停止していると更新されない
- ❌ **実装の複雑さ**
  - Observer、Job、マイグレーション、バッチ再計算処理
  - カラム定義変更時の連携処理
- ❌ **デバッグの難しさ**
  - 非表示カラムの中身が見えにくい
  - 期待値と実際のソート値のズレの調査が困難

**リスク:**
1. **値の正規化の正確性**
   - 数値の0埋め桁数が不足すると正しくソートされない
   - 日付フォーマットの統一性
   - 文字列の大文字小文字の扱い
2. **カラム定義変更時のオーバーヘッド**
   - 数万件のレコードがある場合、再計算に時間がかかる
3. **キューの信頼性**
   - キューワーカーの監視とエラーハンドリング

### オプション選択のガイドライン

### 推奨案の変更について（2026-02-01追記）

当初は実装の容易さから**オプションA**を推奨していましたが、**「ページネーション時のソート順序の不整合」**という重大な欠陥が、大規模データ（数千〜数百万件）閲覧時のユーザー体験を大きく損なうことが判明しました。
特に、実務担当者が「過去の業務記録を日付順に遡って探す」ようなユースケースにおいて、ページ間で日付が前後することは許容されません。

したがって、**Phase 1から「オプションD: 非正規化カラム方式」を採用**することを強く推奨します。

#### オプションAの致命的な問題点
- ページネーションがDBレベル（`LIMIT/OFFSET`）で行われるため、PHP側でソートしても**「現在表示されている50件の中での並び替え」**にしかならない。
- 結果として、ページ1に「2024年のデータ」と「2025年のデータ」が混在し、ページ2にも同様に混在する事態が発生する。
- ユーザーは「日付順」と表示されているにも関わらず、ページをめくっても時系列が揃わないため、目的のレコードに到達できない。

#### オプションDの優位性
- ✅ **完全なグローバルソート:** DBインデックスを使用した正確なソートが可能。
- ✅ **高速なページネーション:** `ORDER BY default_sort_value LIMIT 50` は数百万件規模でも高速（インデックスが効くため）。
- ✅ **検索との両立:** 既存の検索条件と組み合わせてもパフォーマンスへの影響が限定的。

---
#### オプションBを選ぶべき場合
- ✅ SQLに精通したチームメンバーがいる
- ✅ 完全なソート順を保証する必要がある
- ✅ パフォーマンスが最重要
- ❌ **現時点では推奨しない**（保守性の問題）

#### オプションCを選ぶべき場合
- ✅ 台帳定義の数が非常に少ない（1-3個程度）
- ✅ レコード数が少ない（数百件以下）
- ❌ **現時点では推奨しない**（スケーラビリティの問題）

### 検討2: ページネーションの扱い

## 🎯 推奨実装アプローチ

### 全体戦略: 2段階アプローチ

**基本方針:**

4つのオプションの比較分析の結果、以下の2段階アプローチを推奨します:

1. **Phase 1（短期）: オプションA - コレクションレベルソートで迅速な価値提供**
   - 実装が簡単で、早期に機能提供が可能
   - 既存機能への影響が最小限
   - 各台帳グループ内でのソート順を保証

2. **Phase 2（中長期）: オプションD - 非正規化カラムで完全なソリューション**
   - 本番環境での運用実績とフィードバックを踏まえて判断
   - レコード数が増加し、パフォーマンスが問題になった場合に移行
   - 完全なソート順保証が必要と判明した場合に実装

**判断基準:**

| 状況 | 推奨オプション | 理由 |
|------|--------------|------
| 実装開始時（現在） | **オプションA** | 迅速な機能提供、リスク最小化 |
| レコード数 < 10,000件 | **オプションA** | 十分なパフォーマンス |
| レコード数 > 10,000件 | **オプションD検討** | パフォーマンス優位性 |
| 完全なソート順が必須 | **オプションD** | データベースレベルの保証 |
| 開発リソース限定 | **オプションA** | 実装コストが低い |

## ⚠️ リスクと対策（オプションD）

### 1. データ更新のオーバーヘッド
**懸念:** レコード保存時に文字列生成処理が走るため、書き込み性能が若干低下する。
**対策:** 生成ロジックは軽量（文字列結合のみ）であり、WEBサーバーのCPU負荷としては無視できるレベル。ブロッキング要因にはならない。

### 2. カラム定義変更時の再計算
**懸念:** 「基準列」の設定を変更した場合、過去の全レコードの再計算が必要。
**対策:**
- コマンドまたはジョブでバックグラウンド更新を行う。
- 更新中は「ソート順が一時的に不正確になる可能性があります」といったアナウンスを表示する運用カバーも検討。

### 3. ストレージ容量
**懸念:** `TEXT`カラム追加による容量増。
**対策:** 1レコードあたり数百バイト程度。100万件でも数百MB程度であり、現代のストレージコストでは許容範囲。
