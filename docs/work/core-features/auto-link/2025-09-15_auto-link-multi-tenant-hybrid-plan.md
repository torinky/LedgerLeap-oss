# 自動リンク機能 マルチテナント対応計画 (最終議論反映版)

**日付:** 2025年9月15日
**作成者:** Gemini
**ステータス:** 完了

**関連ドキュメント:**
*   [AutoLink機能概要](/docs/function/AutoLink.md)
*   [AutoLinkService概要](/docs/services/AutoLinkService.md)

## 1. 概要

本計画は、LedgerLeapの自動リンク機能（AutoLink）をマルチテナント環境に対応させるための実装計画である。

当初は、`AutoLink`定義をグローバルなリソースとしつつ、生成されるURLは閲覧者のテナントコンテキストに応じて動的に変更する案を検討した。しかし、議論を重ねる中で、その方式では「特定のテナントの台帳へ常にリンクしたい」という要件を満たせないことが判明した。

そのため、本計画では方針を大きく転換し、「**自動リンクの定義はグローバルに管理しつつ、リンク先は特定のテナントに固定できる**」ことを基本方針とする。

さらに、台帳の連番（auto_number）など、単一のテナントに紐付かない項目については、利便性を最大化するためのフォールバック機能として「**テナント横断検索および自動リダイレクト機能**」を実装する。これにより、システムはリンク先の候補を自動で判断し、ユーザーを最も適切なページへ誘導する。

## 2. 目的

*   `AutoLink` 定義をテナントに依存しないグローバルなリソースとして管理できるようにする。
*   内部リンク（台帳リンク）を生成する際に、**リンク先のテナントを明示的に指定・固定**できるようにする。
*   連番リンクやテナント未指定リンクの利便性を向上させ、ユーザーが目的の情報に最短でたどり着けるようにする。
*   管理画面（Filament）のUIを改善し、テナントを意識した直感的で間違いのない設定を可能にする。

## 3. ペルソナとユーザーシナリオ

### 3.1. ペルソナ

*   **佐藤さん（システム管理者）:**
    *   **背景:** LedgerLeap全体の運用を担当。複数のテナント（企業）がシステムを利用しており、それぞれのテナントに合わせた設定と、システム全体で共通化すべき設定を管理している。
    *   **関心事:** テナントごとに異なる業務ルール（製品コードやプロジェクトIDの体系など）を、自動リンク機能で効率的に吸収したい。設定ミスを防ぎ、メンテナンス性を高く保ちたい。

### 3.2. ユーザーシナリオ

*   **シナリオ1: テナント横断の共通ルール設定**
    *   佐藤さんは、全社共通で使われるRedmineのチケット番号（例: `#12345`）が、どのテナントの台帳に書かれても、常に単一のRedmineサーバーのURLにリンクされてほしいと考えている。

*   **シナリオ2: 特定テナントへのリンク設定**
    *   テナントA（A社）には「A社製品コード（`A-PROD-XXX`）」があり、テナントB（B社）には「B社仕様書ID（`B-SPEC-YYY`）」がある。
    *   佐藤さんは、「`A-PROD-XXX`」という文字列が台帳に現れた場合、**常にテナントAの**特定台帳へのリンクが生成されるように設定したい。このルールはテナントBのユーザーが閲覧している場合でも同様に動作する必要がある。
    *   同様に、「`B-SPEC-YYY`」は**常にテナントBの**台帳へリンクさせたい。

## 4. 機能要件

上記シナリオを実現するため、以下の機能要件を定義する。

1.  **`auto_links` テーブルの修正:**
    *   `tenant_id` カラムを `nullable` で追加する。このカラムは、リンク先のテナントを固定するために使用する。

2.  **`AutoLink` モデルの修正:**
    *   `tenant()` リレーション (`BelongsTo`) を定義する。

3.  **`AutoLinkService` の改修:**
    *   `createCustomLink()`: `auto_links.tenant_id` が指定されている場合、そのテナントの**パス形式URL**を生成する。
    *   `createAutoNumberLink()`: 後述の「横断検索ルート」へのURLを生成する。

4.  **Filament管理画面 (`AutoLinkResource`) のUI改善:**
    *   **リンク先テナントの選択:** 内部リンク形式の場合に、「リンク先テナント」を選択するドロップダウンを表示する。
    *   **フォルダ選択UIの改善:** ルートフォルダにテナント名を表示し、識別しやすくする。
    *   **プレビュー機能の改善:** 「リンク先テナント」の選択に応じて、正しいパス形式のプレビューURLを生成する。

5.  **連番・テナント未指定リンクの横断検索機能 (新規):**
    *   `GET /ledgers/lookup/{query}` というテナント非依存のグローバルルートを定義する。
    *   上記ルートにアクセスすると、コントローラーは全テナントから `{query}` に一致する台帳を検索する。
    *   検索結果が **1件** だった場合は、その台帳のテナント付きURL（パス形式）へ自動でリダイレクトする。
    *   検索結果が **複数件** あった場合は、テナント名を含む結果一覧ページを表示する。
    *   検索結果が **0件** だった場合は、「見つかりませんでした」というメッセージを表示する。

## 5. 実装計画 (フェーズ別)

### フェーズ1: 基本機能の実装 (完了済み)

*   **目的:** `AutoLink` にリンク先テナントを紐付けるための基本機能と、それに対応した管理画面のUIを実装する。
*   **実施済みタスク:**
    1.  **データベースとモデルの変更:**
        *   `auto_links` テーブルに `tenant_id` カラムと関連する制約を追加済み。
        *   `AutoLink` モデルに `tenant_id` を追加し、`tenant()` リレーションを定義済み。
    2.  **リンク生成ロジックの修正:**
        *   `AutoLinkService` の `createCustomLink()` メソッドを、`tenant_id` に基づいてリンクを生成するように修正済み。
    3.  **管理画面UIの改修:**
        *   `AutoLinkResource` に、「リンク先テナント」の選択機能、テナント名を考慮したフォルダ選択UI、およびそれらと連動するプレビュー機能を実装済み。

### フェーズ2: 横断検索・リダイレクト機能の実装 (今回対象)

*   **目的:** 連番リンク等のための、テナント横断検索と結果に応じた振り分け機能を実装する。
*   **タスク:**
    1.  **ルートとコントローラーの準備:**
        *   `routes/web.php` にグローバルルート `GET /ledgers/lookup/{query}` を追加し、`LedgerLookupController@searchAllTenants` を呼び出すように設定する。
        *   `LedgerLookupController` に、空のメソッド `searchAllTenants(string $query)` を作成する。
    2.  **横断検索ロジックの実装:**
        *   `searchAllTenants` メソッド内に、全テナントをループして台帳を検索し、結果件数に応じてリダイレクトまたはビューを返す振り分け処理を実装する。
    3.  **ビューの作成:**
        *   検索結果が複数件の場合に表示する、テナント名付きの結果一覧ビューを作成する。
    4.  **`AutoLinkService` との連携:**
        *   `AutoLinkService` の `createAutoNumberLink` メソッドを修正し、新しく作成したグローバル検索ルートを指すように変更する。
