# 添付ファイルサムネイルの非同期生成計画

## 1. 目的
台帳レコード登録時のユーザーエクスペリエンスを向上させるため、現在同期的に行われている添付ファイルのサムネイル生成処理を非同期化します。これにより、画像処理によるUIのブロックを解消し、アプリケーションの応答性を高めます。

## 2. 現状の課題
- **UIのブロック**: 現在、台帳レコードに画像を添付して保存する際、Livewireコンポーネント (`CreateColumn.php`, `ModifyColumn.php`) 内で `createThumbnail` メソッドが直接呼び出され、サムネイルが同期的に生成されています。この処理はHTTPリクエストのライフサイクル内で実行されるため、画像処理に時間がかかるとユーザーインターフェースがブロックされ、操作が遅延する可能性があります。
- **スケーラビリティの懸念**: 大量の画像ファイルがアップロードされる場合、同期処理ではサーバーの負荷が増大し、パフォーマンス問題を引き起こす可能性があります。

## 3. 提案する解決策
Laravelのキューシステムを活用し、サムネイル生成をバックグラウンドジョブとして実行します。これにより、ユーザーはファイルのアップロードと同時に操作を継続できるようになります。

## 4. 影響範囲
- `app/Livewire/Ledger/CreateColumn.php`
- `app/Livewire/Ledger/ModifyColumn.php`
- `app/Jobs/Ledger/GenerateThumbnail.php` (新規作成)
- `app/Models/AttachedFile.php`
- `app/Http/Controllers/AttachedFileDownloadController.php`
- `app/Services/Ledger/ColumnHtmlService.php`
- `app/Enums/AttachedFileStatus.php` (必要に応じてステータス追加)
- `resources/views/components/ledger/form/files.blade.php`
- `tests/` ディレクトリ内の関連テストファイル

## 5. 実装計画 (ステップ・バイ・ステップ)

### ステップ 1: サムネイル生成ジョブの作成 (完了)
- **目的**: サムネイル生成のロジックをカプセル化し、非同期で実行可能にする。サムネイル生成失敗時の再試行の基盤を構築する。
- **タスク**:
    1. `php artisan make:job Ledger/GenerateThumbnail` コマンドを実行し、`app/Jobs/Ledger/GenerateThumbnail.php` を作成します。
    2. `GenerateThumbnail` ジョブの `handle()` メソッド内に、既存の `createThumbnail` メソッド（`CreateColumn.php` や `ModifyColumn.php` から移動）のロジックを移植します。
    3. ジョブは `AttachedFile` モデルのIDを受け取り、そのIDに対応するファイルを処理するようにします。
    4. サムネイル生成に失敗した場合、`AttachedFile` モデルの `status` を `AttachedFileStatus::THUMBNAIL_FAILED` に更新し、詳細なエラーログを記録します。
    5. ジョブの再試行回数制限（例: `public $tries = 3;`）を設定し、無限ループを防ぎます。

### ステップ 2: `AttachedFile` モデルからのジョブディスパッチ (完了)
- **目的**: `AttachedFile` レコードが作成または更新された際に、自動的にサムネイル生成ジョブをキューに投入する。
- **タスク**:
    1. `app/Enums/AttachedFileStatus.php` に `THUMBNAIL_FAILED` ケースを追加します。
    2. `app/Models/AttachedFile.php` に `booted()` メソッドを追加（または既存のものを修正）します。
    3. `booted()` メソッド内で、`created` イベントリスナーを登録します。
    4. `created` イベント内で、`GenerateThumbnail` ジョブをディスパッチします。この際、`AttachedFile` モデルのIDをジョブに渡します。
    5. 必要に応じて、`AttachedFileStatus` Enumに `PENDING_THUMBNAIL_GENERATION` のような新しいステータスを追加し、ジョブディスパッチ時にこのステータスに更新するようにします。

### ステップ 3: 同期サムネイル生成ロジックの削除 (完了)
- **目的**: UIをブロックする同期処理を完全に排除する。
- **タスク**:
    1. `app/Livewire/Ledger/CreateColumn.php` および `app/Livewire/Ledger/ModifyColumn.php` から、`createThumbnail` メソッドの呼び出し、および関連するサムネイル生成ロジックを削除します。
    2. `createThumbnail` メソッド自体がこれらのLivewireコンポーネントにのみ存在する場合は、そのメソッドも削除します。

### ステップ 4: フロントエンドでのサムネイル表示の調整と再試行ロジック (進行中)
- **目的**: サムネイルがまだ生成されていない場合や生成に失敗した場合の表示を考慮し、ユーザーに適切なフィードバックを提供するとともに、表示の機会があれば再度サムネイル作成にチャレンジする。
- **タスク**:
    1. `app/Http/Controllers/AttachedFileDownloadController.php` の `download` メソッドを修正します。
        - `THUMBNAIL_FAILED` ステータスのファイルに対して、サムネイル表示を試みる際に `GenerateThumbnail` ジョブを再ディスパッチするロジックを追加します。ただし、ジョブの `attempts` 回数を確認し、最大試行回数を超えていない場合にのみ再ディスパッチするようにします。
    2. `app/Services/Ledger/ColumnHtmlService.php` の `getFileHtml()` メソッドを修正します。
        - `THUMBNAIL_FAILED` ステータスのファイルに対して、サムネイル表示を試みる際に `GenerateThumbnail` ジョブを再ディスパッチするロジックを追加します（`AttachedFileDownloadController` と同様の簡易的なチェックを含む）。
    3. `resources/views/components/ledger/form/files.blade.php` (FilePondコンポーネント) で、サムネイルが非同期でロードされること、および `THUMBNAIL_FAILED` ステータスの場合の表示（例: ローディングスピナー、デフォルトアイコン）を実装します。

### ステップ 5: エラーハンドリングと監視 (未着手)
- **目的**: 非同期処理の信頼性を確保し、問題発生時に迅速に対応できるようにする。
- **タスク**:
    1. `GenerateThumbnail` ジョブ内で、サムネイル生成に失敗した場合の例外処理を強化し、詳細なエラーログを記録するようにします。
    2. `AttachedFile` モデルの `status` カラムを適切に更新し、サムネイル生成の成功/失敗状態をデータベースで追跡できるようにします。
    3. Laravel Horizonなどのキュー監視ツールを導入し、ジョブの実行状況や失敗を監視できるようにします（もし未導入の場合）。

### ステップ 6: テストコードの更新 (進行中)
- **目的**: 非同期処理への変更に伴い、既存のテストを更新し、新しいジョブのテストを追加する。
- **タスク**:
    1. サムネイル生成をテストしていた既存のフィーチャーテストを修正し、ジョブがディスパッチされること、および `THUMBNAIL_FAILED` ステータスへの更新が正しく行われることをアサートするように変更します。
    2. `GenerateThumbnail` ジョブのユニットテストを新規作成し、ジョブが正しくサムネイルを生成すること、エラーケースを処理すること、および再試行ロジックが期待通りに動作することを確認します。

## 6. 期待される効果
- **ユーザーエクスペリエンスの向上**: 台帳レコードの保存が即座に完了し、ユーザーはサムネイル生成の完了を待つ必要がなくなります。
- **アプリケーションの応答性向上**: HTTPリクエストの処理が高速化され、サーバーの負荷が軽減されます。
- **スケーラビリティの向上**: 大量の画像アップロードにも対応しやすくなります。
- **堅牢性の向上**: サムネイル生成処理が独立したジョブとして実行されるため、メインアプリケーションの安定性に影響を与えにくくなります。
- **視覚的フィードバック**: サムネイル生成の失敗時にも適切なアイコンが表示され、ユーザーはファイルの状況を把握しやすくなります。

## 7. 考慮事項
- **キューワーカーの稼働**: 非同期処理が正しく機能するためには、キューワーカーが常に稼働している必要があります。
- **サムネイルの表示遅延**: サムネイルが生成されるまでにわずかな遅延が発生する可能性があります。この間、ユーザーにはローディングインジケーターやデフォルトのアイコンを表示するなど、適切なフィードバックを提供する必要があります。
- **既存ファイルのサムネイル**: 既存の台帳レコードに添付されているファイルでサムネイルが生成されていないものがある場合、それらに対する対応（例: 一括生成コマンドの作成）も検討する必要があります。
- **再試行の頻度と回数**: サムネイル生成の再試行ロジックは、サーバーへの過剰な負荷を避けるため、適切な頻度と回数に制限する必要があります。本計画ではジョブの `attempts` と簡易的なチェックを含みますが、必要に応じてより高度なレートリミットやバックオフ戦略を検討します。
