## 1. 目的

ユーザーがロールバックを実行する前に、対象バージョンと変更内容を再確認し、必要に応じてコメントを入力するためのインターフェースを定義する。

---

## 2. コンポーネント設計

### 2.1 `App\Livewire\Ledger\RollbackConfirmModal`

新規作成するLivewireコンポーネント。MaryUIの `<x-mary-modal>` を使用する。

#### プロパティ

| プロパティ名 | 型 | 説明 |
|-------------|----|------|
| `show` | `bool` | モーダルの表示状態 |
| `step` | `int` | 現在の確認ステップ (1: 理由入力, 2: 最終実行確認) |
| `ledgerId` | `int` | 対象台帳ID |
| `targetDiffId` | `int` | ロールバック先の差分ID |
| `comments` | `string` | ユーザーが入力するロールバック理由 |
| `confirmed` | `bool` | 最終確認チェックボックスの状態 |

#### イベントリスナー

- `ledger.rollback.open-modal(ledgerId, diffId)`:
  - 台帳IDと差分IDを受け取り、モーダルを初期化して表示する。

## 3. UI/UX 仕様

### 3.1 モーダル構成

1. **タイトル**: 「台帳を以前のバージョンに戻す」
2. **対象バージョン情報**:
   - 「Ver. {targetDiff->version} に戻します」と強調表示。
   - 更新者と更新日時を表示。
3. **警告メッセージ**:
   - 「この操作により、現在の内容は Ver. {ledger->version + 1} として新しく保存されます。」
4. **コメント入力エリア**:
   - ラベル: 「ロールバックの理由（任意）」
5. **アクションボタン**:
   - キャンセル / 戻る / 次へ / ロールバック実行

### 3.2 モックアップイメージ (テキストベース)

#### Step 1: 理由入力ダイアログ
```
+-------------------------------------------------------+
|  台帳を以前のバージョンに戻す                        [X] |
+-------------------------------------------------------+
|  Ver. 3 に戻します                                     |
|  (更新者: 山田 太郎 / 更新日: 2025-01-20 10:00)         |
|                                                       |
|  ロールバックの理由 (任意)                              |
|  [                                                 ]  |
|                                                       |
+-------------------------------------------------------+
|                       [ キャンセル ] [ 次へ (確認) ]     |
+-------------------------------------------------------+
```

#### Step 2: 最終実行確認 (再確認)
誤操作を完全に防ぐため、破壊的な変更であることを再認識させるステップ。

```
+-------------------------------------------------------+
|  最終確認：ロールバックを実行しますか？                [X] |
+-------------------------------------------------------+
|  ⚠️ 警告                                              |
|  この操作を実行すると、現在の内容は Ver. 6 として保存され、    |
|  台帳の状態が Ver. 3 の内容に書き換わります。             |
|                                                       |
|  [ ] 上記の内容を理解し、実行します (チェック必須)          |
|                                                       |
+-------------------------------------------------------+
|                       [ 戻る ] [ ロールバックを実行 ]     |
+-------------------------------------------------------+
```

### 3.3 再確認の挙動

1. **チェックボックス連携**: 「理解しました」のチェックを入れるまで「ロールバックを実行」ボタンを非活性（`disabled`）にする。
2. **ボタン配色**: 最終実行ボタンは `btn-error` または `btn-primary` を使用し、注意を促す。
3. **遷移**: モーダル内のステート切り替え（`$step` 変数等）で実装。

## 5. 考慮事項

### 5.1 差分プレビューの要否

Phase 2では簡素化のため、モーダル内での詳細な差分表示（赤/緑のハイライトなど）は見送り、バージョン番号とメタデータのみの提示とする。ユーザーは既に「更新履歴タブ」で差分を確認した上でボタンを押しているという前提に立つ。

### 5.2 ダイアログの再利用性

将来的に「差分表示画面」からのロールバック導線を追加する場合も、同じイベント `ledger.rollback.open-modal` をディスパッチするだけで再利用可能な設計とする。

### 5.3 入力値の保持

ステップ間（理由入力 → 最終確認）でコメント内容が消えないよう、Livewireのプロパティで確実に保持する。

## 6. 懸念事項と対応策

### 6.1 Livewire状態管理の制約対応

**懸念事項**: Livewireのパブリックプロパティはシンプルな連想配列のみサポートするため、複雑なモデル関係の管理が困難。

**対応策**: プロパティを最小限に抑え、必要時にEloquentクエリで再取得する設計。

```php
class RollbackConfirmModal extends Component
{
    public bool $show = false;
    public int $ledgerId = 0;
    public int $targetDiffId = 0;
    public string $comments = '';
    
    // computed property でモデル取得
    public function getLedgerProperty(): ?Ledger
    {
        return $this->ledgerId ? Ledger::with('latestDiff', 'define.folder')->find($this->ledgerId) : null;
    }
    
    public function getTargetDiffProperty(): ?LedgerDiff
    {
        return $this->targetDiffId ? LedgerDiff::with('modifier')->find($this->targetDiffId) : null;
    }
}
```

**理由**: Livewireの制約により、Eloquentモデルを直接プロパティに保存すると、シリアライゼーションエラーが発生する可能性がある。

### 6.2 ワークフロー状態による動的制約表示

**懸念事項**: ワークフロー有効台帳で担当者以外がロールバックボタンを押した場合の UX が不明瞭。

**対応策**: ボタン無効化とツールチップによる理由表示を実装。

```php
public function canExecuteRollback(): bool
{
    if (!$this->ledger || !$this->targetDiff) {
        return false;
    }
    
    return app(RollbackService::class)->canExecute(auth()->user(), $this->ledger);
}

public function getRollbackDisabledReasonProperty(): string
{
    if (!$this->ledger) return '';
    
    if ($this->ledger->isLocked()) {
        return '承認済みの台帳はロールバックできません';
    }
    
    if ($this->ledger->define->workflow_enabled && $this->ledger->status->isWorkflowPending()) {
        $latestDiff = $this->ledger->latestDiff;
        if ($this->ledger->status === WorkflowStatus::PENDING_INSPECTION && $latestDiff?->inspector_id !== auth()->id()) {
            return '点検担当者のみロールバック可能です';
        }
        if ($this->ledger->status === WorkflowStatus::PENDING_APPROVAL && $latestDiff?->approver_id !== auth()->id()) {
            return '承認担当者のみロールバック可能です';
        }
    }
    
    return '';
}
```

**理由**: ワークフロー有効台帳では担当者制約があるため、明確なフィードバックが必要。

### 6.3 モーダルでの権限エラーハンドリング

**懸念事項**: ロールバック実行時に権限エラーが発生した場合の UX が考慮されていない。

**対応策**: エラー種別に応じた適切なメッセージ表示を実装。

```php
public function execute(): void
{
    try {
        $this->validate([
            'comments' => config('ledgerleap.rollback.validation.comments.rule', 'nullable|string|max:500'),
        ]);
        
        if (!app(RollbackService::class)->canExecute(auth()->user(), $this->ledger)) {
            $this->error('ロールバックの権限がありません');
            return;
        }
        
        $result = app(RollbackService::class)->execute(
            $this->ledger,
            $this->targetDiff,
            auth()->id(),
            $this->comments
        );
        
        $this->dispatch('ledger-rolled-back', 
            ledgerId: $result['ledger']->id,
            previousDiffId: $this->ledger->latest_diff_id
        );
        
        $this->closeModal();
        
    } catch (InsufficientPermissionsException $e) {
        $this->error('権限が不足しています: ' . $e->getMessage());
    } catch (WorkflowConditionException $e) {
        $this->error('ワークフロー制約: ' . $e->getMessage());
    } catch (Exception $e) {
        Log::error('Rollback execution failed', [
            'ledger_id' => $this->ledgerId,
            'target_diff_id' => $this->targetDiffId,
            'user_id' => auth()->id(),
            'error' => $e->getMessage()
        ]);
        $this->error('ロールバック処理中にエラーが発生しました');
    }
}
```

**理由**: 既存の`WorkflowService`で定義されている例外クラスと統一性を保つため。

### 6.4 Toast通知との競合回避

**懸念事項**: 親画面とモーダルの両方でToast通知が発生し、重複表示される可能性。

**対応策**: モーダル内では例外エラーのみ表示し、成功通知は親画面に委ねる設計。

```blade
{{-- モーダルテンプレート内 --}}
@if($errors->any())
    <x-mary-alert 
        type="error" 
        :text="$errors->first()" 
        class="mb-4" />
@endif

{{-- 成功時はイベント発行のみ、Toast表示は親画面で実行 --}}
```

**理由**: MaryUIのToast機能は画面単位で管理されるため、重複回避が必要。
