# W5-2 Phase 2 ロールバック機能 テスト設計書

**最終更新:** 2026-01-25  
**対象:** Roleback機能 (W4-2.1～2.3 実装分)  
**ステータス:** 完了（W5-2.5 実施済み）

## 1. 目的
本ドキュメントは、Phase 2 ロールバック機能の実装（W4-2.1～2.3）の妥当性を検証するためのテストケースを定義し、その実施結果を記録する。

---

## 2. サービス層テスト (`RollbackServiceTest` / `RollbackRiskScenarioTest`)

### 2.1 正常系ケース (完了)
- **[TC-SV-01] ワークフロー無効台帳のロールバック**
    - **期待結果**: 指定バージョンの内容に復元され、バージョンが +1 される。
- **[TC-SV-02] 未承認状態（DRAFT）のロールバック**
    - **期待結果**: ステータス `DRAFT` を維持したまま復元。
- **[TC-SV-03] 点検待ち（PENDING_INSPECTION）のロールバック**
    - **期待結果**: ステータスが `DRAFT` に戻り、担当者のタスクカウンターが減算される。

### 2.2 異常・エッジケース (完了)
- **[TC-SV-04] 承認済みレコードの拒否**
    - **期待結果**: `WorkflowConditionException` がスローされる。
- **[TC-SV-05] 楽観的ロック（バージョン不一致）の検知**
    - **期待結果**: 他のユーザーによる更新（バージョン進捗）を検知し、処理を拒否する。
- **[TC-SV-06] 権限なしアクセス**
    - **期待結果**: `WRITE` 権限がないユーザーの実行を拒否する。
- **[TC-SV-07] 添付ファイルのAuto-healing (自動復旧)**
    - **条件**: 復元対象の `AttachedFile` レコードはあるが、抽出テキスト (`tika_metadata`) が欠落している。
    - **期待結果**: `RollbackService` が異常を検知。ステータスをリセットし `ProcessAttachedFile` ジョブを再投入する。
    - **修正履歴**: `RollbackService` 内でカラムタイプを `'file'` と誤判定していたバグを `'files'` に修正（2026-01-25）。

## 4. 検証時の知見・注意点 (反映済み)
- **Mroonga**: 全文検索を含むテストでは `DatabaseMigrations` が必須。
- **型の一貫性**: カラムタイプ判定は `'files'` (複数形) を使用すること。
- **データ構造**: `AsColumnArrayJson` の特性上、`content` のアサーションはインデックス0からの数値添字配列で行う。
- **マルチテナント整合性**: `LedgerDiff` 等の作成時に `tenant_id` が消失すると履歴が見えなくなる。これを防ぐための「データ不整合検知テスト」を常設すること。
