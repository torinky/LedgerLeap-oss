# W2-1.5 ワークフローUI既存実装調査と統合方針

**最終更新:** 2026-01-11
**対象:** LedgerLeap v12.0 / Cycle 3
**関連Issue:** #38, #37
**関連設計:** [W3-3.1 Approval History Design](./2026-01-11_W3-3-1_approval_history_design.md)

## 1. 目的と背景

### 1.1 背景

Cycle 3 で計画している `#38 編集者情報表示`
に関連して、単に編集者を表示するだけでなく、「誰が承認したか」「現在のステータスは何か」というワークフローの文脈を提示することが、現場リーダーや管理者のユースケースにおいて不可欠である。
本ドキュメントでは、この「承認履歴UI」の定義と、既存のワークフロー実装資産の活用方針をまとめる。

### 1.2 解決したい課題 (ペルソナ別ユースケース)

* **現場リーダー (レビューアー):** 部下の変更内容を承認する際、「誰が申請したか」に加え、「前段で誰が承認したか」といった履歴を確認したい。
* **管理者 (監査):** 過去の変更履歴において、「承認者」「ステータス」「アクション種別（差戻し等）」の完全なトレーサビリティを確保したい。
* **実務担当者:** 自分の申請が現在どうなっているか（ステータス、差戻し理由）を即座に把握したい。

## 2. 実装スコープの定義

承認履歴UIとして、以下の要素をサイドバーの履歴リストに追加・拡張する方針とする（**案B: トレーサビリティ重視** を採用）。

| 要素           | 説明                                                           | 新規/既存                       |
|:-------------|:-------------------------------------------------------------|:----------------------------|
| **ヘッダー**     | バージョン、日時                                                     | 既存維持                        |
| **ステータスバッジ** | ワークフロー状態 (`APPROVED`, `Draft` 等)                             | **新規** (`WorkflowStatus`利用) |
| **アクター情報**   | 編集者（Modify）および承認者（Approve）の名前。<br>Issue #38のポップオーバーで詳細情報を表示。 | **拡張**                      |
| **アクション**    | 操作種別（「作成」「更新」「承認」「差戻し」「ロールバック」）                              | **新規**                      |
| **コメント**     | 申請/承認時のコメント                                                  | **拡張** (承認コメントも表示)          |

## 3. 既存実装の調査結果

### 3.1 関連クラス・コンポーネント

| コンポーネント/クラス                | パス                                                                 | 役割・特徴                                                    | 再利用判断                |
|:---------------------------|:-------------------------------------------------------------------|:---------------------------------------------------------|:---------------------|
| **`WorkflowStatus` Enum**  | `app/Enums/WorkflowStatus.php`                                     | ステータス定義。<br>`label()`, `colorClass()`, `icon()` メソッドを持つ。 | **全面的に再利用 (必須)**     |
| **`WorkflowHistoryList`**  | `resources/views/livewire/ledger/workflow-history-list.blade.php`  | 監査用テーブルコンポーネント。                                          | **再利用不可** (レイアウト不適合) |
| **`LedgerHistoryManager`** | `resources/views/livewire/ledger/ledger-history-manager.blade.php` | 変更履歴サイドバー。                                               | **拡張対象** (これをベースにする) |

### 3.2 `WorkflowHistoryList` の分析

`WorkflowHistoryList` はテーブル形式 (`x-mary-table`) であり、以下のロジックを持つ。

* `PENDING_INSPECTION` → 次の点検者名を表示
* `APPROVED` → **承認者名 (`$diff->approver->name`) を表示**
  このロジックは、サイドバーでの表示においても有用である。

## 4. 統合・再利用の方針

**「ロジック再利用 (Logic Reuse)」アプローチ**を採用する。

### 4.1 UIコンポーネント統合の判断

`LedgerHistoryManager` (サイドバー) と `WorkflowHistoryList` (テーブル)
は、レイアウト目的とインタラクション（サイドバーはクリックで差分比較、テーブルはリンク遷移）が大きく異なるため、*
*コンポーネント自体の共有は行わない**。
無理に統合すると、片方の変更が他方に悪影響を与えるリスクが高い。

### 4.2 具体的な実装方針

1. **Enumの徹底活用:**
    * ステータス表示には必ず `App\Enums\WorkflowStatus` のメソッド (`colorClass`, `label`, `icon`) を使用し、色と文言の一貫性を保証する。

2. **サイドバー (`LedgerHistoryManager`) の拡張:**
    * 既存のバージョンリスト表示部分 (`div`ループ内) を、「承認履歴アイテム」としてリッチ化する。
    * 表示ロジック（承認済みなら承認者を出す等）は `WorkflowHistoryList` を参考にするが、コードは `LedgerHistoryManager`
      固有の Blade テンプレートとして実装する。

3. **アクター表示と Issue #38:**
    * 編集者名、承認者名が表示される箇所には、**共通の「ユーザー情報ポップオーバー」コンポーネント** (Issue #38で作成)
      を適用する。

### 4.3 UI実装イメージ

```html
<!-- サイドバーアイテム案 -->
<div class="flex flex-col gap-1">
    <div class="flex justify-between">
        <span class="font-bold">Ver. 5</span>
        <span class="text-xs text-gray-500">2026-01-12 10:00</span>
    </div>

    <!-- ステータスバッジ (Enum活用) -->
    <div>
        <x-mary-badge :value="$diff->status->label()" class="{{ $diff->status->colorClass() }} badge-xs"/>
    </div>

    <!-- アクター情報 (Issue #38 ポップオーバー適用対象) -->
    <div class="text-sm">
        <i class="fas fa-edit"></i> 佐藤 (編集)
    </div>
    @if($diff->status === WorkflowStatus::APPROVED)
    <div class="text-sm text-success">
        <i class="{{ WorkflowStatus::APPROVED->icon() }}"></i> 鈴木 (部長) が承認
    </div>
    @endif
</div>
```

## 5. 結論

* **スコープ:** 編集者だけでなく、承認者・ステータスも含めた「承認履歴」として実装する。
* **再利用:** `WorkflowStatus` Enum は完全利用。Viewコンポーネントは分離し、サイドバー専用の実装を行う。
* **次のステップ:** Issue #38 にて、この方針に基づき `LedgerHistoryManager` の改修とポップオーバー実装に着手する。
