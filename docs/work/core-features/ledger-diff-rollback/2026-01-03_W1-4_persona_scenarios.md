# W1.4 ペルソナ別シナリオ検討（履歴/差分・ロールバック）

**最終更新:** 2026-01-04  
**対象:** LedgerLeap v12.0 / Branch: `feature/ledger-rollback`  
**ステータス:** Done

---

## 0. 前提
- W1.3の結果: 案C「タブ内コンポーネント埋め込み」＋案D「ドロワー/ダイアログ併用」を推奨方針とする。
  - 基本情報タブ: 既存の即時差分（最新↔直前）と前回編集内容表示を維持。
  - 更新履歴タブ: 専用履歴画面と同一の差分表示Livewireコンポーネントを埋め込み、任意比較・ロールバックまで完結（遅延ロード）。
  - 専用履歴画面: 独立エントリーポイントとして残し、同一コンポーネントを再利用。
  - 必要に応じてドロワー/ダイアログで詳細差分・ロールバック確認を表示。
- 対象ペルソナ: `docs/function/PersonaUseCaseScenario.md` に定義の「実務担当者」「現場リーダー」「管理者」を主対象とする。
- スコープ: ワークフロー無効の台帳における履歴/差分閲覧とロールバック導線。ワークフロー有効の台帳は、本フェーズでは「認可前であれば通常編集で内容・ステータスを戻す」ことを基本とし、専用ロールバックは対象外とする。
- リテラシー前提: 任意の2バージョン比較は概念的に難度が高く、主に高リテラシーな管理者や一部の現場リーダー向けとし、実務担当者・多くの現場リーダーは「ロールバック実行 → ロールバック後に直前と比較して確認」のフローを基本とする。

---

## 1. 実務担当者シナリオ

### 1.1 シナリオA: 直近修正内容の確認（即時差分）
- 導線: 日報等の台帳詳細画面 → 基本情報タブ。
- 操作:
  1. 実務担当者が直前に修正した内容を再確認したい。
  2. 基本情報タブ内の「前回内容」と「現在内容」の差分表示を確認。
- 要件メモ:
  - 既存機能を維持（本計画では仕様の踏襲とバグ回避のみ）。
  - 新しい履歴タブ/専用履歴画面と矛盾しない文言・ラベリングに調整（"直前の変更" と "履歴全体" の区別）。

### 1.2 シナリオB: 過去の特定更新の内容確認（任意比較なし）
- 導線: 台帳詳細 → 更新履歴タブ。
- 操作:
  1. 過去数回分の更新履歴をざっと確認したい。
  2. 更新履歴タブ内の履歴一覧から対象バージョンを選択し、その時点の内容を「当時の状態」として閲覧（差分ハイライトは任意）。
- 要件メモ:
  - 更新履歴タブの埋め込みコンポーネントに「スナップショット表示モード」（差分OFF）を持たせる余地。
  - ページング/スクロール性能（履歴件数が一定数を超えた場合）。

### 1.3 シナリオC: 任意2バージョンの比較（詳細差分・高リテラシー向け）
- 導線:
  - パターン1: 台帳詳細 → 更新履歴タブ（タブ内埋め込みコンポーネントで任意比較）。
  - パターン2: 台帳一覧や他画面から専用履歴画面直行。
- 操作（高リテラシーユーザー、主に管理者/一部リーダー想定）:
  1. 特定の過去版と別の過去版（または現在版）を直接比較したい。
  2. 履歴一覧から2つのバージョンを選択し、フィールド/添付/ステータス差分を確認する。
- 実務担当者向けの推奨フロー:
  - 任意比較は原則利用しなくてもよい想定とし、誤更新時は「ロールバック候補の差分をざっくり確認 → ロールバック実行 → ロールバック後に基本情報タブの即時差分で直前状態と比較して安心する」流れをメインとする。
- 要件メモ:
  - 任意比較UIは設定や権限で露出を制御可能にする（管理者/高リテラシーユーザー向け）。
  - 添付ファイル差分で画像→PDFキー変換を透過的に扱う。
  - Auto Link/タグ/表示レベル/グループ開閉を基本情報タブに合わせて適用。

### 懸念・対応案（実務担当者）
- 懸念1: 基本情報タブの即時差分と履歴タブの詳細差分の使い分けが直感的でない可能性。
  - 対応案:
    - ラベルを明確化（例: 基本情報側は「直前の変更を確認」、履歴タブ側は「履歴・ロールバック候補を確認」）。
    - 任意比較UIはデフォルト非表示／折りたたみとし、必要なユーザーだけが使えるようにする（設定 or 権限）。
    - 初回利用時に簡易チュートリアル/ツールチップを表示。
- 懸念2: 任意比較UIが複雑になり、日常利用には重すぎる可能性。
  - 対応案:
    - 実務担当者向けには「ロールバック → ロールバック後に直前比較」というフローをガイドとして提示し、任意比較はヘルプ内に位置づける。
    - デフォルトは「最新↔直前」をワンクリックで表示、任意比較はオプション扱い。
    - フィルタ/ソートを最小限に抑え、必要に応じて拡張。

## 2. 現場リーダーシナリオ

### 2.1 シナリオD: 部下の更新内容レビューと差分確認
- 導線: マイポータルや一覧 → 台帳詳細 → 更新履歴タブ。
- 操作:
  1. 現場リーダーが部下の編集後台帳を開く。
  2. 基本情報タブで全体の現状をざっと確認。
  3. 更新履歴タブで、直近数回の更新内容を差分で確認（誰がいつ何を変えたか）。
- 要件メモ:
  - 履歴一覧に編集者・編集日時を表示。
  - 差分ビューに編集者情報を含める（可能なら）。

### 2.2 シナリオE: 誤更新のロールバック（ワークフロー無効）
- 導線: 台帳詳細 → 更新履歴タブ（または専用履歴画面）。
- 操作:
  1. 誤った更新を確認した現場リーダーが、過去バージョンへ戻したい。
  2. 対象バージョンを選択し、ロールバック候補として差分をざっくり確認（主に「いつ誰がどの項目を変えたか」レベル）。
  3. ドロワー/ダイアログでロールバック内容と影響範囲を確認し、実行ボタンを押下。
  4. ロールバック後、基本情報タブの即時差分を使って「ロールバック前の状態」と比較し、期待通りに戻っているか確認する。
- 要件メモ:
  - ロールバックはワークフロー無効台帳のみ許可。
  - 権限チェック（フォルダ権限・ロール）と確認ダイアログ。
  - ロールバック後にトースト通知 + 履歴にイベント追記。

### 懸念・対応案（現場リーダー）
- 懸念1: ロールバック権限の線引きが曖昧になるリスク。
  - 対応案:
    - ロール/フォルダ権限に「ロールバック可」フラグを追加または既存権限との対応表を明示。
    - UI上でも「ロールバック可能なユーザーのみボタンを表示」する。
- 懸念2: ロールバック前後の状態把握が難しい。
  - 対応案:
    - ドロワー/ダイアログに「代表的な変更点」を簡潔に示し、詳細はロールバック後に基本情報タブの即時差分で確認する流れをガイドとして提示。
    - 実行後に「ロールバック前の状態への再ロールバック」までは行わず、履歴と差分が残る設計とするかどうかをW2で検討（運用ポリシーと合わせて決定）。

## 3. 管理者シナリオ

### 3.1 シナリオF: 監査目的での履歴・差分確認
- 導線: 管理画面/台帳一覧 → 専用履歴画面。
- 操作:
  1. 管理者が特定台帳の変更履歴を監査したい。
  2. 専用履歴画面で期間・ユーザー・操作種別などで絞り込み。
  3. 差分表示で重要フィールドや添付の変更内容を確認。
  4. 必要に応じて任意の2バージョンを選択して詳細比較（高リテラシー前提）。
- 要件メモ:
  - 操作ログ（`activity_log`、`ledger_diffs`）との整合性。
  - フィルタ/ソート機能の拡張余地。
  - 任意比較機能は主に管理者向けと位置づける。

### 3.2 シナリオG: ロールバック履歴の監査
- 導線: 専用履歴画面または監査画面。
- 操作:
  1. 誰がいつどのバージョンへロールバックしたかを確認。
  2. 必要に応じてロールバック前後の差分を再確認。
- 要件メモ:
  - `ledger_diffs` にロールバックイベント種別・実行者を明示保存。
  - 操作ログとの突合（IDやタイムスタンプで紐付け）。

### 懸念・対応案（管理者）
- 懸念1: 履歴/差分UIが実務担当者向けと監査向けで要件が乖離する可能性。
  - 対応案:
    - UIは共通コンポーネントとしつつ、管理者向けはフィルタ/メタ情報表示を強化する「監査ビュー」オプションを検討。
    - 任意比較や詳細フィルタは管理者ロールにのみフル公開し、一般ユーザーにはシンプルなビューをデフォルトとする。
- 懸念2: ロールバック機能が監査上のリスクとみなされる可能性。
  - 対応案:
    - ロールバック操作には必須コメント（理由）を要求し、監査ログに保存。
    - ロールバック自体を機能フラグで無効化できるように設計（運用ポリシーに応じて制御）。

## 4. 横断的な懸念・対応案

1. **アクセシビリティ（キーボード操作/フォーカス）**
   - 懸念: タブ埋め込みコンポーネント＋ドロワー/ダイアログでフォーカス遷移が複雑化。
   - 対応案:
     - 重要操作（タブ切替、履歴選択、ロールバックダイアログ）はキーボードのみで完結できることをW2/W3で明文化。
     - ドロワー/ダイアログオープン時にフォーカスをヘッダーに移動し、閉じた後は元のトリガー要素へ戻す実装ガイドラインを策定。

2. **パフォーマンス（履歴件数多）**
   - 懸念: 管理者・現場リーダーが大量履歴を扱う際、タブ埋め込みで応答が悪化。
   - 対応案:
     - タブ切替時に初回ロードし、その後はページング/無限スクロールを採用。
     - デフォルトは直近N件のみ表示し、さらに古い履歴は専用履歴画面で確認させるパターンも検討。

3. **導線の分かりやすさ**
   - 懸念: 「基本情報タブの即時差分」「更新履歴タブの詳細差分」「専用履歴画面」が増え、どこで何をするのか混乱する可能性。
   - 対応案:
     - タブ・ボタン文言を役割ベースで整理（例: 「直前の変更を確認」「履歴・差分」「履歴を詳細に確認」）。
     - ヘルプテキスト/ホバー説明を追加し、概要を示す。

## 5. W1.4のアウトプットとW2への引き継ぎ

- W1.4アウトプット:
  - ペルソナ別に「どの画面から」「どのタブ/画面/ドロワーに」「何をするか」を明文化。
  - 懸念事項と対応方針を整理し、W2要件定義で具体化すべき論点をリストアップ。
- W2で具体化する項目（抜粋）:
  - 各シナリオごとの必須機能/任意機能（MVP範囲）の確定。
  - ロールバック権限と監査要件（コメント必須/機能フラグ）を仕様化。
  - アクセシビリティ要件（キーボード/フォーカス/ARIA）をUI設計に落とし込む。
  - パフォーマンス要件（履歴件数とレスポンス時間の目安）を設定し、遅延ロード戦略に反映。
