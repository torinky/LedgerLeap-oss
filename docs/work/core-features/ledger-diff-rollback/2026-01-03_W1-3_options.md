# W1.3 移植対象確定と履歴タブ統合案（ドラフト）

**最終更新:** 2026-01-03  
**対象:** LedgerLeap v12.0 / Branch: `feature/LLM-integration`  
**ステータス:** Draft（レビュー待ち）

---

## 0. 前提・スコープ
- 対象: ワークフロー無効の台帳における差分表示/ロールバック。
- 目的: 基本情報タブの利便性を履歴/差分に移植し、更新履歴タブと台帳履歴画面の統合を視野にUX改善案を比較する。
- 制約: Livewire状態は単一配列、AsColumnArrayJsonで`data_get()`禁止、テナント初期化必須、Mroonga複合インデックス不可、添付差分キー（画像→`.pdf`）を共通ロジックで扱う。

---

## 1. 既存UI/導線の前提整理（抜粋）
- 基本情報タブ: 表示レベル切替、グループ開閉、型別レンダリング、Auto Link、タグ表示、添付表示（処理ステータス表示/再処理導線は閲覧では抑制想定）。
- 基本情報タブの即時差分: 最新↔直前の差分表示と前回編集内容の表示機能が既に存在（軽量/即時性重視）。
- 更新履歴タブ（詳細画面内）: 直近履歴の一覧と差分表示（現状機能は限定的、表示レベル/グループ開閉/Auto Link/タグ/添付差分が不足）。
- 台帳履歴画面（別画面）: 履歴一覧→差分ビュー（任意比較UIは弱め）。

---

## 2. ユーザーシナリオ（ワークフロー無効前提）
- 実務担当者: 最新と直前/任意バージョンを比較し、添付・ステータス差分を確認。
- 現場リーダー: 部下の更新内容をレビューし、必要なら旧版へロールバック（権限があれば即時）。
- 管理者: 監査観点で差分とロールバック履歴を確認（監査ログと突合）。

---

## 3. 案の比較とトレードオフ

### 案A: 「詳細画面内タブ統合」強化型
- 概要: 現行の詳細画面内「更新履歴タブ」を強化し、基本情報タブの表示レベル/グループ開閉/Auto Link/タグ/添付表示を移植。差分・任意比較も同タブで完結。基本情報タブの即時差分/前回編集表示はそのまま活用し、履歴タブ側で任意比較を拡充。
- メリット:
  - 画面遷移なしで完結、操作コスト低。
  - Livewireコンポーネント間で状態共有しやすい。
  - キーボード操作・アクセシビリティを統一しやすい。
- デメリット/リスク:
  - 初期ロード負荷が増大。遅延ロード/キャッシュ必須。
  - 詳細タブが肥大化しUI密度が上がる。
  - 履歴が多い場合のパフォーマンス劣化リスク。

### 案B: 「専用履歴画面への統合」集中型
- 概要: 台帳履歴専用画面に機能を集約し、詳細画面の履歴タブは薄くして専用画面へ導線を張る。比較・ロールバックは専用画面で実行。基本情報タブの即時差分/前回編集表示は維持し、深い比較は専用画面に委譲。
- メリット:
  - パフォーマンス最適化を専用画面で行いやすい（遅延ロード/ページング）。
  - UIを履歴/差分特化で再設計できる。
  - コード責務を分離し、詳細画面の肥大化を防げる。
- デメリット/リスク:
  - 画面遷移が発生し、操作ステップが増える。
  - 詳細画面との表示/操作の一貫性確保が課題。
  - 状態引き継ぎ（表示レベル/フィルタ）のUX設計が必要。

### 案C: 「ハイブリッド」段階型
- 概要: 詳細画面の履歴タブで軽量ビュー（最新↔直前差分 + 移植機能のサブセット）を提供し、詳細比較・任意比較・ロールバックは専用履歴画面で行う。基本情報タブの即時差分/前回編集表示は維持しつつ、履歴タブで軽量任意比較を強化。
- メリット:
  - よく使う最新差分は遷移なしで確認可能。
  - 重い処理は専用画面に委譲し、パフォーマンスを確保。
  - 徐々に移植しやすく、リスクを分散。
- デメリット/リスク:
  - UIが二段構えで、導線設計が複雑化。
  - 機能の二重実装/重複テストコストが発生しやすい。

### 案D: 「ドロワー/ダイアログ＋子コンポーネント」派生型
- 概要: 詳細画面や専用履歴画面から、差分表示・任意比較・ロールバックをドロワー/ダイアログとして表示するLivewire子コンポーネントに切り出す（A/B/Cいずれにも組み合わせ可能）。基本情報タブの即時差分/前回編集表示は温存し、深い比較のみドロワー/ダイアログで展開。
- メリット:
  - 遷移を抑えつつ画面占有を避け、段階的導線にしやすい。
  - 子コンポーネント化で責務分離し、複数導線から再利用できる。
  - ハイブリッド案Cとの親和性が高く、拡張時の差し替えが容易。
- デメリット/リスク:
  - フォーカス管理/アクセシビリティの実装コストが増える。
  - 状態同期の複雑化（親との表示レベル・フィルタ共有）。
  - パフォーマンス配慮（ドロワー初回表示時のデータ取得遅延設計）が必要。

---

## 4. 共通考慮事項（案共通で対応）
- 案C（ハイブリッド）をベースに、案D（サブコンポーネント化）を組み合わせる。
- **コンポーネントの「モード」化**: `readonly`（詳細タブ用、最新更新表示特化）と `interactive`（専用画面用、ロールバック・任意比較可能）の導入。
- **表示状態の引き継ぎ**: 詳細画面での `displayLevel` や `collapsedStates` を遷移先に引き継ぐ（URLパラメータ等を活用）。
- **差分表示のトグル化**: 常に差分を出すのではなく、単に「当時のデータ」として閲覧するモード（差分ハイライトなし）を選択可能にする。
- **添付ファイル整合性の厳密化**: 画像→`.pdf`変換を考慮しつつ、ロールバック実行時に実ファイルの存在チェックを行うロジックを担保。
- **アクセシビリティ**: 既存キーボード操作/フォーカス有無を確認し、履歴ビューでも踏襲または再設計。

---

## 5. 推奨と次ステップ（W1.3→W2への入力）
- 推奨: 案C（ハイブリッド）をベースに、最新差分は基本情報タブの既存即時差分機能を維持、軽量任意比較は履歴タブで強化、深い任意比較/ロールバックは専用履歴画面で実行。ドロワー/ダイアログ子コンポーネント（案D）を併用して導線の柔軟性を確保。
- W1.3で決めること:
  - 採用案: 案C（ハイブリッド）＋ 案D（サブコンポーネント化）の採用。
  - 共通コンポーネントのインターフェース定義（モード、表示レベル、トグル挙動）。
  - 基本情報タブの既設「前回編集」表示との棲み分け（履歴タブへ機能を寄せるか、共存か）。
- W2以降への宿題:
  - コンポーネント間の状態同期（`displayLevel` / `collapsedStates`）の具体的実装設計。
  - ロールバック時の添付ファイル存在チェック（画像→PDFキー変換整合性）のジョブ設計。
  - アクセシビリティ要件の具体化（キーボード/フォーカス）。
  - パフォーマンス要件（遅延ロード/ページング）の数値目標。
  - 管理者向け監査表示（操作ログとの突合）と通知要否。
