# W1.3 移植対象確定と履歴タブ統合案（ドラフト）

**最終更新:** 2026-01-04  
**対象:** LedgerLeap v12.0 / Branch: `feature/ledger-rollback`  
**ステータス:** Approved

---

## 0. 前提・スコープ
- 対象: ワークフロー無効の台帳における差分表示/ロールバック。
- 目的: 基本情報タブの利便性を履歴/差分に移植し、更新履歴タブと台帳履歴画面の統合を視野にUX改善案を比較する。
- 制約: Livewire状態は単一配列、AsColumnArrayJsonで`data_get()`禁止、テナント初期化必須、Mroonga複合インデックス不可、添付差分キー（画像→`.pdf`）を共通ロジックで扱う。

---

## 1. 既存UI/導線の前提整理（抜粋）
- 基本情報タブ: 表示レベル切替、グループ開閉、型別レンダリング、Auto Link、タグ表示、添付表示（処理ステータス表示/再処理導線は閲覧では抑制想定）。
- 基本情報タブの即時差分: 最新↔直前の差分表示と前回編集内容の表示機能が既に存在（軽量/即時性重視）。
- 更新履歴タブ（詳細画面内）: 直近履歴の一覧と差分表示（現状機能は限定的、表示レベル/グループ開閉/Auto Link/タグ/添付差分が不足）。
- 台帳履歴画面（別画面）: 履歴一覧→差分ビュー（任意比較UIは弱め）。

---

## 2. ユーザーシナリオ（ワークフロー無効前提）
- 実務担当者: 最新と直前/任意バージョンを比較し、添付・ステータス差分を確認。
- 現場リーダー: 部下の更新内容をレビューし、必要なら旧版へロールバック（権限があれば即時）。
- 管理者: 監査観点で差分とロールバック履歴を確認（監査ログと突合）。

---

## 3. 案の比較とトレードオフ

### 案A: 「詳細画面内タブ統合」強化型
- 概要: 現行の詳細画面内「更新履歴タブ」を強化し、基本情報タブの表示レベル/グループ開閉/Auto Link/タグ/添付表示を移植。差分・任意比較も同タブで完結。基本情報タブの即時差分/前回編集表示はそのまま活用し、履歴タブ側で任意比較を拡充。
- メリット:
  - 画面遷移なしで完結、操作コスト低。
  - Livewireコンポーネント間で状態共有しやすい。
  - キーボード操作・アクセシビリティを統一しやすい。
- デメリット/リスク:
  - 初期ロード負荷が増大。遅延ロード/キャッシュ必須。
  - 詳細タブが肥大化しUI密度が上がる。
  - 履歴が多い場合のパフォーマンス劣化リスク。

### 案B: 「専用履歴画面への統合」集中型
- 概要: 台帳履歴専用画面に機能を集約し、詳細画面の履歴タブは薄くして専用画面へ導線を張る。比較・ロールバックは専用画面で実行。基本情報タブの即時差分/前回編集表示は維持し、深い比較は専用画面に委譲。
- メリット:
  - パフォーマンス最適化を専用画面で行いやすい（遅延ロード/ページング）。
  - UIを履歴/差分特化で再設計できる。
  - コード責務を分離し、詳細画面の肥大化を防げる。
- デメリット/リスク:
  - 画面遷移が発生し、操作ステップが増える。
  - 詳細画面との表示/操作の一貫性確保が課題。
  - 状態引き継ぎ（表示レベル/フィルタ）のUX設計が必要。

### 案C: 「タブ内コンポーネント埋め込み」型
- 概要: 詳細画面の更新履歴タブ内に、専用履歴画面の差分表示Livewireコンポーネントを直接埋め込む（遅延ロード対応）。基本情報タブの即時差分/前回編集表示は維持し、履歴タブでは同じコンポーネントで任意比較・ロールバックまで完結。専用履歴画面は独立したエントリーポイントとして残し、タブと専用画面で同一コンポーネントを再利用。
- メリット:
  - コンポーネント再利用で実装コスト低減、テスト一元化。
  - 基本情報タブの既存差分機能はそのまま、履歴タブで深い比較も可能。
  - タブ切り替えだけで任意比較・ロールバックまで遷移なし完結。
  - 遅延ロード対応で初期ロード負荷を抑制。
- デメリット/リスク:
  - タブ内に埋め込むと画面占有が大きく、UI密度が上がる。
  - 履歴件数が多い場合のパフォーマンス配慮（ページング/フィルタ）が必要。
  - 基本情報タブの即時差分と履歴タブの詳細差分で、ユーザーがどちらを使うべきか混乱する可能性。

### 案D: 「ドロワー/ダイアログ＋子コンポーネント」派生型
- 概要: 詳細画面や専用履歴画面から、差分表示・任意比較・ロールバックをドロワー/ダイアログとして表示するLivewire子コンポーネントに切り出す（A/B/Cいずれにも組み合わせ可能）。基本情報タブの即時差分/前回編集表示は温存し、深い比較のみドロワー/ダイアログで展開。
- メリット:
  - 遷移を抑えつつ画面占有を避け、段階的導線にしやすい。
  - 子コンポーネント化で責務分離し、複数導線から再利用できる。
  - ハイブリッド案Cとの親和性が高く、拡張時の差し替えが容易。
- デメリット/リスク:
  - フォーカス管理/アクセシビリティの実装コストが増える。
  - 状態同期の複雑化（親との表示レベル・フィルタ共有）。
  - パフォーマンス配慮（ドロワー初回表示時のデータ取得遅延設計）が必要。

---

## 4. 共通考慮事項（案共通で対応）
- 状態管理: Livewire公開プロパティは単一配列。差分DTO生成はサーバー側で統一。
- パフォーマンス: 遅延ロード、ページング、キャッシュ。履歴件数多時のクエリ最適化（Mroonga複合NGに留意）。
- 権限/テナント: ポリシー + フォルダ権限、`tenancy()->initialize()` をFeatureテストで徹底。
- データアクセス: AsColumnArrayJsonで`data_get()`禁止、直接配列アクセス。添付キーは画像→`.pdf`変換を統一ロジックで扱う。
- ロールバック: ワークフロー無効限定で先行。権限チェックと監査ログ必須。適用後にスコアリング/検索インデックス/添付整合ジョブを非同期投入。
- アクセシビリティ: 既存キーボード操作/フォーカス有無を確認し、履歴ビューでも踏襲または再設計。
- **コンポーネント再利用の方針**:
  - 専用履歴画面のLivewireコンポーネントを、更新履歴タブ内にも埋め込む形で再利用。
  - プロパティで埋め込み先を識別（例: `$context = 'tab'` or `'standalone'`）し、必要に応じてUI調整（ヘッダー表示/非表示、ナビゲーション有無など）。
  - 表示レベル・グループ開閉状態はプロパティで親から子に渡し、子から親へイベントで同期。

---

## 5. 推奨と次ステップ（W1.3→W2への入力）
- 推奨: **案C（タブ内コンポーネント埋め込み）＋ 案D（ドロワー/ダイアログ併用）**
  - 基本情報タブの即時差分/前回編集表示は維持（軽量で即時性重視のユースケース対応）。
  - 更新履歴タブには専用履歴画面と同じLivewireコンポーネントを埋め込み、任意比較・ロールバックまで完結（遅延ロード対応でパフォーマンス確保）。
  - 専用履歴画面は独立エントリーポイントとして残し、タブと専用画面でコンポーネントを再利用（実装・テストコスト削減）。
  - 必要に応じてドロワー/ダイアログ（案D）で差分詳細やロールバック確認を表示し、導線の柔軟性を確保。
- W1.3で決めること:
  - 採用案の確定: 案C（タブ内埋め込み）＋ 案D（ドロワー/ダイアログ併用）。
  - 共通コンポーネントのインターフェース定義（埋め込み先に応じた表示モード、`displayLevel` / `collapsedStates` の引き継ぎ）。
  - 基本情報タブの即時差分との棲み分け（簡易差分は基本情報、詳細任意比較は履歴タブという役割分担を明確化）。
  - 履歴タブでの遅延ロード設計（タブ切り替え時に初めてコンポーネントをロード）。
- W2以降への宿題:
  - コンポーネント間の状態同期（`displayLevel` / `collapsedStates`）の具体的実装設計。
  - ロールバック時の添付ファイル存在チェック（画像→PDFキー変換整合性）のジョブ設計。
  - アクセシビリティ要件の具体化（キーボード/フォーカス、タブ遷移とドロワー開閉）。
  - パフォーマンス要件（遅延ロード/ページング）の数値目標。
  - 管理者向け監査表示（操作ログとの突合）と通知要否。
