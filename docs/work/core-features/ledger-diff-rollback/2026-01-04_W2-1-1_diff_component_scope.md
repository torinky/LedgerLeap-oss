## 1. 目的・背景

台帳詳細画面の「基本情報タブ（即時差分）」、「更新履歴タブ（埋め込み）」および「専用履歴画面」において、一貫した差分表示体験・操作性を提供するための共通Livewireコンポーネント（既存の `LedgerDiffViewer` を拡張）の機能範囲とインターフェースを定義する。

### 記載範囲

- 共通差分コンポーネントのインターフェース（入力・出力）
- 実装すべき機能範囲（表示レベル、検索ハイライト、ワークフローステータス、添付）
- UI/UX要件（最新UIトレンドおよびDaisyUIのセオリー準拠）
- 実装・保守方針（基本情報タブとの共通化）

### 記載しない内容

- ロールバック実行ロジック（Phase 2で定義）
- 検索エンジン自体の仕様

---

## 2. 共通コンポーネントとしての位置付け

既存の `LedgerDiffViewer` を、詳細画面の「更新履歴タブ（履歴確認）」と「専用履歴画面」の両方で共通して使用する唯一のコンポーネントとして位置づける。

> [!IMPORTANT]
> **基本情報タブの即時差分との関係**
>
> [W1.3](2026-01-03_W1-3_options.md)および[W1.4](2026-01-03_W1-4_persona_scenarios.md)の検討結果に基づき、以下の役割分担を明確化:
>
> - **基本情報タブの即時差分**: 最新↔直前の軽量・即時性重視の差分表示として**別実装を維持**（既存機能）
> - **本コンポーネント（`LedgerDiffViewer`）**: 更新履歴タブ/専用履歴画面で使用する詳細差分・任意比較用コンポーネント
>
> この分離により、実務担当者は「直前の変更を確認」（基本情報タブ）と「履歴全体・任意比較」（履歴タブ）を明確に使い分けられる。詳細は[W2-1.2](2026-01-04_W2-1-2_role_wording_policy.md)を参照。

### 利用パターン

- **更新履歴タブ**: 任意バージョンのスナップショット表示、または任意2バージョンの比較として利用（遅延ロード対応）。
- **専用履歴画面**: 独立したエントリーポイントとして、同一コンポーネントを再利用。

これらを共通化することで、表示ロジック（`LedgerContentProcessor` 連携）やUIスタイル（DaisyUI）の一貫性を保ち、将来的なメンテナンスコストを抑制する。

## 3. インターフェース (IF)

### 入力プロパティ (Props)

- `ledgerRecord`: 表示対象の台帳本体。
- `comparisonTargetDiff`: 比較対象となる `LedgerDiff` レコード（Nullの場合は差分なし、単一表示）。
- `highlight`: **検索キーワード（ハイライト対象）**。W1.1に基づき、検索ヒット時の可視性を確保。
- `displayLevel`: 表示レベル（1:非表示, 2:簡略, 3:詳細）。**基本情報タブと履歴タブで共通の設定値を引き継ぐ。**
- `collapsedStates`: グループごとの開閉状態。
- `context`: `'details_tab'`, `'history_tab'`, `'standalone'`。コンテキストに応じてボタン導線やヘッダー情報を最適化。

### 出力・イベント

- `groupToggled(groupName, state)`: グループ開閉状態を親に通知し、タブ間で同期可能とする。

## 4. 機能範囲 (Functional Scope)

### 4.1 差分・表示ロジック

- **フィールド差分**: `LedgerContentProcessor` による型別整形（テキスト/数値/日付/選択肢/スコア等）。
- **検索ハイライト**: `highlight` キーワードに基づくヒット箇所の視覚的強調。
- **表示レベル準拠**: ユーザーが選択した表示レベルに基づき、表示・非表示カラムを制御。

### 4.2 表示モード

> [!NOTE]
> [W1.4のシナリオB](2026-01-03_W1-4_persona_scenarios.md#12-シナリオb-過去の特定更新の内容確認任意比較なし)に対応

- **差分モード**: `comparisonTargetDiff` が指定されている場合、2バージョン間の差分をハイライト表示。
- **スナップショットモード**: `comparisonTargetDiff` が Null の場合、差分ハイライトなしで単一バージョンの内容を表示。
  - 基本情報タブと同様の表示レベル・グループ開閉・Auto Link・タグ表示を適用。
  - 実務担当者が「過去の状態をそのまま確認したい」ニーズに対応。

### 4.3 任意比較UI

> [!NOTE]
> **PM承認結果**: 任意比較UIは全ユーザーに公開する

- **デフォルト表示**: 最新↔直前の差分表示（または単一バージョンのスナップショット表示）。
- **任意比較UI**: 全ユーザーに公開し、履歴一覧から任意の2バージョンを選択して比較できるようにする。
  - UI設計: 直感的でわかりやすいバージョン選択インターフェース（チェックボックスまたはドロップダウン）。
  - 初回利用時のガイド: ツールチップやヘルプテキストで使い方を説明し、実務担当者でも迷わず利用できるようにする。
  - 補助機能: 「最新と比較」「直前と比較」などのクイックアクションボタンを提供し、よくある操作を簡単に実行できるようにする。

### 4.4 ワークフローステータス差分（承認履歴）

> [!NOTE]
> **PM承認結果**: 承認履歴として表形式で表示し、表示中の内容がどの区間かを明示する

- **表示形式**: 現状の台帳詳細画面の更新履歴タブ内の表示内容に近い形式を採用。
- **表示内容**: 
  - バージョン番号（Ver. 1, Ver. 2, ...）
  - 更新日時
  - 更新者（アイコン + 名前）
  - ステータス（`WorkflowStatus` Enum: 下書き、申請中、承認済み等）
  - コメント（任意、更新理由やロールバック理由など）
- **差分表示時の強調**: 
  - 比較対象の2バージョンを表内でハイライト（背景色やボーダー）し、「この区間の変更内容を表示中」であることを視覚的に示す。
  - 例: Ver. 3 と Ver. 5 を比較している場合、表内の Ver. 3 と Ver. 5 の行をハイライト。
- **表示位置**: フィールド差分の上部に配置し、変更の文脈（誰がいつ何のために変更したか）を最初に確認できるようにする。

### 4.5 添付ファイル（過去バージョン含む）

- **永続的閲覧**: 添付ファイルはアップロード後システムからは削除されない運用を前提とする。
  - > [!WARNING]
    > 将来的な削除ポリシー導入時には「添付ファイル削除済み」の表示対応が必要（Phase 2以降で検討）。
- **過去バージョンの実体閲覧**: 過去バージョンの差分表示時も、当時の添付ファイルを直接閲覧（File Inspector連携）可能とする。
- **キー変換の透過性**: OCR処理後のファイル名変更（画像→PDF）が発生していても、論理的に同一ファイルであれば「変更なし」として扱う。

## 5. UI/UX デザイン方針 (DaisyUI & Modern Trend)

### デザイン原則

- **DaisyUI セオリー**: `Table`, `Badge`, `Collapse`, `Diff` 等、DaisyUIの標準コンポーネントをベースに、一貫性のあるスタイリングを適用。
- **最新UIトレンド**:
  - ステータス変更には直感的なバッジとタイムラインUIを採用。
  - 数値やテキストの変更箇所は、背景色だけでなく「以前の値を打ち消し線で表示」などの視覚的フィードバックを検討。
  - レスポンシブ性を損なわない、密度感のある情報配置。

### 構成案

- **埋め込みモード**: 余計な装飾を削ぎ落とし、親カードの「コンテンツ」として溶け込む。
- **履歴比較モード**: 左右2画面比較、または行内差分表示をユーザーが直感的に理解できるレイアウト。

## 6. PM確認事項（再検討結果）

以下の事項についてPMの確認・承認が必要です:

1. **表示レベルの同期**:
   - 検討結果: ユーザー設定（LocalStorage等）またはLivewireの親コンポーネント経由で、「基本情報タブ」と「更新履歴タブ」の表示レベルを完全に同期させる。

2. **ワークフローステータスの扱い**:
   - **PM承認結果**: 承認履歴として表形式で表示し、表示中の内容がどの区間（どのバージョン間）の内容かを明示する。
   - 現状の台帳詳細画面の更新履歴タブ内の表示内容に近い形式を採用。
   - 表には以下を含める: バージョン番号、更新日時、更新者、ステータス、コメント（任意）。
   - 差分表示時には、比較対象の2バージョンを表内でハイライトし、「この区間の変更内容」であることを視覚的に示す。

3. **基本情報タブの即時差分との共通化範囲**:
   - 同一コンポーネントで実装するか、別実装とするか。
   - 現状の推奨: 別実装を維持し、表示ロジック（`LedgerContentProcessor`）のみ共通化。

4. **任意比較UIの露出制御方針**:
   - **PM承認結果**: 全ユーザーに公開する。
   - 実務担当者にも任意比較機能を提供し、必要に応じて過去の任意2バージョンを比較できるようにする。
   - UIは直感的でわかりやすい設計とし、初回利用時にはツールチップやヘルプで使い方をガイドする。

5. **スナップショット表示モードの優先度**:
   - Phase 1で実装するか、Phase 2以降に延期するか。
   - 現状の推奨: Phase 1で実装（実務担当者の「過去の状態をそのまま確認したい」ニーズに対応）。

6. **検索ハイライトの優先度**:
   - Phase 1の必須機能とするか、オプション機能とするか。
   - 現状の推奨: Phase 1ではオプション機能として位置づけ、実装コストに応じてPhase 1.5またはPhase 2に延期も検討。
