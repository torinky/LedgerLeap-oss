# 秘密区分表示機能 基本仕様書

**作成日:** 2026-04-29
**最終更新:** 2026-04-30 (Rev.5)
**対象機能:** フォルダ・台帳ごとの「秘密区分」表示機能
**方針:** MVPとして「表示・注意喚起」に留め、アクセス権限制御とは切り離す

> 📄 **詳細仕様書はこちら**: [秘密区分表示機能 詳細仕様書](./2026-04-30_detailed_specification.md)

## 0. 前提：用語の整理（⚠️ 重要）

LedgerLeap のコードベースでは「台帳」という言葉が2層に分かれる。仕様書全体でこの区別を厳守する。

| 用語 | コード上のクラス | テーブル | 説明 |
|------|----------------|---------|------|
| **台帳定義** | `LedgerDefine` | `ledger_defines` | 台帳のテンプレート（カラム定義・ワークフロー設定を含む）。フォルダに属する。 |
| **台帳レコード** | `Ledger` | `ledgers`（Mroonga） | 台帳定義に基づいて登録された個別データ行。ワークフローステータスを持つ。 |
| **フォルダ** | `Folder` | `folders` | 台帳定義を格納する階層構造のコンテナ。 |

### 🔑 設計判断：秘密区分を持つ層

> **MVPでは「台帳定義（LedgerDefine）」と「フォルダ（Folder）」に秘密区分を持たせる。**
> 台帳レコード（Ledger）単位への設定はPhase 2以降とする。

**判断根拠:**

| 視点 | 台帳定義（推奨） | 台帳レコード |
|------|---------------|------------|
| 業務実態 | 「採用評価台帳」全体が極秘→台帳定義単位が自然 | レコードごとに機密性が異なるケースは稀 |
| UX | 台帳定義設定時に1回設定 | レコード登録ごとに設定が必要（操作が多い） |
| 一覧表示 | 台帳定義ヘッダーカードで表示可能 | 全レコード行にバッジが並ぶ（視認性が低下） |
| ワークフロー | `isLocked()` 制約なし（台帳定義はロック概念がない） | 承認済みレコード（`APPROVED`）は `isLocked()=true` で変更不可 |
| Mroongaリスク | InnoDB テーブルへの通常カラム追加 | Mroonga テーブルへのカラム追加（要注意事項あり） |

> **⚠️ 確認事項（実装前に合意が必要）:** 個別レコードごとに機密性が異なるユースケース（例：特定の応募者データのみ極秘）が想定される場合は、Ledger レコード層への追加も検討する。その場合はPhase 2として仕様を別途策定する。

## 2. 各小機能の概要

### 小機能1：マスタデータ管理

#### 1-1. 秘密区分の定義管理（設定ファイルベース）

**方針:** MVPではEnumによるコード固定ではなく、`config/confidentiality.php` で定義を管理する。

**理由:**
- システムの導入組織によって秘密区分の名称や段階が異なる（例：一部の組織では「社内限定・社外秘・極秘」の3段階、別の組織では「公開・社内・秘密・機密・極秘」の5段階など）。
- 設定ファイルであれば、デプロイ時に組織に合わせた柔軟な定義が可能。
- 運用中の追加・変更も設定ファイルの更新とキャッシュクリアで対応でき、DBマスタ化の運用コストをPhase 2まで先送りできる。

#### 1-2. 公開範囲の定義管理（DBベース：組織・ロール参照）

公開範囲は、既存の権限管理で使用している**組織（`organizations`）およびロール（`roles`）テーブル**をデータソースとして活用し、複数選択できる。

**設計方針:**
- 公開範囲はあくまで**ラベル表現**であり、フォルダや台帳定義の開示制限（権限）とは紐づかない。
- 理由：権限管理との混同を防ぎ、ユーザーの概念理解を容易にするため。将来的には権限管理との連携を検討するが、MVPでは独立した表示要素として扱う。
- **既存の権限管理スキーマをそのまま活用**：権限管理で開示制限の実装は既に行っているため、権限管理側の変更は行わない。
- **略称カラムの追加**：`organizations` テーブルおよび `roles` テーブルに `abbreviation`（略称）カラムを追加する（現スキーマには存在しないため）。
  - **⚠️ 運用注意**: マイグレーションと `$fillable` 追加だけでは不十分。管理画面（Organization・Role 編集フォーム）に `abbreviation` 入力欄を追加しないと、運用者が略称を設定できない。（Sprint 3A で対応予定）
- **UIは既存の複数選択コンポーネントを流用**：`<x-mary-choices>` などの既存複数選択UIを使用し、組織名・ロール名の正式名または略称を一覧表示して複数選択できるようにする。

**データソース:**

| データソース | テーブル | 正式名 | 略称 | 備考 |
|-----------|---------|-------|------|------|
| 組織 | `organizations` | `name` | `abbreviation`（新規追加） | `org_id` はAD/LDAP識別子として使用 |
| ロール | `roles` | `description` | `abbreviation`（新規追加） | `name` は Spatie Permission の内部識別子のため、表示には `description` を使用 |

**選択と保存の流れ:**
1. UI上で `<x-mary-choices>` コンポーネントを使用し、テナント配下の `organizations` と `roles` を統合したリストを表示する。
2. 各選択肢には正式名（`name` / `description`）または略称（`abbreviation`）を表示する。
3. 選択結果は `folders.confidentiality_scopes` / `ledger_defines.confidentiality_scopes` にJSON配列として保存する。
4. 保存形式：`{org_ids: [1, 2], role_ids: [3, 4]}` のような構造で、組織IDとロールIDを区別して保持する。

#### 1-3. スタンプ表示スタイルの統一定義

**方針:** 秘密区分ごとの色分けやアイコン表現を廃止し、全区分で統一のスタンプスタイルを適用する。

**理由:**
- すべての秘密区分に対して同じように注意を払う必要がある。色分けやアイコン表現では、ユーザーの注意が「極秘」などの強い色に偏り、「社内制限」などを見落とすリスクがある。
- 典型的な秘密区分スタンプ（太枠・太字・大文字）は、紙の文書において長年実績のある視認性の高い表現であり、デジタル画面でも同じ認知パターンを活用できる。

**統一スタイル仕様:**
- 色は赤（`#dc2626`）のみ使用し、秘密区分に関わらず統一する。
- アイコンは使用しない。
- 文字は太字（`font-weight: 700`）、枠線は太め（`3px`）で、文字サイズは周囲より大きめに設定する。

### 小機能2：フォルダへの秘密区分設定

- **2-1. フォルダ作成時**: `Livewire\Folder\FolderForm` コンポーネントの `create` モードに秘密区分セレクトと公開範囲マルチセレクトを追加。未選択（`null`）を許可。
- **2-2. フォルダ編集時**: 同コンポーネントの `edit` モードで変更可能。`Folder` モデルに `LogsActivity` が適用済みのため、`$fillable` に追加するだけで変更ログが自動記録される。
- **2-3. フォルダ階層における秘密区分の解決・表示**: フォルダツリー表示時、各フォルダが自身の秘密区分を持たない場合は、親フォルダの秘密区分を参照する（**継承**）。ただし、子の設定がある場合は子を優先する（**上書き**）。

**継承・解決ルール:**

| 階層 | 自身の設定 | 親の設定 | 表示される秘密区分 |
|------|-----------|---------|----------------|
| ルートフォルダ | `null` | なし | 表示なし |
| ルートフォルダ | `極秘` | なし | `極秘` |
| 子フォルダ | `null` | `社外秘` | `社外秘`（親を継承） |
| 子フォルダ | `極秘` | `社外秘` | `極秘`（自身が優先） |

**理由:** フォルダ階層は上位ほど広い範囲、下位ほど細かい範囲に情報を分割する傾向がある。より細かい範囲で厳しい情報コントロールを適用できるよう、子の設定を優先する。

### 小機能3：台帳定義への秘密区分設定

- **3-1. 台帳定義作成時**: `Livewire\LedgerDefine\Create` コンポーネントのフォームに秘密区分セレクトと公開範囲マルチセレクトを追加。デフォルトは `null`（未設定）。
- **3-2. 台帳定義編集時**: `Livewire\LedgerDefine\Edit` コンポーネントで変更可能。`LedgerDefine` モデルに `LogsActivity` が適用済みのため自動ログ記録される。
- **3-3. 台帳定義保存時の永続化**: `ledger_defines` テーブルに `confidentiality_level`（string, nullable）と `confidentiality_scopes`（json, nullable）カラムを追加（`InnoDB` テーブルのため通常の `Schema::table` マイグレーションで対応可能）。既存の `column_define` や検索インデックスとは完全に独立する。

### 小機能4：画面全体での秘密区分スタンプ表示

**方針:** 一覧・詳細画面の各所に散らばっていたバッジ表示を廃止し、**全画面の右上に統一されたオーバーレイスタンプ**を表示する。

**理由:**
- バッジが一覧の各行やフォルダツリーの各ノードに散在していると、ユーザーが秘密区分を確認するために目を移動させる必要があり、見落としのリスクが高まる。
- 全画面で右上の同じ位置にオーバーレイ表示することで、どの画面にいても一貫した位置で秘密区分を確認でき、認知負荷を下げられる。
- 紙の文書における「押印」の概念に近いため、ユーザーが直感的に「これは重要な表示だ」と認識しやすい。

#### 4-1. 全画面右上へのオーバーレイスタンプ表示

- スタンプは固定位置（`position: fixed; top-right`）で表示され、スクロールしても追従する。
- `pointer-events: auto` を指定し、ツールチップ表示や保守動線リンクのクリックを受け付ける。
- 表示対象は現在表示中のコンテンツ（フォルダまたは台帳定義）の解決後の秘密区分と公開範囲。

#### 4-2. フォルダ・台帳定義一覧でのスクロール連動スタンプ表示

台帳一覧では、1画面に複数の台帳定義セクションが縦に並び、各セクション内に台帳レコードのテーブルがある。ユーザーがスクロールしている間、**画面内で最も面積を占めている台帳定義セクション**の秘密区分を右上スタンプに反映する。

**実装段階と選択肢:**
- **簡易版（Phase 1）**: `fixed` 配置のスタンプで `currentFolder` の秘密区分を常に表示。実装コストが低く、単一フォルダ表示では十分。
- **完全版（Phase 1.5 / Sprint 3A）**: **Intersection Observer API** を使用し、スクロール中の最前面セクションの秘密区分にスタンプを動的切り替え。複数台帳定義が縦に並ぶ画面で必要。

**Intersection Observer ロジック（完全版）:**
- スクロールイベントのポーリングを避け、ブラウザのネイティブ実装によりメインスレッドの負荷を最小化する。
- **複数閾値（`threshold: [0, 0.25, 0.5, 0.75, 1.0]`）** を設定し、セクションの画面内割合を細かく捉えつつ、イベント発火頻度を制御する。
- **Livewire への通知は変更時のみ** とし、同じセクションがスクロール中に何度も通知されることを防ぐ。

**理由:** ユーザーがスクロール中にどの台帳定義を見ているか、スタンプだけで直感的に把握でき、誤って違う台帳定義の秘密区分で判断してしまうことを防ぐ。パフォーマンス確保のためメインスレッドポーリングを避ける。

#### 4-3. 台帳レコード詳細画面でのスタンプ表示

台帳レコード詳細画面では、紐づく台帳定義の秘密区分と公開範囲を解決して右上スタンプに表示する。
- 台帳定義に秘密区分が設定されていれば、それを表示
- 台帳定義に設定がなく、親フォルダに設定があれば、親フォルダの秘密区分を継承して表示
- どちらも未設定の場合 → 表示なし

**解決ロジック（フォルダ・台帳定義間）:**

```php
// 概念例
function resolveConfidentiality($ledgerDefine): ?array
{
    // 1. 台帳定義自身の設定を確認
    if ($ledgerDefine->confidentiality_level) {
        return [
            'level'  => $ledgerDefine->confidentiality_level,
            'scopes' => $ledgerDefine->confidentiality_scopes,
        ];
    }

    // 2. 親フォルダの設定を確認（再帰的に遡る）
    $folder = $ledgerDefine->folder;
    while ($folder) {
        if ($folder->confidentiality_level) {
            return [
                'level'  => $folder->confidentiality_level,
                'scopes' => $folder->confidentiality_scopes,
            ];
        }
        $folder = $folder->parent;
    }

    return null;
}
```

**理由:** 台帳定義はフォルダに属しており、フォルダ全体に適用された秘密区分を個別の台帳定義が上書きできることで、運用の柔軟性を確保する。未設定時は親フォルダの設定を継承することで、設定漏れを防ぎつつ管理上の冗長性を減らす。

#### 4-4. スタンプツールチップ（由来表示）

スタンプにホバーすると、ツールチップが表示され、現在表示されている秘密区分が**どのフォルダまたは台帳定義から由来するものか**を明確に示す。

**表示ルール:**
- **直接設定の場合**: 「設定元：○○台帳定義」と表示。
- **継承の場合**: 「設定元：△△フォルダ（継承）」と表示。継承元が親フォルダであることを明示する。
- **複数階層継承の場合**: 「設定元：□□フォルダ > △△フォルダ（継承）」と、継承パスを階層的に表示する。

**理由:** 秘密区分は人が維持保守する必要がある。ユーザーが「この秘密区分はどこで設定されているのか」を瞬時に把握できなければ、設定の修正や見直しが行いにくくなる。ツールチップで由来を明示することで、保守作業の効率を高める。

#### 4-5. 保守動線（設定画面への遷移）

秘密区分は人が維持保守する必要があるため、**常に保守のための動線を確保**する。

**設計方針:**
- スタンプツールチップ内に、秘密区分の設定画面への遷移リンクを常に表示する。
- 権限がないユーザーにはリンクを非表示とし、表示だけは可能とする。
- 台帳定義由来の場合 → `ledger-define/{id}/edit` へ遷移
- フォルダ由来の場合 → `folder/{id}/edit` へ遷移

**理由:** 秘密区分の設定が散在していると、管理者が設定を見直す際に探し回る必要が生じ、運用負荷が高まる。スタンプから直接設定画面へ遷移できるようにすることで、設定の保守性を確保し、継続的な運用品質を維持する。

## 4. 非機能要件・制約

### 4.1 基本方針

- **表示（注意喚起）機能として実装**: 既存の権限管理（Spatie Permission + FolderPermission）とは独立。
- **既存機能への影響最小化**: 全文検索（Mroonga）、ソート、フィルタ、ワークフロー、活動ログなどに影響を与えない設計。
- **既存データの互換性**: 既存のフォルダ・台帳定義には秘密区分が `null` のまま。未設定時はスタンプを非表示とし、既存と同じ見た目を維持する。

### 4.2 デザイン・UI制約

- **統一スタイルのみ使用**: 秘密区分に関わらず、全て赤枠・太字・大文字の統一スタンプを使用。色分けやアイコン表現は行わない。
- **固定位置オーバーレイ**: 全画面の右上（`top: 1rem; right: 1rem`）に固定表示。`z-index: 50` で最前面に配置し、`pointer-events: auto` でツールチップ・保守動線のクリックを受け付ける。
- **スクロール連動のパフォーマンス確保**: Intersection Observer API を使用し、メインスレッドでのポーリングを避ける。複数閾値 (`[0, 0.25, 0.5, 0.75, 1.0]`) で検知し、Livewire への通知は変更時のみに制限する。
- **MaryUIコンポーネントの優先使用**: セレクトボックスは `<x-mary-select>`、マルチセレクトは `<x-mary-choices>`、ツールチップは `<x-mary-tooltip>`。
- **翻訳キーの使用**: UIテキストはすべて `__('ledger.xxx')` 形式。自然言語の直書きは禁止。

### 4.3 データモデル概要

#### 追加カラム

| テーブル | カラム名 | 型 | 備考 |
|---------|---------|-----|------|
| `folders` | `confidentiality_level` | string, nullable | 秘密区分コード値 |
| `folders` | `confidentiality_scopes` | json, nullable | 公開範囲（組織ID・ロールIDのJSONオブジェクト） |
| `ledger_defines` | `confidentiality_level` | string, nullable | 秘密区分コード値 |
| `ledger_defines` | `confidentiality_scopes` | json, nullable | 公開範囲（組織ID・ロールIDのJSONオブジェクト） |

- `ledgers` テーブル（Mroonga）への変更は**不要**（秘密区分は台帳定義層で管理）。
- `nullable()` を必須とし、既存データへの互換性を保証する。

#### 略称カラム追加（Rev.5）

現スキーマに略称カラムがないため、以下のマイグレーションを追加する：

| テーブル | 追加カラム | 型 | 備考 |
|---------|-----------|-----|------|
| `organizations` | `abbreviation` | string, nullable | 組織略称。未設定時は `name`（正式名）を表示 |
| `roles` | `abbreviation` | string, nullable | ロール略称。未設定時は `description`（正式名）を表示 |

> 📄 **詳細なマイグレーション・モデル変更は詳細仕様書 §4.4 を参照**

### 4.4 テスト・品質

- **テスト実行環境**: `./vendor/bin/sail test` でのみ実行（ホスト側 `php artisan test` 禁止）。
- **データベーストレイト**: FTSを含まない場合は `RefreshDatabase`、Mroonga関連テストは `DatabaseMigrationsOnce` を使用。本機能は `ledger_defines`/`folders` の InnoDB テーブルのみ操作するため `RefreshDatabase` で可。
- **必須テスト**:
  - `tests/Unit/Services/ConfidentialityLevelServiceTest.php` — 設定ファイル読み込み・解決・フォールバックのテスト
  - `tests/Feature/Livewire/Folder/FolderFormTest.php` — 秘密区分・公開範囲セレクトの保存・バリデーションテスト
  - `tests/Feature/Livewire/LedgerDefine/LedgerDefineFormTest.php` — 台帳定義フォームの同上
  - `tests/Feature/Components/ConfidentialityStampTest.php` — スタンプコンポーネントのレンダリング・継承解決テスト
- **テナント初期化**: Featureテストの `setUp()` で必ず `tenancy()->initialize($tenant)` を呼び出す。

### 4.5 MCP ツールへの影響範囲

- **MVPでは MCP ツールへの変更は行わない**。
- `GetLedgerDetailTool`, `SearchLedgersTool` 等が返すレスポンスへの秘密区分情報の追加は Phase 2 で検討。
- 理由: 秘密区分情報の MCP 経由公開はクライアント向け契約の変更を伴い、`client-facing-contract-maintenance` エージェントによる別途レビューが必要。
