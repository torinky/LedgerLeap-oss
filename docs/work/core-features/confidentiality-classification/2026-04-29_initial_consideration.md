# 秘密区分表示機能 初度検討ドキュメント

**作成日:** 2026-04-29
**対象機能:** フォルダ・台帳ごとの「秘密区分」表示機能

---

## 1. 背景と目的

LedgerLeapは多様な台帳データを管理するシステムであり、アクセス権限（閲覧制限）は既にシステムとして実装済みである。
しかし、ユーザーが一度閲覧した情報をメールやチャットなどで外部に転送する場合、判断材料的乏しく、誤って機密情報を共有してしまうリスクがある。
本検討では、フォルダおよび台帳に「秘密区分（例：公開、社外秘、極秘など）」を視覚的に表示し、ユーザーが転送・共有時の判断を誤りなく行えるようにするための機能を対象とする。

## 2. ペルソナ定義

### ペルソナ1: データガバナンス管理者 / セキュリティ担当者
- **氏名:** 山本 太郎（45歳、情報セキュリティ部）
- **業務:** 全社的なデータのアクセス権限管理、情報漏洩対策の推進。
- **課題:** 現場部門が作成した台帳の機密性が一目でわからず、監査やポリシー適用の確認に時間がかかる。
- **期待:** システム上で各データの秘密区分が明確に定義・可視化され、情報取扱のルールが自然と守られること。

### ペルソナ2: 一般ユーザー（現場の台帳管理者・利用者）
- **氏名:** 鈴木 花子（32歳、人事部 採用担当）
- **業務:** 採用応募者の個人情報（極秘）から、社内向けイベントの案内（社内公開）まで、様々な台帳を管理・共有する。
- **課題:** どの台帳がどこまで共有可能か、パッと見で判断できず不安になる。誤って個人情報の台帳リンクを全社に送ってしまうリスクを恐れている。
- **期待:** 台帳一覧や詳細画面で「これは極秘」「これは公開」といったバッジやアイコンが目立ち、直感的に取り扱いレベルがわかること。

## 3. ユーザーシナリオと要件抽出

### シナリオ1: 機密性の高い新しい台帳の作成
**状況:** 鈴木（一般ユーザー）が、新しい採用候補者の評価台帳を作成する。
**行動:** 台帳作成画面で、秘密区分として「極秘（Strictly Confidential）」を選択して保存する。
**結果:** 作成された台帳の名前の横に、赤色などで目立つ「極秘」バッジが表示されるようになる。
**【抽出要件】**
- フォルダ・台帳の作成/編集画面に「秘密区分」の選択項目を追加する。
- 秘密区分のマスタ（あるいはシステム固定の選択肢）を定義する（例：公開、社内制限、社外秘、極秘）。

### シナリオ2: 日常的な台帳の閲覧と共有判断
**状況:** 鈴木が他の部署のメンバーから「先月のイベントアンケート結果の台帳を見せてほしい」と頼まれる。
**行動:** LedgerLeapで該当のフォルダを開く。フォルダには「社内公開」のバッジがついている。
**結果:** 鈴木は安心して台帳のURLを共有できる。もし「部外秘」となっていた場合は共有を思いとどまることができる。
**【抽出要件】**
- 左側のフォルダツリー、および右側の台帳一覧（カード/リスト）において、秘密区分を示す視覚的インジケーター（アイコンやカラーバッジ）を表示する。
- ユーザーがデータを取り扱う際の心理的ハードルを適切に設定するため、区分ごとにカラーリング（例: 赤＝極秘、緑＝公開）を変える。

### シナリオ3: 台帳内容をコピーしてメール転送时的判断
**状況:** 鈴木（一般ユーザー）が、他部署のメンバーから「先月のイベントアンケート結果をメールしてほしい」と頼まれた。
**行動:** LedgerLeapで該当の台帳を開く。画面には「社内公開」のバッジが表示されているが、台帳の詳細や添付文件的にも「社内公開」かどうかが一目でわからない。バッジを確認して、「この内容ならメールしてよい」と判断する。
**結果:** 鈴木は安心して台帳の内容をコピーしてメール送信できる。もしバッジが「社外秘」や「極秘」であれば、コピーの前に「メール転送禁止では？」と判断できる。
**【抽出要件】**
- 台帳詳細画面（内容表示部分）にも、フォルダや一覧と同じ秘密区分バッジを同一位置上に表示する。
- バッジはコピー元の信息来源（フォルダ・台帳）が明確になるよう、バッジ自体に「機密区分名」のみならず「対象」（例：この台帳、このフォルダ）と紐づいていることを明確にする。
- UI上一続きの操作感で、「バッジを見て→判断して→アクションに移す」を阻断なく行えるデザインとする。

### シナリオ4: フォルダ階層と機密性の矛盾の防止
**状況:** 山本（管理者）がセキュリティ監査を行っている際、「公開」フォルダの中に「極秘」の台帳が混ざっているのを発見する。
**行動:** フォルダ自体を「極秘」に変更しようとする。
**結果:** フォルダの秘密区分が、内部の台帳の秘密区分と矛盾しないように管理される。
**【抽出要件】**
- （高度な要件・要検討）親フォルダの秘密区分と子台帳の秘密区分の間に制約を設けるか（例：親が「社外秘」なら、子は「公開」にできない等）、あるいは単なるラベルとして運用するか、運用ルールのシステム的担保について検討が必要。

### シナリオ5: 複数組織間の秘密情報混同の防止
**状況:** 鈴木（一般ユーザー兼、他組織の連携担当）が、A社との取引台帳とB社との取引台帳を両方管理している。两个組織の「極秘」情報が混在しているが、それぞれどの組織の機密情報かが一目ではわからない。
**行動:** 鈴木が台帳一覧を開く。組織と秘密区分の両方を示すバッジ（例：「A社・極秘」「B社・極秘」）がラベル表示されている。
**結果:** 鈴木は見ている台帳がどの組織の情報を含むかを瞬時に判別でき、A社の極秘情報をB社に誤って転送してしまうことを防止できる。
**【抽出要件】**
- 秘密区分バッジには「どの組織の機密情報か」という組織識別子を附属できる。
- バッジのレイアウトは「組織名＋秘密区分」の組み合わせ、または組織ごとに色を替えるなど、複数の組織が混在する環境下でも瞬時に識別できるデザインとする。
- 組織と秘密区分の両方を一目で確認できるよう、バッジは省略せず十分な情報を表示する（例：「極秘」だけなく「A社・極秘」と表示）。

### シナリオ6: 組織間プロジェクトでの秘密区分の使い分け
**状況:** 鈴木が、X社とのNMR共同研究プロジェクトと、Y社との個別取引プロジェクトを兼務している。それぞれのプロジェクトで「社外秘」の意味が異なり、NMR共同研究の方では「NMRプロジェクト内に限定して共有可」、個別取引の方では「社内限定共有」と異なる制御したい。
**行動:** 鈴木が各プロジェクトの台帳に「社外秘」を設定すると、半自動で組織属性とプロジェクト属性を付与する。バッジには「社外秘」「NMR」「個別取引」などの属性が組み合わせて表示される。
**結果:** 鈴木はバッジを見て、「この台帳の社外秘はNMRプロジェクト内に限る」「この台帳の社外秘は社内限り」と即座に判断でき、適切な共有先選択ができる。
**【抽出要件】**
- 秘密区分单一ではなく、「組織＋プロジェクト＋秘密区分」の3轴で情報を識別できるUIとする。
- バッジだけでなく、台帳詳細やメール通知にも組織・プロジェクト属性を明示する。
- 転送・共有時の注意唤起として、バッジ情報はメール通知や共有確認ダイアログにも反映させる。

## 5. 検討結果から導かれる必要要件サマリー

ペルソナ・シナリオを通じて、以下の要件が必要であることが確認できた。

1. **データモデルの拡張**
   - `folders` テーブルおよび `ledger_defines`（台帳定義）テーブルに `confidentiality_level` カラムを追加する。
   - ⚠️ `ledgers` テーブルはMroongaエンジンのため変更リスクが高い。MVPでは台帳定義層に秘密区分を持たせることで `ledgers` テーブルへの変更を回避する。（詳細は仕様書 §0, §4.4 参照）
2. **UI/UXの改修**
   - 作成/編集フォームへの項目追加（FolderForm、LedgerDefine Create/Edit）。
   - フォルダツリー（`badge-xs`）、台帳定義ヘッダーカード、台帳レコード行の右端バッジ領域、台帳詳細画面ヘッダーへの秘密区分バッジ表示。
   - Mary UI / daisyUI を活用した区分ごとのカラーテーマ（`badge-success`/`badge-info`/`badge-warning`/`badge-error`）。
3. **ビジネスロジックの検討（Next Step）**
   - フォルダ階層における秘密区分の継承や制約ルールは Phase 2 で検討。
   - MVPでは「アクセス権限（パーミッション）」とは切り離し、「表示（注意喚起）」機能のみ実装する。

## 6. シナリオの優先度整理（Rev.2追記）

| シナリオ | MVPスコープ | 理由 |
|---------|-----------|------|
| 1: 機密性の高い台帳の作成 | ✅ 対応 | 台帳定義作成時の秘密区分選択で実現 |
| 2: 日常的な閲覧と共有判断 | ✅ 対応 | フォルダツリー・台帳一覧のバッジ表示で実現 |
| 3: メール転送時の判断 | ✅ 部分対応 | 台帳詳細画面のバッジ表示で実現。転送阻止ダイアログはPhase 2 |
| 4: フォルダ階層と機密性の矛盾防止 | ⏳ Phase 2 | 矛盾検知ロジックが必要。MVPは表示のみ |
| 5: 複数組織間の混同防止 | ⏳ Phase 2 | 組織識別子との複合バッジが必要 |
| 6: 組織間プロジェクトでの使い分け | ⏳ Phase 2 | 3軸(組織+プロジェクト+秘密区分)の複合表示が必要 |
