# 秘密区分表示機能 初回検討ドキュメント

**作成日:** 2026-04-29
**最終更新:** 2026-04-30 (Rev.4対応)
**対象機能:** フォルダ・台帳ごとの「秘密区分」表示機能

---

## 1. 背景と目的

LedgerLeapは多様な台帳データを管理するシステムであり、アクセス権限（閲覧制限）は既にシステムとして実装済みである。
しかし、ユーザーが一度閲覧した情報をメールやチャットなどで外部に転送する場合、判断材料が少なく、誤って機密情報を共有してしまうリスクがある。
本検討では、フォルダおよび台帳に「秘密区分（例：公開、社外秘、極秘など）」を視覚的に表示し、ユーザーが転送・共有時の判断を誤りなく行えるようにするための機能を対象とする。

## 2. ペルソナ定義

### ペルソナ1: データガバナンス管理者 / セキュリティ担当者
- **氏名:** 山本 太郎（45歳、情報セキュリティ部）
- **業務:** 全社的なデータのアクセス権限管理、情報漏洩対策の推進。
- **課題:** 現場部門が作成した台帳の機密性が一目でわからず、監査やポリシー適用の確認に時間がかかる。
- **期待:** システム上で各データの秘密区分が明確に定義・可視化され、情報取扱のルールが自然と守られること。

### ペルソナ2: 一般ユーザー（現場の台帳管理者・利用者）
- **氏名:** 鈴木 花子（32歳、人事部 採用担当）
- **業務:** 採用応募者の個人情報（極秘）から、社内向けイベントの案内（社内公開）まで、様々な台帳を管理・共有する。
- **課題:** どの台帳がどこまで共有可能か、一目で判断できず不安になる。誤って個人情報の台帳リンクを全社に送ってしまうリスクを恐れている。
- **期待:** 台帳一覧や詳細画面で「これは極秘」「これは公開」といった表示が目立ち、直感的に取り扱いレベルがわかること。

## 3. ユーザーシナリオと要件抽出

### シナリオ1: 機密性の高い新しい台帳の作成
**状況:** 鈴木（一般ユーザー）が、新しい採用候補者の評価台帳を作成する。
**行動:** 台帳作成画面で、秘密区分として「極秘」を選択し、公開範囲として「人事部」のみを設定して保存する。
**結果:** 作成された台帳を表示する全画面の右上に、赤い太枠のスタンプで「極秘・人事部」と表示されるようになる。
**【導出要件】**
- フォルダ・台帳の作成/編集画面に「秘密区分」と「公開範囲」の選択項目を追加する。
- 秘密区分はシステム導入組織ごとに異なるため、設定ファイルで定義・変更可能とする。
- 公開範囲は既存の権限管理スキーマ（`organizations`・`roles` テーブル）をデータソースとして活用し、組織名またはロール名の正式名・略称から複数選択できるようにする。

### シナリオ2: 日常的な台帳の閲覧と共有判断
**状況:** 鈴木が他の部署のメンバーから「先月のイベントアンケート結果の台帳を見せてほしい」と頼まれる。
**行動:** LedgerLeapで該当のフォルダを開く。画面右上のスタンプに「社内制限・全社員」と表示されている。
**結果:** 鈴木は安心して台帳のURLを共有できる。もし「極秘・人事部」と表示されていた場合は共有を思いとどまることができる。
**【導出要件】**
- 全画面の右上の同じ位置に、秘密区分と公開範囲を示す統一スタンプをオーバーレイ表示する。
- スタンプは太赤枠・太字・大文字で、すべての秘密区分で同じ視覚的強調を行う（色分けやアイコン表現は行わない）。
- ユーザーがデータを取り扱う際の心理的ハードルを適切に設定するため、どの画面にいても一貫した位置で秘密区分を確認できるようにする。

### シナリオ3: 台帳内容をコピーしてメール転送時の判断
**状況:** 鈴木（一般ユーザー）が、他部署のメンバーから「先月のイベントアンケート結果をメールしてほしい」と頼まれた。
**行動:** LedgerLeapで該当の台帳を開く。画面右上のスタンプに「社内制限・全社員」と表示されている。スタンプを確認して、「この内容ならメールしてよい」と判断する。
**結果:** 鈴木は安心して台帳の内容をコピーしてメール送信できる。もしスタンプが「社外秘・経営層」や「極秘・人事部」であれば、コピーの前に「メール転送禁止では？」と判断できる。
**【導出要件】**
- 台帳詳細画面でも、他の画面と同じく右上に統一スタンプを表示する。
- スタンプには「秘密区分名」と「公開範囲」の両方を含め、情報の取り扱い範囲が明確になるようにする。
- UI上で一貫した操作感で、「スタンプを見て→判断して→アクションに移す」を遮断なく行えるデザインとする。

### シナリオ4: フォルダ階層と機密性の継承・上書き
**状況:** 山本（管理者）がセキュリティ監査を行っている際、「社外秘」に設定されたフォルダの中に、一部の台帳定義が「極秘」に設定されているのを発見する。
**行動:** 確認すると、その台帳定義は特別な人事評価データを扱っており、親フォルダよりも細かい範囲で厳しい管理が必要だった。
**結果:** 親フォルダの「社外秘」設定よりも、台帳定義自身の「極秘」設定が優先され、画面右上には「極秘・人事部」と表示される。
**【導出要件】**
- フォルダ階層において、子フォルダの秘密区分設定が親フォルダの設定を上書きする。
- 台帳定義に秘密区分が設定されていれば、親フォルダの設定よりも台帳定義の設定を優先する。
- より細かい粒度で厳しい情報コントロールを適用できるよう、粒度の細かい側の設定を採用する。

### シナリオ5: 複数組織間の機密情報混同の防止
**状況:** 鈴木（一般ユーザー兼、他組織の連携担当）が、A社との取引台帳とB社との取引台帳を両方管理している。2つの組織の「極秘」情報が混在しているが、それぞれどの組織の機密情報かが一目ではわからない。
**行動:** 鈴木が台帳一覧を開く。画面右上のスタンプには「極秘・A社」と表示されている。
**結果:** 鈴木は見ている台帳がA社向けの機密情報であることを瞬時に判別でき、A社の極秘情報をB社に誤って転送してしまうことを防止できる。
**【導出要件】**
- 公開範囲としてDB内の組織名（`organizations`）を設定できるようにし、スタンプに「秘密区分・組織名」の形で表示する。
- 複数の組織が混在する環境下でも、スタンプを見ればどの組織の情報かを瞬時に識別できるようにする。

### シナリオ6: 組織間プロジェクトでの秘密区分の使い分け
**状況:** 鈴木が、X社とのNMR共同研究プロジェクトと、Y社との個別取引プロジェクトを兼務している。それぞれのプロジェクトで「社外秘」の意味が異なり、NMR共同研究の方では「NMRプロジェクト内に限定して共有可」、個別取引の方では「社内限定共有」と異なる制御をしたい。
**行動:** 鈴木が各プロジェクトの台帳に「社外秘」を設定し、公開範囲としてNMRプロジェクトの場合は「NMR共同研究プロジェクト」を、個別取引の場合は「社内限定」を選択する。
**結果:** 鈴木は画面右上のスタンプを見て、「この台帳の社外秘はNMRプロジェクト内に限る」「この台帳の社外秘は社内限り」と即座に判断でき、適切な共有先選択ができる。
**【導出要件】**
- 秘密区分と公開範囲を組み合わせて、情報の取り扱い範囲を詳細に表現できるようにする。
- 公開範囲はDB内の組織・ロールリストから選択し、あくまでラベル表現にとどめ、フォルダや台帳定義の開示制限（権限）とは紐づかないようにする。
- 将来的には権限管理との連携も検討するが、MVPでは概念の混同を防ぐため独立した表示要素とする。

### シナリオ7: 台帳一覧画面でのスクロール中の秘密区分確認
**状況:** 鈴木が台帳一覧画面をスクロールしており、1画面に「イベントアンケート（社内制限・全社員）」「採用候補者評価（極秘・人事部）」「研修資料（公開・全社員）」など、複数の台帳定義セクションが縦に並んでいる。
**行動:** 鈴木がスクロールすると、画面右上のスタンプが自動的に変化する。現在画面内で最も大きく表示されている「採用候補者評価」のセクションに合わせて、スタンプが「極秘・人事部」に変わる。
**結果:** 鈴木はどの台帳定義を現在見ているのか、スタンプだけで直感的に把握でき、誤って違う台帳定義の秘密区分で判断してしまうことを防げる。
**【導出要件】**
- 台帳一覧画面では、画面内で最も面積を占めている台帳定義の秘密区分を、右上スタンプに自動的に反映する。
- スクロール検知は Intersection Observer API を使用し、パフォーマンスを確保する（メインスレッドの負荷を避けるため）。

### シナリオ8: 秘密区分の由来確認
**状況:** 山本（管理者）が監査を行い、「採用候補者評価」台帳定義に「極秘」が設定されていることを確認した。しかし、この「極秘」は台帳定義自身に設定されているのか、それとも親フォルダの「人事部フォルダ」から継承されているのかがわからない。
**行動:** 山本は画面右上の「極秘・人事部」スタンプにマウスを乗せると、ツールチップが表示され、「設定元：採用候補者評価（台帳定義）」と表示される。
**結果:** 山本はこの「極秘」が台帳定義自身に直接設定されていることを瞬時に把握でき、監査記録に正確に記載できる。もし継承されていた場合は、「設定元：人事部フォルダ（継承）」と表示される。
**【導出要件】**
- スタンプにホバーした際のツールチップで、秘密区分がどのフォルダまたは台帳定義から由来するかを明示する。
- 継承された場合は継承元を階層的に表示し、直接設定の場合は直接設定であることを明示する。
- 保守作業の効率化のため、由来情報は瞬時に把握できるようにする。

### シナリオ9: 秘密区分の保守・設定変更
**状況:** 山本（管理者）が監査の結果、「採用候補者評価」台帳定義の秘密区分を「極秘」から「社外秘」に変更する必要があると判断した。
**行動:** 山本は画面右上の「極秘・人事部」スタンプにマウスを乗せ、ツールチップ内の「設定を変更」リンクをクリックする。リンク先の台帳定義編集画面が開き、秘密区分のフォーム項目まで自動的にスクロールされる。
**結果:** 山本は台帳定義の編集画面で秘密区分を「社外秘」に変更し、保存する。活動ログに変更記録が残り、監査追跡が可能となる。
**【導出要件】**
- スタンプツールチップ内に、秘密区分の設定画面への遷移リンクを常に配置する。
- リンクは権限があるユーザーにのみ表示し、権限がないユーザーには非表示とする。
- 秘密区分は人が維持保守する必要があるため、常に保守のための動線を確保する。

## 5. 検討結果から導かれる必要要件サマリー

ペルソナ・シナリオを通じて、以下の要件が必要であることが確認できた。

1. **データモデルの拡張**
   - `folders` テーブルおよび `ledger_defines`（台帳定義）テーブルに `confidentiality_level`（string, nullable）と `confidentiality_scopes`（json, nullable）カラムを追加する。
   - ⚠️ `ledgers` テーブルはMroongaエンジンのため変更リスクが高い。MVPでは台帳定義層に秘密区分を持たせることで `ledgers` テーブルへの変更を回避する。（詳細は仕様書 §0, §4.4 参照）

2. **マスタデータ管理**
    - 秘密区分はシステムの導入組織によって異なるため、`config/confidentiality.php` 設定ファイルで定義・変更可能とする。
    - 公開範囲は既存の権限管理スキーマ（`organizations`・`roles` テーブル）をデータソースとして活用し、DB内のリストから組織名・ロール名の正式名または略称を複数選択できるようにする。
    - 現スキーマに略称カラム（`abbreviation`）がないため、`organizations` テーブルと `roles` テーブルに追加する。
    - UIは既存の複数選択コンポーネント（`<x-mary-choices>` など）を流用する。

3. **UI/UXの改修**
    - 作成/編集フォームへの項目追加（FolderForm、LedgerDefine Create/Edit）：秘密区分セレクト + 公開範囲マルチセレクト。
    - 全画面右上への統一スタンプオーバーレイ表示：太赤枠・太字・大文字で「秘密区分・公開範囲」を明示。
    - 色分けやアイコン表現は行わず、すべての秘密区分で同じ視覚的強調を適用する。
    - 台帳一覧画面でのスクロール連動：画面内で最も面積を占める台帳定義の秘密区分を右上スタンプに自動反映（Intersection Observer API でパフォーマンス確保）。
    - スタンプツールチップ：ホバー時に秘密区分の由来（どのフォルダまたは台帳定義から設定されたか）を明示。
    - 保守動線：ツールチップ内に設定画面への遷移リンクを常に配置し、保守性を確保。

4. **ビジネスロジックの検討（MVP）**
    - フォルダ・台帳定義間の秘密区分解決ルール：粒度が細かい側が上書き。子フォルダ > 親フォルダ、台帳定義 > 親フォルダ。
    - 公開範囲はあくまでラベル表現であり、開示制限（権限）とは紐づかない。
    - MVPでは「アクセス権限（パーミッション）」とは切り離し、「表示（注意喚起）」機能のみ実装する。

5. **拡張検討（Phase 2以降）**
    - 親フォルダとの秘密区分矛盾警告。
    - 公開範囲と開示制限（権限）の連携・自動制御。
    - 転送・共有時の注意喚起ダイアログ。
    - 管理画面からの秘密区分・公開範囲動的編集。

## 7. 次のアクション
- **基本仕様書 §0 の設計判断（台帳定義 vs 台帳レコード）**を関係者・ユーザーと合意する（[基本仕様書](./2026-04-30_basic_specification.md)参照）。
- **設定ファイルの秘密区分定義**を導入組織の運用ルールに合わせて確定する。
- **DBの組織・ロール略称カラム追加**：`organizations` テーブルと `roles` テーブルに `abbreviation` カラムを追加するマイグレーションを実装・適用する（[詳細仕様書 §1.1](./2026-04-30_detailed_specification.md)参照）。
- **統一スタンプのデザイン確認**（太赤枠・太字・右上オーバーレイ）を関係者と合意する。
- **Intersection Observer によるスクロール検知の動作確認**：台帳一覧画面で複数セクションが混在する場合の挙動をプロトタイプで検証する（[詳細仕様書 §5](./2026-04-30_detailed_specification.md)参照）。
- **ツールチップ・保守動線のUI確認**：スタンプホバー時のツールチップ表示と「設定を変更」リンクの操作性を関係者と確認する（[詳細仕様書 §6](./2026-04-30_detailed_specification.md)参照）。
- 合意後、[詳細仕様書 §4](./2026-04-30_detailed_specification.md) の実装コンポーネントマッピングに沿って実装フェーズに移行する。
- **初回の確認事項**: 個別レコードごとに機密性が異なる業務ユースケースが存在するかどうかを確認する。
