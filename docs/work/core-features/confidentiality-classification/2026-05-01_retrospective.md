# 秘密区分表示機能 MVP — 振り返り

**date**: 2026-05-01
**scope**: 仕様書作成 → 現状調査 → Issue 化 → Sprint 計画
**epic**: #186

---

## 良かったこと（What went well）

1. **既存コードベースの深い調査**
   - Permission/ACL、Folder（NestedSet）、LedgerDefine、IndexManager、キャッシュ戦略まで網羅的に調査した
   - これにより、仕様書の「空中散歩」を防ぎ、実装上のトラップ（z-index 競合、tenant_id 未含キャッシュ、Create→Edit リダイレクトなど）を事前に発見できた

2. **ギャップ分析の体系化**
   - 「決定事項チェックリスト」として 8 カテゴリ・20 項目以上の未確定事項を整理した
   - 技術的負債（WritableFolderRepository のキャッシュキー問題）を新機能に持ち込まないための明示的なチェックができた

3. **既存プロジェクトパターンの流用**
   - 管理者お知らせ機能（#180-#185）の Epic→Sprint 分解構造を参照し、同じフォーマットで Issue を作成した
   - これにより、プロジェクト内の一貫性が保たれ、関係者が構造を理解しやすくなった

4. **インターネットのベストプラクティス調査**
   - GitHub Sub-issues、Jira Epic/Story、漸進的詳細化（Progressive Elaboration）などの類似プロジェクト進め方を調査し、スプリント計画に反映した

---

## 悪かったこと（What caused rework / confusion）

1. **「全て決めてから進む」アプローチ**
   - 最初に 20 項目以上の決定事項チェックリストを提示し、「一度に全て決めてください」という形にしてしまった
   - 結果として、ユーザーに大きな認知負荷を与え、「何も見ないまま一度に全ての課題の決定を下せない」というフィードバックを受けた
   - **根因**: 仕様書作成の流れで「完璧な計画」を目指しすぎた。不確実性を受け入れる文化が不足していた

2. **GitHub Sub-issues の API 非対応**
   - GitHub API / MCP / `gh` CLI では Sub-issues の作成・紐付けが自動化できない
   - 結果として、Epic #186 と Sprint Issue #187-#190 の親子関係は手動で Web UI から設定する必要があった
   - Issue 本文内の「GitHub 追跡」セクションはテキストリンクに留まり、機械的な親子追跡にはならなかった

3. **コンテキスト量の肥大化**
   - 仕様書 4 本 + 現状調査 + Issue 5 本作成を 1 セッションで実行したため、コンテキストが圧迫された
   - セッション分割の判断をユーザーに委ねる形になり、効率が落ちた

4. **モックアップ先行の重要性に後から気づいた**
   - 仕様書作成→ギャップ分析→Issue 作成の流れの中で、UI/UX の不確実性を最後まで放置していた
   - ユーザーからの指摘で初めて「モックアップで確定しつつ進めたい」という需要に気づいた

## 技術要素 vs プロセス要素

### 技術要素
- **NestedSet の `ancestors()` を使用**（親子関係取得）: Folder の継承解決で `parent` プロパティの再帰ではなく `ancestors()` を使う方が安全
- **キャッシュキーに tenant ID を含める**: WritableFolderRepository の教訓を反映
- **JSON 構造の確定**: `{"org_ids":[],"role_ids":[]}` のオブジェクト形式が適切
- **z-index・sticky 要素の競合**: 実際のブラウザ確認が必要

### プロセス要素
- **モックアップ先行**: コードを書く前に UI プロトタイプで画面周りを確定
- **漸進的詳細化**: 一度に全て決めない。スプリントを進めながら設計を固める
- **ユーザー負荷の考慮**: 20 項目のチェックリストを一度に提示しない
- **Epic→Sprint→Sub-task の階層**: 大きな機能は Epic で統括し、Sprint で漸進的に実装

## 次回からの改善アクション

1. **仕様書作成時**: 必ず「どの部分をモックアップで確定するか」を最初に宣言する
2. **決定事項チェックリスト**: 「全て決めてください」ではなく「Sprint 1 でこれらを確定します」と提示する
3. **Issue 作成時**: Epic 作成後、即座に Web UI で Sub-issues を手動設定する（API 非対応を前提）
4. **セッション管理**: Epic レベルの作業は自然な区切り（仕様書完了、Issue 作成完了）でセッション分割を提案する
