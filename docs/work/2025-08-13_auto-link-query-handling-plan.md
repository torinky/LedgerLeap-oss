# 自動リンクからの台帳検索機能 実装計画

## 1. 目的と背景

現状の自動リンク機能、特に自動採番カラムから生成されるリンクは、台帳一覧画面 (`/ledgers`) に遷移し、ユーザーに手動での再検索を強いる仕様となっている。文書番号のように一意性が高い情報へのリンクでは、検索結果が1件だけのリストを表示するのは冗長であり、ユーザー体験を損なう一因となっている。

本計画では、この挙動を改善し、自動リンクから遷移してきたユーザーの意図を汲み取った、よりインテリジェントな検索・表示機能を提供することを目的とする。具体的には、検索結果に応じて詳細ページへ直接リダイレクトする機能と、表示モードを制御するオプションを実装する。

## 2. ユーザーシナリオと機能要件

### 2.1. ユーザーシナリオ

#### シナリオ1：一意な文書番号に完全一致した場合（理想的な体験）
*   **状況:** ユーザーは、自動採番された一意の文書番号「`SPEC-001`」のリンクをクリックする。
*   **期待される動作:** 一覧ページを挟むことなく、**直接「`SPEC-001`」の台帳詳細ページにリダイレクト**される。

#### シナリオ2：検索結果が複数または0件の場合（フォールバック処理）
*   **状況:** ユーザーは、一般的な単語（例: `報告書`）や、存在しない文書番号（`OLD-999`）のリンクをクリックする。
*   **期待される動作:** **台帳一覧ページが表示**され、検索ボックスにはクリックしたキーワードが入力された状態で、複数件の検索結果または「該当なし」のメッセージが表示される。

#### シナリオ3：表示モードで「一覧表示」を強制
*   **状況:** 管理者など、意図的に一覧で結果を確認したいユーザーが、`&mode=list` パラメータが付与されたリンク (`/ledgers?query=SPEC-001&mode=list`) をクリックする。
*   **期待される動作:** たとえ検索結果が1件であってもリダイレクトせず、**台帳一覧ページでその1件を表示**する。

### 2.2. 機能要件

上記のシナリオに基づき、機能要件を以下のように整理する。

1.  **クエリパラメータの受信:** 台帳一覧ページは、URLの `query` と `mode` パラメータを認識できる。
2.  **バックエンドでの事前検索:** `query` パラメータが存在する場合、ページ表示の前にその値で台帳を検索する。
3.  **結果件数に基づく条件分岐:**
    *   `mode` パラメータが `list` の場合は、常に一覧を表示する。
    *   上記以外で、検索結果が**ちょうど1件**だった場合は、その台帳の詳細ページへリダイレクトする。
    *   検索結果が**0件または2件以上**だった場合は、一覧を表示する。
4.  **一覧表示の挙動:**
    *   （リダイレクトされなかった場合）検索ボックスに `query` の値を設定し、検索結果を一覧に表示する。
    *   他の絞り込み条件はリセットし、`query` のみによる検索結果を表示する。

## 3. 実装計画

### ステップ 1: `RecordsTable` Livewireコンポーネントの改修

*   **目的:** URLクエリパラメータを処理し、条件分岐ロジックを実装する。
*   **対象ファイル:** `app/Livewire/Ledger/RecordsTable.php`
*   **タスク:**
    1.  **プロパティの追加:**
        *   URLのクエリパラメータを受け取るため、`#[Url]` 属性を付けた `public ?string $query = null;` と `public ?string $mode = null;` プロパティを追加する。
    2.  **`mount()` メソッドの改修:**
        *   `mount()` メソッド内で、`$this->query` プロパティに値がセットされているかを確認する。
        *   値が存在する場合、`Ledger` モデル（または関連する検索サービス）を使って、`query` の値に一致するレコードを検索する。
        *   検索結果の件数をチェックする。
        *   **もし件数が1件で、かつ `$this->mode` が `'list'` でない場合:**
            *   `redirect()->route('ledger.show', ['id' => $result->first()->id]);` を実行して詳細ページへリダイレクトさせる。
        *   **上記以外の場合:**
            *   検索ボックスに `query` の値を表示するため、`wire:model` でバインドされている別のプロパティ（例: `$searchQuery`）に `$this->query` の値をセットする。
            *   既存のフィルタリングロジックが、この `$searchQuery` プロパティを元に一覧を絞り込むようにする。

### ステップ 2: ビューの修正

*   **目的:** 検索ボックスがLivewireコンポーネントのプロパティと正しく連携するようにする。
*   **対象ファイル:** `resources/views/livewire/ledger/records-table.blade.php`
*   **タスク:**
    1.  台帳一覧の検索ボックスの `<input>` タグに、ステップ1で値をセットしたプロパティ（例: `$searchQuery`）への `wire:model` でのバインディングが設定されていることを確認・実装する。

## 4. テスト計画

### 4.1. フィーチャーテスト

*   **目的:** ユーザーの操作を模倣し、URLパラメータに応じた正しいリダイレクトまたは一覧表示が行われることを確認する。
*   **テストファイル:** `tests/Feature/Livewire/Ledger/RecordsTableQueryTest.php` を新規作成。
*   **テストケース:**
    1.  **一意な結果でのリダイレクト:**
        *   **準備:** 特定の文書番号を持つ台帳レコードを1件作成する。
        *   **実行:** `/ledgers?query={文書番号}` のURLへGETリクエストを送信する。
        *   **検証:** レスポンスが、その台帳の詳細ページへのリダイレクト（HTTPステータス 302）であることをアサートする。
    2.  **複数結果での一覧表示:**
        *   **準備:** 共通のキーワードを持つ台帳レコードを2件作成する。
        *   **実行:** `/ledgers?query={共通キーワード}` のURLへGETリクエストを送信する。
        *   **検証:**
            *   レスポンスが正常（HTTPステータス 200）であることをアサートする。
            *   レスポンスのHTMLに、2件のレコード情報と、検索ボックスにキーワードが入力されていることが含まれていることをアサートする。
    3.  **結果0件での一覧表示:**
        *   **準備:** データベースに存在しないキーワードを用意する。
        *   **実行:** `/ledgers?query={存在しないキーワード}` のURLへGETリクエストを送信する。
        *   **検証:**
            *   レスポンスが正常（HTTPステータス 200）であることをアサートする。
            *   レスポンスのHTMLに、「該当するレコードが見つかりませんでした。」というメッセージが含まれていることをアサートする。
    4.  **`mode=list` での一覧表示強制:**
        *   **準備:** 特定の文書番号を持つ台帳レコードを1件作成する。
        *   **実行:** `/ledgers?query={文書番号}&mode=list` のURLへGETリクエストを送信する。
        *   **検証:**
            *   レスポンスがリダイレクトではなく、正常（HTTPステータス 200）であることをアサートする。
            *   レスポンスのHTMLに、1件のレコード情報が含まれていることをアサートする。
