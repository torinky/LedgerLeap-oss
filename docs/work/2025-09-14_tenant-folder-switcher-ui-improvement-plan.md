# テナント・フォルダスイッチャーUI改善計画書

**日付:** 2025年9月14日
**作成者:** Gemini
**ステータス:** 計画中
**関連ドキュメント:** [マルチテナント実装課題の解決戦略](./2025-09-07_issue-resolution-strategy.md)

## 1. 概要

本ドキュメントは、「[マルチテナント実装課題の解決戦略](./2025-09-07_issue-resolution-strategy.md)」で定義された課題3「**ナビゲーションUIの問題**」に対する、現状分析、改善方針の検討、および具体的な実装計画を定義するものである。

課題3は、トップメニューに設置されたテナント・フォルダ選択機能がツリー形式で表示されるため、UIの幅に収まらず視認性と操作性が低いという問題である。本計画は、この問題を解決し、ユーザーがテナントやフォルダをより直感的かつ効率的に選択できる多階層のドロップダウンメニュー形式への改修を目的とする。

## 2. 現状分析

### 2.1. `TenantSwitcher` コンポーネントの役割
`TenantSwitcher` (Livewireコンポーネント: `app/Livewire/TenantSwitcher.php`) は、共通ヘッダー (`resources/views/layouts/daisyuiNavigation.blade.php`) に配置され、ログインユーザーがアクセス可能なテナントとそのテナント内のフォルダ階層を表示し、選択したテナント/フォルダへの遷移機能を提供する役割を担っている。

### 2.2. 現在のUI実装

*   **Livewireコンポーネント (`app/Livewire/TenantSwitcher.php`):**
    *   `mount` メソッドで `TenantAccessService` を利用し、現在のユーザーがアクセス可能なテナントのIDを取得する。
    *   `tenancy()->central()` コンテキストで全てのテナント情報を取得し、各テナントに対して `is_member` プロパティを設定する。
    *   `is_member` が `true` のテナントに対して、`$tenant->run()` を使ってそのテナントのコンテキストに切り替え、`Folder::get()->toTree()` でフォルダツリーを取得し、`folders_tree` プロパティに格納している。

*   **Bladeビュー (`resources/views/livewire/tenant-switcher.blade.php`):**
    *   DaisyUIの `dropdown` と `menu` クラスを使用して、テナントとフォルダの選択UIの全体構造を定義している。
    *   アクセス可能なテナント (`$tenants->where('is_member', true)`) をループし、各テナントに対して `folders_tree` が存在すれば `<details>` タグで展開可能な形式で表示している。
    *   `@include('livewire.tenant-switcher-folder-tree', ...)` を呼び出してフォルダツリーを再帰的に表示している。

*   **Bladeビュー (`resources/views/livewire/tenant-switcher-folder-tree.blade.php`):**
    *   `@foreach` ループと `<ul>`, `<li>` タグ、そして再帰的な `@include` を使って、フォルダの階層構造をツリー形式で表示している。

### 2.3. 現在の実装が抱える問題点

現在の実装は、HTMLの `<details>` タグと `<ul>`/`<li>` を利用したツリー構造でテナントとフォルダの階層を表示しています。この形式は、以下の問題を引き起こしています。

*   **UIの幅の制約:** ドロップダウンメニューの幅が固定されているため、テナント名やフォルダ名が長い場合、あるいは深い階層のフォルダが存在する場合に、内容がはみ出したり、適切に表示されなかったりする。
*   **視認性の低下:** 多数のテナントや深いフォルダ階層がある場合、ツリー構造全体を一度に把握しづらく、目的の項目を見つけるのに時間がかかる。
*   **操作性の低下:** `<details>` タグの開閉操作が煩雑であり、特にモバイル環境やタッチデバイスでの操作が困難である。また、スクロールバーが表示される場合、目的の項目へのアクセスがさらに手間取る。

## 3. ペルソナとユーザーシナリオからの追加検討

`docs/function/MyPortal.md` から抽出したペルソナとユーザーシナリオを考慮し、ナビゲーションUIの改善において特に重要な点を再確認します。

### 3.1. ペルソナ

*   **ITリテラシーにばらつきのあるユーザー:** 専門用語を避け、直感的に理解できる情報提供が求められる。
*   **初回利用者や役割変更があったユーザー:** ログイン直後に自分の状況を素早く把握し、システムへのオンボーディングを支援する必要がある。
*   **日常業務でシステムを利用するユーザー:** 目的のデータへのアクセスや作業開始までのステップを短縮したい。
*   **管理者:** ユーザーの権限設定やシステム設定を効率的に行いたい。

### 3.2. ユーザーシナリオ

*   **ログイン直後の状況把握:** ユーザーはログイン後、自分がどの組織に所属し、どのような役割を持ち、システム上で何ができるのかを一目で確認したい。
*   **担当フォルダへの迅速なアクセス:** ユーザーは日常的にアクセスするフォルダへ、マイポータルから直接移動したい。
*   **権限の確認:** ユーザーは自分が持っている主要な操作権限を分かりやすい言葉で確認したい。
*   **詳細情報の確認:** ユーザーは必要に応じて、アクセス可能な全フォルダツリーなどの詳細情報を確認したい。
*   **ログイン後の初期画面設定:** ユーザーはログイン後に表示される初期画面を「マイポータル」または「台帳/フォルダ一覧」から選択したい。
*   **ナビゲーションからのアクセス:** ユーザーはいつでもヘッダーナビゲーションからマイポータル画面や他の主要機能にアクセスしたい。

### 3.3. ナビゲーションUI改善への影響

上記のペルソナとユーザーシナリオを考慮すると、テナント・フォルダスイッチャーのUI改善において以下の点が特に重要となります。

*   **直感的な操作性:** ITリテラシーにばらつきのあるユーザーでも迷わずテナントやフォルダを選択できるUIであること。
*   **迅速なアクセス:** 日常的に利用するテナントやフォルダへ素早くアクセスできること。特に、深い階層のフォルダへのアクセスステップを最小限に抑える工夫が必要。
*   **視認性:** 多数のテナントや深いフォルダ階層があっても、全体像を把握しやすく、目的の項目を効率的に見つけられること。
*   **一貫性:** 既存のUI（特にMaryUIやDaisyUI）とのデザイン・操作の一貫性を保つこと。

現在のツリー形式のUIが「UIの幅に収まらず視認性と操作性が低い」という問題は、上記のユーザーシナリオ（特に「迅速なアクセス」「視認性」）に直接影響を与え、ユーザー体験を損なっています。多階層のドロップダウンメニュー形式への改修は、この問題を解決し、ユーザー体験を向上させるための重要なステップです。

## 4. 改善方針の検討

「多階層のドロップダウンメニュー形式に改修する」という要件を満たすために、複数のアプローチを検討します。

### 4.1. A案: MaryUIの `<x-mary-dropdown>` と `<x-mary-menu-item>` を活用する（推奨）

*   **概要:** 現在の「設定」メニューで採用されているMaryUIのドロップダウンコンポーネントを、テナント・フォルダスイッチャーにも適用するアプローチ。MaryUIのメニューアイテムはネスト可能であり、多階層のドロップダウンメニューを構築できる。
*   **メリット:**
    *   **UIの一貫性:** 既存の「設定」メニューとのデザイン・操作の一貫性が保たれるため、ユーザーは新しいUIに戸惑うことなく利用できる。
    *   **開発効率:** 既にプロジェクトで利用実績のあるコンポーネントであるため、学習コストが低く、迅速な実装が可能。
    *   **機能性:** MaryUIのドロップダウンは、開閉状態の管理やイベントハンドリングなど、必要な機能が提供されている。
*   **デメリット:**
    *   MaryUIのコンポーネントの制約を受ける可能性がある（例: カスタマイズの自由度）。
    *   非常に深い階層のフォルダが存在する場合、メニューが画面外にはみ出す可能性がある。
*   **ペルソナ/ユーザーシナリオへの適合:**
    *   **直感的な操作性:** 既存UIとの一貫性により、ユーザーは迷わず操作できる。
    *   **迅速なアクセス:** 多階層メニューにより、目的のテナントやフォルダへ効率的にアクセスできる。
    *   **視認性:** ネストされたメニューにより、階層構造を把握しやすくなる。

### 4.2. B案: DaisyUIの多階層メニューコンポーネントを活用する

*   **概要:** 現在のUIがDaisyUIベースであるため、DaisyUIが提供する多階層メニューのクラスやコンポーネントを利用するアプローチ。
*   **メリット:**
    *   **既存スタイルとの親和性:** プロジェクトの既存のDaisyUIスタイルとの整合性が高い。
    *   **軽量性:** MaryUIのコンポーネントを導入するよりも、純粋なDaisyUIのクラスを利用する方が軽量である可能性がある。
*   **デメリット:**
    *   MaryUIのコンポーネントに比べて、JavaScriptによる動的な制御（開閉状態の管理など）を自前で実装する必要がある場合がある。
    *   MaryUIの「設定」メニューとのデザイン・操作の一貫性が損なわれる可能性がある。
*   **ペルソナ/ユーザーシナリオへの適合:**
    *   A案と同様に、直感的な操作性、迅速なアクセス、視認性の向上が期待できるが、UIの一貫性という点でA案に劣る可能性がある。

### 4.3. C案: カスタムJavaScript/Alpine.jsで多階層メニューを実装する

*   **概要:** Tailwind CSSとAlpine.jsを直接利用し、完全にカスタムな多階層ドロップダウンメニューを実装するアプローチ。
*   **メリット:**
    *   **最大の柔軟性:** デザインや機能のカスタマイズにおいて最も自由度が高い。
    *   **パフォーマンス最適化:** 不要なライブラリのオーバーヘッドを排除し、パフォーマンスを最大限に最適化できる可能性がある。
*   **デメリット:**
    *   **実装コストが高い:** ゼロから実装するため、開発時間と労力が最もかかる。
    *   **保守性:** カスタムコードが増えるため、将来的なメンテナンスコストが高くなる可能性がある。
*   **ペルソナ/ユーザーシナリオへの適合:**
    *   UI/UXの設計次第で最も高い適合性を実現できる可能性があるが、実装コストとリスクも最も高い。

### 4.4. D案: 専用のモーダルダイアログまたはサイドバー形式

*   **概要:** ドロップダウンメニューではなく、テナント・フォルダ選択専用のモーダルダイアログや、左側または右側からスライドインするサイドバー形式のUIを導入するアプローチ。
*   **メリット:**
    *   **広大な表示領域:** 多数のテナントや深いフォルダ階層があっても、十分な表示領域を確保できるため、視認性が大幅に向上する。
    *   **複雑なフィルタリング/検索機能の導入:** 専用のUIであるため、テナントやフォルダの検索、フィルタリングなどの高度な機能を容易に組み込める。
*   **デメリット:**
    *   **UI/UXの変更が大きい:** 既存のナビゲーションフローからの逸脱が大きいため、ユーザーが新しい操作に慣れるまで時間がかかる可能性がある。
    *   **実装コストが高い:** ドロップダウンメニュー形式に比べて、UIの設計と実装に多くの労力が必要となる。
*   **ペルソナ/ユーザーシナリオへの適合:**
    *   **視認性:** 非常に多くの項目から選択する場合に最も優れる。
    *   **迅速なアクセス:** 検索機能などを組み合わせることで、目的の項目へ非常に迅速にアクセスできる可能性がある。
    *   **直感的な操作性:** 新しいUIに慣れる必要はあるが、一度慣れれば直感的に操作できる。

## 5. 推奨案と具体的な実装計画

上記の検討結果を踏まえ、**A案: MaryUIの `<x-mary-dropdown>` と `<x-mary-menu-item>` を活用する**ことを推奨します。

**推奨理由:**

*   UIの一貫性を保ちつつ、既存の課題を効率的に解決できるため、ユーザー体験の向上と開発効率の両立が期待できます。
*   既にプロジェクトでMaryUIの利用実績があり、学習コストが低いことから、最もリスクが少なく、迅速な実装が可能です。
*   ペルソナとユーザーシナリオで重視される「直感的な操作性」「迅速なアクセス」「視認性」をバランス良く満たすことができます。

### 5.1. 実装ステップ

1.  **`app/Livewire/TenantSwitcher.php` の変更:**
    *   `mount` メソッドで取得する `folders_tree` のデータ構造は、MaryUIのメニューコンポーネントに直接渡せる形式（例: ネストされた配列）に整形する必要があるかもしれません。あるいは、Blade側で整形ロジックを記述することも可能です。
    *   **検討:** `folders_tree` の取得ロジックは現状維持し、Blade側でMaryUIのメニュー構造に合うように変換する方が、Livewireコンポーネントの責務をシンプルに保てると考えられます。

2.  **`resources/views/livewire/tenant-switcher.blade.php` の再構築:**
    *   現在の `dropdown` と `menu` の構造を、`x-mary-dropdown` と `x-mary-menu-item` を使った多階層メニューに置き換えます。
    *   テナントごとにドロップダウンメニューを生成し、その中にフォルダの階層をネストされた `x-mary-menu-item` で表示します。
    *   **具体的な構造イメージ:**
        ```blade
        <x-mary-dropdown>
            <x-slot:trigger>
                <!-- 現在のテナント名とアイコンを表示するボタン -->
            </x-slot:trigger>

            <!-- アクセス可能なテナントのリスト -->
            @foreach($tenants->where('is_member', true) as $tenant)
                <x-mary-menu-item :title="$tenant->name ?? $tenant->id" :link="route('my-portal', ['tenant' => $tenant->id])" wire:navigate>
                    @if($tenant->folders_tree->isNotEmpty())
                        <x-slot:submenu>
                            <!-- フォルダ階層を再帰的に表示 -->
                            @include('livewire.tenant-switcher-mary-menu-folder-tree', ['folders' => $tenant->folders_tree, 'tenant' => $tenant])
                        </x-slot:submenu>
                    @endif
                </x-mary-menu-item>
            @endforeach

            <!-- その他のテナント (アクセス不可) -->
            <div class="divider my-1"></div>
            <li class="menu-title"><span>{{ __('ledger.navigation.other_tenants') }}</span></li>
            @foreach($tenants->where('is_member', false) as $tenant)
                <x-mary-menu-item :title="$tenant->name ?? $tenant->id" icon="o-lock-closed" disabled />
            @endforeach
        </x-mary-dropdown>
        ```

3.  **`resources/views/livewire/tenant-switcher-folder-tree.blade.php` の改修:**
    *   新しいBladeビュー `resources/views/livewire/tenant-switcher-mary-menu-folder-tree.blade.php` を作成し、MaryUIの `<x-mary-menu-item>` を使ってフォルダ階層を再帰的に表示するように改修します。
    *   **具体的な構造イメージ (`tenant-switcher-mary-menu-folder-tree.blade.php`):**
        ```blade
        @foreach ($folders as $folder)
            <x-mary-menu-item :title="$folder->title" :link="route('ledgersByFolderId', ['tenant' => $tenant->id, 'folderId' => $folder->id])" wire:navigate>
                @if ($folder->children->isNotEmpty())
                    <x-slot:submenu>
                        @include('livewire.tenant-switcher-mary-menu-folder-tree', ['folders' => $folder->children, 'tenant' => $tenant])
                    </x-slot:submenu>
                @endif
            </x-mary-menu-item>
        @endforeach
        ```

### 5.2. 考慮すべき事項

*   **パフォーマンス:** 非常に多数のテナントや深いフォルダ階層が存在する場合、全てのメニュー項目を一度にレンダリングするとパフォーマンスに影響が出る可能性があります。必要に応じて、Livewireの遅延ロード機能や、メニューの開閉時にのみ子要素をロードするなどの最適化を検討する必要があります。
*   **UX:** 多階層ドロップダウンメニューの操作性（開閉、ホバー/クリック）を考慮し、ユーザーが直感的に使えるように設計する必要があります。MaryUIのデフォルトの挙動を確認し、必要であればAlpine.jsなどで微調整を加えることも検討します。
*   **翻訳キー:** 新たに追加される可能性のあるUI要素のテキストについては、適切な翻訳キー (`lang/**/ledger.php`) を定義する必要があります。
