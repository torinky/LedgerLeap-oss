# マルチテナント・アーキテクチャの再評価と最終決定

**日付:** 2025年8月28日 (2025年8月27日 初版)

## 1. 背景

当初、「シングルDB → マルチDB移行」アプローチで実装計画をFIXした後、ユーザーより「前回の失敗（最初からマルチDB方式）で、多くのテーブルは中央DBに残すべきだと判明した。この事実を踏まえると、シングルDBから始める方が有利ではないか」という重要な指摘があった。これを受け、アーキテクチャのトレードオフを再検討し、過去の失敗原因の分析を行った。

## 2. テーブルの役割分類（再検討結果）

`docs/**` 配下のペルソナ、ユースケース、機能仕様を再精査した結果、各テーブルは以下のように分類するのが最も合理的であると判断した。この分類は、後述する失敗分析と最終的なアーキテクチャ決定においても、その正しさが裏付けられた。

*   **中央DBに配置すべきテーブル (Central Models):**
    *   `tenants`, `domains`: テナント管理のコア。
    *   `users`, `organizations`, `user_organizations`: 全テナントで共通のユーザー・組織構造。
    *   `roles`, `permissions`, `model_has_roles`, `role_has_permissions`: 全テナント共通の権限体系。
    *   `auto_links`: 全社共通ルールの自動リンク。
    *   `notification_types`: システム全体で共通の通知種別。
    *   `jobs`, `job_batches`: 非同期処理のキュー。

*   **テナントごとに分離すべきテーブル (Tenant Models):**
    *   `folders`: テナント（部署・プロジェクト）ごとに独立した情報格納庫。
    *   `ledger_defines`: テナント固有の業務プロセスを定義するテンプレート。
    *   `ledgers`: テナントの業務データそのもの。
    *   `ledger_diffs`: 台帳データの変更履歴。
    *   `attached_files`: 台帳に紐づくファイル。
    *   `tags`, `taggables`: テナント固有の分類ラベル。

*   **両方にまたがるため、慎重な設計が必要なテーブル:**
    *   `role_folder_permissions`: 「中央のロール」と「テナントのフォルダ」を紐付ける権限設定。
    *   `activity_log`: 「誰が（中央のUser）」が「何を（テナントのLedgerなど）」したかの記録。

## 3. アプローチの再評価（当初の議論）

*本セクションは、過去の失敗分析を行う前の、当初の議論を記録として残すものです。*

上記の「ハイブリッド構成」を前提に、両アプローチのトレードオフを再評価した。

| 観点 | アプローチA: シングルDB → マルチDB移行 | アプローチB: 最初からマルチDB |
| :--- | :--- | :--- |
| **開発初期** | **非常に容易**<br>DBが1つのため、DBをまたぐ関係（例: `role_folder_permissions`）をEloquentリレーションで簡単に実装できる。 | **非常に複雑**<br>開発開始と同時に、DBをまたぐ関係の実装方法（サービス層でのデータ結合など）を設計・実装する必要がある。 |
| **テスト** | 初期は容易だが、移行フェーズで複雑化。 | 最初から工夫が必要だが、**調査により解決策は見えている。** |
| **将来の移行** | **極めて高リスクな「書き換え」が発生**<br>Eloquentリレーションで実装した部分を、**DBをまたぐことを前提としたロジックにほぼ全て書き直す**必要がある。これは単なるデータ移行に留まらない、大規模な手戻りとなる。 | **不要**<br>移行プロセス自体が存在しない。 |
| **開発の一貫性** | **低い**<br>途中でアーキテクチャの根幹が変わる。移行後に不要になるコードが発生する。 | **高い**<br>最初から本番と同じ構成で開発できる。コードや設計に一貫性がある。 |

当初は、この評価に基づき「アプローチB: 最初からマルチDBで構築」が有力であると提言していた。

## 4. 過去の失敗分析（コードレベル）

当初の提言に基づき、最終決定の前に、過去の失敗した実装（`feature/multi-tenant-db-separation-attempt` ブランチ）を分析した。

*   **使用ライブラリ:** `stancl/tenancy` (v3) を利用していた。
*   **観測された事象:**
    1.  `stancl/tenancy` の「リソース同期」機能を使い、中央の `User` モデルを各テナントDBに同期（コピー）しようと試みた。
    2.  その過程で、外部キー制約などが原因でマイグレーションが失敗。
    3.  エラー解消のため、場当たり的にテーブル配置（マイグレーションファイルの移動）を変更。その結果、`role_folder_permissions` がテナントDBに置かれるなど、アーキテクチャ全体に矛盾が生じた。
    4.  根本的な問題は解決できず、最終的に実装が頓挫した。

*   **失敗の根本原因:**
    失敗の根本原因は、ライブラリの選定ミスではなく、**`stancl/tenancy` の「リソース同期」機能の制約と複雑さを考慮せずに、`User` のようなコアなモデルを同期しようとしたこと**にある。これがマイグレーションの失敗を招き、場当たり的な対応の連鎖を引き起こし、プロジェクト全体を行き詰まらせたと結論付けられる。

## 5. アーキテクチャの再検討とライブラリ選定

過去の失敗分析と、ユーザーからの「マルチDB（データベース全体の分離）は必須ではなく、ID突き抜けやパフォーマンスが懸念されるテーブルのみ分離できれば良い」という新しい要件に基づき、アーキテクチャとライブラリを再検討した。

*   **新要件:** 単一のデータベース接続内で、`users` 等の主要テーブルは共通テーブルとして保持し、`ledgers` 等のコンテンツ関連テーブルのみ、テナントごとに物理的に分離する。
*   **ライブラリ比較:**
    *   **`spatie/laravel-multitenancy`**: シングルDBで `tenant_id` カラムによる論理分離を前提としており、テーブルの物理的な分離はサポート外。**新要件に適合しない。**
    *   **`stancl/tenancy`**: テナントごとにDB設定を動的に変更する機能が強力。これを応用すれば、DB接続は単一のまま、**テーブルプレフィックスを動的に切り替える**ことで、テーブルの物理的分離が実現可能。**新要E件に適合する。**

## 6. 最終結論: 「単一DB・テーブルプレフィックス方式」の採用

以上の分析と検討を経て、本プロジェクトのマルチテナント・アーキテクチャとして、**「単一DB・テーブルプレフィックス方式」**を正式に採択する。

*   **実装方針:**
    *   **`stancl/tenancy` を継続利用する。**
    *   問題の原因となった**リソース同期機能は一切使用しない。**
    *   テナントのコンテキストに応じて、DB接続設定の**テーブルプレフィックス (`database.connections.mysql.prefix`) を動的に変更**するロジックを実装する。
        *   例: テナントAのコンテキストでは、プレフィックスが `tenant_a_` となる。
*   **テーブル構成:**
    *   **単一DB `ledgerleap` 内に、以下のテーブルが同居する。**
        *   `users`, `organizations` （プレフィックスなしの共通テーブル）
        *   `tenant_a_ledgers`, `tenant_a_folders` （テナントA用のプレフィックス付きテーブル）
        *   `tenant_b_ledgers`, `tenant_b_folders` （テナントB用のプレフィックス付きテーブル）
*   **期待されるメリット:**
    1.  **Eloquentリレーションの維持:** DB接続が常に単一であるため、中央テーブルとテナントテーブル間でのEloquentリレーションがほぼ通常通り機能し、実装の複雑さが劇的に低下する。
    2.  **懸念点の解消:** `ledgers` 等のテーブルは物理的に分離されるため、ID突き抜けや将来のパフォーマンス問題を回避できる。
    3.  **過去の失敗の回避:** 失敗の根本原因であるリソース同期やDBをまたいだ連携が不要になるため、同じ轍を踏むリスクがない。

このアーキテクチャは、これまでの議論で出たすべての要件と懸念を解決する、最も合理的で堅牢な解決策である。
