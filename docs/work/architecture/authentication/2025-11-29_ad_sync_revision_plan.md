## 1. 目的と基本方針

Active Directory (AD) のユーザー属性に基づき、LedgerLeap上に組織階層 (`Organization` ツリー) を自動構築・同期します。
単なる名称一致だけでなく、**組織コード（識別子）**を利用することで、組織の**改名（名称変更）**や**移動（親組織の変更）**を正確に追跡・反映することを目的とします。

### 基本方針
1.  **階層構造の定義:** ADユーザーの属性（例: 会社名、部署名、課名）の順序に基づいて階層を定義します。
2.  **同一性管理:** 各階層レベルにおいて、可能な限り「コード属性（識別子）」と「名称属性（表示名）」をペアで利用し、組織の同一性を管理します。
3.  **整合性維持:** `kalnoy/nestedset` ライブラリの仕様に準拠し、ツリー構造の整合性を常に保ちます。

## 2. データモデル拡張 (スキーマ変更)

ADとの同期用に、不変の識別子を格納するカラムを追加します。

### 2.1. Users テーブル
| カラム名 | 型 | 説明 | 備考 |
| :--- | :--- | :--- | :--- |
| `objectguid` | `string` | ADの `objectGuid` (または `entryuuid`) | **新規追加 (Unique, Nullable)** <br> ユーザーの不変ID |
| `name` | `string` | 表示名 (`cn`, `displayName`) | 同期対象 |
| `email` | `string` | メールアドレス (`mail`) | 同期対象 |

### 2.2. Organizations テーブル
| カラム名 | 型 | 説明 | 備考 |
| :--- | :--- | :--- | :--- |
| `org_id` | `string` | 組織ID (ユーザー入力 / AD/LDAP識別子) | **既存 (Nullable, Unique, Index)** <br> AD側の組織識別子 (`objectGuid` や `extensionAttribute`) を格納 |
| `name` | `string` | 組織名 | 同期対象 (改名反映) |
| `parent_id` | `integer` | 親組織ID | 同期対象 (移動反映) |
| `_lft`, `_rgt` | `integer` | NestedSet用インデックス | 自動管理 |

## 3. 設定構成 (`config/ldap_sync.php`)

階層の各レベルに対して、「コード属性」と「名称属性」のマッピングを定義します。

```php
return [
    'mode' => 'attribute', // attribute | ou
    
    'hierarchy_attributes' => [
        // レベル1: 会社 (コードあり)
        // キー: ADのコード属性, 値: ADの名称属性
        'extensionAttribute1' => 'company', 
        
        // レベル2: 部署 (コードあり)
        'extensionAttribute2' => 'department',
        
        // レベル3: 課 (コードなし、名称のみで管理する場合)
        // 値のみ指定した場合は、その属性値を名称兼コード(識別キー)として扱う
        'ou', 
    ],
    
    // 廃止された組織の削除を行うか
    'delete_missing' => true,
    
    // 削除保護の閾値 (誤って全削除されるのを防ぐ。削除対象が全組織の何%を超えたら停止するか)
    'deletion_threshold_percentage' => 20,
];
```

## 4. 同期アルゴリズム詳細

### 4.1. 同期プロセス概要

1.  **初期化:** 現在の全 `Organization` のIDを「未確認リスト」として保持（削除判定用）。
2.  **ユーザー取得:** ADから同期対象ユーザーを一括取得。
3.  **階層解決 (ループ):** 各ユーザーについて、設定された階層属性をルートから順に処理。
4.  **組織作成/更新:**
    *   コードと名称を用いてDBを検索。
    *   **改名検知:** コード一致かつ名称不一致なら `name` 更新。
    *   **移動検知:** コード一致かつ親ID不一致なら `appendToNode` で移動。
    *   **新規作成:** 該当なしなら新規作成。
    *   処理した組織IDを「確認済みリスト」に追加（「未確認リスト」から除外）。
5.  **ユーザー紐付け:** ユーザーを特定し、最下層の組織に所属させる。
6.  **クリーンアップ:** 「未確認リスト」に残った組織（AD上に存在しなくなった組織）を論理削除。
7.  **ツリー修復:** `Organization::fixTree()` を実行。

### 4.2. 組織特定ロジック (Find Strategy)

各階層レベルでの組織特定は以下の優先順位で行います。

1.  **コード検索 (Primary):**
    *   `external_code` が「ADから取得したコード値」と一致するレコードを検索。
2.  **名称検索 (Fallback - コード属性未設定時):**
    *   コード属性が設定されていないレベルの場合、`parent_id` と `name` が一致するレコードを検索。
3.  **過去コード検索 (Advanced - オプション):**
    *   (将来拡張) AD側に「変更前コード」属性がある場合、それを用いて検索し、ヒットしたら `external_code` を新しいコードに更新する。

## 6. テスト計画

同期ロジックの正確性と安全性を検証するため、以下のテストケースを実装します。

### 6.1. テストケース一覧

| ID | カテゴリ | テスト内容 | 期待値 |
| :--- | :--- | :--- | :--- |
| **TC-01** | 新規作成 | 初回同期。コードと名称を持つ属性セットを与える。 | 指定通りの階層構造で `Organization` が作成され、`external_code` が保存されること。 |
| **TC-02** | ユーザー紐付 | ユーザー属性に対応する組織を作成後、ユーザーを紐付ける。 | `users` テーブルにレコードが作成され、正しい `Organization` と `is_primary=1` で紐付くこと。 |
| **TC-03** | 名称変更 | 既存組織のコードは同じままで、AD側の名称属性を変更して同期。 | `Organization` の `name` が更新され、`id` や `external_code` は維持されること。 |
| **TC-04** | 組織移動 | 既存組織 (子) の親となる属性を変更して同期 (例: 営業部->1課 を 海外部->1課 に変更)。 | 当該組織の `parent_id` が新しい親組織のIDに更新され、ツリー構造が移動すること。 |
| **TC-05** | 廃止(削除) | 既存組織が含まれない属性セットで同期を実行。 | 同期されなかった組織が `SoftDelete` (deleted_atが入る) されること。 |
| **TC-06** | 削除保護 | 全組織の削除など、閾値を超える削除が発生する場合。 | 処理が中断され、エラーログが出力されること (削除されないこと)。 |
| **TC-07** | 整合性修復 | NestedSetの左右値を意図的に壊した状態で同期実行。 | 処理完了後、`fixTree()` により `_lft`, `_rgt` が正しい値に修復されていること。 |
| **TC-08** | 混合キー | コードあり階層(会社)とコードなし階層(課)が混在する場合。 | それぞれのロジックで正しく特定・作成されること。 |

### 6.2. テスト実装方針
*   `Orchestra\Testbench` または Laravel の `TestCase` を使用。
*   LDAPサーバーへの実際の接続はモック化 (`LdapRecord\Laravel\Testing\DirectoryEmulator` 等) するか、または同期ロジック部分をサービスクラス (`AdSyncService`) に切り出して、配列入力でのユニットテストを行うことを推奨します。
*   `AdSync` コマンド自体は、Serviceを呼び出すラッパーとして実装します。
