# LedgerLeapにおけるRedisの役割と必要性の検討

## 1. はじめに

本ドキュメントは、LedgerLeapプロジェクトにおいて、キャッシュおよびキューサーバーとして利用が想定されているRedisの必要性について再検討した結果を記録するものです。

## 2. 調査の背景

現在、開発環境 (`docker-compose.yml`) ではRedisサービスがコメントアウトされており、本番環境 (`docker-compose.prod.yml`) でも有効化されていませんでした。この状況を踏まえ、プロジェクトのアーキテクチャと機能要件に基づき、Redisが本当に必要かどうかを関連ドキュメントから調査・検討しました。

## 3. Redisの主な役割と必要性

ドキュメント (`docs/README.md`, `docs/architecture/overview.md`, `docs/architecture/QueueProcessing.md` 等) を横断的に調査した結果、LedgerLeapプロジェクトにおいてRedisは以下の3つの重要な役割を担っており、特に**非同期処理の実現において不可欠なコンポーネント**であると結論付けました。

### 3.1. 非同期処理（キュー）の実現【最重要】

- **目的**: ユーザー体験を損なう可能性のある、時間のかかる重い処理をバックグラウンドで実行するため。
- **具体的な処理**:
    - **Apache Tikaによるファイル内容のテキスト抽出**
    - **OcrMyPDFによる画像ファイルのOCR処理**
- **必要性**:
    - これらのファイル解析処理は、システムのコア機能である「添付ファイルを含む全文検索」を実現するために必須です。
    - もしRedis（キューサーバー）がない場合、ユーザーはファイルのアップロード時にこれらの重い処理が完了するまで、画面の前で待機しなくてはなりません。これはアプリケーションの応答性を著しく低下させ、ユーザー体験を大きく損ないます。
    - Redisをキューとして利用することで、これらの処理をバックグラウンドの「ジョブ」として非同期に実行できます。これにより、ユーザーはファイルをアップロードした後、すぐに次の操作へ移ることが可能になります。

### 3.2. アプリケーションの高速化（キャッシュ）

- **目的**: 頻繁にアクセスされるデータや、生成に時間のかかる計算結果などを高速なメモリ上に一時保存し、データベースへのアクセスを削減してアプリケーション全体の応答速度を向上させるため。
- **必要性**:
    - `docs/README.md` に記載の通り、本システムは「**数人～数千人規模のユーザー**」と「**数百万件規模の台帳レコード**」という大規模な利用を想定しています。
    - この規模のシステムにおいて、パフォーマンスを維持するためにはキャッシュ戦略が非常に重要です。Redisをキャッシュサーバーとして活用することで、データベースへの負荷を軽減し、快適なレスポンス速度を維持します。

### 3.3. セッション管理（将来的な拡張性）

- **目的**: ユーザーのログイン情報（セッション）を管理するため。
- **必要性**:
    - 将来的にシステムの利用が拡大し、負荷分散のためにアプリケーションサーバーを複数台構成（スケールアウト）にする場合、全サーバーでユーザーのログイン状態を共有する必要があります。
    - Redisをセッションストアとして利用することで、このサーバー間でのセッション共有を容易に実現できます。

## 4. 結論

LedgerLeapプロジェクトのコア機能、想定される利用規模、そして将来的な拡張性を考慮すると、Redisは単なるオプションではなく、**アプリケーションの性能とユーザー体験を支えるために必須のコンポーネント**です。

特に、ファイル内容の全文検索という主要機能を実現するための**非同期処理基盤**として、その役割は不可欠です。

## 5. 今後のアクション

以上の結論に基づき、開発環境と本番環境の両方でRedisを有効化するための設定作業を実施しました。

1.  **PHPライブラリのインストール**: `predis/predis` をインストールし、LaravelがRedisと通信できるようにしました。
    ```bash
    ./vendor/bin/sail composer require predis/predis
    ```

2.  **環境設定の更新**: `.env.example` ファイルを更新し、以下の通りキャッシュ、キュー、セッションの各ドライバーとしてRedisを指定しました。これにより、新規に環境を構築する際、`.env`ファイルがRedisを利用する設定で作成されます。
    - `CACHE_DRIVER=file` → `CACHE_DRIVER=redis`
    - `QUEUE_CONNECTION=sync` → `QUEUE_CONNECTION=redis`
    - `SESSION_DRIVER=file` → `SESSION_DRIVER=redis`
    - `REDIS_HOST=127.0.0.1` → `REDIS_HOST=redis` (Dockerコンテナ名を参照)

3.  **Docker環境の確認**: `docker-compose.yml` および `docker-compose.prod.yml` を確認し、Redisサービスと、それに依存する各サービス(`laravel`, `queue`, `scheduler`)の設定が既に有効化されていることを確認しました。

4.  **設定の反映**: 以下のコマンドを実行し、コンテナを再起動して設定を反映させました。
    ```bash
    ./vendor/bin/sail down && ./vendor/bin/sail up -d
    ```

これにより、アプリケーションはキャッシュ、キュー、セッションのすべてでRedisを利用するようになりました。
