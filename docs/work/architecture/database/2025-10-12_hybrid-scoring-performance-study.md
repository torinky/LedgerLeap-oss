# 作業ドキュメント: ハイブリッドスコアリング機能 実装前性能検討

**日付:** 2025年10月12日
**目的:** 新しく導入するハイブリッドスコアリング機能が、現行のデータベースアーキテクチャにおいて性能上のボトルネックとならないことを確認し、安全な実装方針を確立する。

### 1. 前提となる現行アーキテクチャ

-   **DB構成:** 単一DB構成 (`stancl/tenancy` の `tenant_id` 方式)
-   **ストレージエンジン:** `ledgers` テーブル等は全文検索のため `Mroonga` を使用。その他は `InnoDB`。
-   **最適化方針:** 物理分離やパーティショニングは採用せず、`Buffer Pool`の最適化と、`tenant_id` を含む複合インデックス戦略を基本とする。
-   **性能評価:** 移行元システムの実績（総データ量29GB、最大テナント5.9GB）から、現行アーキテクチャで十分対応可能と判断済み。

### 2. スコアリング機能による新たな負荷要因

実装計画書に基づき、本機能は以下の新たな負荷をデータベースに与える可能性がある。

1.  **スコア計算負荷:**
    -   `activity_log` テーブルを集計し、複数のイベント（作成、更新、閲覧、コメント等）から「活動スコア」を算出する必要がある。
    -   台帳一覧が表示されるたびに、全件リアルタイムでスコア計算を行うと、クエリが多発し著しい性能劣化を引き起こす。

2.  **複合ソート負荷:**
    -   「活動スコア」「新鮮度スコア」「重要度スコア」等を重み付けした複合スコアで検索結果をソートする必要がある。
    -   `ORDER BY (activity_score * 0.4) + (freshness_score * 0.3) + ...` のような動的計算を伴うソートは、インデックスが利用できず非常に低速になる。

3.  **書き込み負荷の増加:**
    -   台帳閲覧（`viewed`）などの新しいイベントを `activity_log` に記録するため、同テーブルへの書き込みが増加し、テーブルが肥大化する可能性がある。

### 3. 性能問題の回避策

上記の負荷要因に対し、以下の対策を講じることで、現行アーキテクチャへの影響を最小限に抑えます。

#### 対策1: スコア計算の非同期化・事前計算（最重要）

-   **方針:** ユーザーのリクエスト時にスコアをリアルタイム計算するのではなく、**Artisanコマンドによるバッチ処理で定期的に事前計算**し、結果を `ledgers` テーブルの専用カラム（`activity_score` 等）に保存します。
-   **実装:**
    1.  `scoring:calculate`のようなArtisanコマンドを作成します。
    2.  このコマンドは、`activity_log` を集計して各台帳の `activity_score` を計算し、`ledgers` テーブルを更新します。
    3.  `Kernel.php` にこのコマンドを登録し、深夜帯など低負荷時に実行されるようにスケジュールします。（例: 毎日午前3時）
-   **効果:**
    -   台帳一覧表示時のDB負荷は、事前計算されたスコアを読み取るだけになり、リアルタイム計算のオーバーヘッドを完全に排除できます。
    -   移行元システムのデータ規模（最大テナントでも数百万レコード）を考慮しても、バッチ処理は現実的な時間で完了可能です。

#### 対策2: 複合スコア専用カラムの導入とインデックス利用

-   **方針:** 複合スコアもリアルタイム計算せず、専用のカラムに事前計算した結果を保存します。
-   **実装:**
    1.  `ledgers` テーブルに `composite_score` のような `DECIMAL` 型のカラムを追加します。
    2.  対策1のバッチ処理の中で、各スコア（活動、新鮮度、重要度）から複合スコアを計算し、このカラムに保存します。
    3.  この `composite_score` カラムにインデックスを作成します。
-   **効果:**
    -   検索結果のソートは `ORDER BY composite_score DESC` のように、インデックスが効いた単純なクエリになります。
    -   複雑な計算を伴うソートを回避し、高速な応答性能を維持します。

#### 対策3: `activity_log` テーブルのインデックス最適化

-   **方針:** `activity_log` テーブルの肥大化に備え、集計クエリが高速に実行されるようインデックスを最適化します。
-   **実装:**
    -   `activity_log` テーブルに `(tenant_id, subject_type, subject_id, created_at)` のような複合インデックスを追加します。
-   **効果:**
    -   テナントごと、かつ特定の台帳に関連するログの絞り込みが高速化され、スコア計算バッチの実行時間短縮に繋がります。
    -   これは既存のアーキテクチャ調査で推奨されているインデックス戦略とも合致しています。

### 4. 結論

上記対策を講じることにより、ハイブリッドスコアリング機能は、**現行の単一DBアーキテクチャ上で性能上の問題を引き起こすことなく実装可能**と判断します。

スコア計算を事前計算・バッチ処理化することが、性能を担保する上での鍵となります。

### 5. 次のステップ

1.  **マイグレーションファイルの作成:**
    -   `ledgers` テーブルへのカラム追加 (`activity_score`, `composite_score`, `is_pinned`, `priority_level`)
    -   `ledger_defines` テーブルへのカラム追加 (`activity_score` 等)
    -   `scoring_configs` テーブルの新規作成
    -   `activity_log` テーブルへのインデックス追加
2.  **Artisanコマンドの実装:**
    -   スコアを事前計算するための `scoring:calculate` コマンドを作成する。
