# データベースアーキテクチャ

このカテゴリには、LedgerLeapのデータベースアーキテクチャ、設計思想、および関連する技術調査に関するドキュメントを格納しています。

---

## 📚 主要ドキュメント

### 1. マルチテナントDBアーキテクチャの検討

- **サマリ:** シングルDB構成と物理DB分離構成を比較検討し、移行元システムの実測データ（総量29GB、最大テナント5.9GB）に基づき、**単一DB + Buffer Pool最適化**で十分対応可能であると結論付けました。
- **詳細:** [./db-architecture-study/](./db-architecture-study/)

### 2. Redisの役割と必要性の検討

- **サマリ:** 非同期処理（Tika, OCR）、キャッシュ、セッション管理の3つの観点からRedisの必要性を再検討し、**アプリケーションの性能とユーザー体験を支える必須コンポーネント**であると結論付けました。
- **詳細:** [./2025-07-22_redis-usage-review.md](./2025-07-22_redis-usage-review.md)

### 3. ハイブリッド型情報価値評価システム

- **サマリ:** 検索結果の表示順を改善するため、活動スコア、新鮮度スコア、重要度スコアなどを組み合わせたハイブリッド型のスコアリングシステムを導入する計画です。
- **詳細:** [./2025-10-08_search-result-scoring-and-sorting-plan.md](./2025-10-08_search-result-scoring-and-sorting-plan.md)

---

## ✅ 現状サマリー (2025-10-11)

- **マルチテナントDBアーキテクチャ:** 物理DB分離は**実施見送り**となり、現行の単一DB（`tenant_id`方式）+ Buffer Pool最適化で実装が進められています。この方針は、移行元システムの実データ分析に基づいたもので、現状のデータ規模と負荷に対して最も合理的であると判断されています。
- **Redis:** 開発・本番環境ともに有効化され、非同期処理とキャッシュの基盤として機能しています。
- **検索スコアリング:** 計画段階であり、実装は今後のタスクとなります。

このカテゴリに含まれるドキュメントと、現在のコードベースの実装方針には、大きな齟齬がないことを確認済みです。
