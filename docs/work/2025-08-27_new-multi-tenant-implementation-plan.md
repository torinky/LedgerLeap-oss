# マルチテナント化 新実装計画

**日付:** 2025年8月27日
**改訂日:** 2025年8月27日

## 1. 背景と目的

当初のDB物理分離を先行する実装計画では、テストの複雑化やエラーのループといった問題が発生しました。この反省に基づき、本計画ではアプローチを全面的に見直し、「**まず単一DB上でテナントの論理的分離を完璧に実現し、その後、物理的なデータベース分離へ安全に移行する**」ことを目的とします。各ステップでの検証を容易にし、リスクを最小限に抑えます。

## 2. 改訂版・実装計画

### ステップ0: ライブラリの技術調査 (`stancl/tenancy`)

**目的:**
前回実装で課題となったライブラリの挙動、特にテスト環境での動作について事前に深く理解し、実装段階での予期せぬ問題を回避します。

**主要タスク:**
1.  **公式ドキュメントの精読とPoC:** `stancl/tenancy` の最新ドキュメントを精読し、主要機能について小規模なPoC（概念実証）コードによる動作確認を行います。
2.  **テナント識別方法の調査:** URLパス (`/{tenant_slug}/`) でテナントを識別する `InitializeTenancyByPath` ミドルウェアの動作とカスタマイズ方法を調査します。
3.  **シングルDBモードの調査:** `BelongsToTenant` グローバルスコープの正確な使い方、中央モデルとテナントモデルの連携方法を検証します。
4.  **DB分離モードへの移行戦略の調査:** `tenants:migrate` の動作や、シングルDBからマルチDBへのデータ移行に関するベストプラクティスを調査します。
5.  **テスト手法の確立:** Pest/PHPUnit環境でテナントコンテキストを扱うための公式な方法を調査・検証し、テストの記述方法を確立します。

**完了の定義:**
*   調査結果をまとめたドキュメント (`docs/development/stancl-tenancy-research.md`) が作成されていること。
*   主要機能のPoCが完了していること。

### ステップ1: テナントの論理的分離基盤の構築 (シングルDB)

**目的:**
全てのデータを単一の中央データベースに保持したまま、アプリケーション層でテナント間のデータが完全に分離されている状態を実現します。

**主要タスク:**
1.  **`stancl/tenancy` の導入と設定:** パッケージをインストールし、URLによるテナント識別を有効化します。（この段階ではテナントDB接続は使用しません）
2.  **`Tenant` モデルの拡張:** `Tenant` モデルに、URLで利用するための `slug` カラム（ユニーク）を追加するマイグレーションを作成します。
3.  **マイグレーションの修正:** `ledgers`, `folders` など、テナントに帰属するすべてのテーブルに `tenant_id` カラムを追加します。
4.  **グローバルスコープの適用:** `BelongsToTenant` トレイト等を関連モデルに適用し、DB操作が自動的に現在のテナントに限定されるようにします。
5.  **ユーザーとテナントの関連付け:** ユーザーがどのテナントに所属しているかを管理するロジックを実装します。

**完了の定義:**
*   アプリケーションがURL (`/{tenant_slug}/`) に応じてテナントを正しく識別できること。
*   全てのデータベースクエリが、現在のテナントのデータのみを操作するようになっていること。

### ステップ2: UIとビジネスロジックの完全対応

**目的:**
シングルDB・論理分離の状態で、全ての機能がテナントを跨いで干渉することなく、正しく動作することを保証します。

**主要タスク:**
1.  **テナント横断階層ナビゲーションの実装:** ユーザーが所属する全てのテナントとその配下のフォルダを一覧表示し、シームレスに切り替えられるUIコンポーネントを実装します。
2.  **全機能の動作確認と修正:** ログイン、マイポータル、台帳一覧、作成・編集、ワークフロー、検索など、全ての機能がテナントコンテキストで正しく動作することを確認・修正します。
3.  **テストの拡充と手動検証:** テナント間のデータが分離されていることを証明する自動テストを記述し、手動でも徹底的に検証します。

**完了の定義:**
*   全ての自動テストがパスすること。
*   階層ナビゲーションを含め、アプリケーションの全機能が複数のテナントで並行して利用しても問題なく動作すること。

### ステップ3: データベース物理分離への移行

**目的:**
アプリケーションの安定稼働を確認した上で、論理的に分離されているデータを物理的に異なるテナントDBへと移行します。

**主要タスク:**
1.  **テナントDBの準備:** `database/migrations/tenant` ディレクトリとマイグレーションを準備し、テナントDB接続を有効化します。
2.  **データ移行コマンドの作成:** 中央DBから各テナントDBへデータを移行するための、冪等性のあるArtisanコマンドを作成します。
3.  **データ移行の実行と検証:** メンテナンスモードでデータ移行を実行し、データが正しく移行されたことを検証します。
4.  **コードのクリーンアップ:** データ移行完了後、不要になった `tenant_id` カラムや関連スコープを削除し、コードを整理します。

**完了の定義:**
*   データ移行コマンドが正常に完了し、アプリケーションがテナントごとの物理DBを参照して動作すること。
*   `tenant_id` に依存していた古いコードがクリーンアップされていること。
