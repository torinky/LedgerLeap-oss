# マルチテナント化 新実装計画 (最終FIX版)

**日付:** 2025年8月28日

## 1. 背景と目的

当初のDB物理分離を先行する実装計画では、テストの複雑化やエラーのループといった問題が発生した。この反省に基づき、本計画ではアプローチを全面的に見直し、「**まず単一DB上でテナントの論理的分離を完璧に実現し、その後、物理的なデータベース分離へ安全に移行する**」ことを目的とする。各ステップでの検証を容易にし、リスクを最小限に抑える。

## 2. 全体アーキテクチャ

*   **フェーズ1 (ステップ1〜2): 論理分離**
    *   **データベース:** シングルDB構成。全データが同じDBに存在する。
    *   **データ分離:** `tenant_id` カラムとグローバルスコープを用いて、アプリケーションレベルでテナントのデータを論理的に分離する。
    *   **ライブラリ:** `stancl/tenancy` を、DB接続を切り替えない「シングルDBモード」で利用する。
*   **フェーズ2 (ステップ3): 物理分離**
    *   **データベース:** マルチDB構成。テナント固有データは、テナントごとの物理DBに分離される。
    *   **データ分離:** DB接続の切り替えによって、物理的にテナントのデータを分離する。
    *   **ライブラリ:** `stancl/tenancy` のDB接続切り替え機能 (`DatabaseTenancyBootstrapper`) を有効化する。

## 3. 改訂版・実装計画

### ステップ1: テナントの論理的分離基盤の構築 (シングルDB)

このステップでは、データベース層の基盤を構築します。UIの変更は含みません。

#### ステップ1-A: テナント管理のコア機能実装

*   **目的:** テナントを識別し、管理するための基本的な仕組みを導入する。
*   **主要タスク:**
    1.  `stancl/tenancy` をインストールし、`config/tenancy.php` を公開する。
    2.  `config/tenancy.php` で、テナント識別方法として `path` を有効化し、`DatabaseTenancyBootstrapper` を無効化（コメントアウト）する。
    3.  `Tenant` モデルと、`name`, `slug` カラムを持つ `tenants` テーブルのマイグレーションを作成する。
    4.  テナントの作成 (`tenants:create {name} {slug}`) と一覧表示 (`tenants:list`) を行うArtisanコマンドを作成する。
*   **完了の定義:**
    *   ✅ Artisanコマンドでテナントの作成・一覧表示ができる。
    *   ✅ `/{tenant_slug}/` のURLにアクセスした際に、`tenancy()->tenant` で対応するテナントのインスタンスが取得できる。

#### ステップ1-B: テナント固有モデルの分離対応

*   **目的:** テナントに直接帰属する主要なモデル（データ）を、論理的に分離する。
*   **主要タスク:**
    1.  以下のテーブルのマイグレーションファイルに `tenant_id` カラムを追加する。
        *   `folders`, `ledger_defines`, `ledgers`, `ledger_diffs`, `attached_files`, `tags`
    2.  上記に対応するEloquentモデルに `Stancl\Tenancy\Database\Concerns\BelongsToTenant` トレイトを追加する。
*   **完了の定義:**
    *   ✅ テナントAのコンテキストで `Ledger::all()` を実行すると、`tenant_id` がAのレコードのみが返却される。
    *   ✅ テナントAのコンテキストで `Ledger::create([...])` を実行すると、自動で `tenant_id` にAのIDが設定される。

### ステップ2: UIとビジネスロジックの完全対応

このステップでは、論理分離されたDB基盤の上で、アプリケーション全体が正しく動作するようにします。

#### ステップ2-A: テスト基盤の整備

*   **目的:** テナントを考慮した自動テストを効率的かつ安定して実行できる環境を構築する。
*   **主要タスク:**
    1.  `stancl/tenancy` のドキュメントに従い、Pest/PHPUnitの `TestCase.php` を設定する。
    2.  テスト内でテナントを初期化し、そのコンテキストで後続のテストを実行するためのヘルパーメソッド (`$this->initializeTenant($tenant)`) を実装する。
*   **完了の定義:**
    *   ✅ テストコード内で、特定のテナントのコンテキストをシミュレートしてテストを実行できる。

#### ステップ2-B: 認証とテナントコンテキストの適用

*   **目的:** ユーザーのログイン状態とテナントのコンテキストを正しく連携させる。
*   **主要タスク:**
    1.  `users` テーブルと `tenants` テーブルを関連付けるための中間テーブル `tenant_user` を作成する。
    2.  ログイン処理を修正し、認証後にユーザーが所属するデフォルトのテナントのURLへリダイレクトするようにする。
    3.  ユーザーをテナントに参加させるためのUIとロジックを実装する（例: 管理画面でのユーザー招待など）。
*   **完了の定義:**
    *   ✅ ログインすると、適切なテナントのダッシュボードにリダイレクトされる。
    *   ✅ ユーザーは、自分が所属していないテナントのURLにアクセスしても閲覧・操作できない。

#### ステップ2-C: テナント横断階層ナビゲーションの実装

*   **目的:** ユーザーが複数のテナントとフォルダをシームレスに切り替えられる、本アプリケーションの核となるナビゲーションUIを実装する。
*   **主要タスク:**
    1.  **トップメニュー（ナビゲーションバー）** に、現在のテナント名を表示し、クリックするとプルダウンメニューが開くUIコンポーネントを実装します。
    2.  プルダウンメニューの第一階層には、ログインユーザーが所属する**全てのテナント**が一覧表示されます。
    3.  各テナント名にカーソルを合わせるかクリックすると、その配下のフォルダ階層が**多層プルダウン（サブメニュー）**として展開されます。
    4.  テナント名またはフォルダ名をクリックすると、対応するコンテキスト（URL）に遷移します。
    5.  このコンポーネントの実装には、中央DBの `tenant_user` テーブルと、現在のテナントコンテキストの `folders` テーブルの両方からデータを取得してレンダリングする必要があるため、`Tenancy::central()` を活用します。
*   **完了の定義:**
    *   ✅ 複数のテナントに所属するユーザーが**トップメニューのプルダウン**を開くと、所属テナントが全て表示される。
    *   ✅ プルダウンからテナントやフォルダを選択すると、正しく画面が遷移し、表示データが切り替わる。

#### ステップ2-D: 全機能のテナント対応とテスト

*   **目的:** 全ての既存機能が、テナントが分離された状態で正しく動作することを保証する。
*   **主要タスク:**
    1.  台帳、フォルダ、ワークフロー、検索、通知など、全ての機能について、既存の自動テストをテナントコンテキストで実行するように修正する。
    2.  テストの失敗を元に、リレーションやデータ取得ロジックを修正する。
    3.  2つのブラウザでそれぞれ別のテナントとしてログインし、互いのデータに干渉できないことを手動テストで網羅的に確認する。
*   **完了の定義:**
    *   ✅ 全ての自動テストがパスする。
    *   ✅ 手動テストで、テナント間のデータが完全に分離されていることが確認できる。

### ステップ3: データベース物理分離への移行

**注意:** このステップは、アプリケーションの全機能が論理分離の状態で完全に安定稼働した後にのみ着手します。

#### ステップ3-A: 移行先の準備

*   **目的:** テナントごとの物理DBを作成し、データを投入できる状態にする。
*   **主要タスク:**
    1.  `config/tenancy.php` の `DatabaseTenancyBootstrapper` を有効化する。
    2.  `database/migrations/tenant` ディレクトリを作成し、テナント固有テーブルのマイグレーションファイルを移動する。
    3.  テナント作成時に、物理DBを作成し、マイグレーションを実行するよう `TenantCreated` イベントリスナーを修正する。
*   **完了の定義:**
    *   ✅ 新しいテナントを作成すると、対応する物理データベース（例: `tenant_xxx`）が作成され、テーブルが構築される。

#### ステップ3-B: データ移行コマンドの実装と検証

*   **目的:** 既存のテナントのデータを、中央DBから新しい物理DBへ安全に移行する。
*   **主要タスク:**
    1.  指定したテナントのデータを、中央DBからテナントDBへコピーするArtisanコマンド (`tenants:migrate-data {tenant_id}`) を作成する。
    2.  コマンドには、冪等性を持たせ、何度実行しても安全なように設計する。
    3.  開発環境でコマンドを実行し、データが欠損・破損なく完全に移行されることを検証する。
*   **完了の定義:**
    *   ✅ 移行コマンド実行後、テナントDBの内容が、移行前の中央DBのデータと完全に一致する。

#### ステップ3-C: コードのクリーンアップ

*   **目的:** 物理分離への移行が完了したことで不要になったコードを削除し、メンテナンス性を向上させる。
*   **主要タスク:**
    1.  テナント固有モデルから `Stancl\Tenancy\Database\Concerns\BelongsToTenant` トレイトを削除する。
    2.  テナント固有テーブルのマイグレーションから `tenant_id` カラムを削除する。
*   **完了の定義:**
    *   ✅ `tenant_id` に依存していたコードが全て削除されている。
    *   ✅ アプリケーションがテナントごとの物理DBを参照して、正常に動作する。
