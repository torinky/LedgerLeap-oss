# マルチテナント化 新実装計画（テーブルプレフィックス方式）

**日付:** 2025年8月28日

## 1. 背景と目的

本ドキュメントは、LedgerLeapプロジェクトに、事業部や子会社といった単位でデータを物理的に分離するマルチテナント機能を導入するにあたり、そのアーキテクチャ選定の経緯、議論、および最終的な決定事項を記録するものである。

当初、機能要件の検討を開始したが、その過程でユーザーから以下の重要な背景と懸念事項が提示された。

*   **移行元の存在:** 既存システム `DocumentCabinet` があり、これもMroongaを利用したDB切り替え型のマルチテナント構成で運用されている。
*   **データ規模:** 移行対象のデータはSQLダンプで40GB、15テナントに及び、規模の偏りも大きい。
*   **シングルDB構成への懸念:**
    *   **IDの枯渇:** 将来的にテーブルの主キーが枯渇するリスク。
    *   **パフォーマンス:** 大規模テナントの活動が、他の小規模テナントの応答性に影響を与えるリスク（ノイジーネイバー問題）。
    *   **リソース集中:** 単一のデータベースサーバーに負荷が集中するリスク。
    *   **法令・コンプライアンス:** ITAR, NISTなどの要件を満たすため、データの物理的な分離が求められる可能性。

これらの要件と懸念を解決することが、本アーキテクチャ選定の最終目的となった。

また、過去の失敗分析から、最初から完全な物理分離を目指すことはリスクが高いと判断された。この反省に基づき、本計画ではアプローチを全面的に見直し、「**まず単一DB上でテナントの論理的分離を完璧に実現し、その後、物理的なデータベース分離へ安全に移行する**」ことを目的とする。これにより、各ステップでの検証を容易にし、リスクを最小限に抑える。

## 2. アーキテクチャ方針（最終決定）

上記の背景と目的、そして過去の失敗分析を踏まえ、本プロジェクトのマルチテナント・アーキテクチャとして**「単一DB・テーブルプレフィックス方式」**を正式に採択する。

*   **データベース:** 単一のDB接続を常に使用する。
*   **データ分離:** `stancl/tenancy`の動的な設定変更機能を活用し、テナントのコンテキストに応じて、Eloquentモデルが参照するテーブルにプレフィックス（例: `tenant_a_`）を動的に付与する。
*   **テーブル構成:**
    *   **中央テーブル:** `users`, `roles` など、全テナントで共有するテーブル。プレフィックスは付与されない。
    *   **テナントテーブル:** `tenant_a_ledgers`, `tenant_b_ledgers` のように、テナントごとに物理的にテーブルが作成される。
*   **DB構成以外の要件:** UI（トップメニューでのテナント切り替え等）や機能スコープ（テナント横断検索は非対応等）に関する要件は、これまでの議論でFIXした内容を維持する。

## 3. 実装計画

### ステップ1: 動的テーブルプレフィックス機構の構築

*   **目的:** テナントに応じてテーブルプレフィックスが動的に切り替わる、本アーキテクチャの心臓部を実装し、検証する。
*   **タスク:**
    1.  **`stancl/tenancy` 設定:** `config/tenancy.php`で`path`によるテナント識別を有効化する。`DatabaseTenancyBootstrapper`は無効（コメントアウト）のままにする。
    2.  **`TenancyServiceProvider`のカスタマイズ:** `TenancyInitialized`イベントをリッスンし、DB接続設定のテーブルプレフィックスを `config(['database.connections.mysql.prefix' => ...])` を使って動的に設定するロジックを追加する。`TenancyEnded`イベントでプレフィックスをリセットする処理も実装する。
    3.  **Artisanコマンド作成:** テナントの基本情報（`tenants`テーブル）を作成するための`tenants:create {name} {slug}`コマンドを作成する。
*   **完了の定義:**
    *   ✅ `tenants:create`コマンドでテナントが作成できる。
    *   ✅ `/{tenant_slug}/` のURLにアクセスすると、LaravelのDB設定のテーブルプレフィックスが動的に`tenant_{slug}_`に変更されることを確認できる。

### ステップ2: モデルとマイグレーションの対応

*   **目的:** 全てのモデルとマイグレーションを、動的プレフィックス方式に対応させる。
*   **タスク:**
    1.  **中央モデルの対応:** `User`, `Role`, `AutoLink`など、中央管理する全モデルに`protected $table = 'users';`のようにプレフィックスが付与されないテーブル名を明示的に指定します。
    2.  **テナントモデルの対応:** `Ledger`, `Folder`などテナント固有モデルは、プレフィックスが自動適用されることを確認します。
    3.  **マイグレーションの整理:** 全てのマイグレーションファイルを`database/migrations`に配置します。
    4.  **テナントテーブル用マイグレーションの作成:** テナントテーブルを作成するマイグレーションを新規に作成します。このマイグレーションは、特定のコマンドによってのみ実行されるようにします。
    5.  **テナントDB構築コマンドの作成:** `tenants:migrate --tenant={id}`のような、指定したテナントのテーブル群（`tenant_xxx_ledgers`等）を作成するためのArtisanコマンドを実装します。
*   **完了の定義:**
    *   ✅ `php artisan migrate`で中央テーブルのみが作成される。
    *   ✅ `php artisan tenants:migrate --tenant=1`で、`tenant_xxx_`で始まるテーブル群が作成される。
    *   ✅ テナントAのコンテキストで`Ledger::all()`が`tenant_a_ledgers`を参照し、`User::all()`が`users`を参照することをテストで確認できる。

### ステップ3: UIとビジネスロジックの完全対応

*   **目的:** アプリケーション全体が、動的プレフィックス方式の上で正しく動作することを保証する。
*   **タスク:**
    1.  **テスト基盤の整備:** テナントコンテキストを切り替えてテストできるヘルパーを実装します。
    2.  **認証とテナント所属:** `users`と`tenants`を紐付ける中間テーブル`tenant_user`を作成し、ログイン後のリダイレクトを実装します。
    3.  **テナント横断ナビゲーションの実装:** トップメニューから多層プルダウンでテナントとフォルダを切り替えられるUIを実装します。DB接続が単一であるため、`Tenancy::central()`は不要で、Eloquentリレーションを駆使して実装できる見込みです。
    4.  **全機能のテストと修正:** 特に、中央モデルとテナントモデル間のリレーション（`Ledger`の作成者`User`など）が正しく動作するかを重点的に確認・修正します。
*   **完了の定義:**
    *   ✅ ログイン、テナント切り替え、台帳操作など、全ての機能が正しく動作する。
    *   ✅ 全ての自動テストがパスする。
    *   ✅ テナント間のデータが完全に分離されていることを手動テストで確認できる。
