# マルチテナント化 実装計画

**日付:** 2025年8月30日（更新）

## 1. 背景と目的

本ドキュメントは、LedgerLeapプロジェクトに、事業部や子会社といった単位でデータを物理的に分離するマルチテナント機能を導入するにあたり、そのアーキテクチャ選定の経緯、議論、および最終的な決定事項を記録するものである。

当初、機能要件の検討を開始したが、その過程でユーザーから以下の重要な背景と懸念事項が提示された。

*   **移行元の存在:** 既存システム `DocumentCabinet` があり、これもMroongaを利用したDB切り替え型のマルチテナント構成で運用されている。
*   **データ規模:** 移行対象のデータはSQLダンプで40GB、15テナントに及び、規模の偏りも大きい。
*   **シングルDB構成への懸念:**
    *   **IDの枯渇:** 将来的にテーブルの主キーが枯渇するリスク。
    *   **パフォーマンス:** 大規模テナントの活動が、他の小規模テナントの応答性に影響を与えるリスク（ノイジーネイバー問題）。
    *   **リソース集中:** 単一のデータベースサーバーに負荷が集中するリスク。
    *   **法令・コンプライアンス:** ITAR, NISTなどの要件を満たすため、データの物理的な分離が求められる可能性。

これらの要件と懸念を解決することが、本アーキテクチャ選定の最終目的となった。

また、過去の失敗分析から、最初から完全な物理分離を目指すことはリスクが高いと判断された。この反省に基づき、本計画ではアプローチを全面的に見直し、「**まず単一DB上でテナントの論理的分離を完璧に実現し、その後、物理的なデータベース分離へ安全に移行する**」ことを目的とする。これにより、各ステップでの検証を容易にし、リスクを最小限に抑える。

## 2. アーキテクチャ方針（当初決定）

上記の背景と目的、そして過去の失敗分析を踏まえ、本プロジェクトのマルチテナント・アーキテクチャとして**「単一DB・テーブルプレフィックス方式」**を採択した。

*   **データベース:** 単一のDB接続を常に使用する。
*   **データ分離:** `stancl/tenancy`の動的な設定変更機能を活用し、テナントのコンテキストに応じて、Eloquentモデルが参照するテーブルにプレフィックス（例: `tenant_a_`）を動的に付与する。
*   **テーブル構成:**
    *   **中央テーブル:** `users`, `roles` など、全テナントで共有するテーブル。プレフィックスは付与されない。
    *   **テナントテーブル:** `tenant_a_ledgers`, `tenant_b_ledgers` のように、テナントごとに物理的にテーブルが作成される。
*   **DB構成以外の要件:** UI（トップメニューでのテナント切り替え等）や機能スコープ（テナント横断検索は非対応等）に関する要件は、これまでの議論でFIXした内容を維持する。

## 3. 実装の経緯と方針転換 (2025-08-30追記)

当初の「テーブルプレフィックス方式」に基づき実装を進めたが、深刻な問題に直面し、公式ドキュメントの再調査を経て、アーキテクチャを根本的に見直す決定に至った。以下にその詳細な経緯を記録する。

### 3.1. 当初アプローチ：動的テーブルプレフィックスの実装

計画に基づき、以下の実装を行った。

1.  **`stancl/tenancy` の設定:**
    *   `config/tenancy.php` を変更し、テナントリゾルバを `PathTenantResolver` に設定。テナントごとのDB接続を切り替える `DatabaseTenancyBootstrapper` は無効化した。

2.  **動的プレフィックス設定:**
    *   `app/Providers/TenancyServiceProvider.php` をカスタマイズ。
    *   `TenancyInitialized` イベントをリッスンし、`config(['database.connections.mysql.prefix' => "tenant_{$tenant->getTenantKey()}_"])` を実行して、リクエスト処理中に動的にテーブルプレフィックスを適用するロジックを実装。
    *   `TenancyEnded` イベントで、プレフィックスを空に戻す処理 (`config(['database.connections.mysql.prefix' => ''])`) を実装した。

3.  **ルーティングの分離:**
    *   テナントに属するルート (`web.php`) と、テナントに依存しない中央ルート (`central.php`) を明確に分離。
    *   `app/Providers/RouteServiceProvider.php` を変更し、`InitializeTenancyByPath` ミドルウェアがテナントルート群にのみ適用されるようにマッピングを調整した。

4.  **テナント作成コマンド:**
    *   `php artisan tenants:create {name} {id}` を実行することで、`tenants` テーブルに新しいテナントレコードを作成するカスタムコマンドを実装した。

### 3.2. 問題の発生と原因調査

上記の実装後、テストテナントを作成し、`curl` コマンドでテナント固有URL (`/a/test`) と中央URL (`/test`) へのアクセスを試みたが、一貫性のないルーティングエラー（404 Not Found, 500 Server Error）に悩まされた。

デバッグの過程で、`TenancyServiceProvider` 内の `makeTenancyMiddlewareHighestPriority()` メソッドが、意図せず `InitializeTenancyByPath` ミドルウェアを**グローバル**に適用してしまい、中央ルートへのアクセス時にテナントが見つからずエラーを引き起こしていることを特定。このメソッドをコメントアウトすることで、ようやくルーティングが安定した。

しかし、この一連の複雑なデバッグ作業は、「我々のアプローチが `stancl/tenancy` の想定する標準的な利用方法から外れているのではないか」という強い疑念を抱かせるきっかけとなった。

### 3.3. 重大な発見と方針転換

公式ドキュメントを改めて徹底的に調査した結果、以下の**極めて重要な事実**が判明した。

> **`stancl/tenancy` のシングルデータベース戦略は、テーブルプレフィックスを使用しない。**

公式に推奨されているシングルデータベース・アーキテクチャは、以下の方法で実現される。

*   **`tenant_id` カラム:** テナントに属する全てのテーブルに `tenant_id` (または任意のカラム名) を追加する。
*   **`BelongsToTenant` トレイト:** 対応する全てのEloquentモデルに `Stancl\Tenancy\Database\Concerns\BelongsToTenant` トレイトを追加する。

このトレイトが、モデルに対する全てのクエリ（SELECT, INSERT, UPDATE, DELETE）に `WHERE tenant_id = 'current_tenant_id'` 句を自動的に付与し、データの論理的な分離を保証する。

我々が実装した「`ServiceProvider` でDB接続設定を動的に書き換える」という方法は、ドキュメントに記載のない**非公式なハック**であり、予期せぬ副作用や将来のバージョンアップでの破綻リスクが極めて高いと判断した。

この発見に基づき、**実装アーキテクチャをテーブルプレフィックス方式から、公式推奨の `tenant_id` カラム方式へ全面的に移行する**ことを決定する。

## 4. 新しい実装計画（`tenant_id`方式）

### ステップ1: クリーンアップと基本設定

*   **目的:** 非公式なプレフィックス方式の実装を完全に撤廃し、公式アプローチの基盤を整える。
*   **タスク:**
    1.  **`TenancyServiceProvider`のクリーンアップ:** `boot()` メソッドに追加した `TenancyInitialized` と `TenancyEnded` のイベントリスナ、および関連するプレフィックス設定ロジックを全て削除する。
    2.  **`config/tenancy.php`の再確認:** 設定が `tenant_id` 方式に適合しているか確認する。特に `bootstrappers` の `DatabaseTenancyBootstrapper` が無効化されていることを再度確認する。
    3.  **ルーティングの確認:** `RouteServiceProvider` の設定は、中央・テナントのルート分離に依然として有効なため、現状を維持する。

### ステップ2: マイグレーションとモデルの対応

*   **目的:** 全てのデータ構造とモデルを `tenant_id` 方式に対応させる。
*   **タスク:**
    1.  **マイグレーションの修正:** テナントに属する全てのテーブル（`ledgers`, `folders`, `users`等）のマイグレーションファイルに `tenant_id` カラムを追加する。`tenants` テーブルへの外部キー制約も設定する。
        *   `$table->string('tenant_id');`
        *   `$table->foreign('tenant_id')->references('id')->on('tenants')->onUpdate('cascade')->onDelete('cascade');`
    2.  **モデルへのトレイト適用:** テナントに属する全てのEloquentモデルに `Stancl\Tenancy\Database\Concerns\BelongsToTenant` トレイトを追加する。
    3.  **データベースの再構築:** `php artisan migrate:fresh` を実行し、新しいスキーマでデータベースを再構築する。
    4.  **テナント作成コマンドの再実行:** `tenants:create` コマンドでテストテナントを再作成する。

### ステップ3: UIとビジネスロジックの適応

*   **目的:** アプリケーション全体が `tenant_id` 方式の上で正しく動作することを保証する。
*   **タスク:**
    1.  **バリデーションルールの修正:** `Rule::unique('posts')` のようなルールは自動的にスコープされないため、`Rule::unique('posts')->where('tenant_id', tenant('id'))` のように、手動でテナントIDの条件を追加する必要がある箇所を特定し、修正する。
    2.  **テスト基盤の整備:** テナントコンテキストを切り替えてテストできるヘルパーを実装する。
    3.  **認証とテナント所属の再設計:** `users` テーブルもテナントに属するため、認証プロセスやユーザー管理機能がテナントコンテキストで正しく動作するかを再検証・修正する。
    4.  **全機能のテストと修正:** 特に、異なるモデル間のリレーションが正しく動作するかを重点的に確認・修正する。

### ステップ4: 総合テストと検証

*   **目的:** テナント間のデータ分離が完全であることを保証し、全機能が正常に動作することを確認する。
*   **タスク:**
    1.  テナントAで作成したデータが、テナントBでは一切参照・操作できないことを確認するテストを作成する。
    2.  中央画面（管理者用の画面など、将来実装）とテナント画面が共存できることを確認する。
    3.  ログイン、テナント切り替え、台帳操作など、全ての機能について手動テストを実施する。
