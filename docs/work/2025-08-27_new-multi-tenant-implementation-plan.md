# マルチテナント化 新実装計画 (最終FIX版)

**日付:** 2025年8月27日

## 1. 背景と目的

当初のDB物理分離を先行する実装計画では、テストの複雑化やエラーのループといった問題が発生した。この反省に基づき、本計画ではアプローチを全面的に見直し、「**まず単一DB上でテナントの論理的分離を完璧に実現し、その後、物理的なデータベース分離へ安全に移行する**」ことを目的とする。各ステップでの検証を容易にし、リスクを最小限に抑える。

## 2. 改訂版・実装計画

### ステップ0: ライブラリの技術調査 (`stancl/tenancy`)

**目的:**
前回実装で課題となったライブラリの挙動、特にテスト環境での動作について事前に深く理解し、実装段階での予期せぬ問題を回避する。

**完了の定義:**
*   本ステップの調査結果は、`docs/development/stancl-tenancy-research.md` に記録されている。

### ステップ1: テナントの論理的分離基盤の構築 (シングルDB)

**目的:**
全てのデータを単一の中央データベースに保持したまま、アプリケーション層でテナント間のデータが完全に分離されている状態を実現する。

**主要タスク:**
1.  **`stancl/tenancy` の導入:** `composer require stancl/tenancy` を実行し、パッケージをインストールする。
2.  **設定ファイルの公開:** `php artisan vendor:publish --provider="Stancl\Tenancy\TenancyServiceProvider"` を実行し、`config/tenancy.php` を公開する。
3.  **基本設定の実施:** `config/tenancy.php` を編集し、URLによるテナント識別を有効化する。また、調査結果に基づき、シングルDBモードで動作するように設定する（`DatabaseTenancyBootstrapper`の無効化など）。
4.  **`Tenant` モデルとマイグレーションの作成:**
    *   `Tenant` モデルを作成する。
    *   `tenants` テーブル用のマイグレーションを作成し、URL識別のための `slug` カラム（ユニーク）を追加する。
5.  **テナント管理コマンドの作成:** テナントの作成・削除を行うための、カスタムArtisanコマンド (`tenants:create` 等) を新規に作成する。
6.  **既存マイグレーションの修正:** `ledgers`, `folders` など、テナントに帰属するすべてのテーブルに `tenant_id` カラムを追加する。
7.  **グローバルスコープの適用:** `BelongsToTenant` トレイト等を関連モデルに適用し、DB操作が自動的に現在のテナントに限定されるようにする。

**完了の定義:**
*   `tenants:create` コマンドで、`slug` を持つテナントが作成できること。
*   アプリケーションがURL (`/{tenant_slug}/`) に応じてテナントを正しく識別できること。
*   全てのデータベースクエリが、意識せずとも現在のテナントのデータのみを操作するようになっていること。

### ステップ2: UIとビジネスロジックの完全対応

**目的:**
シングルDB・論理分離の状態で、全ての機能がテナントを跨いで干渉することなく、正しく動作することを保証する。

**主要タスク:**
1.  **テナント横断階層ナビゲーションの実装:** ユーザーが所属する全てのテナントとその配下のフォルダを一覧表示し、シームレスに切り替えられるUIコンポーネントを実装する。
2.  **全機能の動作確認と修正:** ログイン、ユーザーとテナントの関連付け、マイポータル、台帳一覧、作成・編集、ワークフロー、検索など、全ての機能がテナントコンテキストで正しく動作することを確認・修正する。
3.  **テスト基盤の整備:** `RefreshDatabase`トレイトが利用できない制約に対応するため、テストの実行前後でデータベースをクリーンな状態に保つための、テスト用ヘルパーやカスタムトレイトを実装する。
4.  **自動テストの記述:** 上記で整備したテスト基盤の上で、テナントコンテキストを考慮した自動テストを記述する。その際、`Event::fake()`の使用は避け、必要な場合はテナンシー関連のイベントを除外するなど、調査結果で判明した注意点を遵守する。
5.  **手動テストによる最終検証:** 自動テストを補完するため、複数のテナントとユーザーを用いて、データが完全に分離されていることを手動で最終確認する。

**完了の定義:**
*   全ての自動テストがパスすること。
*   階層ナビゲーションを含め、アプリケーションの全機能が複数のテナントで並行して利用しても問題なく動作すること。

### ステップ3: データベース物理分離への移行

**目的:**
アプリケーションの安定稼働を確認した上で、論理的に分離されているデータを物理的に異なるテナントDBへと移行する。

**主要タスク:**
1.  **テナントDBの準備:** `database/migrations/tenant` ディレクトリとマイグレーションを準備し、テナントDB接続を有効化する。
2.  **データ移行コマンドの作成:** 中央DBから各テナントDBへデータを移行するための、冪等性のあるArtisanコマンドを作成する。
3.  **データ移行の実行と検証:** メンテナンスモードでデータ移行を実行し、データが正しく移行されたことを検証する。
4.  **コードのクリーンアップ:** データ移行完了後、不要になった `tenant_id` カラムや関連スコープを削除し、コードを整理する。

**完了の定義:**
*   データ移行コマンドが正常に完了し、アプリケーションがテナントごとの物理DBを参照して動作すること。
*   `tenant_id` に依存していた古いコードがクリーンアップされていること。
