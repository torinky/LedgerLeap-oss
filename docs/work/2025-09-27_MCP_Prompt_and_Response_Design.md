# MCPプロンプトと応答内容の設計案

**日付:** 2025年9月27日

**関連ドキュメント:** [LedgerLeap MCPサーバー実装計画](./2025-09-27_MCP_Server_Implementation_Plan.md)

---

## 1. 概要

本ドキュメントは、`@docs/work/2025-09-27_Docs_Persona_UseCase_Scenario.md` で定義されたペルソナとユースケースに基づき、`gemini` CLIが `@ledgerleap-api` MCPツールを通じてユーザーに応答する際の、具体的な質問応答例と設計方針を定義する。
これは、今後のプロンプトチューニングおよびMCPツール改修のたたき台となることを目的とする。

## 2. MCPの応答内容検討：ペルソナ別のアプローチ

現在のMCPツールは、主にデータベースの情報をJSON形式で返す。これを各ペルソナのニーズに合わせて加工・要約して提示することで、`gemini`の応答価値を最大化する。

### 2.1. 実務担当者 (Operational Staff)

**ニーズ:** 具体的で、タスク指向。シンプルで分かりやすい情報。

#### 質問例1: 「昨日私が作成した日報を見せて。」

*   **Geminiの内部処理:**
    1.  `@ledgerleap-api search-ledgers-tool` を呼び出す。
    2.  パラメータとして `q: "日報"` と、認証情報から取得した `creator_id` を渡す。（※現状のツールに `creator_id` や期間指定フィルタがないため、これはツールの改善点となる）
    3.  返されたJSONデータ（複数件の可能性がある）を解析する。

*   **Geminiの応答例（あるべき姿）:**
    > 昨日あなたが作成した日報は1件です。
    >
    > *   **件名:** 2025年9月26日 営業日報
    > *   **ステータス:** 承認待ち (点検完了)
    > *   **更新日時:** 2025-09-26 18:30
    >
    > 詳細を確認しますか？

*   **ポイント:**
    *   JSONを生で返すのではなく、要点を抜き出してリスト形式で提示する。
    *   件数やステータスなど、ユーザーが知りたいであろう情報を要約して冒頭に示す。
    *   次のアクション（詳細確認）を促す。

#### 質問例2: 「至急対応が必要な承認待ちのタスクはある？」

*   **Geminiの内部処理:**
    1.  `@ledgerleap-api search-ledgers-tool` を呼び出す。
    2.  パラメータとして、認証情報から取得した担当者IDと `status: "PENDING_APPROVAL"` を渡す。（※これもツールの改善点）
    3.  返されたJSONデータから、期限が近いものや優先度が高いものを抽出する。

*   **Geminiの応答例（あるべき姿）:**
    > はい、あなた宛ての承認待ちタスクが3件あります。特に以下の1件は本日が期限です。
    >
    > *   **台帳:** 経費精算申請書
    > *   **申請者:** 営業部 佐藤さん
    > *   **申請日時:** 2025-09-26 15:00
    > *   **件名:** 交通費精算 (9月分)
    >
    > すぐに内容を確認して承認作業を行いますか？

*   **ポイント:**
    *   単なるリストではなく、「至急」という意図を汲み取り、優先順位をつけて提示する。
    *   誰からの、何の申請であるかを明確にし、判断を助ける情報を提供する。
    
    #### 質問例3: 「契約書Aの最新版と一つ前の版で、どこが変わったか教えて。」
    
    *   **Geminiの内部処理:**
        1.  `@ledgerleap-api search-ledgers-tool` で "契約書A" を検索し、対象の台帳IDを特定する。
        2.  `@ledgerleap-api get-ledger-diffs-tool` を台帳IDで呼び出し、最新と一つ前のバージョンの差分情報を取得する。（※このツールは新規作成が必要）
        3.  各差分情報から添付ファイルIDを特定し、`@ledgerleap-api read-attachment-tool` でそれぞれのファイル内容を読み取る。（※このツールは新規作成が必要）
        4.  2つのファイル内容を比較し、差分を抽出・要約する。
    
    *   **Geminiの応答例（あるべき姿）:**
        > 契約書Aの最新版と一つ前の版を比較しました。主な変更点は以下の3点です。
        >
        > *   **3条1項:** 契約期間が「2025年12月末まで」から「2026年3月末まで」に延長されています。
        > *   **5条2項:** 違約金に関する記述が追加されています。
        > *   **末尾:** 署名欄に「法務部確認印」が追加されています。
        >
        > 差分をハイライトした比較表示をしますか？
    
    *   **ポイント:**
        *   台帳のバージョン管理機能（`LedgerDiff`）と連携し、添付ファイルの変更履歴を追跡・比較する。
        *   契約書のような構造化された文書の場合、変更箇所を条文レベルで具体的に示す。

### 2.2. 管理者 (Administrator / Manager)

**ニーズ:** 網羅的、統計的、監査可能。サマリーや異常値の可視化。

#### 質問例1: 「先週、システム全体でどれくらいの台帳が作成されたか、種類ごとに集計して。」

*   **Geminiの内部処理:**
    1.  `@ledgerleap-api search-ledgers-tool` を期間指定で呼び出す。（※改善点）
    2.  返された全件のJSONデータを取得し、`ledger_define_id` ごとにカウントを集計する。
    3.  `@ledgerleap-api get-ledger-defines-tool` を呼び出し、IDと台帳定義名をマッピングする。

*   **Geminiの応答例（あるべき姿）:**
    > 先週 (2025/09/20〜2025/09/26) に作成された台帳は、合計で **58件** です。
    >
    > 内訳は以下の通りです。
    >
    > *   営業日報: 25件
    > *   議事録: 15件
    > *   経費精算申請書: 10件
    > *   その他: 8件
    >
    > 詳細なリストを表示しますか？

*   **ポイント:**
    *   ユーザーの意図通り、統計情報に加工して提示する。
    *   複数のツール（`search` と `get-defines`）を内部で組み合わせて呼び出し、より付加価値の高い情報を生成する。

#### 質問例2: 「ユーザーAの最近のアクティビティで、何か不審な点はある？」

*   **Geminiの内部処理:**
    1.  `@ledgerleap-api search-activity-log-tool` を `user_id: A` で呼び出す。（※このツールは新規作成が必要）
    2.  返されたログデータを解析し、通常の業務時間外のアクセスや、大量のファイルダウンロード、権限変更などの特異な操作を検出する。

*   **Geminiの応答例（あるべき姿）:**
    > ユーザーAの直近24時間のアクティビティを確認しました。
    >
    > 通常の台帳閲覧・編集操作がほとんどですが、1点確認をお勧めする操作があります。
    >
    > *   **日時:** 2025-09-27 03:15 (深夜)
    > *   **操作:** ファイルの一括ダウンロード (15件)
    > *   **対象フォルダ:** /プロジェクトX/機密資料
    >
    > この操作に心当たりはありますか？ 詳細なログを確認しますか？

*   **ポイント:**
    *   単にログを羅列するのではなく、「不審な点」という観点で情報をフィルタリング・分析して提示する。
    *   監査の観点から、具体的な日時や対象を明記する。
    
    #### 質問例3: 「今月提出されたすべての営業日報のうち、『コンプライアンス』の記載がないものをリストアップして。」
    
    *   **Geminiの内部処理:**
        1.  `@ledgerleap-api search-ledgers-tool` を使い、まず「今月提出された営業日報」の総数を取得する。
        2.  次に、同じ期間・種類の条件に加え、全文検索キーワード `q: "コンプライアンス"` を指定して再度検索を実行する。
        3.  1の結果と2の結果を比較し、2に含まれていない台帳のリストを作成する。
    
    *   **Geminiの応答例（あるべき姿）:**
        > 今月提出された営業日報25件のうち、「コンプライアンス」の記載がなかったものは以下の2件です。
        >
        > *   **提出者:** 営業1課 田中さん (2025-09-15提出)
        > *   **提出者:** 営業2課 鈴木さん (2025-09-22提出)
        >
        > これらの担当者に確認を促しますか？
    
    *   **ポイント:**
        *   添付ファイルの内容を含む全文検索機能を活用し、特定のキーワードが含まれて「いない」文書を効率的に特定する。
        *   監査やルール遵守の確認といった管理業務を支援する。

### 2.3. 開発者 (Developer)

**ニーズ:** 技術的詳細、デバッグ情報、APIの生レスポンス。

#### 質問例1: 「キーワード『テストです』で台帳検索した結果のJSONを整形して見せて。」

*   **Geminiの内部処理:**
    1.  `@ledgerleap-api search-ledgers-tool` を `q: "テストです"` で呼び出す。
    2.  返されたJSONをそのまま整形する。

*   **Geminiの応答例（あるべき姿）:**
    > `@ledgerleap-api search-ledgers-tool` を実行しました。以下が結果のJSONです。
    >
    > ```json
    > {
    >     "ledgers": [
    >         {
    >             "id": 112,
    >             "tenant_id": "tenantc",
    >             "ledger_define_id": 15,
    >             // ... (省略) ...
    >         }
    >     ],
    >     "total": 1
    > }
    > ```

*   **ポイント:**
    *   開発者のニーズに応え、あえて加工せず生データを提供する。
    *   どのツールを呼び出したかを明記することで、開発者がAPIの挙動を理解しやすくする。

#### 質問例2: 「台帳定義IDが15のスキーマ定義を教えて。」

*   **Geminiの内部処理:**
    1.  `@ledgerleap-api get-ledger-defines-tool` を呼び出す。
    2.  返されたJSON配列の中から、`id: 15` のオブジェクトを検索し、抽出する。

*   **Geminiの応答例（あるべき姿）:**
    > 台帳定義ID `15` のスキーマ情報は以下の通りです。
    >
    > ```json
    > {
    >     "id": 15,
    >     "tenant_id": "tenantc",
    >     "folder_id": 1,
    >     "title": "sit",
    >     "column_define": [
    >         {
    >             "id": 0,
    >             "name": "ut",
    >             "type": "chk",
    >             // ... (省略) ...
    >         }
    >     ],
    >     // ... (省略) ...
    > }
    > ```

*   **ポイント:**
    *   大量のデータの中から、ユーザーが要求した特定の情報だけを正確に抜き出して提示する。
    *   開発者がデータ構造をすぐに理解できるよう、JSON形式で整形して見せる。
    
    #### 質問例3: 「台帳ID: 123 に添付されている `invoice.pdf` から抽出されたテキストインデックスの内容を見せて。」
    
    *   **Geminiの内部処理:**
        1.  `@ledgerleap-api get-attachment-details-tool` のような新ツールを、台帳ID `123` とファイル名 `invoice.pdf` を指定して呼び出す。（※このツールは新規作成が必要）
        2.  返されたJSONデータから、`content` プロパティ（抽出されたテキストが格納されている）を抜き出して表示する。
    
    *   **Geminiの応答例（あるべき姿）:**
        > 台帳ID `123` の添付ファイル `invoice.pdf` から抽出され、検索インデックスに登録されているテキストは以下の通りです。
        >
        > ```text
        > 株式会社 LedgerLeap
        > ご請求書
        >
        > 品名: システム利用料
        > 金額: ¥100,000-
        > (TikaまたはOCRによって抽出された最終的なテキスト)
        > ```
    
    *   **ポイント:**
        *   ファイルから抽出され、最終的に全文検索の対象となるテキストデータそのものを確認したい、という開発者のデバッグニーズに応える。
        *   複数のエンジン結果を比較するのではなく、DBに保存された「正」のデータを確認するシナリオに修正。
