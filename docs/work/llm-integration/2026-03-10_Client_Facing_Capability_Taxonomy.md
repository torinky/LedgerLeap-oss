# LedgerLeap client-facing capability taxonomy

**作成日:** 2026年03月10日
**ドキュメント種別:** 作業ファイル（client-facing capability 定義）
**関連Issue:** [#85](https://github.com/torinky/LedgerLeap/issues/85), [#83](https://github.com/torinky/LedgerLeap/issues/83)

## 1. 目的

この文書は、LedgerLeap を **MCP / API 経由で使うクライアント** に見せる能力を、WebUI で観測できる概念だけで整理するための taxonomy です。

ここで扱うのは次の3点です。

1. クライアントに見せる能力名
2. 各能力で扱う入力・出力・業務フロー
3. ペルソナ別の初期 skill セット

## 2. 対象読者

- MCP クライアント
- REST API クライアント
- クライアント上で動く LLM
- クライアント設定を行う運用担当者

## 3. client-facing の SoT

client-facing で説明してよいのは、**WebUI で見える概念** のみです。

| 概念 | client-facing での説明 |
|---|---|
| 台帳定義 | どの種類の記録を扱うかを表すテンプレート |
| レコード | 実際に保存された1件の記録 |
| 列 | レコードに入力・表示される項目 |
| 型 | 文字、日付、数値、選択肢、添付ファイルなどの入力形式 |
| フォルダ | 記録を整理する保存先 |
| タグ | 絞り込みや識別に使うラベル |
| 状態 | 作成中、点検待ち、承認待ち、承認済みなどの進行状況 |
| 添付ファイル | レコードに紐づく資料 |
| 履歴 | 変更・確認・承認の流れ |

## 4. client-facing に出さないもの

次は developer-facing に閉じます。

- DB テーブル / カラム設計
- Mroonga / tenancy / Laravel / Livewire などの内部事情
- cast / queue / service class / model 名
- 実装都合によるイベント名や内部メソッド名

## 5. capability 一覧

## 5.1 ledger-search

**何をする能力か**
必要な記録を検索し、候補を一覧で見つけ、必要なものだけ詳細確認へ進む能力。

**主な入力**

- キーワード
- フォルダ
- タグ
- 期間
- 状態

**標準フロー**

1. まず短いキーワードで候補を探す
2. 候補が多い場合はフォルダ・タグ・期間で絞る
3. 一覧でタイトル、更新日時、状態、保存先を確認する
4. 必要なレコードだけ詳細を開く
5. 添付資料がある場合は、その内容も含めて確認する

**返すべき情報**

- 候補一覧
- レコードの基本情報
- 詳細確認へ進むための識別情報

## 5.2 ledger-create

**何をする能力か**
新しいレコードを作成し、必要な項目を入力して保存・提出する能力。

**主な入力**

- 台帳定義
- 保存先フォルダ
- 必須項目
- 任意項目
- 添付ファイル

**標準フロー**

1. どの台帳定義に登録するかを決める
2. 保存先フォルダを確認する
3. 必須項目を埋める
4. 必要に応じて添付ファイルを追加する
5. 下書き保存または提出を行う

**返すべき情報**

- 保存結果
- 作成されたレコードの状態
- 次に必要な確認や承認の有無

## 5.3 ledger-update

**何をする能力か**
既存レコードの内容を確認し、必要な箇所だけ更新する能力。

**主な入力**

- 対象レコード
- 変更したい項目
- 変更理由（必要時）

**標準フロー**

1. まず対象レコードを特定する
2. 現在の状態と編集可否を確認する
3. 変更したい項目だけ更新する
4. 必要なら理由やコメントを残す
5. 保存後の状態変化を確認する

**返すべき情報**

- 更新後の主要項目
- 状態変更の有無
- 再提出や再確認が必要かどうか

## 5.4 workflow-review

**何をする能力か**
自分に割り当てられた確認・承認タスクを処理する能力。

**主な入力**

- 自分の未処理タスク
- 対象レコード
- コメント
- 承認 / 差し戻しの判断

**標準フロー**

1. 未処理タスクを一覧で確認する
2. 優先度や期限が近いものから詳細を開く
3. 内容と添付資料を確認する
4. 承認・点検完了・差し戻しを選ぶ
5. コメントが必要なら残す

**返すべき情報**

- タスクの現在件数
- 処理対象レコード
- 実行後の状態

## 5.5 activity-audit

**何をする能力か**
誰が、いつ、何をしたかを追跡する能力。

**主な入力**

- 期間
- ユーザー
- フォルダ
- 操作種別

**標準フロー**

1. 対象期間を決める
2. 必要ならユーザーや保存先で絞る
3. 変更・確認・承認などの履歴を見る
4. 追跡したいレコードへ移動する

## 5.6 analytics-report

**何をする能力か**
件数や進捗を集計し、運用状況を把握する能力。

**主な入力**

- 集計期間
- 対象台帳
- 状態
- 作成者または担当者

**標準フロー**

1. 集計したい期間を決める
2. 台帳や状態で対象を絞る
3. 件数・傾向・滞留を確認する
4. 必要なレコードへ掘り下げる

## 6. ペルソナ別の初期 skill セット

| ペルソナ | 初期 skill セット | 主なシナリオ |
|---|---|---|
| 実務担当者 | `ledger-search`, `ledger-create`, `workflow-review` | 日報の作成、過去記録の参照、承認待ちタスク処理 |
| 管理者 | `ledger-search`, `workflow-review`, `activity-audit`, `analytics-report` | 台帳定義の確認、活動監査、運用集計 |
| 現場リーダー | `ledger-search`, `ledger-update`, `workflow-review` | 代理更新、チーム内確認、差し戻し対応 |

## 7. local model 向け記述ルール

client-facing skill / prompt / resource は、次のルールで短く保ちます。

1. 1 capability = 1つの主目的に絞る
2. 一覧 → 詳細の二段階で説明する
3. 必須入力は 3〜6 個程度の短い箇条書きにする
4. エラー説明は「何が足りないか」「次に何を聞くか」を短く示す
5. 実装名ではなく業務語彙を使う

## 8. Sprint 2 の完了観点

- `ledger-search` / `ledger-create` / `ledger-update` / `workflow-review` を業務能力として説明できる
- ペルソナ別の初期 skill セットを説明できる
- client-facing で内部実装事情を出さない境界が定義されている
- local model 向けの短文化ルールが整理されている
