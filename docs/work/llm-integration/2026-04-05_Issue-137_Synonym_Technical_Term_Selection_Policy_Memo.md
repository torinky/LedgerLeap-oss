# Issue 137: 同義語・技術用語候補の分離と選択ポリシー

**対象Issue:** `#137`  
**親Issue:** `#136`  
**関連:** `#138`, `#139`

## 0. サマリー

- 同義語は、日本語として一般的な同じ意味合いの用語として扱う
- 技術用語は、業界内や会社内で使われる用語として扱う
- 同義語は LLM 対話の文脈で広く候補化するが、最終的には大きく刈り込む
- 技術用語は全員が必要に応じて使う補助候補として扱う
- #137 の結論は「検索の正規化を増やすこと」ではなく、「同義語と技術用語の候補をどう選ぶかのルールを固定すること」

## 1. 目的

同義語は増えやすいため、単純な自動展開ではなく、**対話文脈に応じて LLM が検索候補を選ぶ**前提を固める。

このメモでは、**技術用語は全員が必要に応じて使う**という前提に立ちつつ、
同義語は LLM 対話の文脈で大きく刈り込まれやすい候補として扱う。
そのうえで、業務に慣れている人 / 慣れていない人で検索パスとワードの選び方がどう変わるかを定義する。

## 2. 既存前提

- 半角/全角、空白などの正規化は既存検索前処理で対応済み
- そのため、`A2-1` では再実装しない
- `SearchContext` / `SynonymService` は既に存在する
- `SearchRequest` 側でも検索前処理がある

## 3. 候補バケット

| バケット | 役割 | 例 | 扱い |
|---|---|---|---|
| 正準語 | 検索の基準語 | `請求` | 会話文脈の軸 |
| 同義語 | 日本語として一般的な同じ意味合いの用語 | `請求書`, `会計帳票`, `納品書` | LLM 対話では幅広く候補に出すが、最終的には大きく絞る |
| 技術用語 | 業界内や会社内で使われる用語 | `売掛`, `与信`, `入金予定` | 全員が必要に応じて使う補助候補として出す |
| 保留候補 | 文脈不足で採用しない語 | - | trace にのみ残す |

## 4. 選択ポリシー

### 4.1 業務/ドメインに慣れていない人

- まず同義語で入口を作る
- ただし、同義語は LLM 対話の文脈で積極的に刈り込む
- 検索結果を見ながら、必要なときだけ技術用語へ寄せる
- 自然文に近い q を維持し、会話で語彙を少しずつ揃える

### 4.2 業務/ドメインに慣れている人

- 最初から技術用語を使ってよい
- 必要なら同義語で補助して検索幅を広げるが、候補は多く持ちすぎない
- 正準語と技術用語の両方を見ながら、狭くも広くも調整する

### 4.3 共通の考え方

- 技術用語は「一部の人だけの語」ではなく、**全員が必要に応じて使う語**として扱う
- 同義語は対話の中で広く候補化されるが、**最終的にはかなり絞り込む**
- 差は「使うかどうか」ではなく、**どの順で使うか / どこまで刈り込むか** に出る
- 役割よりも、業務への慣れ具合を優先して選択する

## 5. 対話での扱い

1. LLM は会話文脈から候補を選ぶ
2. その時点の業務慣熟度に応じて `q` を構成する
3. どの候補を使ったかは後続対話で分かるようにする
4. 同義語は積極的に候補化するが、迷うものは採用せず、後続の調整余地を残す

## 6. 非ゴール

- 半角/全角揺れの新規実装
- 空白正規化の再実装
- フォルダ名 fragment / 台帳名 fragment
- 添付関連の変更
- 詳細な query trace 仕様の実装（これは `A2-2`）

## 7. 完了条件

- 同義語と技術用語を別グループとして扱う方針が明文化されている
- 同義語は一般語として広く候補化し、対話の中で絞る方針が説明できる
- 業務に慣れている人 / 慣れていない人で優先順位の違いが説明できる
- 既存の正規化は別レイヤーで対応済みとして扱われている
- `A2-2` / `A2-3` に引き継げる粒度になっている

## 8. 次の引き継ぎ

## 9. 結論

`#137` は、コード実装を増やすよりも、**同義語と技術用語の候補をどう選ぶかのルールを固定する作業**として完了させる。

この memo をもって `A2-1` の最終アウトプットとする。
