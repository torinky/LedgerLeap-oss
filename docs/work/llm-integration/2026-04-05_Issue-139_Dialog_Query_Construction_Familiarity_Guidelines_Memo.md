# Issue 139: 対話文脈からの検索クエリ構成ルールとペルソナ案内

**対象Issue:** `#139`  
**親Issue:** `#136`  
**関連:** `#137`, `#138`

## 0. サマリー

- `#139` は、LLM が会話文脈から **どの語を q に採用するか** のルールを決める作業
- 差の軸は「役割」よりも **業務 / ドメインに慣れているかどうか**
- 同義語は一般語として、対話の中で大きく刈り込む
- 技術用語は業界内 / 社内で使われる正式語として、必要に応じて使う
- 正規化の再実装はしない。半角/全角・空白などは既存前処理に委ねる
- `#139` の結論は「会話文脈から q をどう組み立てるかの案内ルールを固定すること」

## 1. 目的

LLM が会話文脈を見て、同義語候補と技術用語候補から検索クエリを構成するシナリオを整理する。

この memo では、**業務に慣れていない人は同義語で入口を作る**、**業務に慣れている人は技術用語を軸にしてよい** という基本方針を定義する。

## 2. 既存前提

- 半角/全角、空白などの正規化は既存検索前処理で対応済み
- そのため、`A2-3` では再実装しない
- `SearchContext` / `SynonymService` / `SearchRequest` は既存資産として前提にする
- `SearchLedgersTool` は `synonym-aware` な `q` を既に扱う

## 3. 検索クエリ構成ルール

### 3.1 業務 / ドメインに慣れていない人

- まず同義語で入口を作る
- 同義語は広く候補化してよいが、最終的には対話の中で絞る
- 技術用語は、ユーザーが理解できる見込みがあるときだけ追加する
- q はできるだけ自然文に近く保つ

### 3.2 業務 / ドメインに慣れている人

- 最初から技術用語を軸にしてよい
- 必要なら同義語を補助として足し、検索幅を少し広げる
- ただし、候補を増やしすぎず、検索意図がぼけないようにする

### 3.3 共通ルール

1. 会話の文脈から、正準語・同義語・技術用語を見分ける
2. その人が業務にどれだけ慣れているかで、採用順を変える
3. 迷う候補は採用しない
4. 後続対話で q を微調整できるように、検索意図を狭く保つ

## 4. ペルソナ案内の考え方

この issue では役割固定ではなく、**業務慣熟度** で案内する。

- 慣れていない人: 同義語を入口にする
- 慣れている人: 技術用語を軸にしてよい
- どちらでも: 何を採用したかを後続で説明できるようにする

## 5. 対話での扱い

1. LLM は会話文脈から候補を選ぶ
2. `#137` で整理した taxonomy を使って、同義語 / 技術用語を切り分ける
3. `#138` の trace を参照しながら、後続対話で q を調整する
4. 同義語は積極的に候補化するが、最終検索語は絞り込む

## 6. 非ゴール

- 半角/全角揺れの新規実装
- 空白正規化の再実装
- フォルダ名 fragment / 台帳名 fragment
- 添付関連の変更
- trace の詳細仕様そのもの（これは `#138`）

## 7. 完了条件

- 業務に慣れている / 慣れていないで q の組み立て方が説明できる
- 同義語は一般語として広く候補化し、対話の中で絞る方針が説明できる
- 技術用語は全員が必要に応じて使う語として扱える
- 既存の正規化は別レイヤーで対応済みとして扱われている
- `#137` / `#138` と矛盾せず、A2-3 の最終アウトプットとして完結している

## 8. 次の引き継ぎ

- `#137`: 同義語・技術用語候補の分離と選択ポリシー
- `#138`: 検索クエリのトレース可視化と説明可能性

## 9. 結論

`#139` は、LLM が会話文脈から **どの語を q に採用するか** を、業務慣熟度ベースで案内するルールを固定する作業として完了させる。

この memo をもって `A2-3` の最終アウトプットとする。
