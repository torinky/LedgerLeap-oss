# 台帳差分表示におけるカラム順序の不整合問題と修正計画

## 1. 概要

台帳定義のカラムの並び順を変更したり、カラムを削除したりした後、該当台帳の古い変更履歴（差分）を表示すると、表示の不整合が発生する問題が確認された。本ドキュメントは、その根本原因と具体的な修正方針を記録するものである。

## 2. 根本原因の技術的詳細

この問題は、過去の台帳データ（`content`）を解釈する際に、どの時点の「カラム定義（`column_define`）」を使用するかの不整合に起因する。

1.  **データ保存形式**:
    台帳の各レコードのデータ (`ledgers.content` および `ledger_diffs.content`) は、カラムIDをキーとした連想配列ではなく、**順序が意味を持つ「数値インデックス配列」** としてJSON形式で保存されている。これは `LedgerDefine::normalizeByColumnDefine()` メソッドの仕様による。

2.  **変更履歴の設計思想**:
    将来的なカラム定義の変更（並び替え、追加、削除）に影響されずに過去の履歴を正確に再現するため、`ledger_diffs` テーブルには、データ本体 (`content`) と共に、**そのデータが記録された時点でのカラム定義のスナップショット (`column_define`)** が保存されている。

3.  **不具合の発生ロジック**:
    現在の差分表示ロジックは、過去の `LedgerDiff` の `content` を表示する際に、本来参照すべき**同レコード内に保存されている `column_define` スナップショットを無視**し、**常に最新の `LedgerDefine` のカラム定義を誤って適用**してしまっている。

    このため、カラムの並び順が変更されると、過去のデータのインデックスと、最新の定義のカラムの対応関係が崩れ、結果として「過去の列に違う値が表示される」という現象が発生する。同様に、カラムが削除された場合も、過去のデータが正しく表示されない。

## 3. 修正方針

差分表示に関わる全てのロジックを修正し、以下の2つの原則を徹底する。

### 3.1. コア方針: 各時点のスナップショットを正しく利用する

差分表示を行う際は、常に表示対象の `LedgerDiff` レコードに保存されている **`column_define` を使って、その `content` を解釈する**ように変更する。これにより、カラムの並び順が変更されても、データと定義の対応関係が常に保たれる。

### 3.2. 新要件: 削除済みカラムのハンドリング

差分比較時に、過去のバージョンには存在したが現在のバージョンでは削除されているカラムを検出し、**「(削除済み)」のように表示する**。これにより、ユーザーはどのカラムが削除されたのかを明確に認識できる。

### 3.3. 実装の詳細

*   **`app/Services/Ledger/LedgerDiffProcessor.php` の修正:**
    *   `prepareContentDiff` メソッドのロジックを全面的に見直す。
    *   新旧両方の `column_define` をマージし、比較対象となる全てのカラムIDのユニークなリストを作成する。
    *   各カラムIDについて、新旧両方の `content` と `column_define` における存在をチェックする。
    *   カラムの状態（`added`, `deleted`, `modified`, `unchanged`）を示すフラグを持つ、より詳細な差分オブジェクトを生成するように変更する。

*   **`app/Livewire/Ledger/LedgerDiffViewer.php` の修正:**
    *   `LedgerDiffProcessor` から返される新しい差分オブジェクトをビューに渡す。
    *   ビュー側 (`ledger-diff-viewer.blade.php`) で、カラムの状態フラグに応じて表示を切り替える。例えば、`deleted` フラグが立っているカラムは、「過去」の列に値を表示し、「現在」の列には「(削除済み)」と表示し、打ち消し線などのスタイルを適用する。

## 4. テスト計画

本修正の品質を保証するため、以下のテストケースを実装する。

*   **ユニットテスト (`LedgerDiffProcessorTest`)**:
    *   **カラム順序変更ケース:** カラムの順序が異なる2つの `column_define` を使って差分が正しく計算されることを検証する。
    *   **カラム削除ケース:** 古い定義には存在するが新しい定義には存在しないカラムについて、生成される差分オブジェクトに `deleted` フラグが正しく設定されることを検証する。
    *   **カラム追加ケース:** 新しい定義にのみ存在するカラムについて、`added` フラグが正しく設定されることを検証する。

*   **フィーチャーテスト (`LedgerDiffViewerTest`)**:
    *   **セットアップ:**
        1.  特定のカラム定義 (`[A, B, C]`) で台帳レコードを作成・保存する (Version 1)。
        2.  台帳定義を修正し、カラムBを削除、カラムCの順序を変更し、新しいカラムDを追加する (`[C, A, D]`)。
        3.  台帳レコードを再度編集・保存する (Version 2)。
    *   **検証:**
        1.  `LedgerDiffViewer` コンポーネントで Version 1 と Version 2 の差分を表示する。
        2.  レンダリングされたHTML内に、カラムBが「(削除済み)」として表示されていることをアサートする。
        3.  カラムAとCの値が、それぞれの列で正しく表示されていることをアサートする。
        4.  カラムDが「(追加)」として表示されていることをアサートする。

## 5. 影響範囲

*   **影響を受ける機能:** 台帳の変更履歴（差分）表示機能全体。
*   **影響を受けない機能:** データの保存処理、台帳の通常表示、権限管理など、差分表示以外の機能への影響はない。本修正は表示ロジックの不整合を解消するものであり、データ構造や永続化ロジックには変更を加えない。
