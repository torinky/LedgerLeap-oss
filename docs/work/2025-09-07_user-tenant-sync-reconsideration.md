# ユーザー・テナント所属自動同期機能の再検討

**日付:** 2025年9月7日
**作成者:** Gemini
**ステータス:** 検討済み

## 1. 概要

本ドキュメントは、ユーザーからの要望である「roleとtenantごとのfolderとの紐付けを元に同期するような機能（ユーザーがロールとフォルダ権限を通じて特定のテナントのリソースにアクセス可能になった場合、そのユーザーを自動的に当該テナントの所属ユーザーとして `tenant_user` テーブルに登録する機能）」について、ペルソナ、ユーザーシナリオ、および技術実装上の観点から俯瞰的に検討し、そのトレードオフを明確にするものです。

## 2. 検討する機能の概要

**機能名:** ロールベーステナント所属自動同期機能

**目的:** ユーザーがロールとフォルダ権限を通じて特定のテナントのリソースにアクセス可能になった場合、そのユーザーを自動的に当該テナントの所属ユーザーとして `tenant_user` テーブルに登録する。また、アクセス権を失った場合は所属を解除する。

**トリガー:**
*   ユーザーにロールが割り当てられた時 (`User::assignRole()`, `User::removeRole()`)
*   ロールにフォルダ権限が割り当てられた時 (`RoleFolderPermission` が作成/更新/削除された時)
*   フォルダのテナントが変更された時 (`Folder::tenant_id` が変更された時)

**ロジック:**
1.  対象のユーザーが、特定のテナントのフォルダに対して何らかの権限を持つロールを持っているかを確認する。
2.  もし持っていれば、そのユーザーをそのテナントに所属させる (`tenant_user` テーブルにレコードを追加する)。
3.  もし権限がなくなった場合は、所属を解除する (`tenant_user` テーブルからレコードを削除する)。

## 3. ペルソナとユーザーシナリオの観点からの検討

### 3.1. ペルソナA: 管理者 / 部門長 (佐藤さん)

*   **現在の課題:**
    *   テナント管理画面でユーザーをテナントに手動で紐付ける必要がある。
    *   ユーザーのロールやフォルダ権限の変更があった際に、`tenant_user` テーブルの同期を意識する必要がある。
    *   ユーザーがどのテナントにアクセスできるかを把握するために、ロールとフォルダ権限を個別に確認する必要がある。
*   **提案機能による解決:**
    *   `tenant_user` テーブルの管理が自動化され、手動での紐付け作業が不要になる。
    *   ユーザーのアクセス権限（ロールとフォルダ権限）を変更するだけで、所属テナントが自動的に更新されるため、管理の手間が削減される。
    *   ユーザーがどのテナントにアクセスできるかという情報が `tenant_user` テーブルに常に最新の状態で反映されるため、管理者はこのテーブルを参照するだけでユーザーの所属テナントを把握しやすくなる。
*   **トレードオフ:**
    *   **メリット:** 管理者の運用負荷軽減、データの一貫性向上。
    *   **デメリット:** 同期ロジックの複雑性増加、パフォーマンスへの潜在的な影響。

### 3.2. ペルソナB: 実務担当者 / 現場リーダー (田中さん)

*   **現在の課題:**
    *   自分がアクセスできるはずのテナントが、テナント切り替えメニューに表示されない場合がある（`tenant_user` テーブルの不整合）。
    *   アクセス権限があるにもかかわらず、テナントに所属していないために、特定の機能が利用できない場合がある。
*   **提案機能による解決:**
    *   テナント切り替えメニューが常に最新のアクセス可能なテナントリストを反映するため、ユーザーは自分がアクセスできるテナントを確実に見つけられるようになる。
    *   アクセス権限があるにもかかわらず機能が利用できないといった、ユーザー体験上の不整合が解消される。
*   **トレードオフ:**
    *   **メリット:** ユーザー体験の向上、システム利用の円滑化。
    *   **デメリット:** なし（ユーザーにとっては透過的な機能であり、直接的なデメリットはない）。

## 4. 技術実装上の観点からの検討

### 4.1. 現在のアーキテクチャと関連性

*   **中央ユーザーモデル:** `User` モデルは中央DBに存在し、`BelongsToTenant` トレイトは適用されていない。
*   **テナント所属管理:** `User` モデルと `Tenant` モデルは `tenant_user` 中間テーブルを介して多対多のリレーションを持つ。
*   **権限管理:** `spatie/laravel-permission` を使用し、`Role` と `Permission` は中央DBに存在する。`RoleFolderPermission` テーブルは「中央のRole」と「テナントのFolder」を紐付ける。`Folder` モデルは `BelongsToTenant` トレイトを使用し、テナントスコープされる。
*   **テナント切り替えUI:** `TenantSwitcher` Livewireコンポーネントが `Auth::user()->tenants` を参照してテナントリストを表示している。

### 4.2. 実装の複雑性

提案機能の実装は、以下の理由により複雑性が高い。

1.  **複数のイベント監視:**
    *   `User` モデルの `roles()` リレーションの `attached`, `detached` イベント。
    *   `RoleFolderPermission` モデルの `created`, `updated`, `deleted` イベント。
    *   `Folder` モデルの `tenant_id` 変更イベント（もしあれば）。
    *   これらのイベントを正確に捕捉し、同期ロジックをトリガーする必要がある。
2.  **双方向の同期ロジック:**
    *   ユーザーにロールが割り当てられた場合、そのロールに紐づくフォルダ権限から所属テナントを特定し、`tenant_user` に追加する。
    *   ロールが解除された場合、`tenant_user` から削除する。
    *   フォルダ権限が変更された場合、影響を受けるユーザーの所属テナントを再評価し、`tenant_user` を更新する。
3.  **パフォーマンスへの影響:**
    *   イベントが頻繁に発生する場合、同期処理がボトルネックになる可能性がある。特に、大量のユーザーやロール、フォルダが存在する場合、同期処理の効率が重要になる。
    *   同期ロジック内でデータベースクエリが多数発生する可能性があるため、Eager Loadingやキャッシュの活用が必須となる。
4.  **冪等性:**
    *   同期処理は、複数回実行されても同じ結果になるように冪等性を担保する必要がある。
5.  **エラーハンドリング:**
    *   同期処理中にエラーが発生した場合のロールバックや通知の仕組みを考慮する必要がある。

### 4.3. 代替案との比較（`tenant_user` テーブルの完全廃止）

前回の議論で「`tenant_user` テーブルを廃止した場合、ユーザーがアクセス可能なテナントを動的に判断するロジックがより複雑になる可能性がある」と述べた。

*   **`tenant_user` テーブル完全廃止のメリット:**
    *   データ重複の排除。
    *   同期ロジックが不要になる。
*   **`tenant_user` テーブル完全廃止のデメリット:**
    *   テナント切り替えUI (`TenantSwitcher`) のロジックを大幅に書き換える必要がある。
    *   ユーザーがアクセス可能なテナントを判断するロジックが複雑化し、パフォーマンスに影響を与える可能性がある（ロールとフォルダ権限を毎回評価するため）。
    *   現在の `stancl/tenancy` の `User` と `Tenant` のリレーションシップの標準的な使い方から逸脱する。

## 5. 最終結論と提案 (更新)

このドキュメントで検討された「`tenant_user`テーブルへの自動同期機能」は、その後の詳細な議論を経て、**最終的に不採用となりました。**

当初は実装の複雑性から「見送り」と判断されましたが、さらに議論を深める中で、「テナントは単なる入れ物であり、アクセス制御は中身のデータ（フォルダ）で行うべき」という、より上位の設計思想が確認されました。

この思想に基づき、`tenant_user`テーブルのような明示的な所属情報に依存せず、`role_folder_permissions`テーブルから動的にアクセス可能なテナントを決定する、新しいアーキテクチャ方針が採択されました。

したがって、本文書で検討された機能は実装されません。
