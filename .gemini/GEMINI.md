
# 作業を進める上での基本原則

- あなたはソフトウェア開発者です。
- 提示される要件に従って、必要機能を抽出し、実装計画を立案し、実行していきます。
- 業務プロセスで登場するペルソナ、ユーザーシナリオを元に要件を抽出し、実装計画の整合性を検証ます。不明確な事項はユーザーに確認します。
- 実装計画は機能確認ができる範囲でステップごとに設定します。
- 実装に先立ち、あなたはユーザーにステップを実行するにあたって考慮が必要な既存コードを要求します。
- ユーザーは既存コードを提供し、あなたは提供されたコードに整合するよう、実装計画で定義された機能を満足するコードを検討して提供します。
- あなたが検討したコードをユーザーが実装します。実装が終わったら機能を確認をユーザーに求めます。
- 検討が進むことでユーザーは実装計画の見直しや作業の追加をする場合があります。この場合は、実装が完了した機能の計画は変更せず、次のステップ以降の実装計画を変更します。
- 計画や作業記録は別のユーザーが検討を引き継ぐために必要な情報のため、計画に作業結果を追記・変更する場合、情報量を下げないように工夫してください。
- このプロジェクトのソースコードはステップに沿って部分的にユーザーから提供されます。あなたは直接変更できないのでユーザーが変更した結果を常に確認する必要があります。ユーザーから提供されたコードスニペットやファイル全体像を元に、あなたは実装を提案しますが、最終的なコードの正確性や完全性はユーザーが検証し、必要に応じて修正・調整を行います。
- 同じファイルがユーザーから提供された場合、最後に提供されたファイルの内容を優先して検討に使用します。
- 1つのステップの完了を合意する際は、実装計画に記載されたタスクのうち、完了した範囲を具体的に示すよう、ユーザーに要求してください。確認できるまでは次のステップに進めてはいけません。
- 実装方針の変更など、重要な決定事項とあなたが判断できた場合は、次のステップに進む前に認識合わせのためユーザーに再確認をしてください。
- あなたは検討のために必要なコードを要求しますが、ユーザーは関連性が高いと判断した他のコードも任意で提供することがあります。
- ユーザーから示される方針やあなたの検討結果が競合してしまった場合に限り、過去の議論よりも、直近のユーザーとのやり取りで確定した仕様や方針を最優先して応答してください。過去のチャットでの決定事項よりも、現在のチャットでユーザーが最終的に確認・指示した内容が絶対です。
- 判断に疑義が生じる場合は、細かい部分であってもユーザーに再確認してください。
- コードの検討開始のタイミングはユーザーが決定します。指示があるまでは計画や結果のドキュメントの合意形成を中心に議論を進めてください。
- 起動した直後はdocsディレクトリを確認し、このシステム全体の理解に努めてください。


# Gemini Added Memories

## LedgerLeap プロジェクト概要

### 目的
添付ファイルを含む全文検索と柔軟な権限管理を特徴とする、Webベースの台帳管理システム。

### コア技術
- **バックエンド:** Laravel (PHP)
- **フロントエンド:** Livewire, Alpine.js, MaryUI (一般ユーザー向け), Filament (管理者向け)
- **データベース:** MySQL/MariaDB + **Mroonga** (全文検索)
- **ファイル内容抽出:** **Apache Tika**
- **開発環境:** Laravel Sail (Docker)

### アーキテクチャ
- 標準的なWebアプリケーション構成（Webサーバー、アプリケーションサーバー、DBサーバー）。
- 非同期処理のために**Redis**と**キューワーカー**を利用。
- ファイル検索のために**Apache Tika**サーバーが独立して動作。

### データベース設計
- `ledgers` (台帳データ) と `ledger_defines` (台帳定義) が中心。
- `folders` による階層的な情報整理。
- `ledger_diffs` で台帳の変更履歴とワークフローの状態を管理。
- 権限管理は `spatie/laravel-permission` をベースに、`role_folder_permissions` でフォルダ単位の詳細な制御を実現。
- `activity_log` で操作履歴を記録。

### API
- **Laravel Sanctum**によるSPA認証とAPIトークン認証を提供。
- 標準的なRESTful API。

### 開発規約
- **フォーマット:** `laravel/pint`
- **命名規則:** Laravelの標準的な規約に準拠（変数:スネークケース, メソッド:キャメルケース, クラス:パスカルケース）。
- **設計思想:** ファットコントローラを避け、サービスクラスにビジネスロジックを分離。FormRequest、APIリソース、Enumを積極的に活用。

## 開発から得られた教訓

これまでのUI改善や機能実装（台帳定義、アクセス制御、ワークフロー）を通じて得られた、技術選定と実装に関する重要な教訓を以下に記録します。

### 1. 状態管理とUIの原則

- **単一の信頼できる情報源 (Single Source of Truth):**
  - Livewireコンポーネントで複雑な状態（例: 台帳定義のカラムリスト）を扱う際は、複数のプロパティに分割せず、単一の配列プロパティ（例: `$columns`）に集約することが極めて重要です。これにより、状態の不整合を防ぎ、ロジックを大幅に簡素化できます。
- **Livewireプロパティの型:**
  - Livewireのpublicプロパティは、UIにバインドする場合、複雑なPHPオブジェクト（例: `ColumnDefine`クラスのインスタンス）ではなく、**シンプルな連想配列**として扱うことが「Property type not supported」エラーを避ける鍵となります。`mount()`時や`addColumn()`時に連想配列に変換・初期化する一手間が、安定動作につながります。

### 2. Livewire, Alpine.js, 外部ライブラリ連携

- **DOM操作のライフサイクル理解:**
  - Livewire、Alpine.js、SortableJSのように複数のライブラリがDOMを操作する場合、それぞれのライフサイクルとDOMの再描画タイミングを理解することが不可欠です。特に`wire:key`は、LivewireがDOM要素を正しく追跡するための生命線です。
- **イベントの制御:**
  - ドラッグイベントやクリックイベントが競合する場合、`x-on:mousedown.stop`のようなAlpine.jsのイベント修飾子が有効です。イベントの伝播を意識的に制御することで、意図しない動作を防げます。
- **LivewireとAlpine.jsの役割分担:**
  - Livewireのリアクティビティだけでは実現が難しいUI（例: 即時フィードバックが必要な部分）は、Alpine.jsを併用してクライアントサイドで補完するのが効果的です。`x-data`, `x-bind`, `@event.window`, `$dispatch`などを活用し、サーバー通信を最小限に抑えつつ、リッチなUIを実現できます。

### 3. Livewireの高度な活用法

- **フックメソッド (`updated[PropertyName]`) の活用:**
  - Livewireの「フック」メソッドは非常に強力です。`columns.0.type`のようなネストされたプロパティの変更を検知するには、命名規則に従った`updatedColumns($value, $key)`メソッドを実装します。`$key`を解析することで、どの要素のどのプロパティが変更されたかを特定し、依存する他のプロパティ（例: `useOptions`）を動的に更新できます。
- **Bladeテンプレートからの直接参照:**
  - Bladeビュー内で`new Model((object)$column)`のようにオブジェクトを再インスタンス化すると、予期せぬ挙動の原因となります。Livewireコンポーネントのプロパティ（例: `$column['useOptions']`）を直接参照することで、状態の一貫性を保つべきです。

### 4. コンポーネント設計とサービスレイヤー

- **汎用コンポーネントの威力:**
  - `ActivityHistoryDisplay`や`PermissionDisplay`のように、特定の機能（履歴表示、権限表示）を汎用的なLivewireコンポーネントとして切り出すことで、コードの再利用性が劇的に向上します。モーダルでの呼び出しや、異なる画面への埋め込みが容易になります。
- **モーダルの活用:**
  - 複雑な操作（例: 担当者選択）や詳細情報の表示は、モーダルコンポーネント（例: `WorkflowAssigneeModal`）に集約すると効果的です。親コンポーネントはイベント（`$dispatch`）でモーダルを呼び出し、モーダルは選択結果をイベントで返す、という疎結合な設計が望ましいです。
- **ビジネスロジックの分離 (サービスクラス):**
  - `WorkflowService`や`PermissionService`のように、複雑なビジネスロジックやDB操作はLivewireコンポーネントからサービスクラスに完全に分離すべきです。これにより、コンポーネントはUIの状態管理とユーザー操作の受付に専念でき、コードの見通しとテスト容易性が向上します。

### 5. デバッグと問題解決

- **地道な追跡:**
  - 複雑なイベントの競合や状態変化の問題では、`dd()`や`console.log()`を要所に仕込み、イベントの伝播範囲やデータの状態を一つずつ追跡する地道なデバッグが最も確実な解決策です。
- **ソースコードは最高のドキュメント:**
  - ドキュメントで解決しない場合、`vendor`や`node_modules`内のライブラリのソースコードを読むことが、内部実装を理解し、問題を解決するための最終手段として非常に有効です。

### 6. その他

- **Enumの積極活用:**
  - `WorkflowStatus`や`FolderPermissionType`のように、状態や種別を表す値はPHPのEnumで定義することで、コードの可読性と堅牢性が向上します。`$casts`での自動変換や、Enum内でのヘルパーメソッド定義も有効です。
- **権限の包含関係:**
  - `ADMIN > APPROVE > INSPECT > WRITE > READ`のような権限の包含関係は、専用のヘルパーメソッド（例: `FolderPermissionType::includes()`）で判定ロジックをカプセル化することで、権限チェックのコードが簡潔かつ一貫性を保ちます。

### 7. テスト環境とFakerに関する教訓

- **Fakerのプロバイダロードの挙動:**
  - Laravelのファクトリは通常Fakerのプロバイダを自動ロードするが、特定のロケール（例: `ja_JP`）では、`text()`, `sentence()`, `word()`のような汎用的なメソッドを提供する`Lorem`プロバイダがデフォルトでロードされない場合がある。
  - `config/app.php`の`faker_locale`設定が、テストデータ生成に大きな影響を与える。
  - 解決策として、ファクトリ内で`Faker\Factory::create('en_US')`のように明示的にFakerインスタンスを生成し、ロケールを固定する方法が有効な場合がある。
- **Laravelユニットテストにおけるデータベース接続とモデルファクトリの競合:**
  - `RefreshDatabase`トレイトを使用しても、`User::factory()->create()`のようなモデルファクトリが、テストケースの実行前にデータベース接続が完全に確立されていない状態で呼び出されると、`Call to a member function connection() on null`エラーが発生することがある。
  - これは、Eloquentモデルの`boot()`メソッド内で、`app()`ヘルパーを介してサービスコンテナからデータベース接続を必要とするサービス（例: `WritableFolderRepository`, `UserService`）が解決されるタイミングと、テスト環境のデータベース準備のタイミングとの間に競合があるため。
  - `User::withoutEvents()`はイベントを無効にするが、モデルのインスタンス化プロセス全体におけるデータベース依存性を解決するわけではない。
  - **暫定的な解決策:** 各テストメソッド内で`User::unguard()`と`User::reguard()`を呼び出すことで、一時的にマスアサインメント保護を無効にし、ファクトリがよりシンプルに動作するように試みた。これにより、`User`モデルの`boot()`メソッド内のイベントリスナーが、`unguard()`と`reguard()`の呼び出しによって影響を受け、データベース接続を必要とする処理がスキップされる可能性がある。
  - **Pestにおけるモック:** `$this->mock()`ではなく、Pestのグローバル関数`mock()`を使用する必要がある。
- **バリデーションルールにおける`$fail`クロージャの挙動:**
  - `Closure $fail`は、アプリケーション環境では`->translate()`メソッドを持つ`PotentiallyTranslatedString`を返す一方、ユニットテスト環境では単なる文字列を返す場合がある。
  - `app()->runningUnitTests()`を使用して、テスト環境とアプリケーション環境で`$fail`の呼び出し方（`->translate()`の有無）を分岐させることで、両方の環境で正しく動作するバリデーションルールを実装できる。

### 8. テスト環境の複雑性への対応

- **テスト環境の分離と依存性注入:**
  - ユニットテストは、対象のユニット（クラスやメソッド）のみをテストし、外部依存性（データベース、ファイルシステム、外部APIなど）を排除することが理想的である。
  - しかし、LaravelのEloquentモデルやファクトリは、フレームワークの多くの機能（データベース接続、イベント、サービスコンテナなど）に深く依存しているため、純粋なユニットテスト環境を構築することが難しい場合がある。
  - このような場合、`RefreshDatabase`トレイトやモック、`withoutEvents()`などのLaravelが提供するテストヘルパーを適切に組み合わせる必要がある。
  - それでも解決しない場合は、テスト対象のコードの設計を見直し、依存性をより明確に分離できるようにリファクタリングすることも検討すべきである。
- **テストフレームワークのバージョンと挙動:**
  - PestやPHPUnitのバージョン、Laravelのバージョンによって、テストヘルパーの挙動や推奨されるテスト方法が異なる場合がある。公式ドキュメントやコミュニティの情報を常に最新に保ち、適切なアプローチを選択することが重要である。
- **デバッグの重要性:**
  - テストが失敗した場合、`dd()`や`Log::info()`、Xdebugなどのデバッグツールを活用し、変数の状態、実行パス、エラー発生箇所を詳細に追跡することが問題解決の鍵となる。特に、フレームワークの内部で何が起こっているかを理解することが重要である。

- LedgerLeapプロジェクトは、Webベースの台帳管理システムであり、全文検索（Mroonga, Apache Tika, OcrMyPDF）、柔軟な権限管理、ワークフロー機能を特徴とする。技術スタックはPHP ^8.4, Laravel ^12.0, MySQL/MariaDB + Mroonga, フロントエンドはMaryUI, DaisyUI, Alpine.js, Tailwind CSS, Vite, Filament, Livewire。開発環境はLaravel Sail (Docker)。開発規約としてlaravel/pintによるフォーマット、ケバブケースを多用する命名規則、PHPDoc推奨のコメント、Gitflowベースのブランチ戦略、Conventional Commits形式の日本語コミットメッセージが定められている。データベースとしてMroongaが必須。
- 添付ファイルのパス管理は `AttachedFilePathHelper` に一元化されている。`ColumnDefine` はEloquentモデルではないため、テストでインスタンス化する際はファクトリではなく直接コンストラクタを使用すること。OCR処理関連のジョブテストでは、`ocrmypdf` コマンドによるファイル生成が `Storage::fake()` 環境下で正しく認識されるよう、テスト内で一時ファイルを作成するなどの工夫が必要。 
- テストで `Bus::fake()` や `Bus::spy()` を使用する際は、外部プロセスによるファイルシステム操作との相互作用を考慮し、テスト設定が適切か確認すること。 
- EloquentモデルとプレーンなPHPオブジェクトの区別を常に意識し、ファクトリの使用やDB操作の期待値を正しく設定すること。データベースの `NOT NULL` 制約があるフィールドは、テストデータで必ず提供すること。

-   **OCR ジョブのデバッグ:**
    *   OCR 処理の無限ループ問題、`jobs` テーブルへのジョブ継続登録問題、Docker コンテナ間のパス指定の重要性、ログ出力の管理、テスト用ファイルのクリーンアップ、変更適用時の Docker コンテナ再起動の必要性。
-   **テストにおけるモックとデータ:**
    *   テストで `Bus::fake()` や `Bus::spy()` を使用する際は、外部プロセスによるファイルシステム操作との相互作用を考慮し、テスト設定が適切か確認すること。
